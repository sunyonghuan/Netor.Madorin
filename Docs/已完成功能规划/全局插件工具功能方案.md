# 全局插件与插件管理界面调整方案

## 1. 现状梳理

### 1.1 已存在的数据表

当前数据库已存在这些和本方案相关的表：

| 表 | 作用 | 与本方案关系 |
|---|---|---|
| `Agents` | 智能体配置 | 已保存 `EnabledPluginIds`、`EnabledMcpServerIds`，说明“智能体绑定插件/MCP”已经落库 |
| `McpServers` | MCP 服务配置 | MCP 已有独立配置表和启用状态 |
| `SystemSettings` | 系统设置 | 可用于保存简单全局开关，但不适合保存复杂插件状态 |

`Agents` 表已有字段：

- `EnabledPluginIds TEXT NOT NULL DEFAULT '[]'`
- `EnabledMcpServerIds TEXT NOT NULL DEFAULT '[]'`

因此，“某个智能体绑定哪些插件/MCP”不是新能力，当前系统已经实现。

### 1.2 已存在的系统能力

当前系统已经具备：

- 插件加载：`PluginLoader` 扫描并加载插件；
- 双插件目录：
  - 全局插件目录：`App.UserPluginsDirectory` / `IAppPaths.UserPluginsDirectory`
  - 工作区插件目录：`App.WorkspacePluginsDirectory` / `IAppPaths.WorkspacePluginsDirectory`
- 插件热插拔：`FileSystemWatcher` 监听插件目录变化；
- 插件卸载/重载能力：`PluginLoader.UnloadPlugin()`、`PluginLoader.ReloadPluginAsync()`；
- AI 侧工具装配：`AIAgentFactory` 已根据智能体配置处理插件和 MCP 工具；
- 设置页：`SettingsWindow` 中已有“工具管理”页面；
- 当前“工具管理”页面：必须先选择智能体，才显示插件和 MCP。

所以本方案不再重复设计“工具如何注入 AI”，重点改为：

1. 插件管理界面重构；
2. 插件列表不再依赖选择智能体；
3. 全局插件开关；
4. 插件卸载入口；
5. 智能体和插件绑定单独处理；
6. 为插件数量很多的场景重新设计布局。

## 2. 本次目标

把设置页中的“工具管理”升级为“插件管理”。

目标：

- 插件作为系统级资源独立展示；
- 不选择智能体也能看到所有本地插件；
- 每个插件显示名称、简介、版本号、插件 ID、工具数量、安装位置；
- 全局目录中的插件可以设置为全局插件；
- 工作区目录中的插件不能设置为全局插件；
- 插件管理页提供“卸载”和“重载”按钮，删除插件作为独立危险操作二次确认；
- 智能体和插件的绑定关系在插件详情中用智能体复选列表处理，不再用下拉框选择单个智能体；
- 页面布局适配插件数量很多的情况。

## 3. 命名调整

当前设置导航中“工具管理”应改名为“插件管理”。

建议调整：

| 当前名称 | 新名称 |
|---|---|
| 工具管理 | 插件管理 |
| 保存工具配置 | 保存绑定配置 |
| 为智能体启用/禁用插件工具和 MCP 服务器工具 | 管理本地插件、全局插件，以及智能体插件绑定 |

说明：

- “工具”是插件暴露给 AI 的能力；
- “插件”才是用户管理、卸载、更新、全局启用的对象；
- MCP 已经有独立设置页，不应在插件管理页中成为主角，只在“智能体绑定”中作为关联项出现即可。

## 4. 界面布局方案

### 4.1 当前问题

当前 `ToolManagementPage` 的主要问题：

1. 必须选择智能体后才显示本地插件，体验不好；
2. 插件列表和智能体绑定混在一起，职责不清；
3. 插件数量多时，单列卡片列表会很长，难查找；
4. 没有搜索、过滤、分组；
5. 没有插件卸载入口；
6. 插件名称、简介、版本号展示不足；
7. “本地插件”和“MCP 服务器”都在同一绑定列表中，后续扩展会越来越乱。

### 4.2 新页面结构

建议插件管理页采用“上方工具栏 + 左侧插件列表 + 右侧详情/绑定”的布局。

```text
┌────────────────────────────────────────────────────────────┐
│ 插件管理                                                    │
│ 管理本地插件、全局插件，以及智能体插件绑定                    │
├────────────────────────────────────────────────────────────┤
│ 搜索框            来源筛选：全部/全局/工作区    刷新/打开目录 │
├───────────────────────┬────────────────────────────────────┤
│ 插件列表               │ 插件详情                            │
│ - 插件 A               │ 名称 / 版本 / 简介 / ID / 路径        │
│ - 插件 B               │ 工具列表                             │
│ - 插件 C               │ [设为全局插件] [卸载] [重载] [删除]    │
│ ...                   │ 智能体绑定复选列表                    │
└────────────────────────────────────────────────────────────┘
```

### 4.3 左侧插件列表

左侧列表必须在页面加载时直接显示插件，不依赖智能体选择。

列表项显示：

- 插件名称；
- 版本号，例如 `v1.2.0`；
- 简短简介，单行省略；
- 来源标记：`全局目录` / `工作区目录`；
- 全局状态标记：`全局启用` / `未全局启用` / `不可全局启用`；
- 工具数量。

插件多时需要支持：

- 搜索：按名称、ID、简介过滤；
- 来源过滤：全部 / 全局目录 / 工作区目录；
- 状态过滤：全部 / 全局启用 / 可全局启用 / 工作区插件；
- 虚拟化列表或至少避免一次性创建过重 UI。

### 4.4 右侧插件详情

选中插件后显示完整详情：

| 信息 | 来源 |
|---|---|
| 名称 | `IPlugin.Name` |
| 简介 | `IPlugin.Description` |
| 版本号 | `IPlugin.Version` |
| 插件 ID | `IPlugin.Id` |
| 工具数量 | `IPlugin.Tools.Count` |
| 工具列表 | `IPlugin.Tools` |
| 安装位置 | 插件目录路径 |
| 来源 | 全局目录 / 工作区目录 |

右侧操作按钮：

- `设为全局插件` / `取消全局插件`；
- `卸载`；
- `重载`；
- `删除插件`；
- `打开目录`。

按钮规则：

| 按钮 | 全局目录插件 | 工作区插件 |
|---|---|---|
| 设为全局插件 | 可用 | 禁用，提示“只有全局目录插件可以设为全局插件” |
| 重载 | 可用 | 可用 |
| 卸载 | 可用，只断开当前插件运行实例，不删除文件 | 可用，只断开当前插件运行实例，不删除文件 |
| 删除插件 | 可用，危险操作，必须二次确认 | 可用，危险操作，必须二次确认 |
| 打开目录 | 可用 | 可用 |

## 5. 全局插件状态保存

### 5.1 不复用 `Agents.EnabledPluginIds`

`EnabledPluginIds` 是智能体级绑定字段，不应该用来保存全局插件。

全局插件是系统级配置，应该独立保存。

### 5.2 推荐新增表 `GlobalPlugins`

新增表：

```sql
CREATE TABLE IF NOT EXISTS GlobalPlugins (
    PluginId TEXT PRIMARY KEY,
    IsEnabled INTEGER NOT NULL DEFAULT 1,
    CreatedTimestamp INTEGER NOT NULL,
    UpdatedTimestamp INTEGER NOT NULL
);
```

说明：

- 只保存插件 ID 和启用状态；
- 插件名称、简介、版本号从当前加载插件读取；
- 插件被删除时可以保留配置，方便重新安装后恢复；
- 不在本方案中设计联网更新表，避免提前过度设计。

### 5.3 新增服务

建议新增 `GlobalPluginService`：

- `GetEnabledPluginIds()`
- `IsEnabled(pluginId)`
- `SetEnabled(pluginId, enabled)`
- `Remove(pluginId)`

该服务只负责全局插件开关，不负责工具注入细节。

## 6. 插件来源识别

当前 `PluginLoader.GetActivePlugins()` 只返回插件实例，不携带目录来源。

为了界面能判断插件是否在全局目录，建议扩展插件加载信息。

新增模型：

```csharp
public enum PluginInstallScope
{
    Global,
    Workspace
}

public sealed record LoadedPluginInfo(
    IPlugin Plugin,
    string DirectoryPath,
    string DirectoryName,
    PluginInstallScope Scope);
```

新增方法：

```csharp
public IReadOnlyList<LoadedPluginInfo> GetLoadedPluginInfos();
```

判断规则：

- 插件目录位于 `_appPaths.UserPluginsDirectory` 下：`Scope = Global`；
- 插件目录位于 `_appPaths.WorkspacePluginsDirectory` 下：`Scope = Workspace`。

保留现有 `GetActivePlugins()`，避免影响已有代码。

## 7. 卸载、重载与删除设计

### 7.1 卸载按钮行为

插件详情中提供“卸载”按钮，但卸载不能删除插件目录。

卸载含义：

- 断开当前插件运行实例；
- 释放进程、文件锁或连接资源；
- 让插件临时从当前运行环境中移除；
- 保留插件文件、全局配置和智能体绑定配置；
- 后续可以继续点击“重载”恢复。

卸载流程：

1. 弹出确认；
2. 调用 `PluginLoader.UnloadPlugin(directoryName)` 释放进程和文件占用；
3. 刷新插件运行状态；
4. 插件文件仍保留在原目录；
5. 全局插件配置和智能体绑定配置不变。

卸载适用场景：

- 插件连接异常，需要先卸载再重载；
- 插件进程卡死，需要释放进程；
- 用户临时停用插件但不想删除文件和配置。

### 7.2 重载按钮行为

重载含义：

- 对当前插件执行卸载再加载；
- 用于插件异常恢复或文件替换后重新加载；
- 不改变全局插件配置；
- 不改变智能体绑定配置。

重载流程：

1. 调用 `PluginLoader.ReloadPluginAsync(directoryName)`；
2. 刷新插件列表和详情；
3. 如果重载失败，在详情区显示错误状态。

### 7.3 删除插件行为

删除是独立危险操作，不能和卸载混为一体。

删除流程：

1. 二次确认；
2. 先调用 `PluginLoader.UnloadPlugin(directoryName)`；
3. 删除插件目录；
4. 清理 `GlobalPlugins` 中的该插件 ID；
5. 从所有智能体的 `EnabledPluginIds` 中移除该插件 ID；
6. 刷新插件列表。

删除适用场景：

- 用户确定不再使用该插件；
- 插件损坏，需要彻底移除后重新安装；
- 插件版本替换流程需要先删除旧目录。

### 7.4 提示文案

卸载：

> 将卸载插件「{Name}」的当前运行实例。插件文件和绑定配置会保留，可稍后重载。

重载：

> 将重新加载插件「{Name}」。如果插件当前异常，系统会先卸载再加载。

全局目录插件：

> 将删除全局插件「{Name}」。删除后所有智能体都无法再使用该插件，相关绑定也会被清理。

工作区插件：

> 将删除当前工作区插件「{Name}」。只影响当前工作区，相关智能体绑定会被清理。

### 7.5 安全处理

- 删除目录前必须先卸载插件，避免进程占用；
- 删除失败时提示用户关闭占用进程或重启软件；
- 不允许删除插件根目录，只能删除某个插件子目录；
- 删除路径必须校验位于全局插件目录或工作区插件目录之下。

## 8. 智能体绑定单独处理

### 8.1 绑定区域放在插件详情中

智能体绑定不应该决定插件列表是否显示，也不应该通过下拉框选择单个智能体。

一个插件可以绑定多个智能体；智能体数量是有限的，适合直接全部列出来，用复选框或开关维护绑定关系。

建议在插件详情中提供“绑定智能体”区域：

```text
绑定智能体
[x] 默认助手
[ ] 代码助手
[x] 文档助手
[ ] 翻译助手

[保存绑定]
```

保存逻辑：

- 当前选中的是插件 A；
- 勾选某个智能体 = 把插件 A 的 ID 加入该智能体 `EnabledPluginIds`；
- 取消勾选某个智能体 = 从该智能体 `EnabledPluginIds` 移除插件 A 的 ID；
- 保存时批量更新发生变化的智能体。

### 8.2 绑定范围

智能体绑定只处理：

- 工作区插件；
- 未设为全局的全局目录插件；
- MCP 服务。

已经设为全局插件的插件：

- 在绑定区域显示“已全局启用”；
- 不需要重复勾选；
- 可以禁用勾选，避免重复绑定。

### 8.3 MCP 绑定另行处理

本轮优先处理本地插件和智能体的绑定。

MCP 已有独立页面，MCP 与智能体绑定可以后续在 MCP 服务页或单独绑定页中用相同的“智能体复选列表”模式处理，不再混入本地插件详情。

### 8.4 绑定关系清理

当插件被卸载：

- 遍历所有智能体；
- 如果 `EnabledPluginIds` 包含该插件 ID，则移除；
- 调用 `AgentService.Update()` 保存；
- 发布 `Events.OnAgentChange` 或插件变化事件。

## 9. MCP 的位置

MCP 已有独立的“MCP 服务”页面，因此插件管理页不负责 MCP 的新增、删除、连接配置。

插件管理页中 MCP 只出现在“智能体绑定”区域，用于配置某个智能体是否启用某个 MCP 服务。

这样职责更清楚：

- `MCP 服务` 页面：管理 MCP 服务本身；
- `插件管理` 页面：管理本地插件本身；
- `智能体绑定` 区域：管理智能体和插件/MCP 的关系。

## 10. 后续联网更新预留

本次界面必须显示版本号，但不实现联网更新。

只预留 UI 位置：

- 插件详情中显示当前版本；
- 预留“检查更新”按钮位置，但首期可隐藏；
- 后续再设计远程清单、下载、校验和替换流程。

不要在首期新增复杂更新表，避免过度设计。

- 只有全局插件目录中的插件可以设为全局插件；

- 插件详情右侧滚动区域已增加右侧 `Padding`，滚动条不再压住文字；
- 全局插件开关已改为 `ToggleSwitch`；
- 开启全局插件后，智能体绑定区域会隐藏，并提示“所有智能体默认可用”；
- 关闭全局开关后，原有智能体绑定仍保留，可继续生效；
- 智能体绑定区域使用双列展示，一行两个智能体；
- 构建验证通过：
  - `dotnet build .\Src\Netor.Cortana.UI\Netor.Cortana.UI.csproj -p:OutDir=.\artifacts\build-verify\ui\`

重复注入验证：

- `AIAgentFactory.AssembleToolProviders()` 会先注入全局插件；
- 注入后把插件 ID 记录到 `injectedPluginIds`；
- 后续处理智能体 `EnabledPluginIds` 时，如果插件 ID 已经在 `injectedPluginIds` 中，会跳过；
- 因此“全局启用 + 已绑定智能体”不会导致同一个插件重复注入。

当前决策：

- 开启全局插件时，不自动删除智能体已有绑定；
- 原因是保留回退路径：关闭全局插件后，原绑定关系仍可继续使用；
- UI 上隐藏绑定区域，避免用户误以为还需要重复绑定。

## 14. 工作区插件是否继续保留的讨论

### 14.1 当前代码验证

当前插件加载器仍然同时扫描两个目录：

- 全局插件目录：`IAppPaths.UserPluginsDirectory`；
- 工作区插件目录：`IAppPaths.WorkspacePluginsDirectory`。

代码位置：

- `PluginLoader.GetPluginRootDirectories()` 返回以上两个目录；
- `PluginLoader.ScanAndLoadAsync()` 会遍历两个目录并加载插件；
- `PluginLoader.ReloadAllPluginsAsync()` 会在工作目录变化时卸载全部插件并重新扫描；
- `PluginLoader` 订阅了 `Events.OnWorkspaceChanged`，工作目录切换会触发插件重载；
- `AppPaths.PluginDirectory` 当前仍指向 `WorkspacePluginsDirectory`，这是旧设计遗留。

当前智能体绑定逻辑：

- 智能体是全局配置；
- `AgentEntity.EnabledPluginIds` 只保存插件 ID，不保存插件目录来源；
- `AIAgentFactory` 装配绑定插件时使用 `pluginLoader.GetActivePlugins()`；
- 如果某个智能体绑定了工作区插件，而切换到另一个工作目录后该插件不存在，则该绑定不会生效；
- 这就是“智能体全局，但工作区插件随工作区变化”的不一致问题。

### 14.2 原工作区插件设计的原因

以前没有“全局插件”和“智能体绑定”的概念时，插件放在工作区目录有一个合理目的：

- 插件能力跟随工作区；
- 切换工作区时，插件也随工作区切换；
- 避免某个工作区的专用插件污染另一个工作区的上下文。

这个设计本质上是“工作区即能力边界”。

### 14.3 当前新机制下的问题

引入全局插件和智能体绑定后，能力边界已经从“工作区”迁移到：

- 插件是否安装在全局目录；
- 插件是否设为全局插件；
- 插件是否绑定到某些智能体。

因此继续把插件放在工作区目录，会带来几个问题：

1. **智能体绑定不稳定**  
    智能体是全局的，但绑定的工作区插件只在某个工作目录存在。切换工作目录后，同一个智能体可能找不到插件。

2. **用户理解成本高**  
    用户需要理解“插件装在哪里”和“智能体在哪里使用”的关系，容易出现绑定了但不可用的情况。

3. **上下文隔离已有更好方式**  
    现在可以通过“只绑定某个智能体”来隔离插件能力，而不是依赖工作区目录隐式隔离。

4. **全局目录更适合联网更新和版本管理**  
    后续插件联网更新、版本显示、卸载、重载都更适合统一管理全局插件目录。

### 14.4 初步结论

倾向于：**新插件安装优先只安装到全局插件目录，工作区插件目录逐步弱化为兼容能力。**

理由：

- 全局插件 + 智能体绑定已经可以解决上下文污染；
- 插件不会因为切换工作目录而消失；
- 一个智能体无论在哪个工作目录都能稳定使用自己的插件；
- 插件更新、卸载、重载和版本管理更集中；
- 用户心智模型更简单：插件是系统能力，是否给哪个智能体使用由绑定决定。

### 14.5 不建议立刻删除工作区插件目录

短期不建议马上删除工作区插件支持，原因：

- 可能已有用户或调试流程还依赖 `.cortana/plugins`；
- `PluginLoader`、`WindowToolProvider`、`AppPaths.PluginDirectory` 仍有旧语义；
- 直接删除会影响兼容性和插件热加载流程；
- 某些一次性、实验性、项目专属插件仍可能适合工作区目录。

### 14.6 建议迁移策略

第一阶段：保留兼容，但 UI 弱化。

- 插件管理页继续显示工作区插件；
- 工作区插件不能设为全局插件；
- 安装入口默认指向全局插件目录；
- 文案提示“推荐安装到全局插件目录，再按智能体绑定”。

第二阶段：新增迁移能力。

- 对工作区插件提供“迁移到全局目录”按钮；
- 迁移后保留智能体绑定关系；
- 若插件 ID 冲突，提示用户选择覆盖、跳过或取消。

第三阶段：工作区插件降级为高级/兼容功能。

- 默认不再自动建议使用工作区插件目录；
- `sys_get_workspace_plugins_directory` 可以标记为兼容路径；
- `AppPaths.PluginDirectory` 可评估改为 `UserPluginsDirectory`，减少新安装走错目录。

最终方向：

- 常规插件：全局目录安装；
- 能力隔离：通过智能体绑定和全局开关控制；
- 工作区插件：仅保留给项目专属、临时实验、兼容旧数据使用。

## 15. 移除工作区插件支持执行计划

### 15.1 决策

当前确认：

- 现有插件没有安装在工作目录中；
- 插件数量不多；
- 插件统一放到全局目录后，可以通过“全局插件开关”和“智能体绑定”控制可用范围；
- 不再需要通过工作区插件目录来隔离上下文；
- 工作区插件反而会造成智能体跨工作目录时插件不可用的问题。

因此决定执行：**移除工作区插件支持，插件统一从全局插件目录加载。**

### 15.2 目标行为

- 插件加载器只扫描 `IAppPaths.UserPluginsDirectory`；

    - 设置为全局插件，所有智能体可用；
    - 不设置全局，只绑定给指定智能体；
- 工作目录切换不再触发插件重载；
- 工作目录不再自动创建 `.cortana/plugins`；
- AI 工具不再推荐或暴露工作区插件目录作为插件安装位置。
