## 1. 主智能体的无状态决策模型

### 1.1 核心设计：主智能体 = 无状态决策者

主智能体**不是**一个长时间运行的 LLM 会话。它是一个**按需唤醒的决策者**——
每次做一个决策（一次 API 调用，几秒完成），然后释放连接。

```
错误理解（会超时）：
  主智能体 = 一个一直保持连接的 LLM 会话
  任务执行 30 分钟 → 连接一直开着 → 超时断掉 → 任务丢失

正确理解（不会超时）：
  主智能体 = 每次被调用做一个决策的函数
  每次调用只有几秒 → 决策完成后连接释放
  子智能体执行期间 → 主智能体不存在，没有连接可以断
  子智能体完成后 → 引擎再次调用主智能体做下一个决策
```

### 1.2 与对话模式的根本区别

| 维度 | 对话模式（有超时风险） | 任务引擎模式（无超时风险） |
|------|----------------------|--------------------------|
| LLM 调用方式 | 一次调用完成所有工作 | 每步一次调用，每次只做一个决策 |
| 连接时长 | 可能几分钟~几十分钟 | 每次只有几秒~十几秒 |
| 工作量大时 | 调用时间变长 → 超时 | 调用次数增多，但每次都很短 |
| 执行长工具时 | LLM 等工具返回 → 连接一直占着 | 工具执行是程序代码，LLM 已释放 |
| 断开后 | 任务丢失，重新对话 | 从文件恢复，接续执行 |

### 1.3 执行时间线示例

```
时间线（30 分钟的复杂任务）：

 0:00  引擎调用主智能体 → "创建需求分析师"（3 秒 API 调用）→ 释放
 0:03  子智能体（需求分析师）与用户对话（每轮 3-5 秒 API 调用 × 5 轮）
 0:30  引擎调用主智能体 → "需求分析完成，创建策划师"（3 秒）→ 释放
 0:33  子智能体（策划师）制定计划（10 秒 API 调用）→ 释放
 0:43  等待用户确认（5 分钟，无 LLM 连接开销）
 5:43  引擎调用主智能体 → "用户确认了，开始步骤 1"（3 秒）→ 释放
 5:46  子智能体（数据采集员）执行步骤 1：
       - API 调用 1: "搜索 NVIDIA 财报"（3 秒）→ 释放
       - 工具执行: web_search（5 秒，程序代码，无 LLM 连接）
       - API 调用 2: "获取详情"（3 秒）→ 释放
       - 工具执行: web_fetch（8 秒，程序代码）
       - API 调用 3: "整理结果"（5 秒）→ 释放
 6:13  引擎调用主智能体 → "步骤 1 完成，开始步骤 2"（3 秒）→ 释放
 ...

整个 30 分钟任务期间：
  - LLM 实际连接时间：总共约 2 分钟（分散在十几次独立调用中）
  - 每次连接最长：十几秒
  - 超时风险：零
```

### 1.4 主智能体每次调用的输入输出

```
每次调用主智能体时：

输入（引擎构建）：
  {
    "system_prompt": "Orchestrator.SystemPrompt.md 的内容",
    "current_state": {
      "phase": "executing",
      "plan_summary": "5 个步骤，已完成 2 个",
      "completed_steps": [
        { "stepId": "step-1", "summary": "已采集 47 条 AI 芯片数据" },
        { "stepId": "step-2", "summary": "已采集 23 条终端设备数据" }

      "current_step": { "stepId": "step-3", "title": "数据分析", "status": "pending" },
      "last_event": "步骤 2 执行完成"
    }
  }

输出（主智能体决策）：
  {
    "action": "execute_step",
    "params": {
      "stepId": "step-3",
      "agentName": "数据分析师",
      "agentInstructions": "你是市场数据分析专家...",

      "context": "前置步骤已采集 NVIDIA/AMD/Intel 芯片数据和终端设备数据..."
    }
  }
```

## 2. 上下文传递机制

### 2.1 问题描述

步骤之间存在数据依赖：
- 步骤 2（数据分析）需要步骤 1（数据采集）的产出数据
- 步骤 3（报告撰写）需要步骤 2 的分析结论
- 主智能体需要知道每步的摘要结果来做调度决策

### 2.2 解决方案：文件传递 + 分层注入

```
               ┌─────────────────────────────────────┐
               │       上下文传递架构                  │
               │                                     │
               │  ┌─────────────┐                    │
               │  │ 主智能体     │ ← 只看摘要         │
               │  │ (编排决策)   │   (result.summary) │
               │  └──────┬──────┘                    │
               │         │                           │
               │    每次调用时注入                     │
               │    plan 摘要 + 最近步骤结果摘要       │
               │         │                           │
               │  ┌──────┼──────────────────────┐    │
               │  │      ▼                      │    │
               │  │ 子智能体  ← 摘要注入 prompt   │    │
               │  │ (执行者)  ← 文件按需读取     │    │
               │  │           (read_file 工具)   │    │
               │  └─────────────────────────────┘    │
               │                                     │
               │  ┌─────────────────────────────────┐│
               │  │        文件系统（持久化层）       ││
               │  │  result.summary.txt (摘要)      ││
               │  │  result.json (完整结果)         ││
               │  │  workspace/ (中间产物)          ││
               │  └─────────────────────────────────┘│
               └─────────────────────────────────────┘
```

### 2.3 三级上下文注入策略

| 级别 | 注入对象 | 内容 | 大小 | 来源 |
|------|---------|------|------|------|
| L1 | 主智能体 | 各步骤的一句话摘要 | 极小（<500 tokens） | result.summary.txt |
| L2 | 子智能体 prompt | 依赖步骤的详细摘要 | 中等（<2000 tokens） | result.summary.txt |
| L3 | 子智能体按需 | 完整数据 | 可能很大 | read_file → result.json / workspace/* |

**为什么这样分层**：
- L1：主智能体只做调度决策，不需要看数据细节，摘要足够
- L2：子智能体开始执行前需要知道上下文背景，注入详细摘要
- L3：子智能体执行过程中如果需要完整数据，通过工具按需读取，避免 context 爆炸

### 2.4 子智能体 prompt 注入示例

步骤 3（数据分析师）开始执行时，引擎构建的 prompt：

```markdown
## 你的身份
你是市场数据分析专家，擅长从原始数据中提取市场洞察。

## 你的任务
分析 AI 芯片和终端设备市场数据，产出以下内容：
- 市场规模与增长趋势
- 竞争格局分析
- 价格趋势分析

## 前置步骤的结果

### 步骤 1 - 数据采集（AI 芯片市场）
已采集 47 条数据记录，包含 NVIDIA/AMD/Intel 近 3 年财报中 AI 相关营收。
完整数据文件：.cortana/tasks/{taskId}/runs/run-20260524-011500-a3f2/steps/step-1/result.json

### 步骤 2 - 数据采集（终端设备市场）
已采集 23 条产品动态信息，涵盖手机/PC/IoT 终端 AI 芯片出货量。
完整数据文件：.cortana/tasks/{taskId}/runs/run-20260524-011500-a3f2/steps/step-2/result.json

## 你的工具
- read_file: 读取文件内容（用于获取完整数据）

## 输出要求
输出 JSON 格式的分析结果，包含 summary（摘要）和 detail（详情）。
```

子智能体根据 prompt 中的摘要信息判断需不需要读完整数据。
如果数据量不大（摘要已包含关键数字），可能直接分析；
如果需要详细数据，调用 `read_file` 按需读取。

## 3. 断点恢复机制

### 3.1 断点可能发生的位置

```
引擎循环中，任何时刻都可能崩溃：

  [A] 调用主智能体 API → 网络断开 → 决策丢失
  [B] 子智能体执行到一半 → 进程崩溃 → 中间状态
  [C] 子智能体完成 → 引擎还没写 result → 结果丢失
  [D] 引擎写了 result → 还没更新 plan.json → 状态不一致
  [E] 引擎更新了 plan.json → 还没更新 checkpoint → 进度偏差
```

### 3.2 WAL（Write-Ahead Log）思路

每个步骤的执行采用"先写意图，后写结果"的模式：

```
执行步骤 N 的完整写入序列：

  ① intent.json     ← 步骤开始前写入（声明"我要开始做这个步骤了"）
  ② workspace/*     ← 执行过程中实时写入中间产物
  ③ result.json     ← 子智能体完成后写入最终结果
  ④ plan.json       ← 更新步骤状态为 completed
  ⑤ checkpoint.json ← 更新总体进度

在 ①-⑤ 任何位置崩溃，恢复时都能从文件系统推断出实际状态。
```

### 3.3 恢复判断逻辑

```csharp
/// <summary>
/// 恢复时对单个步骤的状态推断逻辑。
/// 不只看 plan.json 的记录，还检查实际文件系统。
/// </summary>
public enum InferredStepState
{
    /// <summary>步骤从未开始（无 intent.json）。</summary>
    NeverStarted,

    /// <summary>步骤开始了但未完成（有 intent 无 result）。</summary>
    PartiallyExecuted,

    /// <summary>步骤完成了但 plan 未更新（有 result 但 plan 状态不是 completed）。</summary>
    CompletedButNotRecorded,

    /// <summary>步骤完成且 plan 已更新。</summary>
    FullyCompleted,
}

public InferredStepState InferStepState(string taskRunDir, string stepId, PlanStepStatus planStatus)
{
    var stepDir = Path.Combine(taskRunDir, "steps", stepId);
    var hasIntent = File.Exists(Path.Combine(stepDir, "intent.json"));
    var hasResult = File.Exists(Path.Combine(stepDir, "result.json"));

    return (hasIntent, hasResult, planStatus) switch
    {
        (false, false, _)                          => InferredStepState.NeverStarted,
        (true,  false, _)                          => InferredStepState.PartiallyExecuted,
        (true,  true,  not PlanStepStatus.Completed) => InferredStepState.CompletedButNotRecorded,
        (true,  true,  PlanStepStatus.Completed)   => InferredStepState.FullyCompleted,
        _                                          => InferredStepState.NeverStarted, // 异常兜底
    };
}
```

### 3.4 恢复流程

```
进程重启 → 恢复流程启动：

1. 扫描所有未完成的任务（plan.status != Completed/Cancelled/Failed）
2. 对每个未完成任务：
   a. 加载 plan.json
   b. 对每个 status=Running 的步骤，做文件检查（InferStepState）
   c. 修正 plan.json 中的状态（如果有 CompletedButNotRecorded 的步骤，更新为 completed）
   d. 构建恢复上下文（哪些步骤完成了、哪些中断了、中间产物有什么）
   e. 调用主智能体做恢复决策：
      → "步骤 3 执行到一半，workspace 里有 2/3 的文件，怎么办？"
      → 主智能体决策：接续 / 重做 / 跳过
   f. 根据决策恢复执行循环
```

### 3.5 中间产物落盘策略

子智能体执行过程中，所有有价值的中间产物**实时写入文件系统**：

```
.cortana/tasks/{taskId}/runs/{runId}/steps/{stepId}/
  intent.json           ← 执行意图（必须第一个写入）
  workspace/            ← 子智能体的工作空间
    data-nvidia.json    ← 中间产物（采集到一条就写一条）
    data-amd.json       ← 中间产物
    draft-analysis.md   ← 中间产物（分析草稿）
  result.json           ← 最终结果（子智能体完成后写入）
  result.summary.txt    ← 结果摘要（给后续步骤和 UI 用）
```

**设计原则**：
- 子智能体每完成一个有价值的中间动作，就写入一个文件
- 不要等全部完成再一次性写入
- 这样即使中途崩溃，已完成的中间产物不会丢失
- 恢复时可以告诉新的子智能体："workspace 里已经有这些文件，请接续"

### 3.6 主智能体恢复决策的 prompt 模板

```markdown
## 恢复上下文

任务 "{taskSummary}" 因中断恢复。以下是恢复前的状态：

### 计划进度
{对每个步骤列出状态}

### 需要恢复的步骤：{stepTitle}
- 开始时间：{intent.json 中的 timestamp}
- plan.json 记录状态：running
- 实际文件检查：
  - intent.json: ✅ 存在
  - result.json: ❌ 不存在（未完成）
  - workspace/ 目录内容：
    {列出 workspace 中的文件名和大小}

### 请做出恢复决策

根据 workspace 中的中间产物判断：
A) 创建新子智能体接续执行（告知已有哪些中间产物）
B) 创建新子智能体从头重做（丢弃中间产物）
C) 标记为失败，跳过此步骤继续后续

请输出 JSON 决策。
```

### 3.7 幂等性保障（不可重复操作的处理）

部分操作具有副作用（上传、发邮件、API 写入等），重试/恢复时不能重复执行。

**解决方式：操作标记文件**

```
子智能体执行"上传视频到 B 站"时：

  1. 执行前检查：workspace/uploaded.flag 是否存在？
     → 存在 → 已上传过，跳过
     → 不存在 → 继续执行
  2. 上传成功后：
     写入 workspace/uploaded.flag = { "url": "https://...", "time": "..." }
  3. 如果上传后崩溃，恢复时：
     新的子智能体检查 uploaded.flag → 发现已上传 → 跳过上传步骤
```

**在子智能体 instructions 中明确说明**：

```markdown
## 幂等性要求
本步骤包含不可重复的操作（上传/发送/写入外部系统）。
执行前请检查 workspace/ 中是否已有对应的 .flag 标记文件。
如果标记文件存在，说明该操作已在之前的执行中完成，请跳过。
操作成功后，请立即写入标记文件。
```

## 4. 文件编码规范（避免新旧执行混淆）

### 4.1 问题描述

用户可能对**同一个任务**重复执行多次：
- 第一次执行：计划制定完成，执行到步骤 3 失败 → 用户取消
- 第二次执行：用户修改需求后重新开始
- 第三次执行：基于模板再次执行类似任务

如果所有执行的文件都放在同一个目录下，会出现：
- 读取到旧的 result.json（上次执行的结果被误认为本次的）
- 旧的 checkpoint.json 导致恢复到错误的位置
- 旧的 workspace 文件污染新执行的子智能体上下文

### 4.2 核心解决方案：执行运行 ID（Run ID）

每次启动一个任务的执行（包括首次执行和每次"重新执行"），
都生成一个唯一的 **Run ID**，所有该次执行的文件都隔离在 Run ID 目录下。

```
Run ID 格式：run-{yyyyMMdd}-{HHmmss}-{4位随机hex}

示例：
  run-20260524-011500-a3f2   ← 2026-05-24 01:15:00 启动，随机后缀 a3f2
  run-20260524-143022-7b1e   ← 同一天第二次执行
```

### 4.3 文件目录结构

```
{WorkspaceDir}/
  .cortana/                          ← 系统数据目录（隐藏）
    tasks/
      {taskId}/                      ← 任务级目录（跨执行共享）
        task.json                    ← 任务元信息（标题、创建时间、状态）
        runs/                        ← 所有执行记录
          run-20260524-011500-a3f2/  ← 第 1 次执行
            run.json                 ← 本次运行元信息（开始时间、状态、关联 planVersion）
            plan.json                ← 本次执行的计划（完整快照）
            plan.md                  ← 本次执行的计划摘要（用户查看）
            checkpoint.json          ← 本次执行的断点
            requirements.json        ← 本次执行的需求分析结果
            conversation.json        ← 本次执行的对话历史
            steps/
              step-1/
                intent.json
                workspace/
                  data-nvidia.json
                  data-amd.json
                result.json
                result.summary.txt
              step-2/
                intent.json
                workspace/
                result.json
                result.summary.txt
              step-3/
                intent.json          ← 执行到此步崩溃
                workspace/
                  partial-data.json  ← 中间产物
          run-20260524-143022-7b1e/  ← 第 2 次执行（重新开始）
            run.json
            plan.json                ← 可能是修改后的新计划
            plan.md
            checkpoint.json
            steps/
              step-1/
                ...                  ← 全新的执行，不会读到上次的文件
        templates/                   ← 从此任务保存的模板
          tpl-market-analysis.json
        latest-run                   ← 符号链接或文本文件，指向最新的 run 目录名
```

### 4.4 Run ID 的生命周期

```
新任务创建 → taskId 生成（UUID）→ 创建 .cortana/tasks/{taskId}/ 目录
  → task.json 写入元信息

用户确认计划，开始执行 → 生成 Run ID → 创建 runs/{runId}/ 目录
  → run.json 写入（startTime, status: "running", planVersion: 1）
  → plan.json 写入（本次执行的计划快照）
  → 更新 latest-run 指向此 runId

执行过程中 → 所有文件写入 runs/{runId}/ 下

执行完成 → run.json 更新（status: "completed", endTime, report）
         → task.json 更新（lastRunId, status）

用户重新执行 → 生成新 Run ID → 创建新的 runs/{newRunId}/ 目录
  → 完全隔离，不会读到旧 run 的文件
  → 更新 latest-run 指向新 runId
```

### 4.5 恢复时如何找到正确的执行

```
进程重启 → 恢复流程：

1. 扫描 .cortana/tasks/ 下所有 taskId 目录
2. 对每个 task：
   a. 读取 latest-run → 得到最新的 runId
   b. 读取 runs/{runId}/run.json
   c. 如果 status == "running" → 需要恢复
   d. 只在这个 runId 目录下做文件检查和恢复
3. 不会误读到旧的 run 目录（因为 latest-run 明确指向）
```

### 4.6 重新执行 vs 恢复执行的区别

| 操作 | 触发方式 | 行为 |
|------|---------|------|
| **恢复执行** | 进程重启自动触发 | 读取 latest-run 指向的 runId，接续执行 |
| **重新执行** | 用户点击"重新开始" | 生成新 runId，全新目录，从头开始 |
| **修改后恢复** | 用户暂停→修改计划→继续 | 同一个 runId，plan.json version+1，增量分析 |

### 4.7 旧执行数据的清理策略

```
规则：
  - 最新的 run 永远保留（latest-run 指向的）
  - 已完成（completed）的 run 保留最近 N 个（默认 5，可配置）
  - 已取消/失败的 run 保留最近 3 个
  - 超出保留数量的旧 run → 自动清理（删除目录）
  - 用户可手动清理（UI 提供"清理历史执行"按钮）

清理时机：
  - 新 run 创建时，检查是否需要清理旧 run
  - 应用启动时，后台扫描清理
```

### 4.8 Run ID 在代码中的使用

```csharp
/// <summary>
/// 执行运行 ID 生成器。
/// 格式：run-{yyyyMMdd}-{HHmmss}-{4位随机hex}
/// 保证同一任务的多次执行完全隔离。
/// </summary>
public static class RunIdGenerator
{
    /// <summary>生成新的 Run ID。</summary>
    public static string Generate()
    {
        var now = DateTimeOffset.Now;
        var random = Random.Shared.Next(0, 0xFFFF).ToString("x4");
        return $"run-{now:yyyyMMdd}-{now:HHmmss}-{random}";
    }

    /// <summary>从 Run ID 中解析启动时间（用于排序和清理）。</summary>
    public static DateTimeOffset? ParseStartTime(string runId)
    {
        // run-20260524-011500-a3f2 → 2026-05-24 01:15:00
        if (runId.Length < 20 || !runId.StartsWith("run-")) return null;
        var datePart = runId.Substring(4, 8);  // 20260524
        var timePart = runId.Substring(13, 6); // 011500
        if (DateTime.TryParseExact(
            datePart + timePart, "yyyyMMddHHmmss",
            null, System.Globalization.DateTimeStyles.None, out var dt))
        {
            return new DateTimeOffset(dt, TimeZoneInfo.Local.GetUtcOffset(dt));
        }
        return null;
    }
}

/// <summary>
/// 任务文件路径解析器。
/// 所有文件路径都基于 Run ID 隔离，保证不会跨执行混淆。
/// </summary>
public sealed class TaskFileResolver
{
    private readonly string _workspaceDir;

    public TaskFileResolver(string workspaceDir)
    {
        _workspaceDir = workspaceDir;
    }

    /// <summary>任务根目录。</summary>
    public string GetTaskDir(string taskId)
        => Path.Combine(_workspaceDir, ".cortana", "tasks", taskId);

    /// <summary>执行运行目录。</summary>
    public string GetRunDir(string taskId, string runId)
        => Path.Combine(GetTaskDir(taskId), "runs", runId);

    /// <summary>步骤目录。</summary>
    public string GetStepDir(string taskId, string runId, string stepId)
        => Path.Combine(GetRunDir(taskId, runId), "steps", stepId);

    /// <summary>步骤工作空间。</summary>
    public string GetStepWorkspace(string taskId, string runId, string stepId)
        => Path.Combine(GetStepDir(taskId, runId, stepId), "workspace");

    /// <summary>读取 latest-run 指向的 runId。</summary>
    public string? GetLatestRunId(string taskId)
    {
        var path = Path.Combine(GetTaskDir(taskId), "latest-run");
        return File.Exists(path) ? File.ReadAllText(path).Trim() : null;
    }

    /// <summary>更新 latest-run。</summary>
    public void SetLatestRunId(string taskId, string runId)
    {
        var path = Path.Combine(GetTaskDir(taskId), "latest-run");
        Directory.CreateDirectory(Path.GetDirectoryName(path)!);
        File.WriteAllText(path, runId);
    }
}
```

```csharp
public sealed class TaskExecutionEngine
{
    public async Task ExecuteAsync(string taskId, string runId, CancellationToken ct)
    {
        var runDir = fileResolver.GetRunDir(taskId, runId);
        var plan = await LoadPlan(runDir);

        while (!scheduler.IsAllCompleted(plan) && !ct.IsCancellationRequested)
        {
            // ═══ 1. 构建主智能体上下文 ═══
            var context = BuildOrchestratorContext(plan, runDir);
            // context 包含：当前阶段、各步骤摘要、最近事件

            // ═══ 2. 调用主智能体做决策（几秒的 API 调用） ═══
            var decision = await orchestratorLLM.GetDecision(context, ct);

            // ═══ 3. 根据决策执行 ═══
            switch (decision.Action)
            {
                case "execute_step":
                    await ExecuteStep(taskId, runId, plan, decision, ct);
                    break;

                case "request_user_input":
                    await PauseForUserInput(taskId, runId, decision.Question, ct);
                    break;

                case "task_complete":
                    await FinalizeTask(taskId, runId, plan, decision.Report, ct);
                    return;

                // ... 其他 action
            }

            // ═══ 4. 持久化（每次决策后） ═══
            await SavePlan(runDir, plan);
            await SaveCheckpoint(runDir, plan);
        }
    }

    private async Task ExecuteStep(
        string taskId, string runId, ExecutionPlan plan,
        OrchestratorDecision decision, CancellationToken ct)
    {
        var step = plan.Steps.First(s => s.StepId == decision.StepId);
        var stepDir = fileResolver.GetStepDir(taskId, runId, step.StepId);
        Directory.CreateDirectory(stepDir);

        // ═══ 写入 intent（WAL 第一步） ═══
        await WriteIntent(stepDir, step);

        // ═══ 构建子智能体上下文 ═══
        var subAgentContext = BuildSubAgentContext(plan, step, runId, taskId);
        // 包含：依赖步骤的 result.summary.txt 内容 + 文件路径（按需读取）

        // ═══ 子智能体执行（可能较久，但每次 API 调用都很短） ═══
        var result = await subAgentExecutor.Execute(
            agentName: decision.AgentName,
            instructions: decision.AgentInstructions,
            tools: decision.AgentTools,
            context: subAgentContext,
            workspaceDir: fileResolver.GetStepWorkspace(taskId, runId, step.StepId),
            ct: ct);

        // ═══ 写入 result（WAL 第二步） ═══
        await WriteResult(stepDir, result);
        await WriteResultSummary(stepDir, result.Summary);

        // ═══ 更新 plan（WAL 第三步） ═══
        step.Status = PlanStepStatus.Completed;
        step.ResultSummary = result.Summary;
        step.CompletedAt = DateTimeOffset.UtcNow;

        // ═══ 发布事件 → UI 更新 ═══
        publisher.Publish(Events.OnStepCompleted, new StepCompletedArgs(taskId, step));
    }

    /// <summary>
    /// 构建子智能体的上下文：收集依赖步骤的摘要。
    /// </summary>
    private string BuildSubAgentContext(
        ExecutionPlan plan, PlanStep currentStep, string runId, string taskId)
    {
        var sb = new StringBuilder();
        sb.AppendLine("## 前置步骤的结果");
        sb.AppendLine();

        foreach (var depId in currentStep.DependsOn)
        {
            var depStep = plan.Steps.FirstOrDefault(s => s.StepId == depId);
            if (depStep is null) continue;

            // 读取摘要文件
            var summaryPath = Path.Combine(
                fileResolver.GetStepDir(taskId, runId, depId), "result.summary.txt");
            var summary = File.Exists(summaryPath)
                ? File.ReadAllText(summaryPath)
                : depStep.ResultSummary ?? "(无摘要)";

            sb.AppendLine($"### {depStep.Title}（步骤 {depStep.Sequence}）");
            sb.AppendLine(summary);
            sb.AppendLine();

            // 告知完整数据的文件路径（子智能体按需读取）
            var resultPath = Path.Combine(
                fileResolver.GetStepDir(taskId, runId, depId), "result.json");
            if (File.Exists(resultPath))
            {
                sb.AppendLine($"完整数据文件：{resultPath}");
                sb.AppendLine();
            }
        }

        return sb.ToString();
    }
}
```

### 5.2 恢复执行的完整流程

```csharp
public async Task ResumeAsync(string taskId, CancellationToken ct)
{
    // 1. 找到最新的 run
    var runId = fileResolver.GetLatestRunId(taskId);
    if (runId is null) throw new InvalidOperationException("无执行记录可恢复");

    var runDir = fileResolver.GetRunDir(taskId, runId);
    var plan = await LoadPlan(runDir);

    // 2. 对每个 running 状态的步骤做文件检查
    var recoveryInfo = new List<StepRecoveryInfo>();
    foreach (var step in plan.Steps.Where(s => s.Status == PlanStepStatus.Running))
    {
        var inferred = InferStepState(
            fileResolver.GetStepDir(taskId, runId, step.StepId), step);
        recoveryInfo.Add(new StepRecoveryInfo(step, inferred));

        // 如果文件显示已完成但 plan 未更新 → 直接修正
        if (inferred == InferredStepState.CompletedButNotRecorded)
        {
            var result = await LoadResult(
                fileResolver.GetStepDir(taskId, runId, step.StepId));
            step.Status = PlanStepStatus.Completed;
            step.ResultSummary = result.Summary;
            step.CompletedAt = result.CompletedAt;
        }
    }

    // 3. 对真正需要恢复的步骤（PartiallyExecuted），询问主智能体
    var partialSteps = recoveryInfo
        .Where(r => r.State == InferredStepState.PartiallyExecuted)
        .ToList();

    if (partialSteps.Count > 0)
    {
        var recoveryContext = BuildRecoveryContext(plan, partialSteps, taskId, runId);
        var decision = await orchestratorLLM.GetRecoveryDecision(recoveryContext, ct);
        await ApplyRecoveryDecision(plan, decision, taskId, runId, ct);
    }

    // 4. 保存修正后的 plan
    await SavePlan(runDir, plan);

    // 5. 进入正常执行循环
    await ExecuteAsync(taskId, runId, ct);
}

/// <summary>
/// 检查步骤目录中的实际文件，推断真实执行状态。
/// </summary>
private InferredStepState InferStepState(string stepDir, PlanStep step)
{
    var hasIntent = File.Exists(Path.Combine(stepDir, "intent.json"));
    var hasResult = File.Exists(Path.Combine(stepDir, "result.json"));

    if (!hasIntent) return InferredStepState.NeverStarted;
    if (hasResult && step.Status != PlanStepStatus.Completed)
        return InferredStepState.CompletedButNotRecorded;
    if (hasResult) return InferredStepState.FullyCompleted;
    return InferredStepState.PartiallyExecuted;
}

/// <summary>
/// 构建恢复上下文：列出中断步骤的 workspace 文件状况。
/// </summary>
private string BuildRecoveryContext(
    ExecutionPlan plan, List<StepRecoveryInfo> partialSteps,
    string taskId, string runId)
{
    var sb = new StringBuilder();
    sb.AppendLine("## 恢复上下文");
    sb.AppendLine();
    sb.AppendLine($"任务 \"{plan.TaskSummary}\" 因中断需要恢复。");
    sb.AppendLine();

    foreach (var info in partialSteps)
    {
        var step = info.Step;
        var workspace = fileResolver.GetStepWorkspace(taskId, runId, step.StepId);

        sb.AppendLine($"### 步骤 {step.Sequence}: {step.Title}");
        sb.AppendLine($"- 状态: 执行到一半中断");

        if (Directory.Exists(workspace))
        {
            var files = Directory.GetFiles(workspace);
            if (files.Length > 0)
            {
                sb.AppendLine("- workspace/ 中间产物：");
                foreach (var f in files)
                {
                    var fi = new FileInfo(f);
                    sb.AppendLine($"  - {fi.Name} ({fi.Length / 1024}KB)");
                }
            }
            else
            {
                sb.AppendLine("- workspace/ 为空（子智能体尚未产出任何文件）");
            }
        }
        sb.AppendLine();
    }

    sb.AppendLine("### 请对每个中断步骤做出恢复决策");
    sb.AppendLine("A) resume — 接续执行（告知子智能体已有哪些中间产物）");
    sb.AppendLine("B) redo — 从头重做（清空 workspace）");
    sb.AppendLine("C) skip — 标记为跳过，继续后续步骤");
    sb.AppendLine();
    sb.AppendLine("请输出 JSON 格式的决策数组。");

    return sb.ToString();
}
```

## 6. 设计决策总结

| 编号 | 决策 | 选择 | 理由 |
|------|------|------|------|
| P4-S1 | 主智能体调用模式 | 无状态决策者（每次独立 API 调用） | 彻底消除超时风险 |
| P4-S2 | 上下文传递方式 | 文件系统 + 分层注入（摘要 in prompt + 详情按需 read_file） | 避免 context 爆炸 |
| P4-S3 | 断点恢复策略 | WAL + 文件状态推断 + 主智能体恢复决策 | 任何时刻崩溃都能恢复 |
| P4-S4 | 执行隔离机制 | Run ID 目录隔离（每次执行独立目录） | 避免新旧执行混淆 |
| P4-S5 | Run ID 格式 | `run-{yyyyMMdd}-{HHmmss}-{4hex}` | 可读 + 可排序 + 唯一 |
| P4-S6 | 中间产物策略 | 实时写入 workspace/（不等执行完） | 崩溃时最大化保留已完成工作 |
| P4-S7 | 幂等性保障 | 操作标记文件（.flag） | 不可重复操作不会重复执行 |
| P4-S8 | 旧执行清理 | 保留最近 N 个（默认 completed 5 / failed 3），超出自动清理 | 避免磁盘无限膨胀 |
| P4-S9 | 子智能体获取详细数据 | 通过 read_file 工具按需读取 workspace 和 result.json | 子智能体自主决定需不需要看详情 |
