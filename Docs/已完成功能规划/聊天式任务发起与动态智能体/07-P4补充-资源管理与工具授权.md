## 1. 多任务并行时的资源管理

### 1.1 问题描述

用户可能同时运行多个长任务：
- 任务 A：市场分析报告（5 个步骤，并发 3）
- 任务 B：视频脚本生成（3 个步骤，并发 2）
- 任务 C：代码审查（2 个步骤，并发 1）

每个任务内部已有并发控制（`EngineConfig.MaxConcurrentAgents`），
但**跨任务的全局并发**没有控制——可能同时有 3+2+1=6 个子智能体同时调 LLM API。

风险：
- LLM API rate limit 被打满 → 大量 429 错误 → 所有任务都在重试
- 本地资源（内存/CPU）压力，特别是嵌入/工具调用同时触发时
- 用户体验：所有任务都变慢，不如集中资源先跑完一个

### 1.2 方案建议：全局信号量 + 任务优先级队列

```
核心思路：
  引入一个进程级全局信号量（GlobalLlmSemaphore），控制所有任务共享的 LLM 并发上限。
  每个子智能体在调用 LLM 前必须先获取全局信号量，获取后再执行。

  任务内部的 MaxConcurrentAgents 是该任务的"上限"，
  但实际并发受全局信号量约束——即 min(任务内上限, 全局剩余槽位)。
```

**架构图**：

```
                         ┌─────────────────────────────┐
                         │  GlobalLlmSemaphore          │
                         │  (进程级, 默认上限 = 5)       │
                         └──────────┬──────────────────┘
                                    │
                    ┌───────────────┼───────────────┐
                    │               │               │
              ┌─────▼─────┐  ┌─────▼─────┐  ┌─────▼─────┐
              │ 任务 A     │  │ 任务 B     │  │ 任务 C     │
              │ 内部限 3   │  │ 内部限 2   │  │ 内部限 1   │
              │ SemaphoreA │  │ SemaphoreB │  │ SemaphoreC │
              └────────────┘  └────────────┘  └────────────┘

  子智能体要调 LLM 时：
    1. await taskSemaphore.WaitAsync()     ← 任务内并发控制
    2. await globalSemaphore.WaitAsync()   ← 全局并发控制
    3. 调 LLM API
    4. globalSemaphore.Release()
    5. taskSemaphore.Release()
```

### 1.3 全局并发上限的默认值选择

| 方案 | 默认值 | 理由 |
|------|--------|------|
| **A（推荐）** | 5 | 主流 LLM API（Claude / GPT / Gemini）的默认 RPM 在 50-100 之间。每次子智能体调用含 1-5 次 API 请求（多轮 tool use），5 并发 × 平均 3 次 = 15 RPM 峰值，远低于限流。留出余量给 Chat 模式同时使用。 |
| B | 3 | 保守安全，但多任务时所有任务都很慢 |
| C | 10 | 激进，用户本地配置如果是小厂 API 可能直接打满 |

**推荐 A（默认 5）**，理由：
- 单任务时有效并发 = min(任务内3, 全局5) = 3，和现在一样
- 双任务时有效并发 = 3+2=5（刚好打满全局），两个任务都能跑但速度略降
- 三任务时第三个任务的子智能体需要排队，不会让 API 过载
- 用户可在 engine.json 中调整

### 1.4 优先级策略

当全局信号量满了，多个任务的子智能体排队时，谁先拿到槽位？

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| **FIFO（推荐）** | 先来先服务，谁先 await 谁先拿 | P4 初版使用，简单可靠 |
| 任务优先级 | 用户手动标记优先级（高/中/低） | 远期可选 |
| 前台优先 | 用户当前正在查看的任务优先 | 远期可选 |

**P4 初版使用 FIFO**。理由：
- `SemaphoreSlim` 本身就是 FIFO（.NET 实现保证公平性）
- 不需要额外数据结构
- 远期如果需要优先级，可以替换为 `PriorityQueue<TaskCompletionSource>` 自定义调度

### 1.5 代码骨架

```csharp
/// <summary>
/// 全局 LLM 并发控制器。
/// 所有 TaskExecutionEngine 实例共享同一个 GlobalLlmThrottle。
/// DI 注册为 Singleton。
/// </summary>
public sealed class GlobalLlmThrottle : IDisposable
{
    private readonly SemaphoreSlim _semaphore;
    private int _maxConcurrency;

    public GlobalLlmThrottle(EngineConfig config)
    {
        _maxConcurrency = config.GlobalMaxConcurrentLlmCalls;
        _semaphore = new SemaphoreSlim(_maxConcurrency, _maxConcurrency);
    }

    /// <summary>当前可用槽位数。UI 可读取展示"全局负载"。</summary>
    public int AvailableSlots => _semaphore.CurrentCount;

    /// <summary>当前等待中的请求数。</summary>
    public int QueueLength => _maxConcurrency - _semaphore.CurrentCount;

    /// <summary>
    /// 获取全局 LLM 调用许可。子智能体调 LLM 前必须先获取。
    /// 返回 IDisposable 用于自动释放。
    /// </summary>
    public async Task<IDisposable> AcquireAsync(CancellationToken ct)
    {
        await _semaphore.WaitAsync(ct);
        return new SlotRelease(_semaphore);
    }

    /// <summary>
    /// 动态调整上限（UI 配置修改时调用）。
    /// 注：SemaphoreSlim 不支持缩减，需要重建。扩展可以直接 Release 差值。
    /// 简化做法：仅在无任务运行时允许修改（P4 初版限制）。
    /// </summary>
    public void UpdateMaxConcurrency(int newMax)
    {
        // P4 初版：记录新值，下次创建 SemaphoreSlim 时生效
        // 运行中动态调整作为远期优化
        _maxConcurrency = newMax;
    }

    public void Dispose() => _semaphore.Dispose();

    private sealed class SlotRelease(SemaphoreSlim semaphore) : IDisposable
    {
        public void Dispose() => semaphore.Release();
    }
}
```

**子智能体执行时使用**：

```csharp
private async Task<StepResult> InvokeSubAgentAsync(
    AIAgent subAgent, string query, CancellationToken ct)
{
    // 先获取全局信号量
    await using var slot = await _globalThrottle.AcquireAsync(ct);

    // 再执行 LLM 调用

}
```

### 1.8 与 Chat 模式的隔离

Chat 模式（直接对话）不经过全局信号量——它是用户实时交互，不应被后台任务阻塞。

```
规则：
  - 任务执行引擎的子智能体 → 受全局信号量控制
  - Chat 模式对话 → 不受全局信号量控制（直接调 LLM）
  - 理由：Chat 是用户实时等待的交互，不能因为后台任务满载而卡住

  如果 Chat + 任务同时打满 API rate limit → 任务侧自动重试（已有 RetryPolicy）
```

### 1.9 风险与回退

| 风险 | 缓解 |
|------|------|
| 全局信号量默认 5 对某些低 RPM API 太高 | 引导用户在 engine.json 调低 |
| 任务 A 长时间占满全部槽位 → 任务 B 饿死 | FIFO 保证公平性；远期加优先级 |
| SemaphoreSlim 不支持运行时缩减 | P4 初版限制：仅无任务时可改配置 |

## 2. 执行模板的版本兼容性

### 2.1 结论（用户已确认）

**不存在版本兼容性问题**。策略非常简单：

```
保存模板时：完整保存当时的编排提示词快照
使用模板时：直接用模板中的提示词替换当前主智能体提示词
修改模板时：修改后就是新的，旧版本不保留

即：模板没有"版本"概念，只有"最新状态"。
```

### 2.2 详细说明

| 场景 | 行为 |
|------|------|
| 用户保存模板 | 保存当前提示词全文 + 步骤结构 → `ExecutionTemplate` |
| 用户修改模板 | 直接修改 → 覆盖保存 → 修改前的内容**不保留** |
| 系统升级默认提示词 | 旧模板中的提示词不受影响（它保存的是快照，不是引用） |
| 用户想用新版默认提示词 | 删除旧模板 → 用新默认跑一次 → 满意后重新保存为模板 |

### 2.3 对之前设计的简化

之前 `06` 文档中预留了 `BasePromptVersion` 字段：

```csharp
/// <summary>保存模板时记录基于哪个版本的默认提示词。</summary>
public string BasePromptVersion { get; set; } = "1.0";
```

根据用户确认的策略，**此字段可以删除**。模板不关心基于哪个版本——保存什么就是什么，
没有"升级旧模板到新版本"的需求。

简化后的 `ExecutionTemplate`：

```csharp
public sealed class ExecutionTemplate
{
    public required string TemplateId { get; init; }
    public string Name { get; set; } = string.Empty;
    public string Description { get; set; } = string.Empty;
    public List<string> Tags { get; set; } = [];
    public string SourceTaskId { get; set; } = string.Empty;

    /// <summary>
    /// 主智能体的完整提示词快照。
    /// 使用模板时直接替换主智能体的系统提示词。
    /// 如果为空，使用系统默认万能提示词。
    /// </summary>
    public string OrchestratorPrompt { get; set; } = string.Empty;

    /// <summary>步骤结构模板（精简版）。</summary>
    public List<TemplateStep> Steps { get; set; } = [];

    public DateTimeOffset CreatedAt { get; init; }
    public int UsageCount { get; set; } = 0;
    public string WorkspaceId { get; set; } = string.Empty;

    // ❌ 已删除 BasePromptVersion — 不需要版本管理
}
```

### 2.4 设计哲学

```
模板 = 快照，不是引用。
快照没有版本问题——它就是那一刻的状态。
用户修改它，旧的就不存在了。
简单、清晰、没有隐藏的复杂度。
```

## 3. 子智能体工具授权：接入 MAF 显式授权机制

### 3.1 现有机制回顾

当前系统中已有两层工具管控：

**层级 1：任务发起时的高风险工具黑名单（UI 层）**

```
用户在 WorkflowInputVm 中看到高风险工具列表：
  ☐ C# 脚本执行 (sys_csharp_script)
  ☐ 进程启动 (sys_process)
  ☐ 窗口管理 (sys_window_manager)
  ☐ Office 文档操作 (sys_office)

默认全部不勾选 = 全部允许。
用户勾选 = 屏蔽（黑名单模式）。
```

**层级 2：动态创建子智能体时的工具校验（后端）**

```
CreateSubAgentTool 中：
  1. requiredTools 校验拼写真实性（factory.GetAvailableToolNames()）
  2. BuildDynamicSubAgent 按白名单注入工具（仅 requiredTools 中列的）
  3. DynamicAgentCreationGate 用户审批（可 auto-approve）
```

### 3.2 MAF 框架中的工具授权能力

MAF（Microsoft Agents Framework）中的相关机制：

| 能力 | 说明 | 我们当前使用状况 |
|------|------|----------------|
| `ExternalRequest` / HITL | 工作流运行中暂停等外部响应 | ✅ 已接入（PendingHitlContext） |
| `DynamicAgentCreationGate` | 动态创建子智能体时暂停等用户审批 | ✅ 已接入 |
| Tool-level filtering | SDK Workflow 可配置 tool filter | ❌ 未接入（我们自己做白名单） |
| `IToolAuthorization` 接口 | 每次工具调用前可拦截审批 | ❌ 未接入 |

### 3.3 P4 执行引擎的工具授权设计

**核心原则**：我们可以接入类似 MAF 的显式授权机制，但设计上要分三级控制——
避免用户被频繁打扰（每次工具调用都弹窗确认会让体验崩溃）。

#### 三级工具授权模型

```
┌─────────────────────────────────────────────────────────────────────┐
│                        工具授权三级模型                               │
│                                                                     │
│  Level 0 — 全局禁止（黑名单）                                        │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │ 用户在发起任务时勾选的高风险工具                                │  │
│  │ → 整个任务生命周期内完全不可用                                  │  │
│  │ → 子智能体的 requiredTools 中包含被禁工具 → 创建时直接报错      │  │
│  │ → 不会弹任何确认，直接拒绝                                     │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                     │
│  Level 1 — 首次授权（白名单 + 首次确认）                             │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │ 子智能体创建时声明 requiredTools                                │  │
│  │ → DynamicAgentCreationGate 审批时，用户可以看到工具列表         │  │
│  │ → 用户批准创建 = 授权该子智能体使用这些工具                     │  │
│  │ → 后续该子智能体调用已授权工具时，不再弹窗                      │  │
│  │                                                                │  │
│  │ 这是当前 P2 已有的机制！只是语义上重新定义为"工具授权"。         │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                     │
│  Level 2 — 逐次授权（高敏感操作，每次都确认）                        │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │ 某些特定工具被标记为 RequirePerCallAuthorization                │  │
│  │ → 即使子智能体创建时已批准，每次实际调用时仍暂停等用户确认      │  │
│  │ → 用户看到：工具名 + 参数 + "允许 / 拒绝 / 全部允许"           │  │
│  │                                                                │  │
│  │ 适用场景：                                                     │  │
│  │   - 文件删除（delete_file）— 不可逆                             │  │
│  │   - 发送邮件（send_email）— 有外部副作用                        │  │
│  │   - 支付/转账类操作 — 不可逆 + 有经济后果                       │  │
│  │   - 发布到公开平台（publish_to_xxx）— 有社会影响                │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.4 实现设计

#### A. 工具分类注册

```csharp
/// <summary>
/// 工具的授权等级。注册工具时由工具提供者声明。
/// </summary>
public enum ToolAuthorizationLevel
{
    /// <summary>普通工具，首次授权后不再确认。</summary>
    Normal = 0,

    /// <summary>高风险工具，UI 黑名单中展示供用户禁用。</summary>
    HighRisk = 1,

    /// <summary>极高风险/不可逆工具，每次调用都需要用户确认。</summary>
    RequirePerCallAuth = 2,
}

/// <summary>
/// 工具注册时附带的授权元信息。
/// 由插件/MCP 工具提供者在注册时声明。
/// </summary>
public sealed class ToolAuthorizationMeta
{
    /// <summary>工具名称。</summary>
    public string ToolName { get; init; } = string.Empty;

    /// <summary>授权等级。</summary>
    public ToolAuthorizationLevel Level { get; init; } = ToolAuthorizationLevel.Normal;

    /// <summary>
    /// 逐次授权时的风险描述（展示给用户看的，解释为什么要确认）。
    /// 仅 Level == RequirePerCallAuth 时有效。
    /// </summary>
    public string? RiskDescription { get; init; }

    /// <summary>
    /// 逐次授权时是否允许用户选"后续全部允许"（本任务内不再确认此工具）。
    /// 默认 true — 用户可以选择信任后放行。
    /// </summary>
    public bool AllowBlanketApproval { get; init; } = true;
}
```

#### B. 逐次授权拦截器

```csharp
/// <summary>
/// 工具调用授权拦截器。
/// 子智能体调用工具前经过此拦截器。
/// </summary>
public sealed class ToolCallAuthorizationGate
{
    private readonly ConcurrentDictionary<string, TaskCompletionSource<ToolCallDecision>> _pending = new();

    /// <summary>本任务中已被用户"全部允许"的工具（不再弹窗）。</summary>
    private readonly ConcurrentDictionary<string, HashSet<string>> _blanketApprovedByTask = new();

    /// <summary>
    /// 工具调用前检查。对于 RequirePerCallAuth 级别的工具，暂停等用户确认。
    /// </summary>
    /// <returns>true = 允许执行；false = 用户拒绝。</returns>
    public async Task<bool> CheckAuthorizationAsync(
        string taskId,
        string toolName,
        IDictionary<string, object?> parameters,
        string callerAgentName,
        ToolAuthorizationMeta meta,
        CancellationToken ct)
    {
        // Level 0 / Level 1 不走这里（Level 0 在创建子智能体时已拒绝；Level 1 创建时已授权）
        if (meta.Level != ToolAuthorizationLevel.RequirePerCallAuth)
            return true;

        // 检查是否已被 blanket approve
        if (_blanketApprovedByTask.TryGetValue(taskId, out var approved)
            && approved.Contains(toolName))
            return true;

        // 暂停等待用户决策
        var requestId = $"{taskId}_{toolName}_{Guid.NewGuid():N}"[..32];
        var tcs = new TaskCompletionSource<ToolCallDecision>(
            TaskCreationOptions.RunContinuationsAsynchronously);
        _pending[requestId] = tcs;

        // 发布事件 → UI 弹出授权确认卡片
        publisher.Publish(Events.OnToolCallAuthRequired, new ToolCallAuthArgs(
            TaskId: taskId,
            RequestId: requestId,
            ToolName: toolName,
            Parameters: parameters,
            CallerAgentName: callerAgentName,
            RiskDescription: meta.RiskDescription ?? "此操作需要您确认后才能执行。",
            AllowBlanketApproval: meta.AllowBlanketApproval));

        try
        {
            var decision = await tcs.Task.WaitAsync(ct);
            _pending.TryRemove(requestId, out _);

            switch (decision)
            {
                case ToolCallDecision.Allow:
                    return true;

                case ToolCallDecision.AllowAll:
                    // 用户选了"后续全部允许" → 加入 blanket 名单
                    _blanketApprovedByTask
                        .GetOrAdd(taskId, _ => new HashSet<string>(StringComparer.OrdinalIgnoreCase))
                        .Add(toolName);
                    return true;

                case ToolCallDecision.Deny:
                default:
                    return false;
            }
        }
        catch (OperationCanceledException)
        {
            _pending.TryRemove(requestId, out _);
            return false;
        }
    }

    /// <summary>UI 调用：用户做出决策。</summary>
    public void ResolveDecision(string requestId, ToolCallDecision decision)
    {
        if (_pending.TryRemove(requestId, out var tcs))
            tcs.TrySetResult(decision);
    }

    /// <summary>任务结束时清理。</summary>
    public void ClearTask(string taskId)
    {
        _blanketApprovedByTask.TryRemove(taskId, out _);
        // 取消所有该任务的 pending 请求
        foreach (var (key, tcs) in _pending)
        {
            if (key.StartsWith(taskId))
            {
                tcs.TrySetCanceled();
                _pending.TryRemove(key, out _);
            }
        }
    }
}

/// <summary>用户对工具调用的决策。</summary>
public enum ToolCallDecision
{
    /// <summary>允许本次调用。</summary>
    Allow,

    /// <summary>允许本次 + 后续同工具调用（本任务内不再询问）。</summary>
    AllowAll,

    /// <summary>拒绝本次调用。</summary>
    Deny,
}
```

#### D. 与现有 DynamicAgentCreationGate 的关系

```
创建子智能体时（DynamicAgentCreationGate）：
  → 用户审批包含工具列表
  → 审批 = 授权该子智能体使用这些工具（Level 1 授权）
  → 对于 Normal + HighRisk 级别的工具，审批后不再干预

子智能体实际调用工具时（ToolCallAuthorizationGate）：
  → 仅对 RequirePerCallAuth 级别的工具生效
  → 即使创建时已审批，每次实际执行仍要确认

两个 Gate 串联工作，不冲突：
  DynamicAgentCreationGate → "你可以用这些工具"（准入）
  ToolCallAuthorizationGate → "这次调用需要确认一下"（执行授权）
```

### 3.6 MCP 工具如何声明授权等级

MCP 工具在注册时可以通过扩展元数据声明：

```json
{
  "name": "send_email",
  "description": "发送邮件到指定地址",
  "inputSchema": { ... },
  "x-cortana-auth": {
    "level": "require_per_call_auth",
    "riskDescription": "此操作会向外部发送邮件，发送后不可撤回。",
    "allowBlanketApproval": true
  }
```

如果 MCP 工具没有声明 `x-cortana-auth`，默认为 `Normal`。

### 3.7 与 MAF 框架 HITL 的对接

我们的 `ToolCallAuthorizationGate` 本质上就是 MAF 的 HITL 机制在工具层面的应用。
差异在于：

| 维度 | MAF HITL | 我们的实现 |
|------|----------|----------|
| 触发时机 | SDK 内部 ExternalRequest 事件 | 我们自己在工具调用前拦截 |
| 等待机制 | StreamingRun.SendResponseAsync | TaskCompletionSource 自管理 |
| UI 呈现 | 由应用方实现 | TimelineCard 内嵌确认卡片 |
| 取消语义 | 与 run 生命周期绑定 | 与任务级 CancellationToken 绑定 |

**为什么不直接用 MAF SDK 的 HITL**：
- P4 执行引擎是自建的，不跑在 SDK Workflow 之上
- 子智能体执行是我们自己调度的，不通过 `InProcessExecution.RunStreamingAsync`
- 但设计思路完全一致——都是"暂停执行等人类决策"

**可复用现有代码**：
- `DynamicAgentCreationGate` 的模式（TCS + 事件 + UI 解锁）可直接复制给 `ToolCallAuthorizationGate`
- UI 侧的审批卡片渲染逻辑可复用 `DynamicAgentCreationApprovalVm` 的模式

### 3.8 完整工具授权流程图

```
用户发起任务
    │
    ▼
[Level 0] 任务配置中的黑名单检查
    │  被禁的工具从可用列表中移除
    │  子智能体声明被禁工具 → 创建失败（CreateSubAgentTool 报错）
    │
    ▼
[Level 1] 子智能体创建审批（DynamicAgentCreationGate）
    │  用户看到：智能体名 + 职责 + 所需工具列表
    │  批准 → 授权该子智能体使用列出的工具
    │  拒绝 → 不创建
    │
    ▼
子智能体开始执行
    │
    ├── 调用 Normal/HighRisk 工具 → 直接执行（Level 1 已授权）
    │
    └── 调用 RequirePerCallAuth 工具 → [Level 2] 逐次确认
         │
         ├── 用户选"允许" → 执行本次
         ├── 用户选"后续全部允许" → 执行 + 加入白名单
         └── 用户选"拒绝" → 子智能体收到拒绝信息，自行处理
              │
              └── 子智能体可以：
                  - 换一种方式完成（不用这个工具）
                  - 报告执行失败 → 主智能体决策
```

## 4. 设计决策总结

| 编号 | 决策 | 选择 | 理由 |
|------|------|------|------|
| P4-U9 | 多任务全局并发 | 全局信号量，默认上限 5 | FIFO 公平调度，单任务体验不变，多任务安全 |
| P4-U10 | 优先级策略 | P4 初版 FIFO，远期可选优先级 | 简单可靠，SemaphoreSlim 原生支持 |
| P4-U11 | Chat 模式隔离 | Chat 不受全局信号量控制 | 实时交互不能被后台任务阻塞 |
| P4-U12 | 模板版本策略 | 无版本管理，保存即最新 | 用户确认：修改即替换，旧版不保留 |
| P4-U13 | BasePromptVersion 字段 | 删除 | 无版本管理需求，简化模型 |
| P4-U14 | 工具授权模型 | 三级：全局禁止 / 首次授权 / 逐次确认 | 平衡安全性与用户体验 |
| P4-U15 | 逐次授权实现 | ToolCallAuthorizationGate（TCS 模式） | 复用 DynamicAgentCreationGate 成熟模式 |
| P4-U16 | MAF HITL 对接 | 不直接用 SDK HITL，自建同模式实现 | P4 引擎自管调度，不依赖 SDK Workflow |
| P4-U17 | MCP 工具授权声明 | x-cortana-auth 扩展字段 | 兼容 MCP 标准，不入侵协议 |
