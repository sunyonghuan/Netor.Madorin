## 3. 计划 vs 实际偏差

### 3.1 推迟后已闭环的决策（DT-9 SubMode 过滤）

**计划**：C4 时为 WorkflowTaskListVm 添加 SubModeFilter 属性，工作流 / 群聊两个 tab 显示各自的任务列表。

**实际**：C4 推进时发现 SubModeFilter 涉及数据流大改造（WorkflowTaskListQuery + WorkflowExecutor.ListTasksAsync + 事件订阅过滤），与 C4 主目标（拆 WorkspaceTab）冲突，**推迟到 P1（多智能体编排阶段 7+ 第一项）**。

**P1 闭环（2026-05-16，单 commit 完成）**：

- `WorkflowTaskListQuery` 加 `SubModes` 字段
- `WorkflowTaskRepository.ListByWorkspace` 加 `subModes` 参数 + SQL `SubMode IN (...)` 子句
- `WorkflowExecutor.ListTasksAsync` 透传 SubModes
- `WorkflowTaskListVm` 加 `SubModeFilter` 属性 + `IsMatchingSubMode` 校验（OnTaskStarted 防止跨 SubMode 任务串入）
- `WorkspaceTabVm.OnAttachedAsync` 加 `subModes` 可选参数
- `MainWindow.axaml.cs` 加 2 个静态常量（WorkflowSubModes / GroupChatSubModes）+ 3 处 OnAttachedAsync 调用按 WorkMode 传值
- 验证：Debug 0 错误 / AOT 72MB / 烟雾测试通过

**最终状态**：工作流 tab 仅显示 magentic + parallelanalysis 任务；群聊 tab 仅显示 groupchat 任务；详情区共享（用户选谁就看谁）。架构选项 A（单 VM + SubModeFilter）实施完成，无需拆 2 个独立 VM 实例。

详见 P1 commit message。

### 3.2 方案调整的决策（DT-11 主区切换方式）

**计划**：`<ContentControl Content="{Binding CurrentTabView}" />` + DataTemplate 按 enum 切换。

**实际**：发现 axaml DataTemplate.DataType 在 AOT 模式下的代码生成有限制（参考 Avalonia 11.x 文档），且 ContentControl + DataTemplate 会让 binding 链变得难调试。

**改为**：**3 个具名控件 IsVisible 切换**（与 C2 顶栏 tab 切换风格一致）。三个 tab 内容控件挂在同一个 Panel 下，由 OnTabSwitchClick 显式控制 IsVisible。

**收益**：
- AOT 友好（控件实例化在 axaml InitializeComponent 时完成）
- 调试简单（每个 tab 内容都是独立 x:Name 节点）
- 与 C2 风格一致（学习成本低）

### 3.3 实施期新增的决策（DT-13 自解析 VM 模式）

**背景**：C4 拆分 WorkspaceTab 时，新的 TaskListPanel + WorkflowDetailView 需要共享 VM 引用（用户在左侧点击任务后，右侧主区需要联动）。

**两个选项**：
- **选项 A**：调用方（LeftPanel.axaml + MainWindow.axaml）显式给两个控件赋 DataContext
- **选项 B**：两个控件构造时自行从 DI 容器解析 WorkspaceTabVm（Singleton）

**选 B 的理由**：
- 父容器解耦：LeftPanel.axaml 不需要为子控件单独设置 DataContext
- axaml InitializeComponent 时序解耦：避免 DataContextChanged 事件链耦合
- DI 容器统一管理 VM 生命周期：3 个使用方（TaskListPanel + WorkflowDetailView + MainWindow.OnAttachedAsync）都从容器拿同一实例

**副作用与解决方案**：父容器（LeftPanel.axaml）无法直接给 TaskListPanel 绑 `IsVisible='!MainVm.IsChatMode'`，因为 TaskListPanel 自身 DataContext 已是 WorkspaceTabVm（没有 MainVm 属性）。**解决**：用外层 Panel 包裹（不覆盖 DataContext），让 binding 在 LeftPanelVm 上下文求值（详见 LeftPanel.axaml line 79-91）。

## 4. 架构模式沉淀（4 个核心模式）

界面重设计阶段沉淀了 4 个可复用的架构模式，后续 UI 重构应优先采用。

### 4.1 自解析 VM 模式（DT-13）

**适用场景**：子控件需要访问共享 VM（Singleton），且不希望耦合父容器 DataContext。

**实现**：
```csharp
public partial class TaskListPanel : UserControl
{
    private readonly WorkspaceTabVm _vm;  // readonly 字段，构造时确定

    public TaskListPanel()
    {
        InitializeComponent();
        _vm = App.Services.GetRequiredService<WorkspaceTabVm>();
        DataContext = _vm;  // 自己设 DataContext，父容器不管
    }
}
```

**注意事项**：
- VM 必须是 DI Singleton（否则每个控件拿到不同实例，列表 / 详情联动失效）
- 父容器（LeftPanel.axaml）如果要给本控件设 IsVisible，必须用 Panel/Border 包裹隔离（避免 binding 在错误的 DataContext 上求值）

**已应用**：
- ✅ `TaskListPanel`（C4）
- ✅ `WorkflowDetailView`（C4）

**可推广**：
- `ChatHistoryPanel`（C5 仍是父容器注入 DataContext 模式 - 可在后续 polish 时迁移到自解析）
- 未来任何需要共享 VM 的子控件

### 4.2 父子 binding 隔离模式

**适用场景**：子控件自解析 VM 后，父容器仍想绑 IsVisible / IsEnabled 等属性。

**问题**：直接在子控件上写 `IsVisible="{Binding !MainVm.IsChatMode}"` 会失败 — binding 在子控件自己的 DataContext（如 WorkspaceTabVm）上求值，找不到 MainVm 属性。

**解决**：用外层 Panel 包裹（不覆盖 DataContext，保持父继承）：
```xml
<!-- ❌ 错：binding 在子控件 DataContext 上求值 -->
<controls:TaskListPanel IsVisible="{Binding !MainVm.IsChatMode}" />

<!-- ✅ 对：用外层 Panel 隔离 -->
<Panel IsVisible="{Binding !MainVm.IsChatMode}">
    <controls:TaskListPanel />
</Panel>
```

**已应用**：
- ✅ `LeftPanel.axaml` Tab2 切换 TaskListPanel / ChatHistoryPanel（C5）

### 4.3 DI Singleton VM 模式（DT-11）

**适用场景**：多个视图需要共享同一个 VM 实例（如 MainWindow 触发列表刷新 + LeftPanel 显示列表 + 主区显示详情）。

**实现**：
```csharp
// App.axaml.cs DI 注册
.AddSingleton<WorkspaceTabVm>()
.AddSingleton<MainWindowVm>()
.AddSingleton<LeftPanelVm>()

// 各处使用方
private readonly WorkspaceTabVm _vm = App.Services.GetRequiredService<WorkspaceTabVm>();
```

**与传统 `new ViewModel()` 的区别**：
- ✅ 共享状态：列表加载完成后，所有依赖方自动看到最新数据
- ✅ 生命周期统一：DI 容器管理，进程退出时统一释放
- ✅ 事件订阅安全：VM 内部订阅 EventHub 后，不需要担心多个 VM 重复订阅

**已应用**：
- ✅ `WorkspaceTabVm`（C4，DT-11）
- ✅ `MainWindowVm`（C2）
- ✅ `LeftPanelVm`（C3）

### 4.4 EmptyState 控件复用模式

**适用场景**：任何"列表为空 / 主区无内容"的兜底视图。

**实现**：`EmptyState` 用户控件（C1 引入），支持 4 个属性：
- `IconText`（大图标，如 📋 / 💬）
- `Title`（主标题）
- `Description`（描述文字，支持 `&#10;` 换行）
- `PrimaryActionText` + `PrimaryActionClick`（可选主链接）

**收益**：
- 统一视觉规格（避免每个控件重复定义"灰色斜体 + 居中"样式）
- 主链接打通"空状态 → 创建新内容"路径（如 C4 主区 ✨ 新建工作流任务）

**已应用**：
- ✅ `LeftPanel.Tab2` C3 占位 / C4 Chat 模式占位
- ✅ `WorkflowDetailView` 空任务态（C4，含 ✨ 新建链接）
- ✅ `MainWindow` 群聊 tab 占位（C2 - C3）

## 5. 关键经验教训

### 5.1 ✅ 做对了的

#### E1：方案先行 + 分 5 个 commit 串行
- C0 写完 5 篇文档（README + 4 篇技术文档）后才动手，避免"边写边改"
- 每个 commit 都是独立的视觉里程碑，用户可以中途看到进度
- 每个 commit 都做完整 5 步验证（Debug → AOT → 烟雾测试 → push → SHA 三方校验）

#### E2：commit message 详尽
- 每个 commit 的 message 包含：改动文件清单 / 决策点（含选项对比） / 视觉变化 / 验证结果 / commit 链
- 半年后回看仍能快速理解当时的选择
- 配合本文档食用更佳

#### E3：用户决策即时锁定
- C5 时用户提的 R1/R2/R3 在 30 分钟内讨论 + 锁定 + 启动实施
- 避免"决策悬空"导致开发停滞

#### E4：DI Singleton 模式比 new ViewModel() 强太多
- C4 时如果用 `new WorkspaceTabVm()`，TaskListPanel + WorkflowDetailView 拿到的是不同实例，列表 / 详情联动无法工作
- 改 DI Singleton 后整个数据流自然打通，零额外代码

### 5.2 ⚠️ 经验教训（下次注意）

#### L1：方案文档与实施会有偏差，要预留"实施期新增决策"槽位
- 计划期 DT-1 ~ DT-12 共 12 条，实施期新增 DT-13 + R1/R2/R3 共 4 条
- 偏差率 33%（4/12），符合"软件设计三七开"经验
- **建议**：后续大主题的 04-实施阶段.md 应在 §7 决策清单表预留"实施期新增"栏目

#### L2：用户记忆有偏差，要做调研确认
- C5 时用户提到"ChatHistoryPanel 有改标题功能" — 实际调研发现没有
- 如果直接按用户说法做，会"修复一个不存在的问题"
- **建议**：用户提需求后，开发前先做 10 分钟调研，确认现状 vs 预期

#### L3：Edit 工具对 tab / space 缩进非常敏感
- C5 时多次 Edit 失败，原因是 axaml 文件用 tab 缩进，但 Edit 提供的 old_string 拷贝过来变成 space
- **建议**：复杂的 axaml 改动用 sed -i 直接操作行号（先用 Bash + awk 看精确字节确认）

#### L4：commit message 含反引号要用 -F 文件方式
- C5 commit 时第一次失败（commit message 含反引号 `` ` `` 触发 bash 反引号解析）
- **解决**：Write 消息到临时文件 → `git commit -F <file>` → 删除临时文件

#### L5：AOT publish 后 netstat 残留状态不等于进程存活
- 烟雾测试后 taskkill /F 杀掉 Cortana.exe，netstat 短时间内仍显示端口被占用
- 实际上是 Windows 内核里的 TIME_WAIT 状态，3-5 秒后自动清理
- **建议**：判断进程是否退出以 tasklist 为准，端口检查只做辅助参考
