# 06 - 优缺点与风险分析

## 优势

### 1. 独立更新与部署

STT/TTS 模型和引擎可以独立升级，无需重新编译主程序。
例如将来替换 STT 引擎（Sherpa → Whisper.cpp），只需替换 `Cortana.Stt.exe`。

### 2. 故障隔离

STT/TTS 进程崩溃不会影响主程序稳定性。
Native 内存泄漏、模型加载失败等问题被限制在子进程范围内。

### 3. 内存隔离

Sherpa-ONNX 模型占用大量内存（TTS 约 600MB+，STT 约 200MB+）。
独立进程后，关闭语音服务可完全释放这部分内存。

### 4. 技术栈解耦

子进程可以用任何语言实现（C#、C++、Rust、Python 均可），
只要遵守 WebSocket 协议。为未来技术选型保留最大灵活性。

### 5. 按需启停

用户可通过设置页关闭不需要的语音服务：
- 文字办公场景：关闭全部语音 → 零录音/零内存
- 翻译场景：只开 TTS → 回复读出来
- 快速问答：只开 STT → 语音输入不播报

### 6. 并行开发

三个进程可以由不同开发者独立开发和测试，降低合并冲突。

---

## 劣势

### 1. 系统复杂度上升

从单进程变为三进程，带来进程管理、重连、心跳等新的复杂性。
出问题时需要同时查看三个进程的日志。

### 2. 启动链路变长

主程序启动后需要等待子进程就绪 + WS 握手完成，首次语音交互的可用延迟增加。

### 3. 进程管理负担

需要实现子进程的启动、监控、重启、优雅关闭。
处理子进程异常退出、端口冲突、僵尸进程等边缘情况。

### 4. 调试体验下降

三个进程需要 attach 三个调试器。
断点跨进程不连贯，排查时序问题更困难。

### 5. VoiceOrchestrator 重写风险

`SpeechRecognitionService` 中的编排逻辑（唤醒→问候→识别→续杯）
需要在 `VoiceOrchestrator` 中完全重写为异步 WebSocket 消息驱动。
这是最容易出 bug 的部分。

## 风险矩阵

| 风险项 | 可能性 | 影响 | 严重度 | 缓解措施 |
|---|---|---|---|---|
| 编排时序 bug（续杯/打断逻辑出错） | 高 | 高 | 🔴 严重 | 独立测试每个场景，写详细的状态转换日志 |
| 子进程崩溃后重启风险（循环崩溃） | 中 | 高 | 🔴 严重 | 限制重启次数（5分钟内最多3次），超限标记为不可用 |
| WebSocket 消息丢失或乱序 | 低 | 高 | 🟡 中等 | TCP 保证有序，心跳检测连接断开，重连后重置状态 |
| 端口冲突（12858/12859 被占用） | 低 | 中 | 🟡 中等 | 启动时检测端口占用，日志报错提示用户 |
| 首次启动超时（模型加载慢） | 中 | 中 | 🟡 中等 | 子进程启动后先发 `connected` 再加载模型，或加超时容忍 |
| KWS 仍依赖 sherpa.onnx 包 | 低 | 低 | 🟢 低 | 短期保留包引用，长期考虑 KWS 也独立化 |
| Native AOT 发布体积增大 | 低 | 低 | 🟢 低 | 三个 exe 共享 ONNX Runtime，通过共享目录减少重复 |

---

## 回退方案

如果拆分后出现严重问题无法快速修复：

1. `VoiceServiceExtensions` 中注册旧实现（旧代码在 Step 7 最后才删除）
2. 关闭 STT/TTS 开关，不启动子进程
3. 主程序回退到单进程语音模式

**关键：旧代码在全部测试通过前不删除。** Step 7 中的"清理旧代码"子任务排在最后。

---

## 不做的事情

以下变更不在本次范围内，避免膨胀：

| 项 | 原因 |
|---|---|
| KWS 独立进程 | 太轻量，收益不大，且唤醒需要最低延迟 |
| STT/TTS 进程间直接通信 | 无此需求，所有编排通过主程序 |
| gRPC 替代 WebSocket | JSON 文本帧足够简单，AOT 友好，gRPC 引入额外依赖 |
| 动态端口分配 | 固定端口足够，复杂度不值得 |
| 多实例支持 | 当前不需要运行多个 Cortana 实例 |
