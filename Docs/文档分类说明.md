# Docs 文档分类说明

> 整理时间：2026年5月14日（初次）  
> 更新时间：2026年5月27日（第二次整理）

## 目录约定

### 未来版本策划

路径：`Docs/未来版本策划/`

用于存放：

-   尚未实施的功能方案；
    
-   仍在讨论中的架构规划；
    
-   未完成的执行计划；
    
-   未来版本可能落地的功能设计；
    
-   状态为“未解决”“待执行”“0%”“90%/94% 未完全完成”的文档。
    

功能完成后，应从这里移动到 `Docs/已完成功能规划/`。

### 已完成功能规划

路径：`Docs/已完成功能规划/`

用于存放：

-   已经实现并发布或验证完成的功能方案；
    
-   标记为 `100%` 的执行计划；
    
-   已有 release notes 证明落地的方案文档；
    
-   作为历史设计归档保留的完成文档。
    

### 保留在原位置的文档

以下目录暂时保留原位：

-   `Docs/release-notes/`：发布说明，不移动。
    
-   `Docs/GIT提交说明/`：提交说明归档，不移动。
    
-   `Docs/系统流程与规划/`：当前系统参考文档，例如 `websocket-api.md`、`class-reference.md`、`UI-编译打包发布流程.md`。
    
-   `Docs/memory/`：当前为空或历史迁移残留，暂不处理。

## 后续使用规则

1.  新建未来功能方案，优先放入 `Docs/未来版本策划/`。
    
2.  已完成并验证的功能方案，移动到 `Docs/已完成功能规划/`。
    
3.  如果某个功能只有一部分完成，仍保留在 `未来版本策划`。
    
4.  发布说明和 Git 提交说明不参与该分类移动。
    
5.  当前系统协议、类参考、构建流程等长期参考文档保留在 `系统流程与规划`。
