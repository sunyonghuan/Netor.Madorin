# 05 - 记忆体系设计

> 状态：结构模型第一版已定稿

## 目标

定义长期记忆对象模型、关系网络、遗忘与衰退机制，以及宿主可消费的召回包结构。

## 定位

记忆体系不是聊天历史压缩增强版，而是一套独立的长期记忆网络系统。

它必须同时具备以下能力：

- 能记住事实
- 能抽象经验
- 能形成偏好
- 能选择性遗忘
- 能随时间衰退
- 能在被再次使用时强化
- 能在出现矛盾证据时降权或废弃

## 设计原则

- 宿主发布的是事实流，不是记忆结论
- 所有长期记忆都必须保留来源链
- 自动记忆默认不是永久记忆
- 记忆强度会随时间、使用和冲突动态变化
- 抽象记忆不能脱离片段独立存在
- 规则记忆优先于长期记忆
- 长期记忆插件只能构建自己的记忆，不得反向修改宿主事实

## 记忆形成链

```text
Conversation Event
	-> ObservationRecord
	-> MemoryFragment
	-> MemoryLink
	-> MemoryAbstraction
	-> RecallPackage
	-> Reinforcement / Decay / Forgetting
```

说明：

- `ObservationRecord` 是插件内部的原始观察层，不直接暴露给宿主
- `MemoryFragment` 是第一层可召回记忆
- `MemoryAbstraction` 是跨片段形成的稳定知识、偏好或经验
- `RecallPackage` 是召回时临时组装出的交付对象，不是持久化主体

## 作用域原则

- 长期记忆以 `agentId` 为核心作用域
- `workspaceId` 是过滤维度，不是主作用域
- 同一智能体可跨工作区积累成长性记忆
- 可选 `userId` / `personaId` 作为更细粒度画像维度

## 记忆分型

长期记忆至少包含以下认知类型：

- `episode`：某次具体经历，例如一次任务、一次讨论、一次失败
- `preference`：稳定偏好，例如代码风格、交互习惯、输出偏好
- `procedure`：做事步骤和可复用方法
- `insight`：跨多个事件提炼出的规律、经验、判断
- `failure`：失败模式、踩坑记录、禁忌路径
- `relationship`：实体之间的关联，例如用户、项目、模块、工具之间的关系
- `goal`：跨轮持续存在的目标、计划、待完成事项

这不是为了把模型做复杂，而是为了避免后期把“偏好”“经验”“失败教训”“长期目标”全部塞进同一种对象里，导致检索和衰退策略失真。

## 核心对象

### ObservationRecord

原始观察记录，是插件接到宿主事实流后的私有落盘对象。

建议字段：

- `id`
- `agentId`
- `sessionId`
- `turnId`
- `messageId`
- `eventType`
- `role`
- `content`
- `attachments`
- `timestamp`
- `traceId`
- `sourceFacts`

职责：

- 作为记忆抽取的原始证据层
- 保证后续记忆都可回溯到真实事实
- 支撑误记忆治理、重建索引和重新抽象

### MemoryFragment

最小可召回记忆单元，来源于一个或多个 ObservationRecord 的归纳。

它更接近“经历片段”而不是原始聊天文本。

建议字段：

- `id`
- `agentId`
- `workspaceId`
- `memoryType`
- `title`
- `summary`
- `detail`
- `keywords`
- `tags`
- `entities`
- `sourceObservationIds`
- `sourceSessionIds`
- `sourceTurnIds`
- `importance`
- `confidence`
- `emotionalWeight`
- `novelty`
- `salienceScore`
- `retentionScore`
- `decayRate`
- `accessCount`
- `reinforcementCount`
- `contradictionCount`
- `lastAccessedAt`
- `lastReinforcedAt`
- `expiresAt`
- `state`
- `createdAt`
- `updatedAt`

其中几个关键字段的语义：

- `salienceScore`：当前“有多容易被想起”，综合重要性、时效性、最近使用情况
- `retentionScore`：当前“有多值得长期保留”，偏稳定判断
- `decayRate`：衰退速度，不同类型记忆可不同
- `state`：记忆生命周期状态，不是简单的启用/禁用

### MemoryLink

描述两个记忆对象之间的连接关系，既可连接 Fragment，也可连接 Abstraction。

建议字段：

- `id`
- `agentId`
- `sourceMemoryId`
- `sourceMemoryKind`
- `targetMemoryId`
- `targetMemoryKind`
- `relationType`
- `weight`
- `evidenceCount`
- `confidence`
- `createdAt`
- `updatedAt`

建议支持的关系类型：

- `causes`
- `supports`
- `contradicts`
- `similar_to`
- `derived_from`
- `part_of`
- `precedes`
- `follows`
- `belongs_to_goal`
- `about_entity`

### MemoryAbstraction

跨多个片段提炼出的抽象结论，例如偏好、经验、工作习惯、高频失败模式、长期目标画像。

它更接近“人总结出来的认知”，不是聊天摘要。

建议字段：

- `id`
- `agentId`
- `abstractionType`
- `title`
- `statement`
- `summary`
- `supportingMemoryIds`
- `counterMemoryIds`
- `keywords`
- `tags`
- `importance`
- `confidence`
- `stabilityScore`
- `retentionScore`
- `decayRate`
- `accessCount`
- `reinforcementCount`
- `contradictionCount`
- `lastValidatedAt`
- `lastAccessedAt`
- `expiresAt`
- `state`
- `createdAt`
- `updatedAt`

建议支持的抽象类型：

- `user_preference`
- `agent_preference`
- `procedure_pattern`
- `failure_pattern`
- `project_understanding`
- `entity_profile`
- `goal_profile`
- `stable_fact`

关键约束：

- 抽象记忆必须至少有一条 `supportingMemoryIds`
- 抽象记忆允许存在 `counterMemoryIds`，用于表示出现了反证
- 当反证足够强时，不是直接删除，而是进入降权或废弃状态

### RecallPackage

一次召回给宿主的记忆包，包含命中片段、扩展片段、抽象记忆和召回解释。

建议字段：

- `agentId`
- `queryText`
- `queryIntent`
- `triggerSource`
- `hitFragments`
- `supportingFragments`
- `abstractions`
- `links`
- `suppressedMemories`
- `recallSummary`
- `confidence`
- `generatedAt`

其中：

- `hitFragments` 是直接命中的记忆
- `supportingFragments` 是通过图关系扩展补入的上下文
- `suppressedMemories` 用于记录本次被压制但本来可能入选的记忆，例如过期、低置信度、被规则记忆覆盖

## 生命周期模型

为了支持“像人一样记忆、像人一样遗忘”，记忆对象必须具备明确生命周期。

建议统一状态枚举：

- `candidate`：刚提取出来，尚未稳定
- `active`：可正常参与召回
- `dormant`：保留但不主动高频召回
- `reinforced`：近期被再次使用或再次证实
- `contradicted`：出现明显反证，等待重新判断
- `archived`：历史保留，不再主动参与召回
- `forgotten`：逻辑上遗忘，不再参与正常检索
- `deleted`：人工硬删除，仅用于治理或合规场景

状态迁移原则：

- 新记忆默认先进入 `candidate`
- 多次命中或被宿主使用后进入 `active` 或 `reinforced`
- 长期未使用会退化为 `dormant`
- 持续低价值会进入 `archived` 或 `forgotten`
- 出现冲突证据时进入 `contradicted`

## 遗忘与衰退机制

这部分不是附属功能，而是结构模型的核心组成。

### 1. 自然衰退

每条记忆都具备 `decayRate`，系统周期性降低其 `salienceScore`。

适合快速衰退的对象：

- 一次性任务细节
- 临时上下文
- 已过时的工作状态

适合慢速衰退的对象：

- 用户长期偏好
- 高价值失败教训
- 稳定项目理解

### 2. 强化记忆

以下事件会提升 `salienceScore` 或 `retentionScore`：

- 被再次提及
- 被再次召回且证明有用
- 被新片段再次支持
- 被人工确认或固定

### 3. 选择性遗忘

系统必须允许记忆不是“都记住”，而是“保留有价值的”。

选择性遗忘规则至少包括：

- 低重要性且长期未访问
- 与更高质量记忆重复
- 被新抽象完全覆盖
- 已经过期且无再次强化
- 与规则记忆冲突且被判定为不可靠

### 4. 反证降权

当新事实与旧记忆冲突时，不应马上删除旧记忆，而应：

- 增加 `contradictionCount`
- 建立 `contradicts` 关系
- 降低 `confidence` 与 `salienceScore`
- 必要时把状态改为 `contradicted`

### 5. 人工固定与人工遗忘

系统需要保留人工治理能力：

- `pinned` 或等价策略：阻止自动衰退
- `archived`：人工归档
- `forgotten`：人工要求逻辑遗忘
- `deleted`：合规或用户明确要求物理删除

## 召回模型

召回不是简单做相似度搜索，而是一次受治理的记忆组装过程。

建议分 4 层：

### 1. 候选召回层

- 关键词倒排召回
- 向量相似召回
- 实体关联召回
- 目标或会话上下文召回

### 2. 图扩展层

- 根据 `MemoryLink` 扩展支持片段
- 把因果、相似、前后依赖关系一并补入

### 3. 治理过滤层

- 去重
- 低置信度过滤
- 过期记忆降权
- 与规则记忆冲突时压制
- 对同类型记忆做数量限制

### 4. 输出组装层

- 优先输出抽象记忆
- 保留少量高价值片段作为证据
- 输出召回摘要，而不是把大量原文直接灌给宿主

## 私有数据库与索引结构

第一版就应明确插件内部最少要有哪几类持久化结构。

### 主表

- `observation_records`
- `memory_fragments`
- `memory_links`
- `memory_abstractions`
- `recall_logs`
- `memory_mutations`

### 索引

- 主键索引：按 `id`
- 作用域索引：`agentId + state`
- 时间索引：`updatedAt`、`lastAccessedAt`
- 类型索引：`memoryType`、`abstractionType`
- 倒排索引：`keywords`、`tags`、`entities`
- 图索引：`sourceMemoryId`、`targetMemoryId`
- 向量索引：针对 fragment 和 abstraction 的 embedding

### mutation 日志

为了支持审计、回滚和重建，建议单独记录记忆变动：

- `create_fragment`
- `merge_fragment`
- `create_abstraction`
- `reinforce_memory`
- `decay_memory`
- `archive_memory`
- `forget_memory`
- `delete_memory`

## 结构约束

以下约束建议在第一版就定死，避免后期大改：

- `ObservationRecord` 与 `MemoryFragment` 分层，不能直接拿宿主事实当记忆对象
- `MemoryAbstraction` 必须能回溯到 fragment
- `RecallPackage` 是临时输出对象，不作为主存储模型
- 每条记忆都必须具备时间属性、强度属性和状态属性
- 遗忘优先做逻辑遗忘，物理删除是少数治理场景
- 规则记忆永远不写入长期记忆库本体

## 第一版取舍

第一版不要求实现所有高级能力，但模型必须预留以下位置：

- 记忆衰退
- 反证降权
- 选择性遗忘
- 人工固定与人工删除
- 抽象记忆的来源链
- 图关系扩展召回

这样后续可以只补能力，不需要再推翻结构。
