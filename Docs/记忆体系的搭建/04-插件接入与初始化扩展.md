# 04 - 插件接入与初始化扩展

> 状态：第一版已落地

## 目标

为新标准插件定义初始化扩展槽位和内部事件端点下发方式。

## 当前现状

当前插件 init 参数仅包含：

- `dataDirectory`
- `workspaceDirectory`
- `pluginDirectory`
- `wsPort`

这只能满足旧聊天 WebSocket 接入需求，无法表达新的内部事件流端点。

## 当前已落地槽位

当前插件 init 配置已新增：

- `extensions`

当前结构：

```json
{
	"dataDirectory": "xxx",
	"workspaceDirectory": "xxx",
	"pluginDirectory": "xxx",
	"wsPort": 52841,
	"extensions": {
		"chatWsEndpoint": "ws://localhost:52841/ws/",
		"conversationFeedEndpoint": "ws://localhost:52841/internal/conversation-feed/",
		"conversationFeedProtocol": "conversation-feed",
		"conversationFeedVersion": "1.0.0"
	}
}
```

## 设计原则

- `wsPort` 保留为旧聊天 WebSocket 端口字段
- 新标准插件不再依赖单一 `wsPort` 推导内部事件地址
- 内部事件流通过明确字段传入
- Native / Process SDK 都会把扩展字段解析成可直接使用的运行时属性

## 当前宿主下发链路

- `ExternalProcessPluginHostBase.InitializePluginAsync(...)` 负责组装 `extensions`
- `NativePluginInitConfig` 增加 `extensions` 槽位并走源生成序列化
- `CortanaWsEndpoints` 统一产出 `chatWsEndpoint` 与 `conversationFeedEndpoint`

## 当前插件运行时可见字段

- `PluginSettings.ChatWsEndpoint`
- `PluginSettings.ConversationFeedEndpoint`
- `PluginSettings.ConversationFeedProtocol`
- `PluginSettings.ConversationFeedVersion`

## 当前兼容策略

- 旧字段 `wsPort` 继续保留
- 如果扩展字段缺失，Native / Process SDK 仍会基于 `wsPort` 生成默认 endpoint，避免调试场景直接断裂
