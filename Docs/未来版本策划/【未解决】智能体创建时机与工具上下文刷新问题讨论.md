# 【未解决】智能体创建时机与工具上下文刷新问题讨论

> 状态：未解决，待后续处理
> 日期：2026年5月9日

## 背景

当前发现一个体验问题：当用户更改 MCP 工具、安装新插件、启用/禁用插件，或更新技能目录后，当前已经存在的智能体实例不会自动加载新的插件、技能和 MCP 工具。

用户从界面上看到插件或 MCP 工具已经安装/配置成功，容易认为可以立即使用。但实际上，当前会话中已经构建好的智能体仍然持有旧的工具上下文，只有在重新切换智能体、切换模型、切换厂商，或触发重新构建后，新工具才会生效。

## 当前代码现象

`AIAgentFactory.Build()` 和 `AIAgentFactory.BuildWithSubAgents()` 在创建智能体时会一次性组装：

- `AgentSkillsProvider`
- 内置 `AIContextProvider`
- 插件工具 `PluginContextProvider`
- MCP 工具 `McpContextProvider`
- `ChatOptions`
- `ChatHistoryProvider`

因此，当前的 `AIAgent` 更像是一个“创建时快照”，不是“运行时动态视图”。

`AiChatHostedService.SendMessageAsync()` 当前普通对话路径中使用了类似缓存复用逻辑：

- 如果 `_agent` 已存在，则继续复用；
- 只有 `_agent` 为空时才重新通过 `factory.Build(...)` 构建；
- 有 `@智能体` 提及时，会临时构建带子智能体工具的主 Agent，并在本轮结束后置空。

目前已经会触发重建的配置变化主要包括：

- AI Provider 变化；
- AI Model 变化；
- Agent 变化；
- Workspace 变化。

但插件、MCP、技能目录变化尚未进入统一的 Agent 失效链路。

## 问题核心

需要决定智能体的创建时机：

1. 是否每次调用 AI 都重新创建一次智能体；
2. 是否继续缓存智能体，只在工具上下文变化后重建；
3. 是否需要将插件、MCP、技能目录变化统一纳入 Agent 失效机制。

## 每次调用 AI 都创建智能体的优点

- 实现简单；
- 插件、MCP、技能变化后下一轮一定生效；
- 不需要额外维护版本号或 dirty 状态；
- 可以避免用户误以为工具已生效但实际未加载的问题。

## 每次调用 AI 都创建智能体的风险

不建议无条件每次都完整重建，原因如下：

1. **重复枚举插件和 MCP 工具**
   - 每轮都需要遍历插件、MCP Server、工具列表；
   - 每轮都需要重新做工具冲突检查。

2. **重复构造 ChatClient**
   - `AIAgentFactory.Build()` 内部会重新调用 `driver.CreateChatClient(...)`；
   - 如果底层驱动初始化较重，可能带来不必要开销。

3. **影响运行时状态稳定性**
   - `ChatClient` 重建会影响 token tracking；
   - 如果 Session 处理不当，可能造成会话状态和当前 Agent 不一致。

4. **高频对话场景浪费明显**
   - 用户连续多轮对话时，插件和 MCP 工具大概率没有变化；
   - 每轮都重建属于不必要开销。

## 倾向方案：缓存智能体 + 工具上下文版本号 + 懒重建

更合理的方案是：

> 保留当前 Agent 缓存，但当插件、MCP、技能或工具上下文发生变化时，只标记当前 Agent 已过期；下一次用户发送消息前再重建。

也就是：

1. 平时复用 `_agent`；
2. 插件/MCP/技能变化时递增一个全局工具上下文版本号，或设置 dirty 标记；
3. `SendMessageAsync()` 发送前检查当前 `_agent` 的构建版本是否落后；
4. 如果落后，则重新构建 Agent；
5. 不在变更发生时立即打断当前流式响应；
6. 下一轮对话自动使用新工具。

## 建议的失效粒度

不同变更对 Agent 和 Session 的影响应区分处理：

| 变更类型 | 是否重建 Agent | 是否清空 Session |
|---|---:|---:|
| Provider 切换 | 是 | 是 |
| Model 切换 | 是 | 是 |
| Agent 切换 | 是 | 是 |
| 插件安装/卸载 | 是 | 不一定 |
| 插件启用/禁用 | 是 | 不一定 |
| MCP 工具刷新 | 是 | 不一定 |
| 技能目录变化 | 是 | 不一定 |
| Workspace 切换 | 是 | 是 |

插件、MCP、技能变化通常只需要重建 Agent，不一定需要清空当前 Session，因为用户只是更新了工具能力，历史上下文仍然应该保留。

## 可能的实现方向

后续可以考虑增加一个运行期上下文版本服务，例如：

- `AgentRuntimeContextVersionService`
- `AgentToolContextVersionService`
- `AgentContextInvalidationService`

职责：

- 维护当前工具上下文版本；
- 提供当前版本号；
- 插件/MCP/技能变化时递增版本；
- 可记录最近一次变化原因，便于日志和调试。

`AiChatHostedService` 中记录当前 `_agent` 构建时的版本，例如：

- `_builtToolContextVersion`

发送消息前判断：

- `_agent` 为空：重建；
- 当前工具上下文版本与 `_builtToolContextVersion` 不一致：重建；
- 否则复用。

## UI 提示建议

插件、MCP 或技能变更后，建议给用户明确提示：

- 插件安装成功后：`插件已安装，将在下一次对话中生效。`
- 如果当前正在 AI 响应中：`当前回答仍使用旧工具集，新工具将在下一轮对话中生效。`

这样可以避免用户误解为当前正在输出的这轮回复已经拥有新工具。

## 暂定结论

暂不建议直接改成“每次调用 AI 都重新创建智能体”。

更推荐后续实现：

1. Agent 缓存继续保留；
2. 补齐插件、MCP、技能变化的失效事件；
3. 引入工具上下文版本号或 dirty 标记；
4. 在下一次发送消息前懒重建 Agent；
5. 插件/MCP/技能变化默认不清空 Session。
