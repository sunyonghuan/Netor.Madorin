## 问题一：计划制定完成后，用户看不到计划内容

### 1.1 现象

阶段 2（计划制定）完成后，UI 只在 Feed 中追加一条事件消息：

```text
📋 计划已生成（3 步），等待确认
```

用户完全不知道这 3 步是什么，却要求确认或修改——逻辑上是空洞的。

### 1.2 根本原因

`TaskPlanEventArgs` 只携带 `StepCount`（步骤数量），不携带计划内容。
`OrchestratorAgent.RunPlanningPhaseAsync` 在解析出 `ExecutionPlan` 之后直接返回，
没有把计划内容通过 `onAiMessage` 发给 UI。

### 1.3 解决方案

修改文件：`Src/Netor.Cortana.AI/TaskEngine/Agents/OrchestratorAgent.cs`

修改位置：`RunPlanningPhaseAsync` 方法末尾，`return plan;` 之前。

解析完 JSON、构建出 `ExecutionPlan` 之后，调用 `onAiMessage` 发送一条
Markdown 格式的计划摘要，内容自然流入对话 Feed，用户可直接阅读后确认或修改。

示例输出格式：

```text
好的，我已为您制定了执行计划（共 N 步）：

**步骤 1：[标题]**
[描述]

**步骤 2：[标题]**
[描述]

最终目标：[FinalGoal]

请告诉我是否确认执行，或需要调整哪些步骤。
```

伪代码：

```csharp
if (onAiMessage is not null)
{
    var sb = new StringBuilder();
    sb.AppendLine($"好的，我已为您制定了执行计划（共 {plan.Steps.Count} 步）：");
    sb.AppendLine();
    foreach (var step in plan.Steps)
    {
        sb.AppendLine($"**步骤 {step.Sequence}：{step.Title}**");
        if (!string.IsNullOrWhiteSpace(step.Description))
            sb.AppendLine(step.Description);
        sb.AppendLine();
    }
    if (!string.IsNullOrWhiteSpace(plan.FinalGoal))
        sb.AppendLine($"最终目标：{plan.FinalGoal}");
    sb.AppendLine();
    sb.AppendLine("请告诉我是否确认执行，或需要调整哪些步骤。");
    onAiMessage.Invoke(sb.ToString());
}

return plan;
```

注意：`onAiMessage is not null` 即可作为判断条件，非交互模式（`AutoConfirmPlan`）下跳过。

## 问题二：步骤执行过程中 AI 完全静默，用户无任何过程反馈

### 2.1 现象

执行阶段 Feed 只显示开始和结束两行，中间无论步骤跑多久，Feed 一字不显示：

```text
▶ 步骤 1: 数据采集
✓ 步骤 1 完成: 数据采集
```

子智能体的全部推理过程完全不可见。

### 2.2 根本原因

`OrchestratorAgent.ExecuteStepAsync` 调用 `RunMultiTurnAsync` 时只传了 Logger 回调，
没有 `onAiMessage`。`ExecuteStepAsync` 方法签名本身也没有 `onAiMessage` 参数（接口同理）：

```csharp
var fullResponse = await _runner.RunMultiTurnAsync(
    systemPrompt, userMessage, StepExecutionMaxTurns,
    (turn, text) => _logger.LogDebug(...)  // 只写日志，不推到 UI
    ct);
```

### 2.3 解决方案

修改文件：

- `Src/Netor.Cortana.AI/TaskEngine/Agents/IOrchestratorAgent.cs`
- `Src/Netor.Cortana.AI/TaskEngine/Agents/OrchestratorAgent.cs`
- `Src/Netor.Cortana.AI/TaskEngine/TaskExecutionEngine.cs`

修改内容：

1. `IOrchestratorAgent.ExecuteStepAsync` 增加 `Action<string>? onAiMessage` 参数。
1. `OrchestratorAgent.ExecuteStepAsync` 将 turn 回调改为同时调用 `onAiMessage`：

```csharp
(turn, text) =>
{
    _logger.LogDebug(...);
    if (!text.Contains("[DONE]", StringComparison.OrdinalIgnoreCase)
        && !text.Contains("[CONTINUE]", StringComparison.OrdinalIgnoreCase))
        onAiMessage?.Invoke(CleanLlmMarkers(text));
}
```

1. `TaskExecutionEngine` 调用 `_orchestrator.ExecuteStepAsync` 时传入 `CreateAiMessageCallback(taskId)`。

## 问题三：步骤完成后结果摘要被丢弃，用户看不到结果

### 3.1 现象

步骤完成只显示标题行，没有任何关于步骤做了什么、得到了什么结果的说明：

```text
✓ 步骤 1 完成: 数据采集
```

### 3.2 根本原因

`TaskExecutionEngine` 发布事件时 `ResultSummary` 有值，但 `P4TaskDetailVm` 完全忽略：

```csharp
// TaskExecutionEngine.cs — 发布时有值
_publisher.Publish(Events.OnTaskStepCompleted, new TaskStepEventArgs(
    taskId, ..., step.Title, "completed", result.Summary));

// P4TaskDetailVm.cs — 收到后丢弃
AppendEvent($"✓ 步骤 {args.StepSequence} 完成: {args.Title}", "#73c991");
// args.ResultSummary 存在但从未显示
```

### 3.3 解决方案

修改文件：`Src/Netor.Cortana.UI/ViewModels/Workspace/P4TaskDetailVm.cs`

在 `OnTaskStepCompleted` 回调内，完成事件行之后追加结果摘要：

```csharp
Dispatcher.UIThread.Post(() =>
{
    AppendEvent($"✓ 步骤 {args.StepSequence} 完成: {args.Title}", "#73c991");
    if (!string.IsNullOrWhiteSpace(args.ResultSummary))
        AppendMessage("result", args.ResultSummary);
    UpdateDuration();
});
```

## 问题五：执行阶段 UI 为纯文本行，缺少时间线布局和步骤子过程详情

### 5.1 现象

当前执行阶段所有步骤事件全部渲染为 11px 小字事件行，与设计文档差距很大：

```text
▶ 步骤 1: 数据采集              （蓝色小字）
✓ 步骤 1 完成: 数据采集         （绿色小字）
▶ 步骤 2: 竞品分析              （蓝色小字）
```

缺少：

- 时间线圆点节点（● / ○）
- 步骤内的子过程详情行（AI 执行过程中产生的流式文字）
- 步骤完成后的结果摘要行
- 执行中步骤的流式光标 `▌`

### 5.2 目标效果

参见 `08-P4补充-交互模型重设计.md §2.3`，核心效果如下：

```text
●─ ▶ 步骤 1：数据采集
   └ 正在初始化数据采集专家...
   └ 搜索 NVIDIA 2025 年财报数据...
   └ 找到 47 条相关数据，正在整理...
●─ ✓ 步骤 1 完成
   └ 采集到 47 条数据，涵盖 NVIDIA/AMD/Intel

●─ ▶ 步骤 2：竞品对比分析
   └ 正在对比竞品数据... ▌
○─ ⏳ 步骤 3：等待中
```

### 5.3 需要修改的内容

**5.3.1 数据模型层（`ConversationMessageVm.cs`）**

在现有 Role 基础上，新增或扩展以下支持：

- `Role="step_start"` — 步骤开始行（标题 + 状态图标 + 圆点）
- `Role="step_detail"` — 步骤子过程行（缩进小字，AI 流式输出产生）
- `Role="step_end"` — 步骤完成/失败行（带结果摘要）

或者保留现有 `Role="event"` 不变，改用 `SubRole` / `IndentLevel` 字段区分层级。

**5.3.2 ViewModel 层（`P4TaskDetailVm.cs`）**

`OnTaskStepStarted` 订阅：追加 `step_start` 行（带圆点符号、步骤标题）

`OnWorkflowConversationMessage(role=ai)` 处理中：
当 `CurrentExecutingStepId` 非空时，将 AI 消息作为 `step_detail` 行追加（缩进），
而非独立 AI 消息行。

`OnTaskStepCompleted` 订阅：追加 `step_end` 行 + 结果摘要 `step_detail` 行。

新增 `CurrentExecutingStepId` 属性，在 `OnTaskStepStarted` 时设置，
`OnTaskStepCompleted` / `OnTaskStepFailed` 时清除。

**5.3.3 View 层（`WorkflowDetailView.axaml`）**

在 `ItemTemplate` 中为 `step_start` / `step_detail` / `step_end` 新增渲染模板：

- `step_start`：圆点 `●` + 步骤标题文本，13px，颜色按状态
- `step_detail`：`└` 缩进前缀 + 文本，11px 灰色 `#858585`，左边距 16px
- `step_end`（完成）：`●` + `✓ 步骤 N 完成`，绿色，附带结果摘要 `step_detail` 行
- `step_end`（失败）：`●` + `✗ 步骤 N 失败`，红色

**依赖关系：** 此问题的完整修复依赖问题二（步骤执行过程 AI 消息）同步修复，
两者应一起实现，否则子过程行来源缺失。

## 问题六：对话模式消息缺少左右边距，窗口缩小时右侧文字被压缩

### 6.1 现象

用户消息气泡和 AI 消息区域均撑满 ScrollViewer 可用宽度，视觉上贴着右边界；
窗口缩小时右侧文字换行被遮盖。

### 6.2 根本原因

DataTemplate 内用户气泡 `Border` 和 AI 消息 `StackPanel` 均使用
`HorizontalAlignment="Stretch"` 且无水平 `Margin`，仅依赖 ScrollViewer 的
`Padding="16,10,16,12"`，两侧边距不足。

### 6.3 解决方案

修改文件：`Src/Netor.Cortana.UI/Views/Workspace/WorkflowDetailView.axaml`

- 用户气泡 `Border`：加 `Margin="8,0,8,0"`
- AI 消息 `StackPanel`：加 `Margin="8,0,8,0"`
- 事件行 `TextBlock`：加 `Margin="8,0,8,0"`
- 最终报告 / 失败详情 / 取消提示区块：`Margin` 左右各加 8px

效果：ScrollViewer 16px + 自身 8px = 两侧各 **24px** 总边距，内容不再贴边。

## 问题七：阶段四验证结果展示不完整（Summary 重复 / Issues 丢弃 / Suggestions 断链 / 报告内容空洞）

> 以下四个子问题同属阶段四（验证）展示链，建议一次性一起修复。

### 7.1 Summary 显示两次（内容重复）

引擎执行完成后连续发了两条事件，最终导致同一句 Summary 被显示两遍：

1. `OnWorkflowConversationMessage(role=result, Content=validationResult.Summary)` → FeedItems 内联结果消息
2. `OnTaskEngineCompleted(Reason=validationResult.Summary)` → VM 赋给 `FinalReport` → 底部"最终报告"区域

**解决方案：让两个展示层内容有别。**

- `role=result` 内联消息：保持简短（一句话概要），不变
- `OnTaskEngineCompleted.Reason`：传入完整结构化 Markdown 报告（见问题 7.4），而非只传 Summary

修改文件：`Src/Netor.Cortana.AI/TaskEngine/TaskExecutionEngine.cs`

### 7.2 Issues（问题列表）被丢弃，用户看不到

`TaskValidationEventArgs` 携带了 `Issues` 字段，引擎也正确传入，但 VM 收到后完全忽略：

```csharp
// P4TaskDetailVm.cs — OnTaskValidationCompleted 当前实现
AppendEvent(text, color);
ValidationScore  = args.Score;
ValidationPassed = args.Passed;
// args.Issues 存在但从未处理
```

**解决方案：** 在 `OnTaskValidationCompleted` 回调中，若 `args.Issues` 非空，在验证事件行之后
逐条追加红色缩进事件行：

```text
✗ 验证未通过（65/100）
  └ 步骤3产出的报告缺少竞品分析章节
  └ 数据来源未标注
```

修改文件：`Src/Netor.Cortana.UI/ViewModels/Workspace/P4TaskDetailVm.cs`

### 7.3 Suggestions（改进建议）从引擎到 UI 整条链路断掉

`ValidationResult.Suggestions` 有值，但 `TaskValidationEventArgs` 定义中根本没有该字段，
导致 Suggestions 在事件发布时就被丢弃，UI 永远收不到：

```csharp
// Events.cs — 当前定义缺少 Suggestions
public record TaskValidationEventArgs(
    string TaskId, DateTimeOffset OccurredAt,
    bool Passed, int Score,
    string? Summary = null,
    IReadOnlyList<string>? Issues = null);   // ← 无 Suggestions
```

**解决方案：**

1. `TaskValidationEventArgs` 加 `IReadOnlyList<string>? Suggestions = null` 参数
2. `TaskExecutionEngine` 发布 `OnTaskValidationCompleted` 时传入 `validationResult.Suggestions`
3. Suggestions 纳入底部 FinalReport Markdown 内容展示（见问题 7.4）

修改文件：

- `Src/Netor.Cortana.Entitys/Events.cs`
- `Src/Netor.Cortana.AI/TaskEngine/TaskExecutionEngine.cs`

## 问题八：最终报告浮动在底部，不随对话流滚动

### 8.1 现象

任务完成后，"最终报告"区域作为固定的底部 StackPanel 渲染在 ScrollViewer 内所有
FeedItems 下方。当用户在对话框继续输入时，新消息出现在报告下面，报告始终"飘"在旧
消息末尾和新消息之间，视觉割裂，不符合连续对话流设计。

### 8.2 根本原因

`WorkflowDetailView.axaml` 在 `ItemsControl`（FeedItems 列表）之外单独维护了一个
`IsVisible="{Binding Detail.IsCompleted}"` 的 StackPanel：

```xml
<!-- 最终报告（任务完成后显示） -->
<StackPanel IsVisible="{Binding Detail.IsCompleted}" ...>
    <controls:MarkdownRenderer Markdown="{Binding Detail.FinalReport}" />
</StackPanel>
```

这个块不属于 FeedItems 的 ObservableCollection，因此永远固定在列表末尾，
新消息只能追加在它后面（视觉上是"下面"），无法把报告推上去。

### 8.3 解决方案

1. `ConversationMessageVm.cs`：添加 `IsFinalReport` 计算属性（`Role == "final_report"`）
2. `P4TaskDetailVm.cs`：`OnTaskEngineCompleted` 中追加 `Role="final_report"` 的 FeedItem，
   `Content` 为完整 Markdown 报告文本
3. `WorkflowDetailView.axaml`：
   - 在 DataTemplate 内为 `IsFinalReport` 添加内联卡片渲染分支（`Border` + `MarkdownRenderer`）
   - 删除原来游离的底部报告 StackPanel

## 问题九：步骤执行子智能体无法使用工具，且不知道工作目录

### 9.1 现象

让 AI 写入文件时，执行结果只有 LLM 生成的文本内容，文件没有被实际写入磁盘；
且 AI 回复中常出现"无法确定当前工作目录"之类的说明，说明工作路径未被注入。

### 9.2 根本原因

**工具调用**：`OrchestratorAgent.ExecuteStepAsync` 使用 `SubAgentRunner.RunMultiTurnAsync`，
该方法内部直接调用 `IChatClient.GetResponseAsync`，没有配置任何 `ChatOptions.Tools`，
LLM 回复里的工具调用请求完全被忽略。

**工作目录**：`OrchestratorPrompts.StepExecutor` 系统提示词里没有任何路径占位符，
LLM 不知道应该把文件写到哪里。

### 9.3 解决方案

**修复工作目录注入**：

1. `OrchestratorPrompts.StepExecutor`：添加 `{WorkspaceDirectory}` 占位符，在"工作空间"
   小节中明确告知 AI 当前步骤的工作目录
2. `IOrchestratorAgent.ExecuteStepAsync`：接口签名增加 `string? workspaceDir` 参数
3. `OrchestratorAgent.ExecuteStepAsync`：
   - 接收 `workspaceDir` 参数，替换提示词中的占位符
   - 调用前确保目录存在（`Directory.CreateDirectory`）
4. `TaskExecutionEngine.ExecuteStepWithRetryAsync`：
   - 注入 `TaskFileResolver`（构造函数注入）
   - 调用 `_persistence.GetActiveRunId(taskId)` + `_fileResolver.GetStepWorkspaceDir(...)` 获取路径
   - 将路径传给 `ExecuteStepAsync`

**修复工具调用**：

1. `SubAgentRunner`：新增 `RunWithToolsAsync` 方法，使用
   `AIAgentFactory.BuildDynamicSubAgent(name, systemPrompt, provider, model, requiredTools)` 构建
   带工具的 AIAgent 并调用 `RunAsync`，使 LLM 可真正执行文件写入等操作
2. `OrchestratorAgent.ExecuteStepAsync`：当 `step.RequiredTools.Count > 0` 时
   改走 `_runner.RunWithToolsAsync`，否则保持原有 `RunMultiTurnAsync` 逻辑

修改文件：

- `Src/Netor.Cortana.AI/TaskEngine/Agents/OrchestratorPrompts.cs`
- `Src/Netor.Cortana.AI/TaskEngine/Agents/IOrchestratorAgent.cs`
- `Src/Netor.Cortana.AI/TaskEngine/Agents/OrchestratorAgent.cs`
- `Src/Netor.Cortana.AI/TaskEngine/Agents/SubAgentRunner.cs`
- `Src/Netor.Cortana.AI/TaskEngine/TaskExecutionEngine.cs`

> **状态：已在代码中修复**（本条目为事后补录）
