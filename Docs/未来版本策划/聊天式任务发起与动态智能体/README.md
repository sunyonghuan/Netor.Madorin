# 聊天式任务发起与动态智能体方案（P2 + P3 + P4）

> **文档定位**：本文档是 Madorin 聊天式任务发起 + 动态智能体 + 任务执行引擎方案的**入口与导航页**。详细技术内容拆分到多篇子文档，按顺序阅读即可掌握全部方案。
>
> **当前版本**：v0.2（2026-05-25）
>
> **背景**：阶段 6（commit `513a2c2`）+ 界面重设计 C1-C5 + P1 SubMode 过滤完成后，发现工作流 / 群聊任务发起体验非常糟糕：
>
> - `NewTaskDialog` 表单繁琐（6 项输入：标题 / Input / 子模式 / Manager / Members / 工具）
> - 学习曲线高（新用户不知道 Members 选什么）
> - 错误率高（Members 选错 = 任务跑废）
> - 文件夹无法整体作为附件（必须一个一个文件发）
> - 任务执行过程不可见（只能看列表项状态变化）

## 2. 核心改造点（总览）

### 2.1 工作流 tab 输入框（取代 NewTaskDialog）

```
┌────────────────────────────────────────────────┐
│  [📎附件区]                                    │
│  ────────────────────────────────────────────  │
│  [Manager: deepseek-chat ▼] [子模式: Magentic ▼] [Sub: 5 ▾]  │
│  ┌──────────────────────────────────────────┐  │
│  │ 输入任务描述（# 引用文件  @ 调用智能体）  │  │
│  │ Enter 发送  Shift+Enter 换行...           │  │
│  └──────────────────────────────────────────┘  │
│                              [🛠] [📎] [▶ 发送] │
└────────────────────────────────────────────────┘
```

- 输入即任务描述
- Manager 选择（默认值来自 SystemSettings.Workflow.DefaultManagerId）
- 子模式：Magentic / ParallelAnalysis 二选一
- 子 Agent 数量上限：默认 5（NumericUpDown 可临时改）
- 🛠 工具按钮 Popup（多选，默认勾选全局工具，高风险不勾选）
- 📎 附件按钮（支持文件 + **文件夹** P3 落地）
- 发送：启动任务；任务运行中切换为 ⏸ 暂停 / 停止

### 2.2 群聊 tab 输入框（取代 NewTaskDialog）

```
┌────────────────────────────────────────────────┐
│  [📎附件区]                                    │
│  ────────────────────────────────────────────  │
│  [Agents: 选择 2-N ✓] [code_reviewer ✕] [doc_writer ✕]  │
│  ┌──────────────────────────────────────────┐  │
│  │ 输入讨论话题...                           │  │
│  └──────────────────────────────────────────┘  │
│                              [🛠] [📎] [▶ 发送] │
└────────────────────────────────────────────────┘
```

- 与工作流 tab 视觉一致，但中间换成 Agent 多选 chips
- 没有 Manager 概念（GroupChat 设计就是平等讨论）
- 没有动态创建（Agent 必须从 AgentEntity 表选）

### 2.3 动态 Agent 创建（P2 核心创新）

Manager 内置 MCP 工具：

```csharp
[McpTool("create_subagent", "创建临时子智能体协助任务")]
public string CreateSubAgent(
    string name,           // 子智能体名称（如 "code_reviewer"）
    string instructions,   // 系统提示词
    string responsibility, // 职责描述（用户审批时显示）
    string[]? requiredTools // 必需工具列表（可空）
);
```

- 内存级 `DynamicAgentRegistry`（任务级生命周期）
- 通过 `BuildWithSubAgents` 注入到 Magentic Workflow
- 任务结束自动销毁
- UI 提供「保存为常用 Agent」按钮（决策 TP-3）

### 2.4 Plan 审批（P2 复用阶段 5B HITL）

Manager 在 plan 阶段输出后，弹卡片：

```
🤖 Manager 计划创建 3 个子智能体：
  ① code_reviewer  — 检查代码质量
  ② doc_writer     — 生成 README
  ③ test_runner    — 执行测试

[批准] [修改建议] [取消任务]
```

- 用户批准 → Manager 继续执行
- 修改建议 → Manager 重新规划
- 取消 → 整个任务停止

### 2.5 流程化实时可视化（P3）

- **工作流模式**：时间线视图（左侧时间轴 + 智能体名 + 内容 + 思考 / 工具调用折叠）
- **群聊模式**：聊天气泡（与 Chat 一致 + 头像 + 时间 + 内容 + 思考 / 工具调用气泡）

### 2.6 文件夹附件（P3）

- 文件 + 文件夹拖放（之前只支持文件）
- WorkspaceExplorer 右键 → 「发送到对话」/「发送到工作流」
- 自动过滤 `.exe / .dll / node_modules / .git` 等
- 附件 chips 显示「📁 src/ (12 文件, 230KB)」
- **核心**：Chat 模式也加（之前用户痛点：一个文件夹要重复发文件）

## 4. 核心结论汇总

### 4.1 技术可行性

| 改造点 | SDK 支持 | 风险 |
|---|---|---|
| 聊天式输入框 | 完全可行（复用现有 Chat 输入框样式） | 低 |
| 动态 Agent 创建 | `BuildWithSubAgents` 接受任意 `IEnumerable<AIAgent>`，`ChatClientAgent` 可在内存动态构造 | 低 |
| Plan 审批 | 复用阶段 5B HITL 机制（`ApprovalRequestArgs` + UI 卡片）| 低 |
| 保存常用 | 现有 `AgentService.Add` 直接支持 | 极低 |
| 文件夹附件 | Avalonia `IStorageFolder` 支持 + `Directory.EnumerateFiles` 递归 | 中（需限制递归层数 + 文件过滤）|
| 流程化可视化 | `WorkflowStepStartedEvent` / `WorkflowStepCompletedEvent` 已发出，只需 UI 渲染 | 中（虚拟化滚动 + 富文本）|

### 4.2 业务价值

| 维度 | 当前 | 改造后 |
|---|---|---|
| 新建任务时间（熟悉用户）| ~30 秒 | ~5 秒 |
| 新建任务时间（新用户）| ~2 分钟 | ~10 秒 |
| 错误率 | 中 | 低 |
| 学习曲线 | 高 | 低（与 Chat 一致）|
| 高级用户控制 | 表单精确控制 | Plan 审批 + 提示词工程 |

### 4.3 沉淀价值

- **「保存为常用 Agent」是杀手级 polish**：用户用动态 Agent 跑一次任务，发现某个临时子智能体好用 → 一键保存。半年后这个用户就有了一套自己定制的智能体生态。
- 从「工具」到「平台」的关键一跃。
