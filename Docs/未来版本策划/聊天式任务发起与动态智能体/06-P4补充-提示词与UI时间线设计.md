## 1. 文件保存位置修正

### 1.1 修正说明

之前 `05` 文档中的目录结构有误——执行文件应保存在**工作目录的 `.cortana` 隐藏文件夹**内，
不污染用户的工作目录树。

`.cortana` 是当前软件的系统隐藏目录，用户在文件管理器中默认不可见。

### 1.2 修正后的目录结构

```
{WorkspaceDir}/                      ← 用户的工作目录（项目/文档等）
  .cortana/                          ← 系统隐藏目录（所有 Cortana 运行时数据）
    tasks/
      {taskId}/                      ← 任务级目录（跨执行共享）
        task.json                    ← 任务元信息（标题、创建时间、状态）
        latest-run                   ← 文本文件，内容 = 最新 runId
        runs/                        ← 所有执行记录
          run-20260524-011500-a3f2/  ← 第 1 次执行
            run.json                 ← 本次运行元信息
            plan.json                ← 本次执行的计划快照
            plan.md                  ← 本次执行的计划摘要（用户可查看）
            checkpoint.json          ← 断点
            requirements.json        ← 需求分析结果
            conversation.json        ← 对话历史
            steps/
              step-1/
                intent.json
                workspace/
                  ...                ← 中间产物
                result.json
                result.summary.txt
              step-2/
                ...
          run-20260524-143022-7b1e/  ← 第 2 次执行
            ...
        templates/                   ← 从此任务保存的执行模板
          tpl-{name}.json
    templates/                       ← 全局执行模板（跨任务共享）
      tpl-market-analysis.json
      tpl-video-production.json
    config/                          ← 执行引擎配置
      engine.json                    ← 并发上限、重试策略等
```

### 1.3 关键路径解析

| 路径 | 说明 |
|------|------|
| `{workspace}/.cortana/tasks/{taskId}/latest-run` | 指向当前/最新执行的 runId |
| `{workspace}/.cortana/tasks/{taskId}/runs/{runId}/plan.json` | 某次执行的完整计划 |
| `{workspace}/.cortana/tasks/{taskId}/runs/{runId}/steps/{stepId}/workspace/` | 子智能体中间产物 |
| `{workspace}/.cortana/templates/` | 全局模板（用户保存后跨任务可选） |

## 2. 提示词策略：万能提示词 + 模板直接替换

### 2.1 策略确认

- **默认**：系统提供一个万能编排提示词（覆盖所有场景）
- **模板保存时**：保存当时的完整提示词（可能是万能提示词 + 用户的微调）
- **模板使用时**：直接用模板中保存的提示词**替换**当前主智能体提示词

**这是最简单的做法**——模板就是一份"提示词快照 + 步骤结构"。

### 2.2 模板数据结构更新

public string ResolveOrchestratorPrompt(ExecutionTemplate? template)
{
    // 用户选了模板且模板有自定义提示词 → 直接用模板的
    if (template is not null && !string.IsNullOrEmpty(template.OrchestratorPrompt))
        return template.OrchestratorPrompt;

    // 否则用系统默认万能提示词
    return LoadDefaultOrchestratorPrompt();
}

### 2.4 我的建议（你问有没有更好的方案）

你说"模板保存后直接替换提示词"——这个做法**简单且正确**。

唯一的小建议是：模板保存时，除了保存提示词原文，再保存一个**版本号**。
因为后续你可能迭代默认提示词（修 bug / 加功能），旧模板的提示词不会自动更新。

```csharp

但当前阶段不用实现迁移逻辑，只是**预留字段**。

## 3. 默认编排模式：万能提示词完整内容

以下是根据我们所有讨论内容编写的**系统默认万能提示词**。

文件路径：`Resources/Prompts/Orchestrator.Default.md`

```markdown
# 任务编排主智能体

你是任务执行引擎的编排主智能体（Orchestrator）。

## 核心原则

1. **你只做编排，不做具体工作。** 你是项目经理，不是员工。
2. **所有具体工作都委托给子智能体。** 需求分析、计划制定、步骤执行、结果验证——全部由你创建的子智能体完成。
3. **每次你被调用，只做一个决策。** 做完这个决策就结束，下一个决策等下次被调用时再做。
4. **你不包含任何领域专业知识。** 不管任务是市场分析、视频制作还是代码开发，你的行为模式不变。

## 你的工作流程

你必须按以下四个阶段推进。根据任务复杂度，你可以判断是否跳过某些阶段：
- 简单明确的任务（如"把文件转成 PDF"）→ 可跳过阶段 1，直接制定计划
- 复杂模糊的任务（如"帮我做市场分析"）→ 必须走完四个阶段

### 阶段 1：需求分析

**目的**：弄清楚用户到底要什么。

**你的行为**：
1. 判断用户任务的大类（数据分析 / 内容创作 / 工程开发 / 文件处理 / 其他）
2. 创建一个"需求分析师"子智能体，给它与任务大类相关的分析方法论
3. 把用户输入传给需求分析师
4. 需求分析师可能需要向用户提问 → 你转发问题和答案
5. 收到结构化需求要点后，宣布"需求分析完成"

**跳过条件**：用户输入已经非常明确（有具体步骤、有量化指标、无歧义），
你可以直接跳到阶段 2，把用户输入当作需求分析结果。

### 阶段 2：计划制定

**目的**：把需求拆解成可执行的步骤计划。

**你的行为**：
1. 创建一个"计划制定师"子智能体，给它：
   - 需求分析结果（阶段 1 的输出 或 用户原始输入）
   - 如果有参考模板 → 一并传入模板的步骤结构
2. 收到 JSON 格式的执行计划后，调用 `save_plan` 落盘
3. 向用户展示计划摘要，等待确认
4. 用户要求修改 → 把修改意见转给计划制定师 → 更新计划 → 再次展示
5. 用户确认后，宣布"计划已确认，开始执行"

**计划制定师需要决定的事项**：
- 分几个步骤
- 每步做什么（标题 + 描述）
- 步骤间的依赖关系
- 哪些步骤可以并行
- 每步需要什么类型的子智能体
- 每步需要什么工具
- 哪些步骤耗时长需要用户逐步确认
- 预估总耗时

### 阶段 3：执行

**目的**：按计划逐步完成任务。

**你的行为**：
1. 查看计划中状态为 pending 且依赖已满足的步骤
2. 为步骤创建子智能体：
   - 根据步骤的 agentTypeDescription 生成具体的 instructions
   - 分配步骤要求的工具
   - 注入前置步骤的结果摘要作为上下文
3. 委托执行 → 等待结果
4. 收到结果后：
   - 标记步骤完成，更新进度
   - 如果步骤标记了 requireUserConfirmation → 暂停等用户确认
   - 否则继续下一步
5. 如果步骤失败：
   - 重试次数未满 → 决定重试（可选择修改指令后重试）
   - 重试次数满了 → 暂停并报告用户，等待决策
6. 循环直到全部步骤完成

**并行执行**：
- 如果多个步骤的依赖都已满足且互相无依赖 → 可以同时执行
- 并发上限由系统配置控制（默认 3）

### 阶段 4：验证

**目的**：确保交付物满足需求。

**你的行为**：
1. 创建一个"验证审查员"子智能体
2. 输入 = 全部步骤结果摘要 + 原始需求要点 + 计划目标
3. 如果验证通过 → 生成执行报表 → 宣布任务完成
4. 如果验证发现问题：
   - 问题可自动修复 → 追加补充步骤，回到阶段 3
   - 问题需要用户判断 → 报告用户，等待决策
   - 问题不严重 → 在报表中注明，接受现状

## 决策原则

1. **一次一个决策** — 不要在一次输出中尝试规划多步。做完这步，下次再决定下一步。
2. **依赖事实决策** — 根据 currentPhase、recentEvents、availableSteps 做判断，不要凭空假设。
3. **子智能体 instructions 要具体** — 不要给笼统的指令（如"分析数据"），要写清楚分析什么、怎么分析、输出什么格式。
4. **失败时冷静** — 先判断是否可重试（网络错误/超时 → 重试；逻辑错误 → 修改指令重试或报告用户）。
5. **不要重复已完成的工作** — 如果某步骤已 completed，不要再执行它。
6. **尊重用户选择** — 用户要修改计划、要暂停、要跳过某步骤，照做。

## 特殊场景处理

### 恢复执行
如果 currentPhase 的上下文中包含 `isRecovery: true`，说明这是中断后恢复。
此时你会收到中断步骤的 workspace 文件列表。请根据中间产物判断：
- 已有足够产物 → 指示子智能体接续完成
- 产物不完整/不可信 → 指示子智能体从头重做

### 简单任务快速通道
如果用户输入非常明确且简单（如"把 a.docx 转成 PDF"），你可以：
1. 跳过阶段 1（直接把用户输入当需求）
2. 自己直接生成一个 1-2 步的计划（不创建策划师子智能体）
3. 直接进入执行

判断标准：任务步骤 ≤2 且无歧义 → 可走快速通道。

### 用户中途追加需求
如果在执行阶段收到用户新输入（非确认/非修改计划），判断：
- 是对当前步骤的补充信息 → 传给正在执行的子智能体
- 是新的需求变更 → 暂停执行，回到阶段 2 修改计划

## 4. 子智能体配置 + 并发上限

### 4.1 子智能体 LLM 配置

**确认**：子智能体使用与主智能体**相同的 LLM**（相同 Provider + Model）。

```csharp
/// <summary>
/// 创建子智能体时，继承主智能体的 LLM 配置。
/// </summary>
public AIAgent CreateSubAgent(
    string name, string instructions, List<string> tools,
    AiProviderEntity provider, AiModelEntity model)
{
    // provider 和 model 来自主智能体（用户在 UI 上选择的）
    // 所有子智能体共享同一个 LLM 配置
    return agentFactory.Build(name, instructions, tools, provider, model);
}
```

### 4.2 并发上限

**默认值**：3（同时最多 3 个子智能体并行执行）

**UI 可配置**：在工作流输入框工具栏，类似当前的"智能体数"选择器。

```csharp
/// <summary>引擎配置。</summary>
public sealed class EngineConfig
{
    /// <summary>
    /// 子智能体最大并发数（默认 3，UI 可选 1-10）。
    /// 影响 execute_parallel 时的并行度。
    /// </summary>
    public int MaxConcurrentAgents { get; set; } = 3;

    /// <summary>最大重试次数（默认 3）。</summary>
    public int DefaultMaxRetries { get; set; } = 3;

    /// <summary>重试退避策略（exponential / fixed）。</summary>
    public string RetryBackoffStrategy { get; set; } = "exponential";

    /// <summary>旧执行记录保留数量。</summary>
    public int MaxRetainedRuns { get; set; } = 5;
}
```

**并发控制实现**：

```csharp
/// <summary>
/// 并行执行多个步骤时的并发控制。
/// </summary>
private async Task ExecuteParallelSteps(
    List<StepExecutionRequest> steps, CancellationToken ct)
{
    var semaphore = new SemaphoreSlim(_config.MaxConcurrentAgents);

    var tasks = steps.Select(async step =>
    {
        await semaphore.WaitAsync(ct);
        try
        {
            await ExecuteStepAsync(step, ct);
        }
        finally
        {
            semaphore.Release();
        }
    });

    await Task.WhenAll(tasks);
}
```

## 5. UI 时间线视图细化设计

### 5.1 设计原则

- **不用聊天气泡** — 这是工作流，不是对话
- **不用头像** — 没必要，信息密度更重要
- **时间线形式** — 左侧时间轴 + 圆点节点 + 右侧内容
- **平铺展示** — 所有步骤和事件线性排列，直观清晰
- **只展示重点** — 不输出思考过程和工具调用细节

### 5.2 时间线布局

```

### 5.3 时间线节点类型

| 节点符号 | 含义 | 颜色 | 说明 |
|----------|------|------|------|
| `●` | 关键事件节点 | 蓝色 | 阶段开始、步骤开始/完成、任务完成 |
| `├` | 中间信息节点 | 灰色 | 进度更新、用户消息、重试信息 |
| `●` ✅ | 成功节点 | 绿色 | 步骤完成、任务完成 |
| `●` ⚠ | 警告节点 | 橙色 | 失败重试中 |
| `●` ❌ | 失败节点 | 红色 | 步骤最终失败 |
| `●` ⏸ | 暂停节点 | 灰色 | 等待用户确认/决策 |

### 5.4 时间线事件数据模型

```csharp
/// <summary>
/// 时间线上的单个事件项。
/// 驱动 UI 时间线渲染。
/// </summary>
public sealed class TimelineEvent
{
    /// <summary>事件 ID（用于去重）。</summary>
    public required string EventId { get; init; }

    /// <summary>时间戳。</summary>
    public DateTimeOffset Timestamp { get; init; }

    /// <summary>
    /// 事件类型：
    /// - task_started / task_completed
    /// - phase_started / phase_completed
    /// - step_started / step_completed / step_failed / step_retrying
    /// - step_progress
    /// - user_message / agent_message
    /// - plan_created / plan_confirmed
    /// - waiting_user
    /// </summary>
    public string EventType { get; init; } = string.Empty;

    /// <summary>
    /// 节点级别：
    /// - "primary" → ● 圆点（关键事件）
    /// - "secondary" → ├ 竖线分支（中间信息）
    /// </summary>
    public string NodeLevel { get; init; } = "primary";

    /// <summary>关联的步骤 ID（如果有）。</summary>
    public string? StepId { get; init; }

    /// <summary>关联的步骤序号。</summary>
    public int? StepSequence { get; init; }

    /// <summary>标题行（加粗显示）。</summary>
    public string Title { get; init; } = string.Empty;

    /// <summary>详情文本（正常字重）。</summary>
    public string? Detail { get; init; }

    /// <summary>
    /// 状态标记：
    /// - "running" → 蓝色
    /// - "completed" → 绿色
    /// - "failed" → 红色
    /// - "retrying" → 橙色
    /// - "waiting" → 灰色
    /// - null → 默认蓝色
    /// </summary>
    public string? Status { get; init; }

    /// <summary>进度百分比（step_progress 事件用）。</summary>
    public int? ProgressPercent { get; init; }

    /// <summary>内嵌卡片内容（计划确认卡片、执行报表卡片等）。</summary>
    public TimelineCard? Card { get; init; }
}

/// <summary>
/// 时间线中的内嵌卡片（计划确认、报表等需要用户操作的卡片）。
/// </summary>
public sealed class TimelineCard
{
    /// <summary>
    /// 卡片类型：
    /// - "plan_confirmation" → 计划确认卡片（含步骤列表 + 确认/修改按钮）
    /// - "step_result" → 步骤结果卡片（展示结果摘要）
    /// - "execution_report" → 执行报表卡片（完成后展示）
    /// - "error_decision" → 错误决策卡片（失败时让用户选择）
    /// </summary>
    public string CardType { get; init; } = string.Empty;

    /// <summary>卡片标题。</summary>
    public string Title { get; init; } = string.Empty;

    /// <summary>卡片内容行（每行一条信息）。</summary>
    public List<string> ContentLines { get; init; } = [];

    /// <summary>操作按钮列表。</summary>
    public List<TimelineCardAction> Actions { get; init; } = [];
}

/// <summary>卡片操作按钮。</summary>
public sealed class TimelineCardAction
{
    /// <summary>按钮文字。</summary>
    public string Label { get; init; } = string.Empty;

    /// <summary>按钮类型（primary / secondary / danger）。</summary>
    public string Style { get; init; } = "primary";

    /// <summary>点击后的动作标识（由 ViewModel 处理）。</summary>
    public string ActionId { get; init; } = string.Empty;
}
```

### 5.8 自动滚动策略

```csharp
/// <summary>
/// 时间线自动滚动逻辑：
/// - 新事件到来时，如果用户没有手动向上滚动 → 自动滚到底部
/// - 用户手动向上滚动查看历史 → 停止自动滚动
/// - 用户滚回底部 → 恢复自动滚动
/// </summary>
private bool _autoScrollEnabled = true;

private void OnNewTimelineEvent()
{
    if (_autoScrollEnabled)
    {
        Dispatcher.UIThread.Post(() =>
            _scrollViewer.ScrollToEnd());
    }
}

private void OnScrollChanged(ScrollChangedEventArgs e)
{
    var atBottom = _scrollViewer.Offset.Y >=
        _scrollViewer.Extent.Height - _scrollViewer.Viewport.Height - 20;
    _autoScrollEnabled = atBottom;
}
```

- 完成后：显示"保存模板" / "新建任务"

## 6. 设计决策总结

| 编号 | 决策 | 选择 | 理由 |
|------|------|------|------|
| P4-U1 | 文件保存位置 | `{workspace}/.cortana/` 隐藏目录 | 不污染用户工作目录 |
| P4-U2 | 提示词策略 | 万能提示词（默认）+ 模板直接替换 | 最简单，模板就是提示词快照 |
| P4-U3 | 子智能体 LLM | 与主智能体相同（同 Provider + Model） | 简化配置，统一质量 |
| P4-U4 | 并发上限 | 默认 3，UI 可选 1-10 | 平衡性能和资源占用 |
| P4-U5 | UI 时间线形式 | 时间轴 + 圆点节点 + 平铺内容 | 工作流≠对话，信息密度优先 |
| P4-U6 | 不用聊天气泡/头像 | 用步骤编号 + 智能体名标注 | 减少渲染压力，聚焦内容 |
| P4-U7 | 计划概览 | 可折叠面板（执行中默认折叠） | 节省空间，时间线是主区 |
| P4-U8 | 模板版本号 | 保存时记录 BasePromptVersion | 预留升级迁移能力 |
