# 04 — P4 方案设计：任务执行引擎（Task Execution Engine）

> **阶段定位**：P4 是架构重构版，结束后系统从"SDK 编排消费者"升级为"自主控制的任务执行引擎"。
> 对标 Claude Code / OpenAI Codex / GitHub Copilot 的工作机制，但不限于代码——通用长时间任务执行平台。
>
> **核心价值**：解决对话模式中"任务没完成就断掉、必须重新对话"的痛点，把长任务持久化——断了能恢复、每步有进度、用户全程掌控。
>
> **重构说明**：本方案从需求出发重新设计，不受之前阶段（2B → P3）代码结构约束。
> 底层可复用能力（AIAgentFactory、DynamicAgentRegistry、事件系统、数据库服务）尽量保留，
> 但编排层（WorkflowExecutor 核心）和 UI 层（详情视图）预计需要重写。
> 具体保留/重写范围在方案确认后、进入实施阶段时再做精确评估。
>
> **工期预估**：6-8 周（5-7 个 commit）

## 0. 范围 / 边界

### 0.1 IN（P4 必做）

| 模块 | 内容 |
|---|---|
| 执行引擎 | 自主控制的计划驱动执行循环（替代 SDK 黑盒编排） |
| 主智能体 | 纯编排器角色——不做具体工作，所有分析/计划/执行/验证都委托给子智能体 |
| 计划系统 | 显式执行计划：生成 → 用户确认 → 落盘 → 执行 → 进度追踪 |
| 计划落盘 | JSON（程序用） + Markdown（用户看），断点恢复 |
| 对话式计划 | 工作流 tab 输入框支持多轮对话（需求分析 → 计划制定 → 确认） |
| 执行调度 | 主智能体根据步骤内容决定：顺序 / 并行 / 混合 / 条件跳转 |
| 暂停/修改/恢复 | 用户暂停 → 修改计划 → 主智能体增量分析 → 从合适位置恢复 |
| 重试机制 | 步骤失败自动重试（含网络重试）+ 重试过程 UI 展示 |
| 耗时步骤确认 | 复杂步骤逐步执行 + 等用户确认 → 简单步骤自动连续 |
| 执行模板 | 满意的执行流程可保存为模板 → 下次选用模板重新规划 |
| UI 进度 | 实时计划进度面板 + 重点内容展示（不输出思考/工具调用细节） |

- ❌ Chat 模式改造（Chat 保持独立，不影响）
- ❌ 单元测试全覆盖（→ P5 稳定性阶段）
- ❌ 跨设备/跨进程计划同步（→ 远期）
- ❌ 执行模板版本管理（→ 远期）

## 1. 核心设计原则

### 1.1 主智能体 = 纯编排器（最重要的设计决策）

**主智能体不做任何具体工作**。它是 CEO，不是员工。

| 主智能体做什么 | 主智能体不做什么 |
|---|---|
| 理解用户意图（但不亲自分析需求细节） | ❌ 亲自分析专业领域需求 |
| 判断需要什么类型的子智能体 | ❌ 亲自编写执行计划的具体内容 |
| 创建子智能体（给合适的提示词 + 工具） | ❌ 亲自执行步骤（写报告/采集数据等） |
| 把子智能体的输出串联、传递 | ❌ 亲自验证结果的专业正确性 |
| 做调度决策（继续/重试/暂停/跳过/并行/顺序） | ❌ 包含任何专业领域知识 |
| 与用户对话沟通（传达子智能体的分析结果） | ❌ 自己思考领域问题 |

**为什么这样设计**：

1. **提示词不会冗余** — 主智能体的提示词永远只描述编排流程，不包含任何领域知识。
   不管用户是做市场分析、视频剪辑还是 3D 建模，主智能体的提示词不变。
2. **领域知识交给子智能体** — 需求分析需要专业知识，主智能体创建一个"需求分析专家"
   子智能体，给它领域相关的 instructions，让它去分析。
3. **可扩展** — 新领域只需新的子智能体模板，不改主智能体。
4. **context 不膨胀** — 主智能体只看子智能体的摘要输出，不看全量执行过程。

**类比**：

```
主智能体 = 项目经理
  "我需要一份市场分析报告"
  → 创建[需求分析师]："请分析用户的市场分析需求，整理需求要点"
  → 需求分析师输出需求要点
  → 创建[项目策划师]："根据这些需求要点，制定执行计划"
  → 项目策划师输出执行计划
  → 用户确认计划
  → 创建[数据采集员]："按步骤1的要求采集数据"
  → 创建[数据分析师]："根据采集到的数据进行分析"
  → 创建[报告撰写员]："根据分析结果撰写报告"
  → 创建[质量审查员]："检查报告是否满足需求"
  → 汇总输出
```

### 1.2 子智能体的生命周期

```
创建 → 接收任务指令 → 执行（可能多轮对话/工具调用）→ 输出结果摘要 → 销毁

特殊情况：
- 需求分析子智能体 → 可能需要和用户交互（主智能体转发）
- 计划制定子智能体 → 输出需要用户确认
- 执行子智能体 → 出错时由主智能体决定重试还是换一个
```

### 2.3 典型场景示例

#### 场景 A：市场分析报告

```
用户: "帮我做一份 2026 年 AI 硬件市场分析报告"

─── 主智能体编排（阶段1: 需求分析） ───
主智能体(内部决策): "这是市场分析任务，我需要创建一个需求分析专家"
主智能体 → create_agent("需求分析师", instructions="你是市场分析领域的需求分析专家...", tools=[])
主智能体 → delegate_task("需求分析师", "分析用户的市场分析需求，向用户提出澄清问题")

需求分析师 → 用户: "请问：
  1. 关注哪些细分领域？
  2. 需要哪些数据源？
  3. 报告格式？篇幅？"
用户: "AI 芯片和终端设备，公开数据，Markdown，5000 字"
需求分析师 → 主智能体: 需求要点结构化输出

─── 主智能体编排（阶段2: 计划制定） ───
主智能体(内部决策): "需求已明确，我需要一个项目策划专家来制定计划"
主智能体 → create_agent("项目策划师", instructions="你是项目执行计划的制定专家...", tools=[])
主智能体 → delegate_task("项目策划师", "根据以下需求要点制定执行计划: {...}")

项目策划师 → 主智能体: 执行计划 JSON
主智能体 → save_plan(plan_json)   // 落盘
主智能体 → 用户(UI): "执行计划如下，是否确认？"
用户: "步骤3 加一个价格趋势分析"
主智能体 → delegate_task("项目策划师", "用户要求步骤3加价格趋势分析，请更新计划")
项目策划师 → 主智能体: 更新后的计划
主智能体 → save_plan(updated_plan)
用户: "确认"

─── 主智能体编排（阶段3: 执行） ───
主智能体(内部决策): "步骤1 和步骤2 可以并行，各自有子任务"
主智能体 → create_agent("数据采集员-芯片", instructions="你是 AI 芯片领域的数据采集专家...", tools=[web_search, web_fetch])
主智能体 → create_agent("数据采集员-终端", instructions="你是终端设备市场的数据采集专家...", tools=[web_search, web_fetch])
// 并行执行步骤1 + 步骤2
主智能体 → update_progress(step1, "running", "正在采集 AI 芯片市场数据...")
主智能体 → update_progress(step2, "running", "正在采集终端设备市场数据...")
...
// 步骤3 依赖步骤1+2，等待后顺序执行
主智能体 → create_agent("数据分析师", instructions="你是市场数据分析专家...", tools=[])
主智能体 → delegate_task("数据分析师", "根据以下采集数据进行市场分析: {step1_result + step2_result}")
...

─── 主智能体编排（阶段4: 验证） ───
主智能体 → create_agent("质量审查员", instructions="你是报告质量审查专家...", tools=[])
主智能体 → delegate_task("质量审查员", "检查报告是否满足以下需求: {...}")
质量审查员 → 主智能体: "报告质量合格，但建议补充 xxx 数据"
主智能体(决策): "审查通过，输出最终报告"
```

## 3. 执行流程详细设计

### 3.1 四阶段流程

```
                    ┌─────────────────────┐
                    │    用户输入任务       │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │  阶段 1: 需求分析    │ ← 子智能体执行，可多轮对话
                    │  (RequirementsPhase) │
                    └──────────┬──────────┘
                               │ 需求确认
                    ┌──────────▼──────────┐
                    │  阶段 2: 计划制定    │ ← 子智能体执行，用户可修改
                    │  (PlanningPhase)     │    计划落盘（JSON + MD）
                    └──────────┬──────────┘
                               │ 用户确认计划
                    ┌──────────▼──────────┐
              ┌────►│  阶段 3: 执行        │ ← 按计划调度子智能体
              │     │  (ExecutionPhase)    │    顺序/并行/混合
              │     └──────────┬──────────┘
              │                │
              │     ┌──────────▼──────────┐
              │     │  步骤出错？          │
              │     └──┬──────────┬───────┘
              │        │是        │否
              │  ┌─────▼─────┐   │
              │  │ 自动重试   │   │
              │  │ 或报告用户 │   │
              │  └─────┬─────┘   │
              │        │ 用户修改计划
              │        │ 恢复执行
              └────────┘         │
                                 │ 全部步骤完成
                    ┌────────────▼────────┐
                    │  阶段 4: 验证        │ ← 子智能体检验结果
                    │  (ValidationPhase)   │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │  输出执行报表        │
                    │  可选：保存为模板    │
                    └─────────────────────┘
```

### 3.2 阶段 1: 需求分析（RequirementsPhase）

**主智能体的行为**：
1. 理解用户输入的任务类型（不深入分析具体领域）
2. 创建一个"需求分析师"子智能体，给它：
   - 与任务领域相关的 instructions（如"你是市场分析需求专家"）
   - 分析方法论（如"请向用户澄清以下维度：范围、数据源、格式、约束"）
3. 把用户的输入转发给需求分析师
4. 需求分析师可能需要和用户多轮对话（主智能体充当转发者）
5. 需求分析师输出结构化的需求要点后，阶段 1 结束

**用户的体验**：
- 看到的是和主智能体对话（实际上是子智能体在分析）
- 可以回答澄清问题、补充需求
- 主智能体告知"需求分析完成，下面制定执行计划"

**主智能体不做的事**：
- 不亲自分析"AI 芯片市场有哪些细分领域"
- 不亲自判断"5000 字需要几个数据源"
- 这些都是需求分析师子智能体的工作

### 3.3 阶段 2: 计划制定（PlanningPhase）

**主智能体的行为**：
1. 创建一个"计划制定师"子智能体，给它：
   - 需求分析阶段的输出（需求要点）
   - 计划制定的方法论 instructions
   - 如果用户选了模板 → 把模板结构也传给它
2. 计划制定师输出 JSON 格式的执行计划
3. 主智能体调用 `save_plan` 落盘
4. 主智能体把计划摘要展示给用户
5. 用户确认 / 要求修改
6. 如果修改 → 把修改意见传给计划制定师 → 更新计划 → 再次展示
7. 用户确认后 → 阶段 2 结束

**计划中的关键信息（由计划制定师决定）**：
- 需要几个步骤
- 每步做什么（标题 + 描述）
- 哪些步骤可以并行、哪些必须顺序
- 步骤间的依赖关系
- 每步需要什么类型的子智能体
- 每步需要什么工具
- 哪些步骤耗时长需要用户确认
- 预估总耗时

**主智能体不做的事**：
- 不亲自拆解步骤
- 不亲自判断并行还是顺序
- 这些都是计划制定师的工作

### 3.4 阶段 3: 执行（ExecutionPhase）

**主智能体的行为**：
1. 读取已确认的计划
2. 通过步骤调度器确定当前可执行的步骤
3. 为每个步骤创建对应的子智能体
4. 委托任务给子智能体
5. 监控执行结果
6. 做调度决策：
   - 步骤完成 → 更新进度 → 检查下一步
   - 步骤失败 → 自动重试 / 报告用户
   - 需要用户确认 → 暂停等待
   - 全部完成 → 进入阶段 4
7. 步骤间数据传递：把前一步的 ResultSummary 注入后续步骤的上下文

**执行模式（由计划决定，调度器执行）**：

| 模式 | 说明 | 示例 |
|------|------|------|
| sequential | 一个接一个 | 数据分析 → 报告撰写 |
| parallel | 子任务全部并发 | 同时采集芯片数据和终端数据 |
| mixed | 循环中部分步骤并行 | 步骤 1+2 并行 → 步骤 3 顺序 → 步骤 4+5 并行 |
| await_user | 执行前等用户说"开始" | 耗时的视频剪辑步骤 |

**子智能体的执行方式**：
- 默认：单 Agent 直接调用 `IChatClient`（最轻量）
- 可选：步骤内部需要多 Agent 讨论时，用 SDK `GroupChat`
- 可选：步骤内部需要多 Agent 并行分析时，用 SDK `ParallelAnalysis`
- 由主智能体在创建子智能体时决定

### 3.5 阶段 4: 验证（ValidationPhase）

**主智能体的行为**：
1. 创建一个"验证专家"子智能体
2. 把执行结果 + 原始需求要点 + 计划目标传给它
3. 验证专家检查：
   - 每个步骤的输出是否符合预期
   - 最终交付物是否满足需求
   - 是否有遗漏
4. 如果验证通过 → 生成报表 → 完成
5. 如果验证发现问题 → 主智能体决定：
   - 创建补充步骤（追加到计划）→ 回到阶段 3
   - 接受现状（问题不严重）→ 完成
   - 报告用户（需要用户判断）

    /// 主智能体根据此描述动态生成子智能体的 instructions。

    /// </summary>

    /// <summary>最大重试次数（默认 3）。</summary>
    public int MaxRetries { get; set; } = 3;

    // ──── 运行时状态（执行过程中填充） ────

    /// <summary>步骤执行状态。</summary>
    public PlanStepStatus Status { get; set; } = PlanStepStatus.Pending;

    /// <summary>实际分配的子智能体名称（运行时填充）。</summary>
    public string? AssignedAgentName { get; set; }

    /// <summary>开始时间。</summary>

    /// <summary>完成时间。</summary>

    /// <summary>错误信息（失败时）。</summary>
    public string? ErrorMessage { get; set; }

    /// <summary>已重试次数。</summary>
    public int RetryCount { get; set; } = 0;

    /// <summary>进度百分比（0-100，长步骤可中途更新）。</summary>
    public int ProgressPercent { get; set; } = 0;

## 5. 主智能体提示词设计

### 5.1 设计策略

主智能体的提示词 = 用户选择的 Agent 的 Instructions **被替换**为系统内置的编排流程提示词。

- Agent 的身份/名称保留（用于 UI 显示和日志）
- Agent 配置的 Provider / Model 保留（决定用哪个 LLM 来做编排）
- Agent 原始的 Instructions **不使用**

**理由**：编排流程必须严格控制（不能让 LLM 自由发挥跑偏）。
用户选 Agent 的真正目的是选"哪个 LLM 来做编排"（如选 Claude Opus 做复杂编排）。

### 5.2 提示词结构（骨架）

文件路径：`Resources/Prompts/Orchestrator.SystemPrompt.md`

```markdown
# 任务执行引擎 — 编排主智能体

你是任务执行引擎的编排主智能体（Orchestrator）。
你的唯一职责是**编排**——你不做任何具体工作，所有工作都由你创建的子智能体完成。

## 你的角色定义

你是项目经理，不是员工。你的工作是：
- 判断当前阶段需要什么样的子智能体
- 创建子智能体（给它合适的 instructions + 工具）
- 把子智能体的输出串联起来
- 做调度决策（继续/重试/暂停/跳过/并行/顺序）
- 与用户沟通（转达子智能体的分析结果、接收用户反馈）

你绝对不做的事：
- 不亲自分析专业领域需求（创建需求分析师去做）
- 不亲自编写执行计划（创建策划师去做）
- 不亲自执行具体步骤（创建执行者去做）
- 不亲自验证结果（创建审查员去做）
- 你的 instructions 中不包含任何领域专业知识

## 工作流程

你必须严格按以下四个阶段推进，不能跳过或乱序。

### 阶段 1：需求分析
1. 理解用户任务的大类（市场分析？视频制作？数据处理？代码开发？）
2. 创建一个"需求分析师"子智能体：
   - instructions = 与任务大类相关的需求分析方法论
   - 不需要工具（纯对话分析）
3. 把用户输入转发给需求分析师
4. 需求分析师可能需要向用户提出澄清问题 → 你负责转发
5. 收到结构化需求要点后，告知用户"需求分析完成"

### 阶段 2：计划制定
1. 创建一个"计划制定师"子智能体：
   - instructions = 项目执行计划制定方法论
   - 输入 = 阶段 1 的需求要点{{如果有模板：+ 参考模板结构}}
2. 收到 JSON 格式的执行计划后，调用 save_plan 落盘
3. 把计划摘要展示给用户
4. 用户可能要求修改 → 把修改意见转给策划师 → 更新计划 → 再次展示
5. 用户确认后，标记计划为 Confirmed

### 阶段 3：执行
按计划逐步执行。你的循环：
1. 查看计划中的下一步（或下一批可并行的步骤）
2. 为每个步骤创建对应的子智能体：
   - instructions = 根据步骤的 AgentTypeDescription 生成
   - tools = 步骤的 RequiredTools
   - 上下文 = 之前步骤的 ResultSummary
3. 委托任务 → 等待结果
4. 更新步骤状态（进度/完成/失败）
5. 如果失败：
   - 重试次数未满 → 自动重试
   - 重试次数满了 → 暂停并报告用户
6. 如果步骤标记了 RequireUserConfirmation → 等用户确认
7. 循环直到全部步骤完成

### 阶段 4：验证
1. 创建一个"验证审查员"子智能体
2. 输入 = 所有步骤结果 + 原始需求 + 计划目标
3. 如果验证通过 → 生成执行报表 → 完成
4. 如果验证发现问题 → 决定追加步骤 or 报告用户

## 你的工具

- `create_agent(name, instructions, tools)` — 创建子智能体
- `delegate_task(agentName, taskDescription, context?)` — 委托任务
- `update_progress(stepId, status, progressPercent?, summary?)` — 更新进度
- `request_user_input(question)` — 向用户提问（阶段 1/2 中使用）
- `save_plan(planJson)` — 保存/更新计划到文件
- `mark_step_complete(stepId, summary, detail?)` — 标记步骤完成
- `mark_step_failed(stepId, error)` — 标记步骤失败

## 输出规范

- 与用户的沟通用自然语言，简洁明了
- 子智能体的 instructions 要具体、聚焦，不要大而全
- 进度更新频率：每步开始/完成/失败时各一次，长步骤中途可酌情更新
- 不要输出你自己的内部思考过程（用户看不懂也不需要看）
```

### 5.3 子智能体提示词由主智能体动态生成

主智能体在 `create_agent` 时为每个子智能体生成专门的 instructions。举例：

## 6. 暂停 / 修改 / 恢复 / 重试机制

### 6.1 暂停触发方式

| 触发方式 | 行为 |
|----------|------|
| 用户点击暂停按钮 | CancellationToken 取消当前子智能体 → 计划状态改为 Paused |
| 步骤执行失败且重试耗尽 | 自动暂停 → 等待用户决定 |
| 耗时步骤完成后 | RequireUserConfirmation=true → 等待用户确认继续 |

### 6.2 暂停期间可执行的操作

| 操作 | 说明 |
|------|------|
| 修改计划步骤 | 编辑标题/描述/执行模式/删除/新增步骤 |
| 与主智能体对话 | 输入修改意见 → 主智能体创建策划师子智能体重新分析 |
| 查看已完成步骤的结果 | 展示每步的 ResultSummary |
| 手动标记步骤为"跳过" | 不想执行某步 |
| 恢复执行 | 用户确认修改后的计划 → 恢复执行 |

### 6.3 恢复执行的增量分析（方案 C）

用户修改计划并恢复执行时，主智能体需要做增量分析。
具体做法：主智能体创建一个"计划差异分析师"子智能体来判断。

```
计划差异分析师的输入：
  - 修改前的计划（版本 N）
  - 修改后的计划（版本 N+1）
  - 各步骤的执行结果

计划差异分析师的分析：
  1. 哪些已完成的步骤受到了修改影响？
     → 步骤内容被修改 → 需要重做（Status 重置为 Pending）
     → 步骤内容未变 → 保留结果（Status 保持 Completed）
  2. 新增的步骤放在哪里？
  3. 删除的步骤对后续步骤有依赖影响吗？
  4. 需要重新传递数据的步骤有哪些？

输出：
  - 需要重做的步骤列表
  - 可保留的步骤列表
  - 更新后的依赖关系
```

## 8. 执行引擎核心代码结构（骨架）

### 8.1 执行引擎主类

```csharp
/// <summary>
/// P4 任务执行引擎核心。
/// 驱动整个四阶段流程：需求分析 → 计划制定 → 执行 → 验证。
/// 不直接做任何 LLM 调用，所有工作委托给主智能体和子智能体。
/// </summary>
public sealed class TaskExecutionEngine
{
    // 依赖注入
    private readonly IOrchestratorAgent _orchestrator;
    private readonly IStepScheduler _scheduler;
    private readonly IPlanPersistence _persistence;
    private readonly IEventPublisher _publisher;
    private readonly RetryPolicy _retryPolicy;

    /// <summary>
    /// 启动完整的任务执行流程。
    /// </summary>
    public async Task ExecuteFullLifecycleAsync(
        string taskId,
        string userInput,
        string? templateId,
        CancellationToken ct)
    {
        // ── 阶段 1: 需求分析 ──
        var requirements = await _orchestrator.RunRequirementsPhaseAsync(
            taskId, userInput, ct);
        await _persistence.SaveRequirementsAsync(taskId, requirements, ct);
        _publisher.Publish(Events.OnPhaseCompleted, new PhaseCompletedArgs(
            taskId, "requirements"));

        // ── 阶段 2: 计划制定 ──
        ExecutionTemplate? template = null;
        if (!string.IsNullOrEmpty(templateId))
            template = await _persistence.LoadTemplateAsync(templateId, ct);

        var plan = await _orchestrator.RunPlanningPhaseAsync(
            taskId, requirements, template, ct);
        await _persistence.SavePlanAsync(plan, ct);
        _publisher.Publish(Events.OnPlanCreated, new PlanCreatedArgs(taskId, plan));

        // 等待用户确认（可能多轮修改）
        await WaitForPlanConfirmationAsync(taskId, plan, ct);

        // ── 阶段 3: 执行 ──
        await RunExecutionPhaseAsync(taskId, plan, ct);

        // ── 阶段 4: 验证 ──
        await _orchestrator.RunValidationPhaseAsync(taskId, plan, ct);

        // 完成
        _publisher.Publish(Events.OnTaskCompleted, new TaskCompletedArgs(taskId, plan));
    }

    /// <summary>
    /// 从断点恢复执行。
    /// </summary>
    public async Task ResumeFromCheckpointAsync(
        string taskId, CancellationToken ct)
    {
        var checkpoint = await _persistence.LoadCheckpointAsync(taskId, ct);
        if (checkpoint is null) throw new InvalidOperationException("无断点可恢复");

        var plan = await _persistence.LoadPlanAsync(taskId, ct);
        if (plan is null) throw new InvalidOperationException("计划文件丢失");

        // 校验版本一致性
        if (checkpoint.PlanVersion != plan.Version)
            throw new InvalidOperationException("计划版本不一致，可能需要重新确认");

        // 根据当前阶段恢复
        switch (checkpoint.CurrentPhase)
        {
            case "executing":
                await RunExecutionPhaseAsync(taskId, plan, ct);
                await _orchestrator.RunValidationPhaseAsync(taskId, plan, ct);
                break;
            case "validating":
                await _orchestrator.RunValidationPhaseAsync(taskId, plan, ct);
                break;
            default:
                // requirements / planning 阶段断开 → 重新走
                await ExecuteFullLifecycleAsync(
                    taskId, plan.Requirements?.OriginalInput ?? "", plan.SourceTemplateId, ct);
                break;
        }
    }

    /// <summary>
    /// 执行阶段主循环。
    /// </summary>
    private async Task RunExecutionPhaseAsync(
        string taskId, ExecutionPlan plan, CancellationToken ct)
    {
        plan.Status = PlanStatus.Executing;
        await _persistence.SavePlanAsync(plan, ct);

        while (!_scheduler.IsAllCompleted(plan) && !ct.IsCancellationRequested)
        {
            // 1. 获取当前可执行的步骤批次
            var readySteps = _scheduler.GetReadySteps(plan);

            if (readySteps.Count == 0)
            {
                if (_scheduler.IsWaitingUser(plan))
                {
                    await WaitForUserConfirmationAsync(taskId, plan, ct);
                    continue;
                }
                throw new InvalidOperationException("调度死锁：无可执行步骤");
            }

            // 2. 执行（可能并行）
            var stepTasks = readySteps.Select(step =>
                ExecuteStepWithRetryAsync(taskId, plan, step, ct));
            await Task.WhenAll(stepTasks);

            // 3. 持久化进度 + 断点
            await _persistence.SavePlanAsync(plan, ct);
            await _persistence.SaveCheckpointAsync(taskId, new ExecutionCheckpoint
            {
                PlanVersion = plan.Version,
                CurrentStepId = readySteps.Last().StepId,
                CurrentPhase = "executing",
                SavedAt = DateTimeOffset.UtcNow,
            }, ct);
        }
    }

    /// <summary>
    /// 带重试的单步骤执行。
    /// </summary>
    private async Task ExecuteStepWithRetryAsync(
        string taskId, ExecutionPlan plan, PlanStep step, CancellationToken ct)
    {
        step.Status = PlanStepStatus.Running;
        step.StartedAt = DateTimeOffset.UtcNow;
        _publisher.Publish(Events.OnStepStarted, new StepStartedArgs(taskId, step));

        while (true)
        {
            try
            {
                // 主智能体为这个步骤创建子智能体并委托执行
                var result = await _orchestrator.ExecuteStepAsync(taskId, plan, step, ct);

                step.Status = PlanStepStatus.Completed;
                step.CompletedAt = DateTimeOffset.UtcNow;
                step.ResultSummary = result.Summary;
                step.ResultDetail = result.Detail;
                step.ProgressPercent = 100;

                _publisher.Publish(Events.OnStepCompleted, new StepCompletedArgs(taskId, step));
                await _persistence.SaveStepResultAsync(taskId, step.StepId, result, ct);

                // 需要用户确认？
                if (step.RequireUserConfirmation)
                {
                    step.Status = PlanStepStatus.WaitingUser;
                    _publisher.Publish(Events.OnStepWaitingUser, new StepWaitingArgs(taskId, step));
                }
                break;
            }
            catch (Exception ex) when (ex is not OperationCanceledException)
            {
                step.RetryCount++;
                if (step.RetryCount <= step.MaxRetries && IsRetryableError(ex))
                {
                    step.Status = PlanStepStatus.Retrying;
                    _publisher.Publish(Events.OnStepRetrying, new StepRetryingArgs(
                        taskId, step, step.RetryCount, ex.Message));

                    var delay = CalculateBackoff(step.RetryCount);
                    await Task.Delay(delay, ct);
                    continue;
                }

                // 重试耗尽
                step.Status = PlanStepStatus.Failed;
                step.ErrorMessage = ex.Message;
                _publisher.Publish(Events.OnStepFailed, new StepFailedArgs(taskId, step));

                // 暂停计划等用户决定
                plan.Status = PlanStatus.Paused;
                await _persistence.SavePlanAsync(plan, ct);
                await WaitForUserDecisionOnFailureAsync(taskId, plan, step, ct);
                break;
            }
        }
    }
}
```

### 8.2 步骤调度器

```csharp
/// <summary>
/// 步骤调度器：根据依赖关系和执行模式决定下一步。
/// </summary>
public interface IStepScheduler
{
    /// <summary>获取当前可执行的步骤批次（依赖满足 + 不在等待中）。</summary>
    IReadOnlyList<PlanStep> GetReadySteps(ExecutionPlan plan);

    /// <summary>检查计划是否全部完成。</summary>
    bool IsAllCompleted(ExecutionPlan plan);

    /// <summary>检查是否有步骤在等待用户确认。</summary>
    bool IsWaitingUser(ExecutionPlan plan);
}

public sealed class StepScheduler : IStepScheduler
{
    public IReadOnlyList<PlanStep> GetReadySteps(ExecutionPlan plan)
    {
        var ready = new List<PlanStep>();

        foreach (var step in plan.Steps)
        {
            if (step.Status is not (PlanStepStatus.Pending or PlanStepStatus.WaitingDeps))
                continue;

            // 检查依赖是否全部满足
            var allDepsCompleted = step.DependsOn.All(depId =>
                plan.Steps.Any(s => s.StepId == depId
                    && s.Status is PlanStepStatus.Completed or PlanStepStatus.Skipped));

            if (allDepsCompleted)
            {
                ready.Add(step);
            }
        }

        return ready;
    }

    public bool IsAllCompleted(ExecutionPlan plan)
        => plan.Steps.All(s =>
            s.Status is PlanStepStatus.Completed or PlanStepStatus.Skipped or PlanStepStatus.Cancelled);

    public bool IsWaitingUser(ExecutionPlan plan)
        => plan.Steps.Any(s => s.Status == PlanStepStatus.WaitingUser);
}
```

### 10.2 使用流程

```
用户新建任务 → 输入框旁边显示"📋 选用模板"按钮
  → 弹出模板列表（按标签/搜索筛选）
  → 用户选择模板
  → 模板结构传给阶段2的计划制定子智能体
  → 计划制定师参考模板 + 新需求 → 生成新计划（不是死板复制）
  → 正常进入计划确认流程
```

### 10.3 模板与计划的关系

```
模板 ≠ 计划
模板 = 计划的结构参考（步骤框架 + 执行模式 + 智能体分工模式）
计划制定子智能体拿到模板后：
  - 参考：步骤框架、并行/顺序安排、智能体类型分工
  - 重新生成：步骤具体描述、具体数据源、具体工具参数
  - 可能增删步骤（新任务和模板不完全一样）
```

## 12. 步骤间数据传递

### 12.1 机制

步骤间的数据传递通过 `ResultDetail` 字段实现。

```
步骤1 完成 → ResultDetail = "采集到的 47 条数据记录的 JSON"
步骤3 开始 → 主智能体把步骤1的 ResultDetail 注入步骤3子智能体的上下文

具体流程：
  主智能体在 ExecuteStepAsync 时：
  1. 查找当前步骤的 DependsOn 列表
  2. 把依赖步骤的 ResultDetail 收集起来
  3. 作为 context 参数传给 delegate_task
  4. 子智能体的 instructions 中包含"你的输入数据来自前置步骤: {context}"
```

### 12.2 context 压缩策略（防止 token 膨胀）

对于跨多步依赖的长 context：
- 主智能体只传 ResultSummary（摘要版）给子智能体
- 需要完整数据时，子智能体通过工具读取 `results/step-{stepId}.json` 文件
- 主智能体自身的 context 只包含步骤摘要（不包含详情），避免 context window 爆炸

## 13. 实施计划（Commit 拆分）

### Commit P4-1：执行引擎核心 + 计划模型（后端）

**范围**：数据模型 + 执行循环 + 步骤调度器 + 重试机制

**验收**：给定一个 ExecutionPlan，引擎能按依赖顺序逐步执行，
并行步骤能并发，失败能重试，超时能暂停。

### Commit P4-2：计划落盘 + 断点恢复（后端）

**范围**：IPlanPersistence 实现 + JSON/Markdown 文件读写 + 断点恢复

**验收**：模拟进程重启，能从 JSON 文件恢复计划状态，已完成步骤不重跑。

### Commit P4-3：主智能体提示词 + 子智能体动态创建（后端）

**范围**：Orchestrator 系统提示词 + IOrchestratorAgent 实现 + 四阶段流程

**验收**：用户输入任务 → 主智能体创建需求分析师 → 多轮对话 → 创建策划师 → 生成计划 → 执行。

### Commit P4-4：UI 计划进度面板 + 对话区域改造（前端）

**范围**：WorkflowDetailView 重写 + TaskDetailVm 改造 + PlanStepItemVm

**验收**：计划进度面板实时更新，对话区显示需求分析/计划讨论，执行期显示摘要。

### Commit P4-5：暂停 / 修改 / 恢复 + 增量分析（前后端）

**范围**：暂停按钮 + 计划编辑 + 增量分析恢复 + 重试状态展示

**验收**：用户暂停 → 修改计划 → 增量分析 → 恢复执行。

### Commit P4-6：执行模板（前后端）

**范围**：模板保存/加载/使用 + UI 入口

**验收**：任务完成后保存为模板 → 新任务选用模板 → 基于模板生成新计划。

### Commit P4-7：集成测试 + 边界处理

**范围**：端到端测试 + 异常场景覆盖 + 性能验证

**验收**：完整场景跑通 + 长时间任务不卡 UI + 断点恢复正确。

## 15. 决策记录

| 编号 | 决策 | 选项 | 理由 |
|------|------|------|------|
| P4-D1 | 主智能体职责 | **纯编排**（不做具体工作，一切委托子智能体） | 避免提示词冗余；领域知识交给专业子智能体；主智能体提示词永远不变 |
| P4-D2 | 主智能体提示词 | 替换为固定流程提示词（不用 Agent 原始 Instructions） | 编排流程必须严格控制 |
| P4-D3 | 计划落盘格式 | JSON（程序用）+ Markdown（用户看） | 兼顾可恢复性和可读性 |
| P4-D4 | 对话式计划制定 | 在工作流 tab 输入框中对话 | 不引入新 UI 入口 |
| P4-D5 | SDK 编排的定位 | 降级为步骤内部引擎（可选） | 步骤间编排由自建引擎控制 |
| P4-D6 | 暂停修改后的恢复 | 主智能体创建差异分析子智能体做增量分析 | 最智能，避免无谓重做 |
| P4-D7 | UI 输出内容 | 只输出重点（进度 + 摘要 + 错误） | 避免渲染性能压力 |
| P4-D8 | 执行模板定位 | 结构参考（计划制定师参考模板但重新生成） | 每次任务都不一样 |
| P4-D9 | 与现有代码关系 | 从需求出发设计，不受现有代码约束 | 编排层 + UI 层预计重写；底层能力尽量复用 |
