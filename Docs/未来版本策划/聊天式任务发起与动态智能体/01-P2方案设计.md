## 1. 架构选型（已锁定 TP-1 ~ TP-11）

### 1.1 UI 层：聊天式输入框

| 选项 | 描述 | 工作量 | 风险 |
|---|---|---|---|
| **A（选用）** | 拷贝 Chat 输入框样式 + 改造为 Workflow/GroupChat 专用 | 1 周 | 低 |
| B | 抽 `TaskInputPanel` 用户控件三处共享（Chat + Workflow + GroupChat）| 2 周 | 中（抽象成本高）|
| C | 直接在 WorkflowDetailView / GroupChatDetailView 内嵌（不抽组件）| 0.5 周 | 中（重复代码 3 份）|

**选 A 的理由**：
- 复用现有 Chat 输入框样式 90%，改造成本最小
- 不引入新抽象，避免 P2 期间过度设计
- 后续 P3+ 若需要再抽组件，那时需求更清晰

**注**：P3 文件夹附件如果要让 Chat 模式也支持，那时会涉及 Chat 输入框改动，那个时机再评估是否抽组件。

### 1.2 动态 Agent 注册：内存 Dictionary

| 选项 | 描述 | 工作量 | 风险 |
|---|---|---|---|
| **A（选用）** | 进程内 `ConcurrentDictionary<TaskId, List<AIAgent>>` Singleton DI | 0.5 周 | 极低 |
| B | SQLite 持久化 + 进程内缓存 | 1.5 周 | 中（与"任务结束销毁"语义冲突）|

**选 A 的理由**：
- 临时 Agent 本就是任务级生命周期（决策 TP-3 A）
- 持久化由「保存为常用 Agent」按钮触发，那时走 `AgentService.Add` 持久化到 `AgentEntity` 表

# 子智能体创建指南

你是 Magentic Manager，可以通过 `create_subagent` 工具创建临时子智能体。

## 创建原则

1. **先 plan 后 create**：分析任务后列出需要的子智能体清单
2. **职责分离**：每个子智能体只做一件事
3. **数量克制**：通常 2-5 个，最多不超过 {{MaxSubAgents}} 个
4. **命名规范**：英文小写下划线（如 code_reviewer / doc_writer）

## 示例工作流

任务："审查 main.cs 并生成代码质量报告"

Plan:
- code_analyzer: 分析代码风格、复杂度、潜在 bug
- security_checker: 检查安全漏洞
- report_writer: 综合前两个结果生成报告

调用：
\`\`\`
create_subagent(
  name="code_analyzer",
  instructions="你是 C# 代码分析专家...",
  responsibility="分析代码质量与复杂度",
  requiredTools=["read_file"]
)
\`\`\`
