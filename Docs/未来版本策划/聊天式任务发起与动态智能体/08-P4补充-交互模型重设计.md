# 08 — P4 补充：交互模型重设计（对话驱动 + 双模态切换）

> **文档定位**：基于实际测试反馈，对 P4 任务执行引擎的 **UI 交互模型** 进行重新设计。
> 本文档覆盖以下变更：
>
> 1. 废弃审批卡片模式，改为对话驱动
> 2. 双模态界面切换（对话模式 ↔ 执行模式）
> 3. 危险工具授权的浮动意图面板
> 4. 执行结果展示（摘要 + 文件列表）
> 5. 文件保存位置策略
> 6. 任务生命周期重定义（永续对话循环）
>
> **创建日期**：2026-05-25
>
> **前置文档**：`04-P4方案设计`、`06-P4补充-提示词与UI时间线设计`、`07-P4补充-资源管理与工具授权`
>
> **废弃内容**：本文档生效后，以下既有设计 **废弃**：
> - P2-4 动态子智能体创建审批卡片（`DynamicAgentCreationApprovalVm`、三按钮交互）
> - P2-4 HITL 审批卡片的"批准/修改/拒绝"按钮式计划确认
> - 任务完成即终态的假设

## 1. 问题背景（实测发现）

### 1.1 现有交互模式的问题

当前 P4 的 UI 交互是 **"弹卡片确认"** 模式：

```
用户输入任务 → [后台分析] → 弹出审批卡片（计划详情 + ✓批准/✎修改/✕拒绝）
                           → 弹出子智能体创建审批卡片（提示词/工具 + ✓批准/✕拒绝）
```

实测问题：

| 问题 | 影响 |
| ------ | ------ |
| 审批卡片打断对话流 | 用户无法以自然语言讨论和修改需求 |
| 子智能体创建要求用户审批 | 用户不关心内部调度细节，过度干预 |
| 计划确认只能通过按钮 | 无法表达"大致OK但步骤2改一下"这种混合意图 |
| 任务完成即终态 | 用户对结果有疑问或不满意时没有继续交互的入口 |

### 1.2 期望的交互体验

> **用户的整个体验就是"跟一个项目经理对话"。**
>
> 讨论需求 → 对话；确认方案 → 对话中自然表达；看执行进度 → 进度面板；
> 看完结果 → 回到对话继续聊；不满意 → 继续对话讨论修改再跑一轮；
> 满意了 → 对话中说"可以了"。
>
> **卡片/按钮/审批 = 打断对话自然流动的中间层，应当移除。**

## 2. 核心设计：双模态界面

### 2.1 两种模态定义

工作流 Tab 的界面在任务生命周期中只有 **两种模态**，自动切换：

| 模态 | 界面形态 | 触发切入 | 触发切出 |
| ------ | --------- | --------- | --------- |
| **对话模式** | 纯数据流对话界面 + 输入框 | 任务创建时 / 执行完成时 / 用户暂停时 | LLM 识别到用户"确认执行"意图 |
| **执行模式** | 时间线进度面板 | AI 判断用户确认了执行意图 | 所有步骤完成 / 用户暂停 |

### 2.2 对话模式界面规格

**与普通 Chat 模式共用 Feed 组件，但不显示头像。** 用户消息与 AI 消息通过气泡/背景色区分，
不依赖分隔线。这是当前代码已实现的样式。

```
┌──────────────────────────────────────────────────────────────┐
│                     工作流 - 对话模式                           │
│                                                              │
│  ╔══════════════════════════════════════════════════════╗   │
│  ║ 帮我做一份 AI 芯片市场分析报告                         ║   │
│  ╚══════════════════════════════════════════════════════╝   │
│    （用户消息：半透明蓝色边框气泡，右对齐）                     │
│                                                              │
│  好的，我分析了一下你的需求，有几个问题想确认：                   │
│  1. 报告覆盖哪些地区？                                        │
│  2. 目标受众是投资者还是技术团队？                               │
│  3. 需要包含竞品对比吗？                                       │
│    （AI 消息：无气泡纯文本，左对齐）                            │
│                                                              │
│  ╔══════════════════════════════════════════════════════╗   │
│  ║ 全球市场，给投资者看的，要有竞品对比                    ║   │
│  ╚══════════════════════════════════════════════════════╝   │
│                                                              │
│  明白了。我初步规划如下（共 4 步）：                            │
│  **步骤 1：数据采集**                                        │
│  从公开财报中采集全球 AI 芯片市场数据（NVIDIA/AMD/Intel）       │
│                                                              │
│  **步骤 2：竞品对比**                                        │
│  横向对比主要厂商的市场份额、性能和价格...                       │
│                                                              │
│  确认开始执行吗？还是需要调整？                                  │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 输入想法...                                          │    │
│  └──────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
```

关键特征：

- **无头像**：不显示 AI / 用户头像，聚焦内容
- **气泡区分**：用户消息用半透明蓝色边框气泡，AI 消息为纯文本，视觉上自然区分
- **计划内容以 AI 消息形式展示**：计划生成后直接在对话流中呈现步骤列表（见问题一修复方案）
- **输入框始终在底部**：用户随时可以输入

### 2.3 执行模式界面规格

沿用 doc 06 §5 的时间线设计（时间轴 + 圆点节点），并在每个步骤下方追加
**子过程详情行**，让用户清晰看到步骤内部发生了什么。

```
┌──────────────────────────────────────────────────────────────┐
│                     工作流 - 执行模式                           │
│                                                              │
│  ●─ ▶ 步骤 1：数据采集                          （蓝色，加粗） │
│     └ 正在初始化数据采集专家...                  （灰色小字）   │
│     └ 搜索 NVIDIA 2025 年财报数据...             （灰色小字）   │
│     └ 搜索 AMD 市场份额报告...                   （灰色小字）   │
│     └ 找到 47 条相关数据，正在整理...             （灰色小字）   │
│  ●─ ✓ 步骤 1 完成                               （绿色）      │
│     └ 采集到 47 条数据，涵盖 NVIDIA/AMD/Intel    （灰色小字）   │
│                                                              │
│  ●─ ▶ 步骤 2：竞品对比分析                       （蓝色，加粗） │
│     └ 正在初始化市场分析师...                    （灰色小字）   │
│     └ 对比 NVIDIA vs AMD 市场份额...             （灰色小字）   │
│     └ ▌（流式输出中...）                         （绿色光标）  │
│                                                              │
│  ○─ ⏳ 步骤 3：市场趋势预测                      （灰色，等待） │
│  ○─ ⏳ 步骤 4：撰写报告                          （灰色，等待） │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 输入框（灰显，执行中不可输入，暂停后恢复）              │    │
│  └──────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
```

**步骤行规格：**

| 元素 | 说明 |
| ------ | ------ |
| `●─` | 实心圆点，步骤激活中（蓝色） |
| `○─` | 空心圆点，步骤等待中（灰色） |
| 步骤标题行 | 13px，颜色按状态变化（蓝色=执行中 / 绿色=完成 / 红色=失败） |
| `└ 子过程行` | 11px 灰色小字，缩进显示，每次 AI 有输出时追加一行 |
| 完成后结果行 | `└ [结果摘要]`，显示步骤的 ResultSummary |
| 流式光标 `▌` | 步骤执行中时，子过程行末尾显示绿色光标，表示 AI 正在工作 |

**子过程行来源：**

步骤执行期间，子智能体每轮输出（`onAiMessage` 回调）追加为一条缩进子过程行，
过滤掉 `[DONE]` / `[CONTINUE]` 标记后直接显示文本。
步骤完成后，`ResultSummary` 作为最后一条缩进行追加。

关键特征：

- 步骤粒度可展开/折叠（默认展开当前执行中的步骤，已完成步骤可折叠）
- 子过程行让用户清楚看到 AI 在做什么，不再是黑箱
- 输入框默认灰显（执行中不接受输入），暂停后恢复
- 用户可通过暂停按钮中断执行

### 2.4 模态切换时机

```
对话模式 ──(LLM 识别到"确认执行"意图)──→ 执行模式
                                           │
                                           ├── 执行完成 → 自动切回对话模式
                                           ├── 用户暂停 → 自动切回对话模式
                                           └── 危险工具触发 → 执行模式内弹浮动面板（§4）
```

## 3. 对话驱动的意图识别

### 3.1 用户意图分类

对话模式中，用户的每条输入经过 LLM 意图识别，分为以下类别：

| 意图 | 示例 | 引擎行为 |
| ------ | ------ | --------- |
| `confirm_execute` | "开始"、"执行吧"、"确认"、"OK"、"就这样" | 切入执行模式 |
| `modify_plan` | "步骤2改成并行"、"加一步政策风险分析" | AI 修改计划后继续对话 |
| `question` | "政策风险是指什么？"、"步骤3具体做什么？" | AI 回答问题，留在对话 |
| `specify_path` | "报告保存到 D:/Reports/" | 记录路径偏好，继续对话 |
| `satisfied` | "可以了"、"满意"、"没问题了"、"结束吧" | 任务标记完成 |
| `cancel` | "不做了"、"取消" | 任务取消 |
| `other` | 任何不属于上述类别的输入 | AI 自行判断如何回应 |

### 3.2 意图识别实现策略

```
用户输入 → 编排主智能体（Orchestrator）
          ↓
        主智能体同时承担两个职责：
        1. 判断用户意图（基于对话上下文）
        2. 生成回复内容

        主智能体的输出格式：
        {
          "intent": "modify_plan",
          "response": "好的，已调整计划：...",
          "updatedPlan": { ... }   // 仅当 intent=modify_plan 时存在
        }
```

不需要单独的"意图分类器"——主智能体在理解上下文的过程中自然能判断意图。

### 3.3 对话上下文管理

对话模式中的所有消息（用户输入 + AI 回复）构成完整的对话历史。
切入执行模式后，执行过程的记录（步骤摘要）也会追加到上下文中。
切回对话模式时，AI 拥有完整的历史：需求讨论 + 执行过程 + 执行结果。

```
上下文结构：
  [对话] 用户：帮我做一份 AI 芯片市场分析报告
  [对话] AI：有几个问题想确认...
  [对话] 用户：全球市场，给投资者看的
  [对话] AI：规划如下：1. 采集数据 2. 竞品对比 3. 趋势预测 4. 撰写报告
  [对话] 用户：开始
  [执行] 步骤1 完成：采集到 47 条数据（摘要）
  [执行] 步骤2 完成：NVIDIA 78% / AMD 12% / Intel 8%（摘要）
  [执行] 步骤3 完成：市场趋势预测结论（摘要）
  [执行] 步骤4 完成：报告已生成（摘要）
  [对话] AI：执行完成，以下是结果...
  [对话] 用户：第三章数据来源是什么？    ← AI 可引用步骤1的结果回答
```

## 4. 危险工具授权：浮动意图面板

### 4.1 工具风险分级

| Level | 分类 | 行为 | 示例 |
| ------- | ------ | ------ | ------ |
| 0 | 安全 | 自动执行，不通知 | 文本生成、数据计算 |
| 1 | 低风险 | 自动执行，时间线记录 | 网页搜索、文件读取 |
| 2 | 高风险 | **暂停 + 浮动意图面板** | 文件写入、发送消息、API 调用 |
| 3 | 极高风险 | **暂停 + 浮动意图面板（无"全部授权"选项）** | 文件删除、支付、执行系统命令 |

### 4.2 浮动意图面板设计

当 Level 2-3 工具被调用时，在输入框上方浮出意图面板：

```
┌──────────────────────────────────────────────────────────────┐
│                     执行模式 - 时间线进度                       │
│                                                              │
│  [步骤1] ██████████ 完成 ✓                                   │
│  [步骤2] 执行中...                                            │
│                                                              │
│  ⚠️ 步骤2 请求调用 [发送邮件]                                  │
│     收件人: team@company.com                                  │
│     主题: "AI 芯片市场分析报告 v1"                              │
│     内容摘要: 包含 4 章分析结论 + 3 张图表...                   │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│  ┌─── 浮动意图面板 ───────────────────────────────────────┐  │
│  │                                                        │  │
│  │   ✓ 确认执行                                           │  │
│  │   ✓✓ 本任务中该工具全部授权                              │  │
│  │   ✕ 拒绝本次调用                                        │  │
│  │   ┌──────────────────────────────────────────────┐     │  │
│  │   │ 输入其他想法...                                │     │  │
│  │   └──────────────────────────────────────────────┘     │  │
│  │                                                        │  │
│  └────────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 常规输入框（灰显，等授权决策完成后恢复）                │    │
│  └──────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
```

### 4.3 面板选项定义

**Level 2（高风险但可恢复）：**

| 选项 | 行为 |
| ------ | ------ |
| ✓ 确认执行 | 执行该工具调用，继续后续步骤 |
| ✓✓ 本任务中该工具全部授权 | 执行，且后续本任务内**同一工具**的调用不再询问 |
| ✕ 拒绝本次调用 | 不执行，将"用户拒绝使用 [工具名]"反馈给子智能体，**AI 自主决定替代方案** |
| 自定义输入 | LLM 意图识别，执行相应操作（如"改一下收件人为 boss@..."）|

**Level 3（极高风险/不可逆）：**

| 选项 | 行为 |
| ------ | ------ |
| ✓ 确认执行 | 执行（建议 UI 增加二次确认提示） |
| ✕ 拒绝本次调用 | 不执行，AI 自主替代方案 |
| 自定义输入 | 同 Level 2 |

> **Level 3 不提供"全部授权"选项**，每次调用必须单独确认。

### 4.4 面板交互规则

| 规则 | 说明 |
| ------ | ------ |
| 出现时机 | Level 2-3 工具被调用时自动浮出 |
| 消失时机 | 用户做出任何选择后立即收起 |
| 不遮挡时间线 | 其他不受阻塞的步骤继续执行 |
| 位置 | 输入框正上方，浮动层 |
| 本质 | 输入框的"增强态"——需要决策时输入框变形为决策面板 |

### 4.5 拒绝后的 AI 行为

```
子智能体收到反馈："用户拒绝调用 [工具名]，请用其他方式完成当前步骤"

AI 可能的替代决策（由 LLM 自主判断）：
  · 换用低风险替代工具（如：拒绝发邮件 → 保存到本地文件）
  · 将结果输出到报告正文中让用户自行处理
  · 跳过当前子步骤，在摘要中说明原因
  · 向主智能体汇报无法完成，由主智能体重新规划
```

### 4.6 "全部授权"的粒度

- **粒度**：本任务内**该工具**全部授权
- **作用域**：从授权时刻起，到本任务结束（包括重新执行的循环）
- **存储**：内存级，任务结束即清除（不持久化）
- **示例**：用户对 `file_write` 选择"全部授权"后，本任务内所有后续的文件写入操作自动执行，但 `send_email` 仍然会询问

## 5. 子智能体创建：自动执行 + 对话告知

### 5.1 废弃内容

以下 P2-4 实现的组件 **废弃**：

| 组件 | 文件 | 废弃原因 |
| ------ | ------ | --------- |
| 动态子智能体创建审批卡片 | `DynamicAgentCreationApprovalVm.cs` | 用户不需要审批内部调度细节 |
| 审批闸 | `DynamicAgentCreationGate.cs` | 不再需要阻塞等待用户决策 |
| 三按钮交互（批准/全部批准/拒绝）| `WorkflowDetailView.axaml` 审批区域 | 改为自动执行 |
| 审批事件 | `Events.cs` 中 `OnDynamicAgentCreationRequested/Resolved` | 不再需要 |

### 5.2 新的展示方式

子智能体的创建过程在时间线中以**普通文本行**展示：

```
正在创建数据采集专家，负责从公开财报中提取 AI 芯片数据...
正在创建市场分析师，负责横向对比竞品数据...

[步骤1] ████████░░ 80%  数据采集专家正在工作...
```

- 无审批，无阻塞
- 用户可以看到创建了什么、负责什么
- 但不需要（也不应该）干预创建过程

## 6. 执行结果展示

### 6.1 结果展示位置

执行完成后自动切回对话模式，结果作为对话流中的一个 **"富内容块"** 内联展示：

```
── 执行结果 ────────────────────────────────────────────────

全部 4 个步骤已完成，耗时 3 分 42 秒

┌─ 摘要 ──────────────────────────────────────────────────┐
│ 全球 AI 芯片市场规模预计 2026 年达 XX 亿美元，            │
│ NVIDIA 占据 78% 份额，AMD 和 Intel 分别为...             │
│ 主要风险集中在中美贸易政策和供应链瓶颈...                  │
│                                          [展开完整内容 ▼] │
└─────────────────────────────────────────────────────────┘

┌─ 产出文件 ──────────────────────────────────────────────┐
│  📄 reports/AI芯片市场分析报告.md       12 KB    [打开]  │
│  📊 data/processed/竞品数据对比.xlsx    48 KB    [打开]  │
│  📈 charts/市场趋势图.png              156 KB    [打开]  │
│                                                         │
│                          [在资源管理器中显示 📁]          │
└─────────────────────────────────────────────────────────┘

有什么需要调整的吗？
```

### 6.2 结果展示规则

| 元素 | 规则 |
| ------ | ------ |
| 摘要 | 默认折叠显示前几行，"展开完整内容" 按钮展开全文 |
| 文件列表 | 图标 + 相对路径 + 文件大小 + [打开] 按钮 |
| [打开] | 调用系统默认程序打开（.md→编辑器、.xlsx→Excel、.png→看图）|

## 9. ~~长任务执行：异步工具协议与轮询调度~~ ⚠️ 已废弃

> **状态：已废弃，不实现。**
>
> 本节设计的"异步工具协议"（`IAsyncToolProtocol` / `AsyncToolCheckpoint` / 轮询调度器）在软件架构层面不适配当前运行环境：
>
> - 当前架构中，**引擎从不直接调用工具**。工具由 LLM 在推理过程中自主调用（MCP/内置工具），引擎仅管理 LLM 调用的生命周期。"引擎 → 工具.StartAsync() → 轮询 QueryAsync()" 这条路径根本不存在。
> - 工具的异步特征（如返回 job_id 后轮询）是工具自身与 LLM 之间的协商，不需要也不应该由引擎层介入。
> - 强行在引擎层定义工具协议，反而无法适配语义上具备异步特征但未按固定接口实现的现有工具。
>
> **决策**：本节所有接口、类型、调度逻辑均不实现。原有决策表 D-12 / D-13 同步废弃。

---

### 9.1 问题背景（已废弃，仅保留历史记录）

部分任务步骤耗时极长（分钟到小时级），例如：

| 步骤 | 耗时 | 特征 |
| ------ | ------ | ------ |
| 视频剪辑 | 30 分钟 | 本地工具后台执行 |
| 文件上传（多文件并发） | 1-2 小时 | 网络 I/O，进度可查询 |
| 平台处理/发布 | 10-30 分钟 | 远程服务异步处理 |
| 发布验证 | 5-10 分钟 | 需等待平台就绪后检查 |

**核心约束**：LLM 不能长时间保持连接（超时会断开），但任务必须持续推进。

**解决方案**：LLM 只在需要决策时被唤醒（秒级），工具在后台执行（分钟-小时级），引擎用纯程序代码轮询工具状态来衔接两者。

### 9.2 工具的两种执行模式

| 模式 | 调用方式 | 适用场景 | 耗时 |
| ------ | ------ | ------ | ------ |
| **同步**（现有） | 调用 → 等待 → 返回结果 | web_search、file_read、文本生成 | 秒级 |
| **异步**（新增） | 调用 → 返回 handle → 后台执行 → 轮询/回调获取结果 | 视频剪辑、文件上传、远程任务 | 分钟 - 小时级 |

### 9.3 异步工具协议

异步工具需实现三个方法：

```csharp
/// <summary>
/// 异步工具协议。长耗时工具实现此接口，引擎通过轮询驱动执行。
/// </summary>
public interface IAsyncToolProtocol
{
    /// <summary>
    /// 启动后台任务。
    /// </summary>
    /// <param name="parameters">工具参数。</param>
    /// <param name="ct">取消令牌。</param>
    /// <returns>任务句柄 + 预估耗时。</returns>
    Task<AsyncToolStartResult> StartAsync(
        IDictionary<string, object> parameters, CancellationToken ct);

    /// <summary>
    /// 查询后台任务状态。引擎定时调用此方法轮询进度。
    /// 此方法必须是轻量级的（毫秒级返回），不能阻塞。
    /// </summary>
    /// <param name="taskHandle">StartAsync 返回的句柄。</param>
    /// <param name="ct">取消令牌。</param>
    /// <returns>当前状态 + 进度 + 结果（完成时）。</returns>
    Task<AsyncToolQueryResult> QueryAsync(string taskHandle, CancellationToken ct);

    /// <summary>
    /// 取消后台任务（用户暂停/取消时调用）。
    /// </summary>
    Task CancelAsync(string taskHandle, CancellationToken ct);
}

public sealed class AsyncToolStartResult
{
    /// <summary>后台任务唯一标识（引擎保存到 checkpoint 用于恢复）。</summary>
    public required string TaskHandle { get; init; }

    /// <summary>预估耗时（引擎据此计算轮询间隔）。</summary>
    public TimeSpan EstimatedDuration { get; init; }

    /// <summary>任务描述（UI 展示用，如"正在上传 video_1.mp4"）。</summary>
    public string DisplayText { get; init; } = string.Empty;
}

public sealed class AsyncToolQueryResult
{
    /// <summary>
    /// 任务状态。
    /// </summary>
    public AsyncToolStatus Status { get; init; }

    /// <summary>进度百分比 0-100（UI 进度条用）。</summary>
    public int Progress { get; init; }

    /// <summary>进度描述文本（UI 展示用，如"1.2GB / 2.8GB"）。</summary>
    public string ProgressText { get; init; } = string.Empty;

    /// <summary>预估剩余时间（可选，UI 展示"预计剩余 45 分钟"）。</summary>
    public TimeSpan? EstimatedRemaining { get; init; }

    /// <summary>任务结果（仅 Status=Completed 时有值）。</summary>
    public string? Result { get; init; }

    /// <summary>错误信息（仅 Status=Failed 时有值）。</summary>
    public string? ErrorMessage { get; init; }
}

public enum AsyncToolStatus
{
    /// <summary>已提交，尚未开始。</summary>
    Pending,

    /// <summary>执行中。</summary>
    Running,

    /// <summary>已完成。</summary>
    Completed,

    /// <summary>失败。</summary>
    Failed,

    /// <summary>已取消。</summary>
    Cancelled,
}
```

### 9.4 引擎轮询调度策略

异步工具启动后，引擎进入"等待-轮询"模式。轮询间隔根据 `EstimatedDuration` 智能调整：

| 预估耗时 | 轮询间隔 | 理由 |
| --------- | --------- | ------ |
| < 1 分钟 | 每 5 秒 | 快速任务，尽早感知完成 |
| 1 - 10 分钟 | 每 30 秒 | 平衡响应性和资源开销 |
| 10 - 60 分钟 | 每 60 秒 | 长任务不需要高频查询 |
| > 1 小时 | 每 5 分钟 | 超长任务（大文件上传）|

```csharp
/// <summary>
/// 根据预估耗时计算轮询间隔。
/// </summary>
public static TimeSpan ComputePollInterval(TimeSpan estimatedDuration)
{
    return estimatedDuration.TotalMinutes switch
    {
        < 1 => TimeSpan.FromSeconds(5),
        < 10 => TimeSpan.FromSeconds(30),
        < 60 => TimeSpan.FromSeconds(60),
        _ => TimeSpan.FromMinutes(5),
    };
}
```

轮询过程中的关键约束：

| 约束 | 说明 |
| ------ | ------ |
| 不持有 LLM 连接 | 轮询是纯程序代码，不调用 LLM |
| UI 实时更新 | 每次查询结果推送到时间线（进度百分比 + 描述文本）|
| 用户可暂停/取消 | 引擎调用 `CancelAsync` 通知工具停止 |
| 进度持久化 | 每次查询后更新 checkpoint（崩溃后可恢复）|
| 不阻塞其他步骤 | 异步工具等待期间，无依赖的其他步骤正常调度执行 |

### 9.5 LLM 参与点：只在决策时唤醒

一个完整的长任务（2-3 小时），LLM 实际只被唤醒几次：

```
时间轴（视频制作发布任务示例）：

0:00   LLM唤醒(3s) → 决策："调用 video_edit 工具" → 释放连接
0:00 - 0:30  引擎轮询 video_edit 进度（纯程序，无 LLM）
0:30   工具完成 → 引擎调度 → 发现步骤2/3/4 就绪 → 同时启动

0:30   LLM唤醒(3s) → 决策步骤2："调用 upload_A" → 释放
0:30   LLM唤醒(3s) → 决策步骤3："调用 upload_B" → 释放
0:30   LLM唤醒(3s) → 决策步骤4："生成封面图" → 释放

0:32   步骤4 完成 ✓（同步工具，秒级）
0:30 - 1:45  引擎轮询 upload_A / upload_B（纯程序，无 LLM）

1:15   步骤3 完成 ✓（upload_B 先完成）
1:45   步骤2 完成 ✓（upload_A 完成）

1:45   LLM唤醒(3s) → 决策步骤5："验证发布状态" → 释放
1:55   步骤5 完成 ✓
1:55   LLM唤醒(3s) → 决策步骤6："汇总结果" → 释放
1:58   步骤6 完成 ✓ → 任务结束 → 切回对话模式

─────────────────────────────────────
LLM 唤醒次数：7 次
LLM 总连接时间：约 21 秒
任务总耗时：约 2 小时
```

### 9.6 并行调度与异步工具的配合

异步工具等待期间，步骤状态为 `Running`。调度器只看 `dependsOn` 中的步骤是否 `Completed`，**不关心步骤是同步还是异步完成的**。

#### 场景：多文件并发上传 + 无依赖步骤提前执行

```
执行计划：
  步骤1：视频剪辑                    dependsOn: []
  步骤2：上传到平台A（3个文件并发）   dependsOn: [步骤1]
  步骤3：上传到平台B（2个文件并发）   dependsOn: [步骤1]
  步骤4：生成封面图                  dependsOn: [步骤1]     ← 不依赖上传
  步骤5：发布验证                    dependsOn: [步骤2, 步骤3]
  步骤6：结果总结                    dependsOn: [步骤4, 步骤5]
```

调度行为：

```
步骤1 完成 → 调度器检查就绪步骤：
  步骤2 dependsOn[步骤1] ✓ → 就绪，启动
  步骤3 dependsOn[步骤1] ✓ → 就绪，启动
  步骤4 dependsOn[步骤1] ✓ → 就绪，启动

步骤2/3 在后台上传（异步工具），步骤4 同时执行（同步工具）→ 步骤4 先完成

步骤4 完成 → 调度器检查就绪步骤：
  步骤5 dependsOn[步骤2, 步骤3] → 步骤2 Running, 步骤3 Running → 不就绪
  步骤6 dependsOn[步骤4, 步骤5] → 步骤5 还没开始 → 不就绪

步骤2 完成 → 调度器检查：步骤5 dependsOn 中步骤3 仍 Running → 不就绪
步骤3 完成 → 调度器检查：步骤5 dependsOn 全部 Completed → 就绪，启动

步骤5 完成 → 步骤6 dependsOn[步骤4 ✓, 步骤5 ✓] → 就绪，启动
```

#### 步骤内部多文件并发

步骤2"上传到平台A"本身包含多个并发子任务：

```
步骤2：上传到平台A
  executionMode: "parallel"
  subTasks:
    - subTask-a: 上传 video_1.mp4 (800MB)  → 异步工具 → 后台上传
    - subTask-b: 上传 video_2.mp4 (1.2GB)  → 异步工具 → 后台上传
    - subTask-c: 上传 video_3.mp4 (600MB)  → 异步工具 → 后台上传

  引擎同时启动 3 个异步工具实例
  各自独立轮询进度
  全部 subTask 完成后 → 步骤2 标记 Completed
```

UI 时间线展示：

```
[步骤1] ██████████ 完成 ✓  视频剪辑完成
[步骤2] ████████░░ 72%     上传到平台A
          ├── video_1.mp4  ████████░░ 89%   预计剩余 2 分钟
          ├── video_2.mp4  ██████░░░░ 61%   预计剩余 12 分钟
          └── video_3.mp4  ██████████ 完成 ✓
[步骤3] ██████░░░░ 55%     上传到平台B
[步骤4] ██████████ 完成 ✓  封面图已生成
[步骤5] ⏳ 等待中            依赖步骤 2, 3
[步骤6] ⏳ 等待中            依赖步骤 4, 5
```

### 9.7 断点恢复：异步工具状态持久化

进程崩溃/关闭时，异步工具的运行状态必须持久化到 checkpoint，重启后可恢复轮询。

```csharp
/// <summary>
/// ExecutionCheckpoint 扩展：记录异步工具状态。
/// </summary>
public sealed class AsyncToolCheckpoint
{
    /// <summary>步骤 ID。</summary>
    public string StepId { get; set; } = string.Empty;

    /// <summary>子任务 ID（如果是 parallel 步骤内的子任务）。</summary>
    public string? SubTaskId { get; set; }

    /// <summary>工具名称。</summary>
    public string ToolName { get; set; } = string.Empty;

    /// <summary>后台任务句柄（恢复轮询的关键）。</summary>
    public string TaskHandle { get; set; } = string.Empty;

    /// <summary>上次查询到的进度。</summary>
    public int LastProgress { get; set; }

    /// <summary>上次查询时间（用于计算下次轮询时间）。</summary>
    public DateTimeOffset LastQueryAt { get; set; }

    /// <summary>轮询间隔。</summary>
    public TimeSpan PollInterval { get; set; }
}
```

恢复逻辑：

```
进程重启 → 读取 checkpoint → 发现有 AsyncToolCheckpoint 记录
  ↓
对每个异步工具记录：
  1. 调用 tool.QueryAsync(taskHandle) 获取当前状态
  2. 状态为 Running → 恢复轮询循环
  3. 状态为 Completed → 读取结果，标记步骤完成，触发调度器
  4. 状态为 Failed → 进入重试策略
  5. 状态为 Cancelled/不可达 → 标记步骤为 Pending（重新执行）
```

### 9.8 与 GlobalLlmThrottle 的关系

异步工具的轮询过程**不占用 LLM 信号量**（因为不调用 LLM）。只有在以下时刻才需要获取 LLM 信号量：

- 步骤开始时，唤醒 LLM 做执行决策
- 异步工具完成后，如果需要 LLM 做结果分析/下一步决策

```
GlobalLlmThrottle 管辖范围：
  ✅ 主智能体决策调用
  ✅ 子智能体执行调用
  ✅ 需求分析/计划制定/验证
  ❌ 异步工具轮询（纯程序代码，不占 LLM 槽位）
  ❌ 同步工具执行（如 web_search 本身不调 LLM）
```
