# 03 - 通信协议

## 总体原则

本方案不新增专用 STT/TTS/KWS WebSocket 协议。

语音插件通信复用现有插件体系：

| 通信内容 | 复用机制 |
|---|---|
| 生命周期 | Process 插件协议 `get_info` / `init` / `destroy` |
| 控制命令 | Process 插件协议 `invoke` |
| 长期/流式事件 | 插件通过 `init` 中已有 `wsPort` 回连宿主，或复用现有插件事件上报通道 |
| 前端广播 | 继续由 EventHub → `Netor.Cortana.Networks` 广播，事件名尽量不变 |

## 为什么不新增语音专用协议

原方案定义了：

- `ws://localhost:12858/` STT 服务协议；
- `ws://localhost:12859/` TTS 服务协议。

但现有 Process 插件协议已经提供：

- 插件进程启动与销毁；
- `plugin.json` 描述；
- `get_info` 工具清单；
- `init` 运行时配置；
- `invoke` 命令调用；
- `wsPort` 回连宿主能力。

继续新增语音专用端口和协议会带来重复的生命周期管理、重连、心跳、端口占用处理，不符合“能不添加新协议就不添加新协议”的原则。

## Process 插件协议复用边界

现有 `invoke` 是请求-响应模型，并且存在调用超时。因此：

> `invoke` 只用于短命令：启动、停止、配置、查询状态。  
> `invoke` 不承载长期监听、持续录音、流式 partial、字幕流、播放完成事件。

长期事件由插件主动上报给宿主，再由主程序映射到 EventHub。

## 插件生命周期

```text
宿主启动插件进程
  │
  ├─ get_info：读取插件元数据和工具清单
  ├─ init：传入 dataDirectory / workspaceDirectory / pluginDirectory / wsPort
  ├─ invoke：按需调用 start / stop / configure 等工具
  ├─ 插件通过 wsPort 回连宿主，上报语音事件
  └─ destroy：宿主关闭或卸载插件时释放资源
```

## 能力声明

推荐在插件元数据中增加 `capabilities` 字段：

```json
{
  "id": "cortana.voice.tts.melo",
  "name": "Melo TTS 语音合成",
  "version": "1.0.0",
  "runtime": "process",
  "command": "Cortana.Voice.Tts.Melo.exe",
  "tags": ["voice", "tts"],
  "capabilities": ["voice.tts"]
}
```

短期如果不修改插件 schema，可使用 `tags` 做能力发现：

| 能力 | tags |
|---|---|
| KWS | `voice`, `kws` |
| STT | `voice`, `stt` |
| TTS | `voice`, `tts` |

长期建议使用 `capabilities`，避免 `tags` 语义不够严格。

## KWS 插件工具约定

| 工具名 | 参数 | 返回 | 说明 |
|---|---|---|---|
| `voice_kws_start` | 可选配置 JSON | 成功/失败 | 开始监听唤醒词，立即返回 |
| `voice_kws_stop` | 无 | 成功/失败 | 停止监听 |
| `voice_kws_configure` | 阈值、关键词、模型配置 | 成功/失败 | 更新配置，通常下次 start 生效 |
| `voice_kws_status` | 无 | 状态 JSON | 查询当前状态 |

KWS 事件：

| 插件事件 | EventHub 映射 |
|---|---|
| `voice.kws.detected` | `Events.OnWakeWordDetected` |
| `voice.kws.started` | 可选日志/状态 |
| `voice.kws.stopped` | 可选日志/状态 |
| `voice.kws.error` | 日志 + 可选错误状态 |

## STT 插件工具约定

| 工具名 | 参数 | 返回 | 说明 |
|---|---|---|---|
| `voice_stt_start` | 可选配置 JSON | 成功/失败 | 开始一次识别，会立即返回 |
| `voice_stt_cancel` | 无 | 成功/失败 | 取消当前识别 |
| `voice_stt_configure` | 识别阈值、超时、模型配置 | 成功/失败 | 更新识别配置 |
| `voice_stt_status` | 无 | 状态 JSON | 查询当前状态 |

STT 事件：

| 插件事件 | EventHub 映射 | 说明 |
|---|---|---|
| `voice.stt.partial` | `Events.OnSttPartial` | 中间识别文本 |
| `voice.stt.final` | `Events.OnSttFinal` | 最终识别文本 |
| `voice.stt.stopped` | `Events.OnSttStopped` | 当前识别结束 |
| `voice.stt.error` | 日志 + 可选错误状态 | 识别异常 |

## TTS 插件工具约定

| 工具名 | 参数 | 返回 | 说明 |
|---|---|---|---|
| `voice_tts_speak` | 完整文本 | 成功/失败 | 合成并播放完整文本，建议立即返回，完成靠事件通知 |
| `voice_tts_enqueue` | 一句话文本 | 成功/失败 | 流式入队 |
| `voice_tts_finish` | 无 | 成功/失败 | 标记文本输入结束 |
| `voice_tts_stop` | 无 | 成功/失败 | 立即停止合成和播放 |
| `voice_tts_greeting_play` | 可选问候语 | 成功/失败 | 停止旧播放并播放问候语 |
| `voice_tts_greeting_regen` | 新问候语 | 成功/失败 | 重新生成问候语缓存 |
| `voice_tts_configure` | 语速、音色、模型配置 | 成功/失败 | 更新合成配置 |
| `voice_tts_status` | 无 | 状态 JSON | 查询当前状态 |

TTS 事件：

| 插件事件 | EventHub 映射 | 说明 |
|---|---|---|
| `voice.tts.started` | `Events.OnTtsStarted` | 开始播放 |
| `voice.tts.subtitle` | `Events.OnTtsSubtitle` | 当前播放句子 |
| `voice.tts.completed` | `Events.OnTtsCompleted` | 播放完成 |
| `voice.tts.error` | 日志 + 可选错误状态 | 合成/播放异常 |

## 事件载荷建议

事件载荷保持简单，优先复用已有 `VoiceTextArgs` / `VoiceSignalArgs` 所需信息。

```json
{
  "type": "voice.stt.partial",
  "text": "正在识别的内容",
  "sourcePluginId": "cortana.voice.stt.sherpa",
  "sessionId": "可选轮次 ID"
}
```

```json
{
  "type": "voice.tts.subtitle",
  "text": "当前播放的句子",
  "sourcePluginId": "cortana.voice.tts.melo",
  "sessionId": "可选轮次 ID"
}
```

## AOT 要求

语音插件仍需遵守项目 AOT 约束：

- JSON 序列化使用 `System.Text.Json` Source Generator；
- 不依赖运行时反射发现复杂类型；
- Native 音频与模型调用需要明确处理资源释放；
- stdout 只能输出 Process 插件协议响应，日志必须走 stderr 或日志文件。

## 兼容性要求

- `Netor.Cortana.Networks` 当前订阅的语音 EventHub 事件尽量不变；
- 前端收到的 `stt_partial` / `stt_final` / `tts_subtitle` 等广播尽量不变；
- 旧 `OnTtsEnqueue` / `OnTtsFinish` 可暂时保留，但插件化后主流程优先直接调用 TTS 插件工具。
