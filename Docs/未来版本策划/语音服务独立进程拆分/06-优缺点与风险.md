# 06 - 优缺点与风险分析

## 优势

### 1. 主程序体积显著缩小

KWS/STT/TTS 模型都随插件发布，主程序发布包不再内置 `sherpa_models`，整体体积可大幅下降。

### 2. 语音能力可独立安装和更新

用户可以只安装需要的语音插件：

- 只要语音输入：安装 KWS + STT；
- 只要朗读：安装 TTS；
- 全部不用：不安装任何语音插件。

更新模型时只替换对应插件包，不需要重新发布主程序。

### 3. 支持多模型并存

同一类能力可以安装多个插件：

- `madorin.voice.stt.sherpa`；
- `madorin.voice.stt.whispercpp`；
- `madorin.voice.tts.melo`；
- `madorin.voice.tts.azure`。

用户可在设置中选择当前使用的插件。

### 4. 故障隔离

语音模型、Native Runtime、音频设备等异常被限制在插件进程内。插件崩溃不会直接带崩主程序。

### 5. 技术栈解耦

Process 插件可以用 C#、C++、Rust、Python、Go 等实现，只要遵守现有插件协议。

### 6. 复用现有协议和插件基础设施

不新增 STT/TTS 专用 WebSocket 协议，不新增固定端口，减少重复维护成本。

---

## 劣势

### 1. 插件能力发现复杂度增加

主程序需要从普通工具插件中识别语音能力插件，需要新增 `VoiceCapabilityRegistry` 或等价机制。

### 2. 事件上报链路需要规范

Process 插件 `invoke` 不适合承载流式事件，因此需要明确插件如何通过现有 `wsPort` 或事件通道上报 KWS/STT/TTS 事件。

### 3. 调试链路变长

语音链路会跨主程序、插件进程、宿主 WebSocket、EventHub、前端广播，排查问题时需要更多日志关联。

### 4. 插件包管理要求更高

模型随插件发布后，需要处理：

- 插件包体积大；
- 模型文件缺失；
- 插件版本和模型版本不一致；
- 插件更新时占用文件无法覆盖。

### 5. KWS 插件化风险高于原方案

KWS 是长期后台监听能力，对启动时机、音频设备占用和唤醒延迟更敏感。应放在 STT/TTS 稳定后再拆。

---

## 风险矩阵

| 风险项 | 可能性 | 影响 | 严重度 | 缓解措施 |
|---|---|---|---|---|
| 事件通道设计不清导致 partial/subtitle 丢失 | 中 | 高 | 🔴 严重 | 明确插件事件上报格式，主程序统一映射 EventHub |
| `invoke` 被误用为长任务导致 30 秒超时 | 中 | 高 | 🔴 严重 | 文档明确：invoke 只做启动/停止/配置，流式结果走事件 |
| 语音插件崩溃后循环重启 | 中 | 高 | 🔴 严重 | 增加重启限流，超限标记插件不可用 |
| 插件能力识别错误 | 中 | 中 | 🟡 中等 | 长期引入 `capabilities` 字段，短期 tags 只作为兼容 |
| 插件缺失导致语音不可用 | 高 | 中 | 🟡 中等 | 设置页明确提示安装对应能力插件，主程序保持纯文字可用 |
| KWS 插件化导致唤醒延迟变高 | 中 | 中 | 🟡 中等 | KWS 最后拆分，保留回退方案并测试延迟 |
| 模型包过大导致下载/更新体验差 | 高 | 中 | 🟡 中等 | 语音插件单独发布，支持按需下载 |
| 插件更新时文件被占用 | 中 | 中 | 🟡 中等 | 更新前 destroy 插件并等待进程退出，必要时下次启动替换 |
| 主程序移除模型后回退困难 | 中 | 中 | 🟡 中等 | 在一个版本周期内保留旧内置语音分支或兼容开关 |

---

## 回退方案

### 阶段性回退

在 TTS/STT/KWS 分阶段拆分时，每一步都保留旧实现：

1. TTS 插件异常：回退旧 `TextToSpeechService`；
2. STT 插件异常：回退旧 `SpeechRecognitionService` 中的识别部分；
3. KWS 插件异常：回退旧 `WakeWordService`。

### 最终回退

如果插件化语音链路出现严重问题：

1. 设置中禁用对应语音能力插件；
2. 主程序进入纯文字模式；
3. 下一版本重新安装稳定语音插件；
4. 在完全验证前，不删除旧语音实现代码。

---

## 不做的事情

| 项 | 原因 |
|---|---|
| 新增固定 STT/TTS WebSocket 端口 | 现有插件协议和 `wsPort` 已够用 |
| 新增 gRPC 协议 | 引入额外依赖和 AOT 复杂度 |
| 主程序识别具体模型格式 | 模型细节应封装在插件内部 |
| 一次性拆完 KWS/STT/TTS | 风险过高，按 TTS→STT→KWS 逐步迁移 |
| 强制所有语音插件使用 C# | Process 插件应保持语言中立 |

---

## 成功判定

插件化方案完成后应满足：

- 主程序不再默认发布 `sherpa_models/KWS`、`sherpa_models/STT`、`sherpa_models/TTS`；
- 未安装语音插件时，主程序可正常以纯文字模式运行；
- 安装 TTS 插件后可朗读 AI 回复；
- 安装 STT 插件后可语音输入；
- 安装 KWS 插件后可关键词唤醒；
- 更换 STT/TTS/KWS 插件不需要重新编译主程序；
- 前端 WebSocket 语音事件保持兼容。
