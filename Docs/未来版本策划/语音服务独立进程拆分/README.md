# 语音服务插件化拆分方案

> 创建日期：2026-04-20  
> 调整日期：2026-05-06  
> 状态：方案调整为“语音能力插件化”，等待按新计划执行

## 背景

当前语音模块（KWS/STT/TTS）作为类库直接集成在主进程中，主程序需要携带语音模型和 Sherpa-ONNX 等运行时依赖，导致：

- 主程序发布包体积大；
- 更换 STT/TTS/KWS 模型需要跟随主程序一起发布；
- 语音引擎崩溃或 Native 依赖异常会影响主进程稳定性；
- 无法按需安装、更新或替换单个语音能力。

原方案计划把 STT/TTS 拆为固定的 `Madorin.Stt.exe` / `Madorin.Tts.exe` 子进程，KWS 留在主进程。
经重新评估，新的目标是：

> **KWS、STT、TTS 全部独立为 Process 子插件，模型跟随插件包发布；主程序只保留语音编排和插件能力适配层。**

这样主程序不再绑定具体语音模型或语音引擎，整体发布体积可显著缩小，语音能力可以单独热插拔或更新。

## 核心决策

| 组件 | 新决策 | 理由 |
|---|---|---|
| KWS 关键词唤醒 | **Process 插件** `voice.kws` | 主程序彻底移除 KWS 模型依赖，支持替换唤醒模型 |
| STT 语音转文字 | **Process 插件** `voice.stt` | 不同识别模型可独立安装、更新、替换 |
| TTS 文字转语音 | **Process 插件** `voice.tts` | 不同合成模型可独立安装、更新、替换 |
| 主程序 | **只保留编排层** | 负责唤醒→识别→AI→合成的流程，不关心具体模型 |
| 通信方式 | **复用现有插件协议** | 能不新增协议就不新增协议，避免重复维护 |
| 模型分发 | **跟随插件包** | 主程序发布包不再内置 `sherpa_models` |

## 设计原则

1. **主程序不关心具体模型**  
   主程序只识别 `voice.kws` / `voice.stt` / `voice.tts` 能力，不知道 Sherpa、Whisper、Melo、VITS 等具体实现。

2. **插件负责模型与引擎**  
   模型目录、ONNX Runtime、音频设备、缓存、模型版本都由插件内部管理。

3. **不新增专用语音协议**  
   控制命令复用现有 Process 插件协议，事件尽量复用现有宿主 WebSocket / EventHub 链路。

4. **前端事件保持兼容**  
   `OnSttPartial`、`OnTtsSubtitle` 等 EventHub 事件名尽量不变，Networks 层和前端 WebSocket 广播保持透明。

5. **优先从低风险能力开始拆分**  
   先 TTS，后 STT，最后 KWS。
