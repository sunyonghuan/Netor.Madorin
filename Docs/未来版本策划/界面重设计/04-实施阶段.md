# 04 - 实施阶段

> 本文档把界面重设计拆分到 5 个独立可验证的 commit，每个 commit 都要 Debug build + AOT publish + 推 github + netor 双远程 + SHA 校验。
> 上级文档：[README.md](./README.md)

---

## 0. 整体路线图

```
C1 ── 设计 Token 收口（纯样式，0 业务）
   ↓
C2 ── 顶栏 3 个 tab + 未保存确认对话框（核心架构）
   ↓
C3 ── LeftPanel + 文件目录改造（左侧重组）
   ↓
C4 ── 工作台拆分 + 任务列表挪左侧（数据流改造）
   ↓
C5 ── 右侧 Popup 干掉 + 顶栏「最近 ▼」+ ChatHistoryPanel 改造（最终收尾）
```

依赖关系：严格 C1 → C2 → C3 → C4 → C5，不允许并行。

每个 commit 之间保留 ≥ 4 小时间隔（允许手动跑回归测试）。

## 6. 跨 commit 共性事项

### 6.1 每个 commit 必走流程

1. 创建分支或在 master 直接开发（建议直接 master，与阶段 6 一致）
2. 编辑文件
3. `dotnet build -c Debug` 0 错误
4. `dotnet publish -c Release -r win-x64` AOT 72MB ± 2MB
5. 手动启动 Madorin.exe，跑当 commit 的验证清单
6. `git add` + 详细 commit message（按 §1.5 模板）
7. `git push github master`
8. `git push netor master`
9. SHA 三方校验（本地 / github / netor 都一致）
10. 更新本文档 §0 路线图标完成

### 6.2 commit message 模板（每个 commit 都遵循）

```
{type}({scope}): {summary}

{背景说明，2-3 句}

详见 docs/未来版本策划/界面重设计/04-实施阶段.md §N。

改动文件（N 文件，+X -Y）：
{逐文件说明改动}

决策点：
- DT-X：{决策标题} = {选项}（理由）

验证：
- Debug build：0 错误
- AOT publish：72MB ± 2MB
- 视觉对比：{无回归 / 改动说明}

未完成项：
- {推到下个 commit 的事项}
```

### 6.3 应急处理

| 情况 | 处理 |
|---|---|
| C2 弹对话框崩溃 | git revert C2 + 修复后重 push |
| C4 拆分后某事件丢失订阅 | 修复 + 在原 commit 上 `git commit --amend`（**仅 push 前**），已 push 则补 hotfix commit |
| C5 硬编码迁移破坏视觉 | C5 拆为 C5a（功能改动）+ C5b（色值迁移） |
| 任意 commit AOT 失败 | 立即 revert，先在 Debug 验证再 publish |

### 6.4 跨 commit 共享代码

- `MainWindowVm` 在 C2 引入后，C3-C5 都会依赖它，不要在子 commit 重新定义
- `DesignTokens` 在 C1 引入后，C2-C5 优先用 `{StaticResource}`，不再硬编码
