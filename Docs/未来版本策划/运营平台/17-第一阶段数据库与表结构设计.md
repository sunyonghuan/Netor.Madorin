# 17. 第一阶段数据库与表结构设计

## 1. 设计目标

第一阶段采用 SQLite 起步，但数据库模型必须直接对齐 `Netor.Database` 的实体基类，后续可平滑切换到 SQL Server。

核心目标：

- 复用 Netor 账号、管理员、订单、交易、钱包体系。
- 个人用户阶段只做最基础功能。
- 插件、技能、智能体、解决方案统一抽象为资源表 `Asset`。
- 资源包文件第一阶段直接存本地文件系统。
- 不引入 Repository 层，业务直接使用 `PlatformDbContext`。
- 后台页面尽量与 `Netor.Operates.Entitys` 的字段和关系保持一致，方便复用 Layui 页面。

## 2. 数据库选型

### 2.1 第一阶段：SQLite

- 部署简单。
- 适合 MVP 和单机验证。
- 不要求单独数据库服务。

### 2.2 后续阶段：SQL Server

当用户量、订单量和审计需求上升后，再切换 SQL Server。

### 2.3 ORM：EF Core

使用 EF Core 管理模型、关系和迁移。

## 3. 关键兼容原则

- 复用 `Netor.Database.Abstractions` 的 `Base` 体系。
- `Base` 主键是 `ID : string`，不是 `Guid Id`。
- 所有外键字段优先使用 `string` / `string?` 对齐主键。
- 时间统一使用 UTC。
- 金额统一使用 `decimal(18,2)`。
- 枚举统一用整数存储。

## 4. 实体结构总览

### 4.1 账户体系

- `Account : AccountBase`
- `Manager : AccountBase`
- `AccountRole : RoleBase`
- `ManagerRole : RoleBase`
- `AccountProperty : AccountPropertyBase<Account>`
- `ManagerProperty : AccountPropertyBase<Manager>`
- `AccountRolePair : AccountRolePairBase<Account, AccountRole>`
- `AccountWallet : WalletBase<Account>`

### 4.2 订单与交易体系

- `Order : OrderBase<Account>`
- `Transaction : TransactionBase<Account>`

### 4.3 资源市场体系

- `Asset`
- `AssetVersion`
- `Category`
- `PricingPlan`
- `Subscription`
- `DownloadRecord`

## 5. 表设计要点

### 5.1 `Assets`

资源主表，统一承载插件、技能、智能体和解决方案。

字段重点：

- `ID`：字符串主键
- `Name`
- `Slug`
- `DeveloperName`
- `ShortDescription`
- `Description`
- `Tags`
- `IconUrl`
- `CoverUrl`
- `Type`
- `Status`
- `CategoryId`
- `CurrentVersionId`
- `IsFeatured`
- `DownloadCount`
- `PublishedAtUtc`

索引：

- `Slug` 唯一
- `(AccountId, TimeStamp)`
- `(AssetId, TimeStamp)`
### 5.2 `AssetVersions`

- `AssetId : string`
- `VersionName`
- `ReleaseNotes`
- `ManifestJson`
- `PackageHash`
- `PackageSize`
- `FilePath`

索引：

- `(AssetId, VersionName)` 唯一

### 5.3 `Categories`

- `ID`
- `Name`
- `Slug`
- `Description`
- `SortOrder`
- `IsVisible`

索引：

- `Slug` 唯一

### 5.4 `PricingPlans`

- `AssetId : string`
- `Name`
- `PlanType`
- `Price`
- `Currency`
- `DurationDays`
- `IsActive`

索引：

- `(AssetId, PlanType)`

### 5.5 `Subscriptions`

- `AccountId : string`
- `AssetId : string`
- `PricingPlanId : string`
- `OrderId : string?`
- `Status`
- `StartedAtUtc`
- `ExpiresAtUtc`
- `CanceledAtUtc`

索引：

- `(AccountId, Status)`
- `(AccountId, AssetId, Status)`

### 5.6 `DownloadRecords`

- `AccountId : string`
- `AssetId : string`
- `AssetVersionId : string`
- `SubscriptionId : string?`
- `IpAddress`
- `UserAgent`

索引：

- `(AccountId, CreatedTime)`
- `(AssetId, CreatedTime)`

## 6. 文件系统存储

第一阶段文件直接落地本地磁盘，数据库只保存相对路径。

建议目录：

```text
Data/
  packages/
    plugins/
    skills/
    agents/
    solutions/
```

## 7. 当前实现状态

- `PlatformDbContext` 已继承 `SqlLiteDbContextBase<Account, AccountRole, Manager, ManagerRole, AccountProperty, AccountRolePair, Order, Transaction, AccountWallet>`。
- 平台资源表已按 Netor 账户体系重新建模。
- 外键已统一调整为字符串主键对齐。
- 平台初始化数据已包含 demo 用户、管理员、分类、资源和系统设置。

## 8. 结论

第一阶段的数据库设计不再是独立的自造模型，而是以 Netor 基类为核心，兼顾市场资源表和后台复用能力。这样后续页面、权限、订单和交易模块才能真正复用现有脚手架。

第一阶段数据库以个人用户和资产订阅为核心。表结构应围绕：

- 用户。
- 资产。
- 版本。
- 文件。
- 价格。
- 订单。
- 订阅。
- 下载。
- 管理员。

使用 EF Core + SQLite 快速起步，同时通过字符串主键、UTC 时间、decimal 金额、Provider 配置切换和迁移约束，为后续迁移 SQL Server 留出空间。
