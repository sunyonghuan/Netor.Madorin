## 2. 主智能体构建：AIAgentFactory.Build / BuildWithSubAgents / BuildSubAgent

### 2.1 三个构建方法对照

| 路径 | 用途 | 入参 |
| --- | --- | --- |
| `Build(agent, provider, model)` | 普通单 Agent 对话 | 一个 AgentEntity |
| `BuildWithSubAgents(mainAgent, mainProvider, mainModel, mentions, providerService, modelService)` | 主 Agent + @ 提及的子 Agent 工具模式 | 主 Agent + List&lt;AgentMention&gt; |
| `BuildSubAgent(agent, provider, model, ...)` | 构建被 @ 的轻量子 Agent | 单个 AgentEntity，**不带历史/记忆/技能** |

### 2.2 元数据设置差异

| 路径 | Name | Id | Description | Instructions |
| --- | --- | --- | --- | --- |
| `Build()` | ✅ `agent.Name` | ❌ 未设置 | ❌ 未设置 | ✅ ChatOptions.Instructions |
| `BuildWithSubAgents()` | ✅ `mainAgent.Name` | ✅ `mainAgent.Id` | ❌ 未设置 | ✅ 同上 |
| `BuildSubAgent()` | ✅ `agent.Name` | ❌ 未设置 | ✅ `agent.Description` | ✅ 同上 |

**结论**：三条路径在 Id / Description 设置上不一致。第一阶段必须先**统一三处都显式设置 `Id = agent.Id` / `Name = agent.Name` / `Description = agent.Description`**，否则：

- Workflow 内 `AIAgentHostExecutor.IdFor(agent) = "{Name}_{Id}"`，`Id` 缺失会让 Executor 名变成纯 `Name_<空>`；
- Magentic 内部 `team.FirstOrDefault(a => a.Name == nextSpeaker)` 依赖稳定 Name；
- 阶段 2 Coordinator 想根据 `agent.Description` 自动选候选时拿不到值。

参见 SDK 源码：`AIAgentExtensions.GetDescriptiveId()`（`01-SDK源码研究笔记.md` §11）。

### 2.3 BuildWithSubAgents 当前流程

```text
主智能体
  ↓
判断主模型是否支持 FunctionCall（InteractionCapabilities.FunctionCall）
  ├── 不支持 → 跳过所有工具/插件/MCP/子智能体注入，仅 logger.LogInformation，无任何用户提示
  └── 支持
        ↓
      将被 @ 提及的子智能体构建为轻量 AIAgent（BuildSubAgent）
        ↓
      subAgent.AsAIFunction(options: { Name, Description })
        ↓
      SubAgentContextProvider 注入为 AIContext.Tools
        ↓
      主智能体通过 function call 调用子智能体（query: string 单参数）
```

**关键边界**：

1. 主模型不支持 FunctionCall 时所有 @ 子 Agent 被**静默丢弃**。多智能体编排所有阶段的前置条件都是"主模型支持 FunctionCall"，否则需要在失败降级路径给用户提示。
2. `AsAIFunction` 默认产生**单 `query: string` 参数**，附件传递必须走自定义委托（详见 SDK 笔记 §7）。
3. 项目当前 BuildSubAgent 不传 session，符合 SDK 强约束"session 跨 Agent 不可复用"（SDK 笔记 §13）。

## 3. AIContextProvider 体系

| 名称 | 作用 | 启用条件 |
| --- | --- | --- |
| `ProjectSettingsProvider` | 注入项目设置上下文 | 总是 |
| `LongMemoryContextProvider` | 注入长期记忆 | 主 Agent，依赖 `agentid/sessionid/turnid/usermessageid/agentname/sessiontitle/traceid/currenttask` |
| `AgentSkillsProvider` | 加载用户技能目录 + 工作区技能目录 | 总是 |
| `SubAgentContextProvider` | 注入子 Agent 工具函数 | `BuildWithSubAgents` 路径 |

**重要**：`AgentSkillsProvider` 是 **`Microsoft.Agents.AI` 1.x SDK 自带的 sealed 类**，不是项目自研。详见 SDK 笔记 §9。

### 3.1 全局插件 Scope 过滤

全局插件需同时满足两个条件才会注入：

1. 插件的安装范围是 `PluginInstallScope.Global`（即来自全局插件目录）。
2. 该插件 ID 出现在 `GlobalPluginService.GetEnabledPluginIds()` 启用列表中。

仅启用未安装到全局目录的插件不会生效。

智能体绑定插件通过 `AgentEntity.EnabledPluginIds` 控制；同一插件已通过全局通道注入时会按 `injectedPluginIds` 自动去重。

### 3.2 子 Agent 不会注入的能力

`BuildSubAgent` 的注释明确：

```text
构建子智能体（轻量）：仅携带 instructions + plugins + MCP，不带历史/memory/skills。
```

不会注入：

- `AgentSkillsProvider` 技能目录
- `ProjectSettingsProvider` 项目设置上下文
- `LongMemoryContextProvider` 长期记忆上下文
- `ChatHistoryDataProvider` 历史记录 Provider
- `TokenTrackingChatClient` token 统计包装器（注意：意味着子 Agent 调用的 token 不进入主 Agent 工厂级状态）

## 4. 历史与记忆约束

核心文件：

- `Src/Netor.Madorin.AI/Providers/ChatHistoryDataProvider.cs`
- `Src/Netor.Madorin.AI/Memory/LongMemoryContextProvider.cs`
- `Src/Netor.Madorin.AI/Providers/TokenTrackingChatClient.cs`

### 4.1 Session.StateBag 键全集

`AiChatHostedService` 在不同时机向 `Session.StateBag` 写入以下键：

| 键名 | 写入位置 | 用途 |
| --- | --- | --- |
| `sessionid` | `LoadOrCreateSessionAsync` / `NewSessionAsync` | 会话 ID，`ChatHistoryDataProvider`、`LongMemoryContextProvider` 都依赖它 |
| `providerid` / `providername` | `UpdateSessionSelectionState` | 当前 AI 提供商 ID / 名称 |
| `agentid` / `agentname` | `UpdateSessionSelectionState` | 当前主智能体 ID / 名称，`LongMemoryContextProvider` 依赖 |
| `modelid` / `modeldbid` | `UpdateSessionSelectionState` | 模型显示名（OpenAI model id） / 数据库主键 |
| `sessiontitle` | `AddOrUpdateSessionAsync` 内部 | 会话标题，`LongMemoryContextProvider` 依赖 |
| `turnid` / `traceid` | `SendMessageAsync` 每轮开头 | 单轮编号 / 跨服务追踪 ID |
| `usermessageid` | `SendMessageAsync` 每轮开头 | 用户消息预生成 ID |
| `currenttask` | `SendMessageAsync` 每轮开头 | 当前用户问题正文（截断 500 字内） |
| `assistantmessageid` | `SendMessageAsync` 写入 contents 后 | 助手消息预生成 ID |
| `isnewsession` | `ChatHistoryDataProvider.CreateNewSessionAsync` | 私有标记，触发"首轮 AI 摘要标题" |

### 4.2 ChatHistoryDataProvider 持久化机制

`StoreChatHistoryAsync(...)` 已经针对工具调用链做了：

- **时间戳归一化**：以"同 session 已存在的最大 `CreatedTimestamp` + 1ms"为基准，按 `ResponseMessages` 原始顺序强制单调递增（见 `StoreChatHistoryAsync` 中 `baseTs + index` 写法）。OpenAI 协议要求 `assistant(tool_calls)` 与 `tool` 邻接关系不被排序破坏。
- **孤立 tool 消息告警**：本轮入库后若发现 `Role=tool` 但同批没有任何带 `functionCall` / `toolCall` 的 assistant 消息，会输出"工具链条警告"。
- **每条消息独立 ID**：不再把 assistant 消息强行统一为同一 messageId。

末尾两个副作用：

- `CompactAndReplaceAsync(...)`：段落压缩（自研逻辑，不是 SDK `Compaction.*`）。
- `GenerateAndUpdateTitleAsync(...)`：首轮 AI 摘要标题，仅 `isnewsession=true` 时触发。

**多智能体并发风险**：上述时间戳归一化策略基于"同 session 顺序写入"假设。如果未来 Concurrent 让多个 Agent 同时写同一个 session，多线程同时计算 `baseTs` 会冲突。详见 [05-风险与规避.md](./05-风险与规避.md) §3。

### 4.3 长期记忆依赖

`LongMemoryContextProvider` 使用 `StateBag` 中的：

- `agentid` / `agentname`
- `sessionid` / `sessiontitle`
- `turnid` / `traceid`
- `usermessageid` / `currenttask`

多 Agent 共用同一 session 时，子智能体可能拿到主智能体记忆，**不符合记忆隔离预期**。第一阶段轻量子 Agent 不带 memory，规避此问题。

## 5. Token 使用追踪现状

核心文件：`Src/Netor.Madorin.AI/Providers/TokenTrackingChatClient.cs`、`AIAgentFactory.cs`

### 5.1 工厂级状态

```csharp
public class AIAgentFactory
{
    public TokenTrackingChatClient? ChatClient { get; private set; }
    private long _lastInputTokens;
    private long _maxContextTokens;
    public event EventHandler<TokenUsageChangedEventArgs>? TokenUsageChanged;
}
```

`CreateTrackingClient` 每次都会 `Interlocked.Exchange` 覆盖 `_lastInputTokens`，意味着：

| 场景 | 当前是否会发生 | 影响 |
| --- | --- | --- |
| 串行多 Agent（切换提供商/模型/@子智能体导致 ChatClient 重建） | **已经发生** | 上一个 Agent 的 token 用量被新 Agent 覆盖；多智能体编排串联调用时 token 不可累加，UI 进度条只反映最近一次调用 |
| 真正并行多 Agent | 阶段 4 之前不会发生 | 多线程同时 `Interlocked.Exchange` 同一字段，进度条数据完全乱序 |

### 5.2 Workflow.AsAIAgent 的影响

将 Workflow 包装为 AIAgent 后，**工厂级 `TokenTrackingChatClient` 只会包住外层 ChatClient**，Workflow 内部成员 Agent 各自的 ChatClient 不会进入这个跟踪器。意味着：

- 阶段 2 起，UI 进度条接受"显示主 Agent / Coordinator 的 token，不显示子 Agent token"的现状；
- 阶段 5 想准确显示"全队总 token"，必须由 Orchestrator 内置 `TokenAggregator`，从 Workflow 事件 / 子 Agent 响应中提取 `UsageDetails`，再回写 `AIAgentFactory.TokenUsageChanged`。

## 9. 对话事件总线与 PluginBus 现状

> 这是工作模式分离方案能否成立的关键事实层。Memory 插件已经按 `conversation` topic 在生产环境运行，任何不兼容变更都会破坏现有用户的长期记忆。详细的分流方案见 [07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md)。

### 9.1 事件层（EventHub）

核心文件：

- `Src/Netor.Madorin.Entitys/Events.cs`（事件定义）
- `Src/Netor.Madorin.AI/AiChatHostedService.cs`（事件发布点）

`AiChatHostedService.SendMessageAsync` 单轮对话发布的对话事实事件：

### 9.4 历史回放（按需查询）

`PluginBusConversationHistoryDispatcher.ReplayAsync` 读取的 SQL：

```sql
SELECT m.Id, m.SessionId, m.Role, m.Content, m.CreatedTimestamp,
       m.ModelName, m.AgentId AS MessageAgentId, m.AgentName AS MessageAgentName,
       s.Categorize AS WorkspaceId, s.AgentId AS SessionAgentId, s.RawDiscription
FROM ChatMessages m
LEFT JOIN ChatSessions s ON s.Id = m.SessionId
WHERE m.CreatedTimestamp >= @Since
ORDER BY m.CreatedTimestamp
LIMIT @PageSize + 1
```

**重要**：当前回放数据源**只读 `ChatMessages` 表**。如果按 [06-工作模式独立模块设计.md](./06-工作模式独立模块设计.md) 把 Workflow 数据落在独立表（`OrchestrationTask` / `OrchestrationStep` / `OrchestrationMessage`），现有回放路径**完全无法触达 Workflow 历史**——这正是工作模式必须有自己独立 history dispatcher 的原因。

### 9.5 长期记忆供应（请求-响应，逆向）

```text
LongMemoryContextProvider.InvokingAsync
    │  从 Session.StateBag 取 agentId/sessionId/turnId/usermessageid/agentname/sessiontitle/traceid/currenttask
    ▼
PluginBus 发送 memory.context.supply.request
    ▼
Memory 插件 MemorySupplyRequestHandler
    ▼
返回 memory.context.supply.response
    ▼
LongMemoryContextProvider 注入 AIContext.Instructions
```

供应请求字段集（`MemoryContextSupplyRequest`）依赖**单 Agent 模型**：一个 `agentId`、一个 `sessionId`、一个 `turnId`。Workflow 模式下多 Agent 协作时，必须为每个 Member Agent 各自构造独立的供应请求（如果他们需要长期记忆），不能用 Workflow 任务级别的统一标识。

### 9.6 Memory 插件订阅与入库

核心文件：`Plugins/Src/Madorin.Plugins.Memory/Services/MemoryConversationEventHandler.cs`

```text
MemoryPluginBusConnection (WS 客户端)
    │  订阅 conversation / memory / model topics
    ▼
MemoryConversationEventHandler.TryHandle
    │  分流：history.batch → IngestExportBatch
    │       conversation.user.message / conversation.turn.completed → 入库 ObservationRecord
    │       conversation.assistant.delta → 累加到 _assistantStreams 缓冲区
    │       conversation.turn.completed (success) → 落地 buffered assistant text
    ▼
IMemoryStore.InsertObservation / BulkInsertObservations
    │
    ▼
ObservationRecord 表
```

**关键事实**：

1. Memory 插件按 `assistantMessageId` 维护 `_assistantStreams` 字典缓冲流式 token，依赖"一个 turn = 一个 assistantMessageId"的语义。
2. Memory 入库的 `agentId/agentName` 是**单 Agent 字段**，没有 Manager / Member 概念。
3. Memory 入库的 `role` 只有 `user` / `assistant` / `tool`，没有 `manager` / `moderator` / `member` 角色。
4. Memory 插件目前**只订阅 conversation topic**；没有 `workflow` 这个 topic，也没有处理 workflow.* 事件的代码。

### 9.7 工作模式必须独立的根本理由

| 问题 | 如果硬塞进 conversation 事件 | 工作模式独立后 |
| --- | --- | --- |
| 多 Agent 字段 | `agentId` 字段错配 | `WorkflowEvent.Participants` 完整列表 |
| 流式 buffer | `_assistantStreams` 字典爆炸（多 Member 同时 stream） | PluginBus 不发流式（决策 3-A） |
| 历史数据源 | 无法跨 ChatMessages 与 OrchestrationTask 表 | 独立 dispatcher 独立查询 |
| 角色语义 | `role` 字段被污染 | `WorkflowEvent.Step.Action` 独立枚举 |
| 内存噪音 | Manager 内部规划文本被记忆 | Memory 仅订阅 `workflow.task.completed` 拿最终 FinalReport |
| 兼容性 | 任何字段变动都破坏 Memory 插件 | 新事件族向后扩展，老事件族不动 |

**结论**：工作模式必须使用新的 PluginBus topic（`workflow`），独立的事件族，独立的 history dispatcher。详见 [07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md)。

### 9.8 版本协调要求

### 10.2 标题生成完整链路

```text
首次提交消息
  ↓
ChatHistoryDataProvider.CreateNewSessionAsync(session, agent)
  ├── session.StateBag.SetValue("isnewsession", "True")     ← 私有标记
  └── AddOrUpdateSessionAsync(session, "新会话", agent)      ← 写库占位标题
  ↓
首轮对话流式响应完成
  ↓
StoreChatHistoryAsync(context)（末尾）
  ├── 检测 session.StateBag.GetValue<string>("isnewsession") == "True"
  ├── 翻转为 False
  └── 异步触发 GenerateAndUpdateTitleAsync(agent, sessionId, firstText, aiResponse)
       ├── ResolveCompactionClient(agent) 取压缩用 LLM 客户端
       ├── (client as TokenTrackingChatClient)?.SuppressUsage()  ← 避免污染主进度条
       ├── prompt: "标题生成助手...20字内...直接输出"
       ├── client.GetResponseAsync(prompt)
       ├── 清洗引号 + 截断 32 字
       ├── UPDATE ChatSessions SET Title=..., UpdatedTimestamp=... WHERE Id=@Id
       └── publisher.Publish(OnSessionTitleUpdated, new SessionTitleUpdatedArgs(sessionId, title))
  ↓
UI ChatHistoryPanel 订阅 OnSessionTitleUpdated → UpdateSessionTitle(sessionId, title)
```

**关键设计点**：

- 标题生成**异步、不阻塞主响应**：用户看到 AI 回复时标题可能还是"新会话"，几秒后自动刷新。
- LLM 调用复用主 Agent 的客户端，但通过 `SuppressUsage()` 隔离 usage 上报。
- 失败静默：catch 异常仅 LogWarning，保留占位标题。

### 10.4 启动时自动恢复策略

核心文件：`Src/Netor.Madorin.AI/AiChatHostedService.cs`

```csharp
private async Task LoadOrCreateSessionAsync(CancellationToken ct)
{
    var categorize = appPaths.WorkspaceDirectory.Md5Encrypt();
    var recentSessionId = dbContext.ExecuteScalar<string>(
        """
        SELECT Id FROM ChatSessions
        WHERE IsArchived = 0 AND Categorize = @Categorize
        ORDER BY IsPinned DESC, LastActiveTimestamp DESC LIMIT 1
        """,
        cmd => cmd.Parameters.AddWithValue("@Categorize", categorize));

    _session = await _agent.CreateSessionAsync(ct);

    if (!string.IsNullOrWhiteSpace(recentSessionId))
    {
        _sessionId = recentSessionId;   // 自动恢复最近会话
    }
    else
    {
        _sessionId = Guid.NewGuid().ToString("N");
    }

    _session.StateBag.SetValue("sessionid", _sessionId);
    UpdateSessionSelectionState(_session);
}
```

**关键事实**：Chat 启动时**自动选中最近会话**，这是 Chat 范式（"等用户继续上次的对话"）。Workflow 不沿用此行为（详见 §10.7）。

### 10.5 会话切换：ResumeSessionAsync 极简实现

```csharp
// AiChatHostedService.ResumeSessionAsync
public async Task ResumeSessionAsync(string sessionId)
{
    if (_agent is null) return;

    _session = await _agent.CreateSessionAsync(CancellationToken.None);
    _sessionId = sessionId;
    _session.StateBag.SetValue("sessionid", sessionId);
    UpdateSessionSelectionState(_session);
    factory.ResetTokenStats();
}
```

**重要发现**：`AgentSession` 本身**无状态**，只是 ID 标识。历史消息由 `ChatHistoryDataProvider` 在每次 LLM 调用前**从 SQL 注入**。所以恢复极简——创建 session、设 sessionId、结束。

UI 切换时分两步走：

```csharp
// MainWindow.SwitchToSession
private void SwitchToSession(string sessionId, string title)
{
    HistoryLabel.Text = title;
    HistoryPanel.CurrentSessionId = sessionId;
    MessageList.Items.Clear();

    // 1. 后端切换：恢复 session 上下文
    _ = Task.Run(() => chatService.ResumeSessionAsync(sessionId));

    // 2. 前端拉历史消息渲染
    var messages = messageService.GetBySessionId(sessionId);
    // ... 渲染到 MessageList
}
```

### 10.6 标题生成的 LLM 客户端复用

`ResolveCompactionClient(agent)` 是 `ChatHistoryDataProvider` 内部的辅助方法，作用：

- 优先取专门配置的"压缩 / 摘要"用 ChatClient（如系统设置中指定的 small model）
- 兜底取主对话的 `TokenTrackingChatClient`

`(client as TokenTrackingChatClient)?.SuppressUsage()` 是关键：

- 避免标题 / 摘要 / 压缩这些副作用调用上报 usage 污染主对话进度条
- 复用同一连接 / 鉴权配置，无需额外配置

### 10.7 Workflow 必须保留差异的点

Chat 模式自动恢复"最近会话"是合理的，因为 Chat 永远是"等用户输入"。Workflow 不能照搬：

| 行为 | Chat | Workflow（必须不同） |
| --- | --- | --- |
| 启动时自动选中 | 最近会话 | **不选中任何任务**（任务可能正在跑或已完成） |
| 同时打开多少个 | 1 个 active session | 多个完成任务可同时浏览（互不影响） |
| 切换语义 | 切换"当前对话上下文"（等用户继续输入） | 仅"打开任务卡片浏览"（不会影响后端运行） |
| 切换后台动作 | `ResumeSessionAsync` 重新绑定 session | 仅 UI 加载，无后端动作（除非任务正在运行需要订阅事件流） |

详细 Workflow 对应方案见 [06-工作模式独立模块设计.md](./06-工作模式独立模块设计.md) §4（数据模型对齐）和 §7（恢复流程状态分流）。
