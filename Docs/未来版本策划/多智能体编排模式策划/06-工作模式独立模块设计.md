# 06 - 工作模式独立模块设计

> **目的**：定义 Madorin 工作模式（Workflow Mode）作为独立模块的数据模型、UI 入口、与 Chat 模式的联动协议、以及 Workflow 模式内部的执行流程。  
> **依赖**：[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md)、[02-当前代码现状.md](./02-当前代码现状.md)、[03-编排模式与边界约束.md](./03-编排模式与边界约束.md)、[04-实施阶段.md](./04-实施阶段.md)。  
> **配套**：[07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md)（PluginBus 协议 / Memory 插件接入）。

---

## 1. 模块定位

工作模式（Workflow Mode）是 Madorin 的**第二个顶层入口**，与对话模式（Chat Mode）并列。两者在交互范式、数据模型、UI 呈现、协议层面**全部独立**。

### 1.1 顶层入口结构

```text
Madorin 主窗口
  ├── 顶部 Tab 1：对话（Chat）        ← 现状
  │     └── 左侧会话列表 + 右侧聊天界面
  │
  └── 顶部 Tab 2：工作台（Workflow）  ← 新增（阶段 3B+）
        ├── 左侧任务列表（按时间倒序）
        └── 右侧任务详情卡片
```

### 1.2 顶层入口选项决策回顾

| 选项 | 描述 | 状态 |
| --- | --- | --- |
| A | 顶部 Tab 切换 | ✅ 已确定（决策 3） |
| B | 侧边栏菜单 | 否定 |
| C | 对话内浮动按钮 | 否定 |

理由：顶部 Tab 是和用户契约最强的视觉信号——"这是一个不同的工作区"，避免被误以为只是 Chat 的变体。

## 2. 模块边界与与 Chat 的关系

### 2.1 完全独立的资源

| 资源 | Chat | Workflow |
| --- | --- | --- |
| 数据库表 | `ChatSessions` / `ChatMessages` | `OrchestrationTask` / `OrchestrationStep` / `OrchestrationParticipant` / `OrchestrationMessage` |
| EventHub 事件 | `conversation.*` | `workflow.*` |
| PluginBus topic | `conversation` | `workflow` |
| 历史回放 dispatcher | `PluginBusConversationHistoryDispatcher` | `PluginBusWorkflowHistoryDispatcher` |
| 后端服务 | `AiChatHostedService` / `IAgentOrchestrator` | `IWorkflowExecutor` |
| Memory 入库 EventHandler | `MemoryConversationEventHandler` | `MemoryWorkflowEventHandler` |
| UI 入口 | "对话" Tab | "工作台" Tab |
| 取消语义 | 取消当前一轮 | 取消整个任务 |
| 流式 token | WebSocket 推送 + Memory 入库 | WebSocket 推送（仅 UI），不进 PluginBus |

### 2.2 共享的资源

| 资源 | 共享方式 |
| --- | --- |
| `AgentEntity` | 同一套 Agent 定义，Chat 与 Workflow 都引用 |
| 工具绑定（插件 / MCP） | 同一套 `EnabledPluginIds` / `EnabledMcpServerIds` |
| 工作区目录 / 系统设置 | 同一套 `WorkspaceDirectory` / `SystemSettingsService` |
| `AIAgentFactory` | Chat 用 `Build()` / `BuildWithSubAgents()`；Workflow 用 `Build()` + `BuildSubAgent()` 后 `Workflow.AsAIAgent()` |
| 长期记忆系统 | Chat 自动按消息粒度入库；Workflow 仅按 task.completed 入库 |
| traceId | 跨模式可串联，方便审计 |

### 2.3 双向联动协议

**Chat → Workflow**（阶段 5B）：

```text
1. 用户在 Chat 输入消息
2. AiChatHostedService 启发式判断（关键词 / LLM 分类）
3. 触发 SystemNotice 卡片："这看起来是一个复杂任务，是否切到工作模式？"
4. 用户点击确认：
   - 取消当前 Chat 一轮
   - 把用户消息文本作为 OrchestrationTask.InitialInput
   - 跳转到工作台 Tab，预填任务表单
   - 任务记录 SourceSessionId = 触发的 ChatSession.Id
```

**Workflow → Chat**（阶段 5B）：

```text
1. 任务完成后任务卡片显示"附加到对话"按钮
2. 用户点击后选择目标会话：
   - 默认选项：原 SourceSessionId 对应的 Chat 会话
   - 备选：当前打开的 Chat 会话
   - 备选：新建 Chat 会话
3. 把 OrchestrationTask.FinalReport 作为一条 assistant 消息插入目标会话
4. 通过现有 conversation.* 链路触发 Memory 自动入库（避免双重入库——见 §6.4）
```

**任务进行中**：

- 用户**不能**像 chat 一样随时插话
- 只能"取消"或"等待 HITL 节点（阶段 5B+）"
- UI 显示"任务运行中"占位，提供轮询进度

## 3. 数据模型

### 3.1 OrchestrationTask 表

> **设计原则**：与 `ChatSessionEntity` 字段命名 / 含义对齐（见 [02-当前代码现状.md](./02-当前代码现状.md) §10.1），UI 层可以复用 ChatHistoryPanel 的列表 / 排序 / 置顶 / 归档 / 标题刷新代码模式。

```sql
CREATE TABLE OrchestrationTask (
    Id              TEXT     PRIMARY KEY,         -- GUID
    Title           TEXT     NOT NULL,            -- 用户填写或 LLM 生成的任务标题，与 ChatSession.Title 对齐（256 字以内）
    Summary         TEXT     NULL,                -- 任务摘要（1024 字内），任务列表预览用，与 ChatSession.Summary 对齐
    IsTitleAutoGenerated INTEGER NOT NULL DEFAULT 0,  -- 1=用户未填，需 LLM 兜底；0=用户已填或已生成完成
    Mode            TEXT     NOT NULL,            -- "discussion" / "execution"
    SubMode         TEXT     NOT NULL,            -- "groupchat" / "magentic" / "parallelanalysis" / "handoffexecution"
    Status          TEXT     NOT NULL,            -- "pending" / "running" / "paused" / "completed" / "failed" / "cancelled"
    StartedAt       INTEGER  NOT NULL,            -- Unix ms
    CompletedAt     INTEGER  NULL,                -- Unix ms
    LastActiveTimestamp INTEGER NOT NULL,         -- Unix ms，与 ChatSession.LastActiveTimestamp 对齐，列表排序主键；
                                                  -- 启动 / 单步完成 / 任务结束 / 标题更新 时全部更新
    FinalReport     TEXT     NULL,                -- Markdown 最终报告
    TraceId         TEXT     NOT NULL,            -- 与 ChatSession 的 traceId 一致以便串联审计
    CreatedBy       TEXT     NOT NULL,            -- 触发任务的来源："user" / "chat-suggested" / "duplicated" / "schedule"
    SourceSessionId TEXT     NULL,                -- 触发该任务的 Chat 会话 ID（如有）
    SourceTaskId    TEXT     NULL,                -- 决策 10：复制为新任务时记录原任务 ID（追踪用，不影响执行）
    ManagerAgentId   TEXT     NULL,                -- Magentic Manager / GroupChat Moderator
    ManagerAgentName TEXT     NULL,
    InitialInput    TEXT     NOT NULL,            -- 任务初始描述
    InitialAttachmentsJson TEXT NULL,            -- JSON 数组，附件路径与描述
    ErrorMessage    TEXT     NULL,                -- 失败时的错误摘要
    WorkspaceId     TEXT     NOT NULL,            -- 工作区标识（与 ChatSession.Categorize 同源），列表 WHERE 主条件
    IsPinned        INTEGER  NOT NULL DEFAULT 0,  -- 置顶（列表优先），与 ChatSession.IsPinned 对齐
    IsArchived      INTEGER  NOT NULL DEFAULT 0,  -- 归档（默认列表不显示），与 ChatSession.IsArchived 对齐
    TotalTokenCount INTEGER  NOT NULL DEFAULT 0,  -- 累计 token 用量（输入+输出），与 ChatSession.TotalTokenCount 对齐
    OverridesJson   TEXT     NULL,                -- 阶段 5B+ 起：保存 WorkflowTaskOverrides（MaxRounds 等覆盖）
    UpdatedTimestamp INTEGER NOT NULL             -- 与 BaseEntity 一致
);

CREATE INDEX IX_OrchestrationTask_Status ON OrchestrationTask(Status);
CREATE INDEX IX_OrchestrationTask_LastActiveTimestamp ON OrchestrationTask(LastActiveTimestamp);
CREATE INDEX IX_OrchestrationTask_Pinned_Active ON OrchestrationTask(IsPinned DESC, LastActiveTimestamp DESC);
CREATE INDEX IX_OrchestrationTask_SourceSessionId ON OrchestrationTask(SourceSessionId);
CREATE INDEX IX_OrchestrationTask_WorkspaceId ON OrchestrationTask(WorkspaceId);
CREATE INDEX IX_OrchestrationTask_SourceTaskId ON OrchestrationTask(SourceTaskId);
```

#### 与 ChatSession 字段对齐说明

| 维度 | ChatSession | OrchestrationTask | 是否完全对齐 |
| --- | --- | --- | --- |
| 工作区隔离 | `Categorize`(MD5) | `WorkspaceId`(MD5) | ✅ 同源命名不同 |
| 标题 | `Title` (256) | `Title` (256) | ✅ |
| 标题自动生成标记 | `StateBag["isnewsession"]`（运行时） | `IsTitleAutoGenerated`（持久化） | 🔶 持久化更可靠 |
| 摘要 | `Summary` (1024) | `Summary` (1024) | ✅ |
| 排序主键 | `LastActiveTimestamp` | `LastActiveTimestamp` | ✅ |
| 置顶 | `IsPinned` | `IsPinned` | ✅ |
| 归档 | `IsArchived` | `IsArchived` | ✅ |
| Token 累计 | `TotalTokenCount` | `TotalTokenCount` | ✅ |
| 压缩缓存 | `CompactedContext` / `CompactedAtCount` | — | ❌ Workflow 不需要（任务一次性执行） |
| 详细描述 | `RawDiscription` (4096) | `InitialInput` (无长度限制) | 🔶 语义不同 |

### 3.4 OrchestrationMessage 表

```sql
CREATE TABLE OrchestrationMessage (
    Id           INTEGER  PRIMARY KEY AUTOINCREMENT,
    TaskId       TEXT     NOT NULL,
    StepId       TEXT     NOT NULL,
    Sequence     INTEGER  NOT NULL,        -- (TaskId, StepId, Sequence) 三元组定序
    Role         TEXT     NOT NULL,        -- "user" / "assistant" / "tool" / "system"
    AuthorName   TEXT     NULL,
    Content      TEXT     NOT NULL,
    CreatedAt    INTEGER  NOT NULL,
    AttachmentsJson TEXT  NULL,
    FOREIGN KEY (TaskId) REFERENCES OrchestrationTask(Id) ON DELETE CASCADE,
    FOREIGN KEY (StepId) REFERENCES OrchestrationStep(Id) ON DELETE CASCADE
);

CREATE INDEX IX_OrchestrationMessage_TaskId_Sequence ON OrchestrationMessage(TaskId, Sequence);
CREATE INDEX IX_OrchestrationMessage_StepId ON OrchestrationMessage(StepId);
```

### 3.5 关键设计决策

| 决策 | 理由 |
| --- | --- |
| `(TaskId, StepId, Sequence)` 三元组定序 | 不依赖时间戳归一化，避免 [05-风险与规避.md](./05-风险与规避.md) §风险 3 的并发问题 |
| `OrchestrationMessage` 与 `OrchestrationStep` 1:N | 一个步骤可能产生多条消息（如 Magentic Manager 一轮内发 facts/plan 多条） |
| `ParentStepId` 支持嵌套 | Magentic 重规划时，新计划下的步骤挂在重规划步骤之下 |
| `SummaryJson` 而不是完整 content | 步骤级摘要用于检索，完整内容在 `OrchestrationMessage` |
| `WorkspaceId` 显式列 | 便于按工作区切换检索（与 ChatSession 一致） |
| `SourceSessionId` 可空 | 既支持"从 Chat 跳转过来"，也支持"工作台直接发起"两种入口 |
| `SourceTaskId` 可空 | 决策 10：支持"复制为新任务"功能，记录原任务以便审计追踪 |
| `TraceId` 与 ChatSession 一致 | 跨模式审计：从 Chat 跳到 Workflow 仍能拼出完整链路 |
| `IsTitleAutoGenerated` 持久化标记 | 决策 6：与 Chat 的 `StateBag["isnewsession"]` 不同，宿主重启后仍能续接 LLM 标题生成 |
| `LastActiveTimestamp` / `IsPinned` / `IsArchived` 对齐 Chat | 决策 7：UI 列表逻辑可复用 ChatHistoryPanel 的列表 / 排序 / 置顶 / 归档代码模式 |

### 3.6 任务列表标准查询语句

与 ChatHistoryPanel 完全同构（[02-当前代码现状.md](./02-当前代码现状.md) §10.3），UI 层可抽出公共 Query Helper：

```sql
-- 默认列表（分页 30 条 / 页）
SELECT * FROM OrchestrationTask
WHERE IsArchived = 0 AND WorkspaceId = @ws
ORDER BY IsPinned DESC, LastActiveTimestamp DESC
LIMIT @limit OFFSET @offset;

-- 按状态筛选（"运行中"/"已完成"/"已失败"标签页）
SELECT * FROM OrchestrationTask
WHERE IsArchived = 0 AND WorkspaceId = @ws AND Status IN (@statuses)
ORDER BY IsPinned DESC, LastActiveTimestamp DESC
LIMIT @limit OFFSET @offset;

-- 归档列表
SELECT * FROM OrchestrationTask
WHERE IsArchived = 1 AND WorkspaceId = @ws
ORDER BY LastActiveTimestamp DESC
LIMIT @limit OFFSET @offset;
```

`LastActiveTimestamp` 更新时机：

| 时机 | 是否更新 LastActiveTimestamp |
| --- | --- |
| `StartTaskAsync`（任务创建） | ✅ = StartedAt |
| `workflow.step.completed` | ✅ |
| `workflow.task.completed` / `failed` / `cancelled` | ✅ = CompletedAt |
| `workflow.task.title.updated`（LLM 异步生成完毕） | ✅ |
| 用户置顶 / 取消置顶 / 归档 / 取消归档 | ❌（这些是元数据操作，不算"活跃"） |
| 用户重命名标题 | ✅（视为活跃） |

### 4.3 任务执行流程

```text
1. StartTaskAsync(request)
   ├── 校验：MemberAgentIds 不为空、Manager Agent 配置（如适用）、模型支持 FunctionCall（如适用）
   ├── 写表：INSERT OrchestrationTask(Status=running)
   ├── 写表：INSERT OrchestrationParticipant(s)
   ├── 发事件：workflow.task.started
   ├── 构建 Workflow（按 SubMode 分支）：
   │     ├── BuildGroupChatWorkflow (3B)
   │     ├── BuildMagenticWorkflow (4B)
   │     ├── BuildParallelAnalysisWorkflow (4B)
   │     └── BuildHandoffExecutionWorkflow (5B+ 待评估)
   │
   ├── 启动 InProcessExecution.RunStreamingAsync(workflow, initialInput)
   │
   └── 异步循环 run.WatchStreamAsync()：
         ├── AgentRunUpdateEvent → 流式累积，仅 UI WS 推送（不进 PluginBus）
         ├── ExecutorCompletedEvent → 落 OrchestrationStep + OrchestrationMessage(s) + 发 step.completed 事件
         ├── RequestInfoEvent (HITL, 5B+) → 落 paused + 发 task.paused 事件 + 等待 ResumeAsync
         └── WorkflowOutputEvent → 落 task.completed + finalReport + 发 task.completed 事件

2. CancelTaskAsync(taskId)
   ├── 取消对应 Workflow run 的 CancellationToken
   ├── 写表：UPDATE OrchestrationTask(Status=cancelled)
   └── 发事件：workflow.task.cancelled
```

### 4.6 任务标题生成（决策 6-A：用户填可选 + LLM 兜底）

模式与 Chat 的 `GenerateAndUpdateTitleAsync`（[02-当前代码现状.md](./02-当前代码现状.md) §10.2）完全对齐。

#### 4.6.1 标题来源优先级

```text
WorkflowTaskRequest.Title 是否为空？
  ├── 非空 → 用户已填，直接使用，IsTitleAutoGenerated = 0
  └── 空    → 第一阶段占位 = InitialInput.Truncate(32)
              IsTitleAutoGenerated = 1
              任务完成后异步触发 LLM 生成
```

#### 4.6.2 占位标题写入时机

```csharp
public async Task<string> StartTaskAsync(WorkflowTaskRequest request, CancellationToken ct)
{
    var taskId = Guid.NewGuid().ToString("N");
    var now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();

    var isTitleAutoGenerated = string.IsNullOrWhiteSpace(request.Title);
    var initialTitle = isTitleAutoGenerated
        ? request.InitialInput.Truncate(_options.TaskTitleMaxLength)
        : request.Title.Truncate(_options.TaskTitleMaxLength);

    var task = new OrchestrationTaskEntity
    {
        Id = taskId,
        Title = initialTitle,
        IsTitleAutoGenerated = isTitleAutoGenerated ? 1 : 0,
        Mode = MapMode(request.SubMode),
        SubMode = MapSubMode(request.SubMode),
        Status = WorkflowTaskStatus.Running,
        StartedAt = now,
        LastActiveTimestamp = now,
        WorkspaceId = request.WorkspaceId ?? appPaths.WorkspaceDirectory.Md5Encrypt(),
        SourceSessionId = request.SourceSessionId,
        SourceTaskId = request.SourceTaskId,
        ManagerAgentId = request.ManagerAgentId,
        // ...
    };
    _taskRepo.Insert(task);

    _publisher.Publish(WorkflowEvents.OnWorkflowTaskStarted, new WorkflowTaskStartedArgs(...));
```

#### 4.6.3 异步生成实现（任务完成后触发）

```csharp
// WorkflowExecutor 内部，task.completed 事件发布后调用
private async Task TryGenerateTitleAsync(OrchestrationTaskEntity task, string finalReport)
{
    if (task.IsTitleAutoGenerated == 0) return;          // 用户已填，跳过
    if (string.IsNullOrWhiteSpace(finalReport)) return;  // 没有最终报告

    try
    {
        // 复用 Chat 模式的客户端解析方式
        var client = ResolveCompactionClient();  // 与 ChatHistoryDataProvider.ResolveCompactionClient 同实现，公共组件
        if (client is null) return;

        using var _ = (client as TokenTrackingChatClient)?.SuppressUsage();

        var prompt = new List<AIChatMessage>
        {
            new(ChatRole.System,
                "你是工作任务标题生成助手。根据任务描述和最终报告，提取关键词并总结成一个简短的任务标题。" +
                $"要求：不超过 {_options.TaskTitleMaxLengthForLlm} 个字，不要引号，不要标点，直接输出标题文本。"),
            new(ChatRole.User,
                $"任务描述：{task.InitialInput.Truncate(200)}\n\n" +
                $"最终报告：{finalReport.Truncate(800)}")
        };

        var completion = await client.GetResponseAsync(prompt);
        var title = completion?.Text?.Trim();
        if (string.IsNullOrWhiteSpace(title)) return;

        // 清理引号和多余空白（与 Chat 一致）
        title = title.Trim('"', '"', '"', '「', '」', '\'').Truncate(_options.TaskTitleMaxLength);

        _taskRepo.UpdateTitle(task.Id, title, isAutoGenerated: false);

        // ★ 事件分流：独立事件，与 Chat 的 OnSessionTitleUpdated 不混淆
        _publisher.Publish(WorkflowEvents.OnWorkflowTaskTitleUpdated,
            new WorkflowTaskTitleUpdatedArgs(task.Id, title, ...));

        logger.LogInformation("Workflow 任务 {TaskId} 标题已由 AI 生成：{Title}", task.Id, title);
    }
    catch (Exception ex)
    {
        logger.LogWarning(ex, "为 Workflow 任务 {TaskId} 生成 AI 标题失败，保留占位标题。", task.Id);
    }
}
```

#### 4.6.4 LLM 客户端复用约束

| 维度 | Chat（`GenerateAndUpdateTitleAsync`） | Workflow（`TryGenerateTitleAsync`） | 复用方式 |
| --- | --- | --- | --- |
| 客户端解析 | `ResolveCompactionClient(agent)` | 同实现 | ✅ 提到公共组件 `IChatCompactionClientResolver` |
| Usage 隔离 | `SuppressUsage()` | 同 | ✅ |
| 失败行为 | LogWarning + 保留占位 | LogWarning + 保留占位 | ✅ |
| 事件 | `OnSessionTitleUpdated` | `OnWorkflowTaskTitleUpdated`（新增） | ❌ 独立事件，避免 Memory 插件误处理 |
| Prompt 内容 | 用户问题 + AI 回复 | 任务描述 + 最终报告 | ❌ 适配 Workflow 语义 |
| 触发时机 | 首轮对话完成（`IsNewSessionStateKey == True`） | 任务完成（`task.IsTitleAutoGenerated == 1`） | ❌ 各自触发 |

`WorkflowTaskTitleUpdatedArgs` 与事件契约详见 [07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md) §2.3.8。

### 4.8 启动时孤儿任务清理（决策 9-A）

宿主启动时（IHostedService.StartAsync）扫描数据库中 Status 处于 `running` / `paused` 但内存无对应 Run 的任务：

```csharp
public async Task StartAsync(CancellationToken ct)
{
    var orphans = _taskRepo.QueryByStatuses(new[]
    {
        WorkflowTaskStatus.Running,
        WorkflowTaskStatus.Paused,
        WorkflowTaskStatus.Pending
    });

    var now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
    foreach (var orphan in orphans)
    {
        _taskRepo.UpdateCompleted(
            orphan.Id,
            finalReport: null,
            status: WorkflowTaskStatus.Failed,
            errorMessage: "宿主重启导致任务丢失，请重新启动该任务。",
            completedAt: now,
            lastActiveTimestamp: now);

        _publisher.Publish(WorkflowEvents.OnWorkflowTaskFailed, new WorkflowTaskFailedArgs(...));

        logger.LogWarning("孤儿任务 {TaskId} 标记为 failed（宿主重启清理）", orphan.Id);
    }

    if (orphans.Count > 0)
    {
        _publisher.Publish(Events.OnSystemNotice, new SystemNoticeArgs(
            Title: "Workflow 任务恢复",
            Content: $"检测到 {orphans.Count} 个任务因上次宿主重启中断，已标记为失败。可在工作台查看详情或使用'复制为新任务'功能重新启动。",
            Severity: "warning"));
    }
}
```

阶段 5B+ 接入 SDK Checkpoint 持久化后，可改为"先尝试从 Checkpoint 恢复，失败再标记 failed"。

### 5.5 实时进度更新

UI 通过 WebSocket 订阅 `workflow` topic：

| WS 事件 | UI 行为 |
| --- | --- |
| `workflow.task.started` | 任务列表插入新项（顶部），新任务高亮 |

| `workflow.task.planned` | Magentic 任务下方插入"计划已生成"占位 |
| `workflow.step.started` | 步骤行变为"运行中" |
| `workflow.step.completed` | 步骤行变为"完成" + 显示耗时与 token |
| `workflow.task.completed` | 任务卡片状态变为"已完成"，最终报告区域出现；列表对应项移到顶部（LastActiveTimestamp 已更新） |
| `workflow.task.failed` | 任务卡片状态变为"失败"，错误信息展示 |
| `workflow.task.cancelled` | 任务卡片状态变为"已取消" |
| `workflow.task.paused` (5B+) | 任务卡片显示批准卡片，列表状态图标变 ⏸ |
| `workflow.task.resumed` (5B+) | 任务卡片回到运行中状态 |

注意：UI 不订阅 PluginBus；UI 通过宿主**单独的内部 WS 通道**接收事件。PluginBus 只对插件开放（详见 [07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md)）。

- `workflow.task.title.updated` → 仅找到对应 list item，更新 TextBlock.Text；不重新拉列表
- `workflow.step.completed` → 仅更新当前打开的任务详情区，不影响其他任务
- `workflow.task.started` / `workflow.task.completed` → 列表项重新插入到正确位置（按 LastActiveTimestamp 排序）

#### 5.8.3 关键控件选型

| 场景 | 选型 | 理由 |
| --- | --- | --- |
| 任务列表 | `ItemsControl` + `VirtualizingStackPanel` | 30+ 任务下保持 60fps |
| 任务详情区 Step 列表 | `ItemsRepeater`（Avalonia 12 内置） | 长任务（Magentic 13+ 步）虚拟化 |
| 任务详情按状态切换 | `ContentControl` + `DataTemplate` | 不同状态走不同模板，VM 驱动 |

| Markdown 报告 | 复用 `MarkdownRenderer`（[02] §10.x） | 与 Chat 视觉对齐 |
| Markdown 报告滚动 | 任务详情区内嵌独立 `ScrollViewer` | 与列表 ScrollViewer 解耦 |
| 多任务并行打开 | **不支持**，单详情区共享（决策 12-A） | 避免渲染压力，符合"一次只看一个"的工作模式心智 |

#### 5.8.4 数据流示例：任务标题异步更新

```csharp
// WorkflowApiController（阶段 5B+）
public async Task<IActionResult> AttachToConversation(
    string taskId,
    string? targetSessionId,
    CancellationToken ct)
{
    var task = await _workflowExecutor.GetTaskAsync(taskId, ct);
    if (task.Status != WorkflowTaskStatus.Completed) return BadRequest();

    var sessionId = targetSessionId
        ?? task.SourceSessionId
        ?? await _chatService.CreateNewSessionAsync(...);

    // 通过现有链路插入消息（不直接写表），让事件总线自然发出
    await _chatService.AppendAssistantMessageAsync(sessionId, new AppendOptions
    {
        Content = $"[来自 Workflow 任务 \"{task.Title}\"]\n\n{task.FinalReport}",
        EventType = "workflow.attached",  // 标记来源，方便审计
        SourceTaskId = taskId
    }, ct);

    return Ok();
}
```

### 6.4 防止双重入库

Workflow 任务完成事件 `workflow.task.completed` 已经触发 Memory 写入一条 ObservationRecord（`EventType=workflow.task.completed`）。

阶段 5B+ 用户"附加到对话"后会再次通过 conversation 链路触发 Memory 写入一条 ObservationRecord（`EventType=conversation.turn.completed`，`Content` 来自 FinalReport）。

**这是有意保留的两条记录**：

- 第一条记录"任务级别完成"事实，用于 Workflow 历史回溯
- 第二条记录"用户认可并将报告纳入对话"行为，用于 Chat 历史回溯

两条记录的 `Id` 不同（一条是 `taskId`，一条是 `assistantMessageId`），不会冲突。详见 [07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md) §4.4。

## 7. 任务恢复与持久化

> 本节是 Workflow 对应 Chat 模式 `LoadOrCreateSessionAsync` + `ResumeSessionAsync`（[02-当前代码现状.md](./02-当前代码现状.md) §10.4 / §10.5）的差异化实现。Workflow 不沿用 Chat 的"自动选中最近会话"逻辑（决策 8-A）。

### 7.1 与 Chat 的根本差异

| 维度 | Chat | Workflow |
| --- | --- | --- |
| 启动时自动选中 | 最近会话 | **不选中任何任务**（决策 8-A） |
| 切换会话语义 | 切换"当前对话上下文"，绑定后端 session | 仅"打开任务卡片浏览"，无后端 session 绑定 |
| 同时可打开数 | 1 个 active session | 浏览多个完成任务无冲突 |
| 切换后端动作 | `ResumeSessionAsync` 重新绑定 AgentSession + 注入历史 | 仅 UI 加载，运行中任务额外 `SubscribeAsync` 跟随事件 |
| 历史消息加载方式 | 每次 LLM 调用时 SQL 注入 | 一次性查询渲染到任务卡片 |
| 取消语义 | 取消当前一轮 | 取消整个任务 |

### 7.2 启动时孤儿任务清理（决策 9-A）

宿主启动时的清理逻辑见 §4.8。关键点：

- 扫描 `Status IN (Running, Paused, Pending)` 的任务
- 全部标记为 `Failed`，错误信息明示"宿主重启导致任务丢失"
- 发 SystemNotice 告知用户，提示可"复制为新任务"重新启动
- **不自动选中任何任务**，用户主动打开工作台 Tab 时才看到列表

阶段 5B+ 接入 SDK Checkpoint 持久化后，可改为"先尝试从 Checkpoint 恢复，失败再标记 failed"。

### 7.3 打开任务的状态分流（核心流程）

```text
用户点击任务列表项 / API 调用 GetTaskDetailAsync(taskId)
  ↓
读取 OrchestrationTask 状态：
  │
  ├── completed / failed / cancelled
  │     └── 只读浏览模式
  │         ├── 一次性查询 Participants + Steps + Messages
  │         ├── UI 渲染任务卡片（Header + Steps + FinalReport）
  │         ├── 按钮：[复制为新任务] [附加到对话(完成时)] [删除]
  │         └── ❌ 不调用 SubscribeAsync（任务已结束，没有事件流）
  │
  ├── running
  │     ├── 内存 _runs 字典有该 taskId
  │     │     └── 实时跟随模式
  │     │         ├── 加载已完成的 Participants + Steps + Messages
  │     │         ├── 调用 SubscribeAsync(taskId) 订阅事件流
  │     │         ├── UI 实时刷新步骤与状态
  │     │         └── 按钮：[取消任务]
  │     │
  │     └── 内存无该 taskId（理论上启动时已清理，仅防御）
  │           └── 异常恢复：标记 failed → 切换为只读浏览
  │
  ├── paused（阶段 5B+ HITL）
  │     ├── 内存有 Run
  │     │     └── 暂停恢复模式
  │     │         ├── 加载历史步骤
  │     │         ├── 显示 RequestInfo 详情（计划批准卡片）
  │     │         └── 按钮：[批准 / 修改 / 拒绝 / 取消]
  │     │           └─→ 触发 ResumeAsync(taskId, response)
  │     │
  │     └── 内存无 Run → 同 running 异常恢复
  │
  └── pending（理论上不存在持久化态，仅启动扫描时短暂）
        └── 启动扫描已转换为 failed
```

### 7.4 GetTaskDetailAsync 实现

```csharp
public async Task<OrchestrationTaskDetail?> GetTaskDetailAsync(string taskId, CancellationToken ct)
{
    var task = _taskRepo.Get(taskId);
    if (task is null) return null;

    var participants = _participantRepo.GetByTaskId(taskId);
    var steps = _stepRepo.GetByTaskId(taskId);
    var messages = _messageRepo.GetByTaskId(taskId);

    var isLive = _runs.ContainsKey(taskId);   // 内存中是否有活跃 Run

    return new OrchestrationTaskDetail
    {
        Task = task,
        Participants = participants,
        Steps = steps,
        Messages = messages,
        IsLive = isLive,                       // UI 决定是否要 SubscribeAsync
        CanCancel = isLive && (task.Status == WorkflowTaskStatus.Running || task.Status == WorkflowTaskStatus.Paused),
        CanResume = isLive && task.Status == WorkflowTaskStatus.Paused,
        CanDuplicate = true,                   // 任何状态都可以复制为新模板
        CanAttachToConversation = task.Status == WorkflowTaskStatus.Completed && !string.IsNullOrWhiteSpace(task.FinalReport)
    };
}
```

### 7.5 任务详情打开的 UI 流程

```csharp
// WorkflowTaskListPanel.OnItemClick(taskId)
private async void OnTaskItemClick(string taskId)
{
    var detail = await _executor.GetTaskDetailAsync(taskId, CancellationToken.None);
    if (detail is null) return;

    // 1. 一次性渲染任务详情
    TaskDetailView.Render(detail);

    // 2. 仅在任务 live 时订阅事件流
    if (detail.IsLive)
    {
        _eventSubscription?.Dispose();  // 取消上一个订阅
        _eventSubscription = SubscribeWorkflowEventsAsync(taskId);
    }
    else
    {
        _eventSubscription?.Dispose();
        _eventSubscription = null;
    }
}

private IAsyncDisposable SubscribeWorkflowEventsAsync(string taskId)
{
    var cts = new CancellationTokenSource();
    _ = Task.Run(async () =>
    {
        await foreach (var evt in _executor.SubscribeAsync(taskId, cts.Token))
        {
            Dispatcher.UIThread.Post(() => TaskDetailView.ApplyEvent(evt));
        }
    });
    return new CancellationDisposable(cts);
}
```

### 7.6 阶段 5B：基于 SDK Checkpoint 的进程内恢复

```csharp
public sealed class WorkflowExecutor
{
    private readonly InMemoryCheckpointManager _checkpointManager = new();

    public async Task<string> StartTaskAsync(WorkflowTaskRequest request, CancellationToken ct)
    {
        var run = await InProcessExecution
            .OffThread
            .WithCheckpointing(_checkpointManager)
            .RunStreamingAsync(workflow, initialInput, ct);

        _runs[taskId] = new RunRegistryEntry(run, ...);

```

### 7.7 阶段 5B：HITL 暂停与恢复

```text
1. Workflow.RequirePlanSignoff(true) → 任务运行到计划生成时挂起
2. SDK 发出 RequestInfoEvent
3. IWorkflowExecutor 捕获 → 落 OrchestrationTask(Status=paused) + 更新 LastActiveTimestamp
4. 发 workflow.task.paused 事件 + RequestInfo 详情
5. UI 任务卡片显示批准卡片（只有打开该任务详情的窗口才看到）
6. 用户点击批准 / 修改 / 拒绝 → 调用 IWorkflowExecutor.ResumeAsync(taskId, response)
7. SDK Run.ResumeAsync → 任务继续运行
8. 发 workflow.task.resumed 事件
```

**关键设计点**：

- 浏览器关闭后宿主进程内的 Run 仍存活；用户重新打开浏览器能看到 paused 任务并选择恢复
- 如果宿主进程重启则任务丢失（参见 §7.2 启动扫描）
- 多用户场景下 paused 任务的批准操作以**第一个响应**为准（不在本次设计范围）

- 远程 Workflow（任务在远程服务器执行）→ 不支持
- 工作流的可视化拖拽编辑器 → 不支持，Workflow 是 SDK 内部工作流，不暴露给用户编辑
- 子模式之间的链式组合（Magentic 之后接 GroupChat）→ 不支持，每个任务只用一个子模式
- 任务级别的 token 成本预算与超限打断 → 阶段 6+ 评估
- 任务跨工作区移动 → 不支持，`WorkspaceId` 一旦写入不可变更

如有这些需求，请单独立项评估。
