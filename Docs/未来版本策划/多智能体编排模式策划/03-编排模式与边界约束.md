# 03 - 编排模式与边界约束

> **目的**：定义 Madorin 多智能体编排的双层模式分类、各模式定位、运行边界、共享约束、历史保存策略与事件分流原则。  
> **依赖**：[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md)、[02-当前代码现状.md](./02-当前代码现状.md)。  
> **配套**：[06-工作模式独立模块设计.md](./06-工作模式独立模块设计.md)（Workflow 数据模型 / UI / 与 Chat 联动）、[07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md)（PluginBus 协议 / Memory 插件接入）。

## 1. 双层模式分类

### 1.1 顶层：WorkspaceMode（工作区模式 / 顶层入口）

```text
WorkspaceMode（顶层入口，UI 上是独立 Tab）
  │
  ├── Chat Mode（对话模式 / 现状）
  │     ├── 单 Agent 对话                          ← 默认
  │     ├── @ 子智能体工具调用（ToolDelegation）    ← 阶段 1 增强
  │     └── 客服分流 Handoff                       ← 阶段 3A
  │
  └── Workflow Mode（工作模式 / 新增）
        ├── Discussion（讨论模式）
        │     └── GroupChat                       ← 阶段 3B
        │
        └── Execution（执行模式）
              ├── Magentic                        ← 阶段 4B
              ├── ParallelAnalysis（Concurrent）  ← 阶段 4B
              └── 专家接管 Handoff（待评估）       ← 阶段 5B
```

### 1.2 两层范式的根本差异

| 维度 | Chat Mode | Workflow Mode |
| --- | --- | --- |
| 用户参与节奏 | 每轮都参与（一问一答） | 提交后长时间不参与 |
| 响应时长预期 | 秒级（用户盯着等） | 分钟到小时级（用户切走干别的） |
| 输出形式 | 流式 token，回复完成即结束 | 计划 → 步骤 → 中间产物 → 最终报告 |
| 中间过程可见性 | 默认不可见 | **必须可见**（任务卡片 + 步骤列表） |
| 数据结构 | 线性 `ChatMessage` 列表 | 树/图：任务 → 步骤 → 多 Agent 发言 |
| 取消语义 | 取消当前一轮 | 取消整个任务 / 暂停 |
| 失败处理 | 直接告知 + 用户重问 | 重试 / 重规划 / 部分降级 |
| token 成本预期 | 单轮可预估 | 大幅波动（Magentic 可能 13+ 次调用） |
| session 概念 | 一次聊天会话 | 一次任务执行（OrchestrationTask） |
| 持久化粒度 | 消息级（ChatMessages 表） | 任务级（OrchestrationTask 表） |
| WS 事件 topic | `conversation`（兼容现状） | `workflow`（新增） |
| Memory 入库 | 自动按消息入 ObservationRecord | 自动按 `workflow.task.completed` FinalReport 入库 |

### 1.3 数据与协议彻底隔离

| 资源 | Chat | Workflow |
| --- | --- | --- |
| 数据库表 | `ChatSessions` / `ChatMessages` | `OrchestrationTask` / `OrchestrationStep` / `OrchestrationParticipant` / `OrchestrationMessage`（新增） |
| EventHub 事件 | `conversation.*` | `workflow.*`（新增） |
| PluginBus topic | `conversation` | `workflow`（新增） |
| 历史回放 dispatcher | `PluginBusConversationHistoryDispatcher` | `PluginBusWorkflowHistoryDispatcher`（新增） |
| Memory 入库 EventHandler | `MemoryConversationEventHandler` | `MemoryWorkflowEventHandler`（新增） |
| UI 入口 | "对话"主页 | "工作台"独立 Tab |
| Session 关系 | `ChatSession.Id` | `OrchestrationTask.Id`，可选携带 `SourceSessionId` 指向触发的 Chat 会话 |

详细 schema 与协议见 [06](./06-工作模式独立模块设计.md) 与 [07](./07-事件分流与插件兼容设计.md)。

## 2. SDK 编排模式枚举与执行策略

这是**实现层**的枚举，仅在 Orchestrator 内部使用，不直接暴露给用户。用户看到的是顶层 WorkspaceMode + 子模式选择。

### 2.1 OrchestrationMode

| 枚举值 | 归属顶层模式 | SDK 对应 | 阶段 |
| --- | --- | --- | --- |
| `None` | Chat | `ChatClientAgent.RunStreamingAsync` | 现状 |
| `ToolDelegation` | Chat | `BuildWithSubAgents` + `AsAIFunction` | 现状 + 阶段 1 增强 |
| `HandoffChat` | Chat | `HandoffWorkflowBuilder` + `EnableReturnToPrevious` | 阶段 3A |
| `GroupChat` | Workflow / Discussion | `GroupChatWorkflowBuilder` | 阶段 3B |
| `Magentic` | Workflow / Execution | `MagenticWorkflowBuilder` | 阶段 4B |
| `ParallelAnalysis` | Workflow / Execution | `BuildConcurrent` + 自定义 aggregator | 阶段 4B |
| `HandoffExecution` | Workflow / Execution | `HandoffWorkflowBuilder` + 多步执行 | 阶段 5B（待评估） |

### 2.2 AgentExecutionStrategy（执行策略）

| 枚举值 | 含义 | SDK 对应 |
| --- | --- | --- |
| `Sequential` | 串行（默认） | `BuildSequential` 或 Workflow 内部默认 |
| `Concurrent` | 并行执行多个 Agent | `BuildConcurrent` |

**关键约束**：

- `Concurrent` **不是** `OrchestrationMode` 枚举值，它是 `Magentic` / `GroupChat` / `ParallelAnalysis` 内部的执行策略选项之一。
- `ParallelAnalysis` 作为 `Workflow / Execution` 下的独立子模式存在，UI 上是任务卡片而非 Chat 气泡；其内部本质是 `BuildConcurrent` + 自定义 aggregator。
- `HandoffChat` / `HandoffExecution` 都不支持 `Concurrent`（本质是串行链路）。

### 2.3 模式 × 策略矩阵

| 模式 \ 策略 | Sequential | Concurrent |
| --- | --- | --- |
| `None` | ✅ 默认 | ❌ N/A |
| `ToolDelegation` | ✅ 默认 | ❌ 受 `AsAIFunction` 单 session 限制（SDK 笔记 §7） |
| `HandoffChat` | ✅ | ❌ |
| `GroupChat` | ✅ 默认 | ⚠️ 阶段 5B 起：单轮内多 Agent 可并行（需自定义 Manager） |
| `Magentic` | ✅ 默认 | ⚠️ 阶段 5B 起：分派阶段可并行 |
| `ParallelAnalysis` | ❌ | ✅ 强制 Concurrent |
| `HandoffExecution` | ✅ | ❌ |

## 3. Chat 模式：各 OrchestrationMode 定位

### 3.1 `None`：单 Agent 普通对话

**定位**：项目现状，不动。

**实现**：`AIAgentFactory.Build(agent, provider, model)` → `ChatClientAgent.RunStreamingAsync`。

### 3.2 `ToolDelegation`：@ 子智能体工具模式

**定位**：Chat 内的轻量多 Agent，"主 Agent 调专家工具"的体验仍是一问一答。

**适用**：

- 用户显式 @ 1-3 个子智能体
- 任务可分解为简单的"专家咨询"
- 不需要重规划与失败检测

**实现**：`AIAgentFactory.BuildWithSubAgents` + `SubAgentContextProvider`（项目当前实现）+ 阶段 1 增强的 `OrchestrationInstructionsProvider`（Coordinator instructions）。

**边界**：

- 主模型必须支持 FunctionCall（`InteractionCapabilities.FunctionCall`）；否则 §2.3 阶段 0 改造后会给用户提示并退回单 Agent 模式。
- 子 Agent 不带历史/记忆/技能（`BuildSubAgent` 注释明示）。
- 主 Agent 工具调用链会进入主会话历史（ChatMessages），UI 默认只显示最终 assistant 文本。
- 事件 / 历史 / Memory 入库**完全走 `conversation` topic**，与现状一致。

### 3.3 `HandoffChat`：客服分流场景

**定位**：Chat 内的辅助模式，triage 智能体识别问题域后路由到专家，**用户感知不到换人，体验仍是 chat**。

**适用**：

- 客服式分流（计费 / 技术 / 售后）
- 主智能体能力不足时转给备用智能体
- 工具能力隔离

**不适用**：

- 多任务拆解（用 Workflow / Magentic）
- 长时执行（用 Workflow / HandoffExecution，阶段 5B 待评估）
- 多智能体会议（用 Workflow / GroupChat）

**实现路径**：

```csharp
using Microsoft.Agents.AI.Workflows;

#pragma warning disable MAAIW001
var workflow = AgentWorkflowBuilder
    .CreateHandoffBuilderWith(triageAgent)
    .WithHandoffs(triageAgent, billingAgent, technicalAgent)
    .EnableReturnToPrevious()  // 后续轮次直接路由到上次专家
    .Build();
#pragma warning restore MAAIW001

AIAgent handoffAgent = workflow.AsAIAgent(...);
// 仍走 _agent.RunStreamingAsync 主链路，发 conversation.* 事件
```

**边界**：

- ⚠️ Handoff 仍标记 `[Experimental("MAAIW001")]`。
- 通过 `handoff_to_<agent_id>` 工具调用实现，主模型必须支持 FunctionCall。
- 默认 Handoff 提示已经够用，通常无需 `WithHandoffInstructions(...)`。
- 事件 / 历史 / Memory 入库**走 `conversation` topic**——Handoff 的本质是"一段对话内换了响应者"，仍是 Chat 范式。
- 切换瞬间不暴露给用户（除非明确希望让用户感知，由 instructions 控制）。

## 4. Workflow 模式：各子模式定位

> **统一约束**：所有 Workflow 子模式都通过 `Workflow.AsAIAgent()` 包装为 AIAgent（[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §6），但**不走 `_agent.RunStreamingAsync` 主链路**。Workflow 任务通过独立的 `IWorkflowExecutor` 服务调度，发 `workflow.*` 事件，落 `OrchestrationTask` 表。详见 [06](./06-工作模式独立模块设计.md)。

### 4.1 `GroupChat`：讨论模式

**定位**：多角色围绕问题开会、形成结论；用户提交话题后看会议纪要。

**适用**：

- 用户明确要求"讨论一下方案"
- 需要不同角色提出意见（架构师 / 实现者 / 测试者 / 风险审查者）
- 讨论完后才执行，而非边讨论边执行

**实现路径**：

```csharp
using Microsoft.Agents.AI.Workflows;

var workflow = AgentWorkflowBuilder
    .CreateGroupChatBuilderWith(agents => new RoundRobinGroupChatManager(agents)
    {
        MaximumIterationCount = 3   // ← 必须显式覆盖默认 40
    })
    .AddParticipants(architect, implementer, tester, reviewer)
    .WithName("DesignReview")
    .Build();

AIAgent groupChatAgent = workflow.AsAIAgent(
    name: "GroupChatOrchestrator",
    description: "...");
```

**边界**：

- ⚠️ `MaximumIterationCount` 默认 **40**（SDK 笔记 §4），必须显式覆盖。
- 第一阶段建议最大 3 轮，每个智能体只发 1-2 轮。
- 最终只输出主持智能体或最后一条消息作为 `OrchestrationTask.FinalReport`，**所有中间消息只入 `OrchestrationMessage` 表**，不进 ChatMessages。
- 讨论阶段默认不启用高风险写操作工具（详见 §7.2）。
- PluginBus 只发 `workflow.task.started` / `workflow.step.completed` / `workflow.task.completed`，**不发流式 delta**（决策 3-A）。

#### 4.1.1 小组成员声明优先级

由于第一阶段不允许扩展 `AgentEntity`（[02-当前代码现状.md](./02-当前代码现状.md) §6.1），小组成员按以下优先级解析：

| 优先级 | 来源 | 实施阶段 |
| --- | --- | --- |
| 1 | 用户在 Workflow 任务表单中显式选择的成员 | 阶段 3B |
| 2 | `workspace/.cortana/groupchat.json` 工作区配置 | 阶段 3B |
| 3 | `OrchestrationGroupEntity` 独立实体 | 阶段 5+（待评估） |

`groupchat.json` 示例：

```json
{
  "groups": [
    {
      "id": "design-review",
      "moderator": "<agentId>",
      "members": ["<agentId>", "<agentId>"]
    }
  ]
}
```

### 4.2 `Magentic`：复杂任务执行模式

**定位**：LLM 管理者驱动的"计划-执行-检查-总结"循环，是 Workflow 模式的旗舰。

**适用**：

- 任务需要拆分为多个子任务
- 需要根据中间结果动态调整计划
- 用户问的是"做一件复杂的事情"而不是"咨询专家"

**实现路径**：

```csharp
using Microsoft.Agents.AI.Workflows;

#pragma warning disable MAAIW001
var workflow = new MagenticWorkflowBuilder(managerAgent)
    .AddParticipants(researcher, coder, tester)
    .WithMaxRounds(5)              // ← 第一阶段必须显式
    .WithMaxResets(2)
    .WithMaxStalls(3)              // 默认值，明示
    .RequirePlanSignoff(false)     // ← 阶段 4B 必须 false（HITL 在阶段 5B 才接入）
    .Build();
#pragma warning restore MAAIW001

AIAgent magenticAgent = workflow.AsAIAgent(
    name: "MagenticOrchestrator",
    description: "...");
```

**边界**：

- ⚠️ `RequirePlanSignoff` 默认 **true**，不显式设 false 会卡死等待人工批准（SDK 笔记 §3.1）。
- ⚠️ 单任务至少 13 次 LLM 调用（SDK 笔记 §3.2），UI 必须在提交前给一次性 token 成本提示。
- 阶段 4B 不支持 HITL，第一阶段最大轮次建议 5、最大重置 2。
- Manager 计划文本、Member 中间响应仅入 `OrchestrationMessage` 表，**绝不进 ChatMessages**。
- PluginBus 发 `workflow.task.planned` / `workflow.task.replanned` / `workflow.step.completed` 等事件，Memory 插件仅订阅 `workflow.task.completed` 自动写入最终 FinalReport。

### 4.3 `ParallelAnalysis`：独立并行分析

**定位**：用户有 N 个独立小问题想同时问；或对同一个问题想要 N 个不同视角的并行回答。

**适用**：

- 多专家独立分析（物理 + 化学 + 生物视角同一现象）
- 多版本草稿对比
- A/B 方案并行评估

**实现路径**：

```csharp
using Microsoft.Agents.AI.Workflows;

var concurrent = AgentWorkflowBuilder.BuildConcurrent(
    agents: members,
    aggregator: messages =>
    {
        // BuildConcurrent 默认返回每个 Agent 最后一条；自定义合并为单条聚合文本
        var combinedText = string.Join("\n\n", messages
            .Where(list => list.Count > 0)
            .Select(list => $"--- {list[^1].AuthorName} ---\n{list[^1].Text}"));
        return new List<ChatMessage> { new(ChatRole.Assistant, combinedText) };
    });

AIAgent parallelAgent = concurrent.AsAIAgent(...);
```

**边界**：

- ⚠️ `BuildConcurrent` 默认 aggregator 返回 `List<ChatMessage>`（每个 Agent 最后一条），**不是单条字符串**（SDK 笔记 §2）；必须自定义 aggregator。
- UI 上是任务卡片，每个 Agent 一栏并列展示，最终合并报告在底部。
- 执行使用 `InProcessExecution.OffThread`（不是 `InProcessExecution.Concurrent`，后者是"多个 run 真正并发"的环境配置，与单 run 内 fan-out 是两件事，详见 SDK 笔记 §12）。
- 同一 Workflow 内 fan-out 由 SDK 保证调度，但**多 Agent 共享 Workflow 内部 session 时不可写主 Chat session**，避免触发 [05-风险与规避.md](./05-风险与规避.md) §风险 3。

### 4.4 `HandoffExecution`：专家接管（阶段 5B 待评估）

**定位**：与 `HandoffChat` 区别在于专家接管后执行多步任务，最终给出汇总报告。

**暂缓理由**：阶段 4B 内可以用 `Magentic` 满足"专家接管 + 多步执行"需求；独立的 `HandoffExecution` 在 5B 之前没有不可替代的用例。

## 5. 历史保存策略（双层模式版）

### 5.1 双层数据流

```text
Chat 模式：
  AiChatHostedService → ChatHistoryDataProvider.StoreChatHistoryAsync
                      → ChatMessages 表（时间戳归一化 + 工具链邻接保证）
                      → conversation.* 事件
                      → Memory.ObservationRecord 表（按消息粒度自动入库）

Workflow 模式：
  IWorkflowExecutor → OrchestrationTask / OrchestrationStep / OrchestrationMessage 表
                    → workflow.* 事件
                    → Memory.ObservationRecord 表（仅 workflow.task.completed 的 FinalReport）
```

两条数据流**互不写对方的表**，互不发对方的事件。

### 5.2 Chat 模式历史保存（不变）

`ChatHistoryDataProvider.StoreChatHistoryAsync(...)` 已经针对工具调用链做了：

- **时间戳归一化**：以"同 session 已存在的最大 `CreatedTimestamp` + 1ms"为基准，按 `ResponseMessages` 原始顺序强制单调递增（[02-当前代码现状.md](./02-当前代码现状.md) §4.2）。
- **孤立 tool 消息告警**。
- **每条消息独立 ID**。

末尾两个副作用：

- `CompactAndReplaceAsync(...)`：段落压缩。
- `GenerateAndUpdateTitleAsync(...)`：首轮 AI 摘要标题，仅 `isnewsession=true` 时触发。

**Workflow 模式不触发**：Workflow 任务结果只入 `OrchestrationTask.FinalReport`，不进 `ChatMessages`，因此不会触发上述两个副作用——压缩与标题逻辑只属于 Chat 主会话。

### 5.3 Workflow 模式历史保存（新增）

`IWorkflowExecutor` 在任务执行过程中：

| 时机 | 落地 |
| --- | --- |
| 任务开始 | `INSERT OrchestrationTask` + 发 `workflow.task.started` |
| 计划生成（Magentic） | `INSERT OrchestrationStep(action="plan")` + 发 `workflow.task.planned` |
| 单步开始 | `INSERT OrchestrationStep(action=...)` + 发 `workflow.step.started` |
| 单步完成 | `UPDATE OrchestrationStep` + `INSERT OrchestrationMessage(s)` + 发 `workflow.step.completed` |
| 重规划 | `INSERT OrchestrationStep(action="replan")` + 发 `workflow.task.replanned` |
| 任务完成 | `UPDATE OrchestrationTask(Status, FinalReport, CompletedAt)` + 发 `workflow.task.completed` |
| 任务取消 / 失败 | 同上但 Status 不同，发 `workflow.task.cancelled` / `workflow.task.failed` |

中间步骤**永远不进 ChatMessages**，避免污染 Chat 历史与压缩/标题逻辑。

### 5.4 多智能体并发风险（缓解后）

[02-当前代码现状.md](./02-当前代码现状.md) §4.2 提到的"时间戳归一化在并发下失效"问题，因数据隔离而**根本性缓解**：

- Workflow 任务的中间消息根本不进 `ChatMessages`，不会触发归一化策略。
- 同一 ChatSession 内仍是单线程顺序写入，归一化策略前提成立。
- `OrchestrationMessage` 表是新设计，采用 `(TaskId, StepId, Sequence)` 三元组定序，不依赖时间戳归一化。

详见 [05-风险与规避.md](./05-风险与规避.md) §风险 3（已降级为🟢低）。

### 5.5 中间消息存储策略切换点

| 阶段 | Chat 中间消息 | Workflow 中间消息 |
| --- | --- | --- |
| 阶段 0-1 | 主 Agent 工具调用链入 ChatMessages（现状） | N/A |
| 阶段 2A | 同上 | N/A |
| 阶段 3A | 同上 + Handoff 切换记录入 ChatMessages | N/A |
| 阶段 3B 起 | 同上 | **入 OrchestrationMessage 表** |

### 5.6 AgentOrchestrationResult 字段去向（双层模式版）

#### Chat 模式（阶段 2A+）

| 字段 | UI 是否可见 | 是否入库 | 默认去向 |
| --- | --- | --- | --- |
| `FinalText` | ✅ 流式输出 | ✅ ChatMessages | 经 `IAiOutputChannel` 广播；由 `ChatHistoryDataProvider` 持久化 |
| `Steps` | ❌ 默认不可见 | ❌ 不入库 | 日志（`logger.LogDebug`）+ 内存 |
| `UsedAgentIds` | ✅ 附在 FinalText 尾部 | ❌ 不入库 | 日志 + 内存 |
| `Warnings` / `Failures` | ✅ 必要时附在尾部 | ❌ 不入库 | 日志（`logger.LogWarning`）+ 内存 |
| `TokenUsage` | ✅ 累计后回写 `AIAgentFactory.TokenUsageChanged` | ❌ 不入库 | Orchestrator 聚合后上报 |

#### Workflow 模式（阶段 3B+）

| 字段 | UI 是否可见 | 是否入库 | 默认去向 |
| --- | --- | --- | --- |
| `FinalReport` | ✅ 任务卡片"最终报告"区 | ✅ `OrchestrationTask.FinalReport` | 发 `workflow.task.completed` 事件；Memory 自动写入 ObservationRecord |
| `Steps` | ✅ 任务卡片"执行过程"折叠区 | ✅ `OrchestrationStep` 表 | 每步发 `workflow.step.completed` 事件 |
| `Messages` | ✅ 每步展开后可见 | ✅ `OrchestrationMessage` 表 | 不发 PluginBus 流式（决策 3-A） |
| `Participants` | ✅ 任务卡片顶部"参与者"列表 | ✅ `OrchestrationParticipant` 表 | `workflow.task.started` 事件携带 |
| `Warnings` / `Failures` | ✅ 任务卡片"告警"区 | ✅ `OrchestrationStep.ErrorMessage` | `workflow.task.failed` 携带 |
| `TokenUsage` | ✅ 任务卡片"成本"区 | ✅ `OrchestrationStep.TokenUsage` | 每步累计 |

## 6. 子智能体工具与技能边界

### 6.1 工具边界（不变）

当前系统已经具备工具分配能力（[02-当前代码现状.md](./02-当前代码现状.md) §7），不需要在多智能体编排层重复设计：

```text
工具能力边界 = 全局插件（Scope=Global ∧ EnabledIds 包含）
              + 当前智能体绑定插件（AgentEntity.EnabledPluginIds）
              + 当前智能体绑定 MCP（AgentEntity.EnabledMcpServerIds）
```

Chat 与 Workflow 共用同一套工具绑定结果，**Orchestration 层只记录实际使用者、步骤和风险，不重新管理插件/MCP 绑定**。

### 6.2 安全约束

- 高风险工具应优先绑定到专用智能体，而不是设为全局工具。
- 主智能体调用子智能体时，应在最终结果或后续 Step 记录中标记实际执行者。
- 后续如引入任务级授权，可以在现有工具绑定结果之上再做临时收窄。
- 发布、删除、批量写入、外部网络、命令执行等高风险行为仍建议由宿主侧确认或审计。
- **GroupChat 讨论阶段**：默认不启用高风险写操作工具，避免讨论本身意外修改外部状态。
- **Magentic Manager Agent**：默认不绑定执行类工具，Manager 只做规划和总结。

### 6.3 技能边界

技能边界与工具不同：

- 技能通过 `AgentSkillsProvider(skillDirs)` 从用户技能目录和工作区技能目录统一注入。
- **`AgentSkillsProvider` 是 Microsoft.Agents.AI 1.x SDK 的 sealed 类**（[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §9）。
- 当前主智能体能够看到同一套技能目录，无法可靠约束"某个技能只能由某个智能体使用"。

技能约束作为后续课题：

| 阶段 | 处理 |
| --- | --- |
| 阶段 1-4 | 接受技能全局可见现状，通过 instructions 引导智能体选择合适技能 |
| 阶段 5+ | 评估自定义 `AgentSkillsSource` 子类，按 AgentId 过滤（`AgentSkillsSource` 是 abstract，可继承） |
| 阶段 5+ | 再考虑技能元数据 `allowedAgentIds` / `deniedAgentIds`、UI 管理、技能冲突 |

## 7. 附件与多模态输入传递策略

当前 `AiChatHostedService` 会把附件转换为 `AIContent` 发给主 Agent。

### 7.1 阶段化传递策略

```text
阶段 1（Chat / ToolDelegation）：附件直接进入主 Agent
  └── 子智能体是否看到附件，由主 Agent 在调用工具时自行摘要或转述
       └── 工具签名包含 attachmentPaths / attachmentDescriptions（详见 §7.2）

阶段 3B 起（Workflow）：附件通过 OrchestrationTask.InitialInput 显式携带
  └── IWorkflowExecutor 决定哪些 Member Agent 可访问哪些附件
       └── 多个 Member 并发时禁止同时写同一文件（[05-风险与规避.md] §风险 9）

阶段 5+：再考虑文件锁、附件权限和资源引用追踪
```

### 7.2 子智能体工具参数 schema（Chat 模式）

由于 `AsAIFunction` 默认只产生**单个 `query: string` 参数**（[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §7），不能通过 options 添加附件参数。

**必须**走自定义委托：

```csharp
using Microsoft.Extensions.AI;

async Task<string> InvokeAgentWithAttachmentsAsync(
    [Description("子智能体要回答的具体问题")] string query,
    [Description("附件绝对路径数组（图片用工作区拷贝后的路径）")] string[]? attachmentPaths,
    [Description("主 Agent 对每个附件的简短描述")] string[]? attachmentDescriptions,
    CancellationToken cancellationToken)
{
    var subSession = await subAgent.CreateSessionAsync(cancellationToken);
    var messages = BuildMessagesWithAttachments(query, attachmentPaths, attachmentDescriptions);
    var response = await subAgent.RunAsync(messages, session: subSession, cancellationToken: cancellationToken);
    return response.Text;
}

var fn = AIFunctionFactory.Create(InvokeAgentWithAttachmentsAsync, new AIFunctionFactoryOptions
{
    Name = subAgent.Name,
    Description = subAgent.Description
});
```

**理由**：

- 主 Agent 看到的图片是 `DataContent`（二进制），如果只让模型"自行转述"，子 Agent 完全拿不到原图。
- 主 Agent 看到的文档是文本路径，如果转述时省略了路径，子 Agent 无法再读原文件。
- 子 Agent 自身的工具（如读文件、读图片）需要绝对路径作为输入参数才能工作。

## 8. 取消、超时、失败降级与编排参数承载位置

### 8.1 基础规则

多智能体编排必须内置失败边界，不能只依赖模型自行停止：

- 每轮编排必须有 `MaxRounds`
- 每个子任务必须有 `Timeout`
- 每次执行最多允许固定数量的子任务
- 用户取消时必须取消所有未完成子任务
- 单个子智能体失败不应默认导致全局失败，除非该步骤被标记为 required
- 最终回复需要说明未完成项、失败原因和降级结果
- Magentic 重规划次数必须受限，避免无限规划
- **主模型未启用 `InteractionCapabilities.FunctionCall` 时**，不应静默丢弃 @ 子智能体；应至少在最终回复中给出一段提示，或退回单 Agent 模式

### 8.2 取消语义双层化

| 顶层模式 | 取消按钮含义 | 实现 |
| --- | --- | --- |
| Chat | 取消当前一轮 | 沿用现有 `CancellationTokenSource` 链路 |
| Workflow | 取消整个任务 | `IWorkflowExecutor.CancelTaskAsync(taskId)` → 取消 SDK Workflow run → 发 `workflow.task.cancelled` |

Workflow 任务进行中**不能像 Chat 一样插话**；只能"取消"或"等待 HITL 节点（阶段 5B+）"。

### 8.3 编排参数承载位置

在不允许第一阶段改 `AgentEntity` 前提下：

| 参数 | 阶段 1 承载位置 | 阶段 2A / 3B 承载位置 | 阶段 5+ |
| --- | --- | --- | --- |
| `MaxRounds` / `MaxSubTasks` / `Timeout` | `AgentOrchestrator` 内置常量 | `SystemSettingsService` 系统配置项（`orchestration.maxRounds` 等） | `AgentEntity` 或独立 `OrchestrationProfileEntity` |
| 默认 Chat 模式策略 | Coordinator instructions 内置规则 | `SystemSettingsService` + 用户在 Chat 输入时的 @ 提及 | `AgentEntity.DefaultOrchestrationMode` |
| Workflow 子模式选择 | N/A | 用户在 Workflow Tab 任务表单显式选择 | + Chat→Workflow 智能推荐 |
| GroupChat 小组成员 | `@` 提及列表（Chat 内）/ 任务表单（Workflow 内） | + `workspace/.cortana/groupchat.json` | `OrchestrationGroupEntity` |
| Magentic 计划批准（HITL） | **强制 `RequirePlanSignoff(false)`** | 同左 | UI 增加批准卡片 + `Run.ResumeAsync` |
| 高风险工具白名单 | 复用现有插件 / MCP 绑定 | `SystemSettingsService` + 每次任务运行期收窄 | 独立授权模型 |

所有阈值默认值由 `AgentOrchestrator` / `IWorkflowExecutor` 在第一阶段以常量形式提供，第二阶段开始迁移到 `SystemSettingsService`，避免硬编码遗留。

## 9. 参与智能体记录

### 9.1 双层模式策略

| 顶层模式 | 阶段 | 来源 | 可信度 |
| --- | --- | --- | --- |
| Chat | 阶段 1 | Coordinator instructions 要求模型在最终回复尾部简短说明 | ⚠️ 模型自述，可能错报 |
| Chat | 阶段 2A+ | 宿主侧 `AgentOrchestrationResult.UsedAgentIds`（来自实际工具调用记录） | ✅ 真实 |
| Workflow | 阶段 3B+ | 宿主侧 `OrchestrationParticipant` 表（任务启动时确定 + 步骤执行时补充） | ✅ 真实 |

### 9.2 阶段 1 不可靠的原因

模型可能：

- 没调用某个子智能体却声称调用了
- 调用了工具但漏写参与者
- 工具失败后仍声称成功

**因此从阶段 2A 开始**，应由 Orchestration 层根据实际执行步骤生成 `UsedAgentIds`，最终回复 UI 文本中的"参与智能体"应来自宿主侧记录，而不是完全依赖模型文本。

## 10. Chat ↔ Workflow 切换策略

### 10.1 Chat → Workflow

| 时机 | 策略 |
| --- | --- |
| 用户在 Chat 输入复杂任务 | 启发式或 LLM 分类识别后，弹"是否切到工作模式？"卡片（阶段 5+） |
| 用户主动点击"切到工作模式"按钮 | UI 把当前 Chat 输入内容作为 `OrchestrationTask.InitialInput`，跳转到 Workflow Tab 创建新任务表单 |
| 任务记录 `SourceSessionId` | 指向触发的 Chat 会话，方便审计与回溯 |

### 10.2 Workflow → Chat

| 时机 | 策略 |
| --- | --- |
| 任务完成 | UI 在任务卡片提供"附加到对话"按钮 |
| 用户点击 | 把 `OrchestrationTask.FinalReport` 作为一条系统/助手消息插入到原 Chat 会话（如有 `SourceSessionId`）或新建会话 |

- 用户**不能**像 chat 一样随时插话
- 只能"取消"或"等待 HITL 节点（阶段 5B+）"
- UI 显示"任务运行中"占位，并提供轮询进度

## 12. 事件分流原则（与 Memory 插件兼容）

> 这是 Workflow 模式数据隔离的协议层保证。详细协议、字段、入库策略见 [07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md)。

### 12.1 PluginBus topic 分配

| Topic | 现状 | 用途 | 兼容性 |
| --- | --- | --- | --- |
| `conversation` | 已有 | Chat 模式 4 个 `conversation.*` 事件 + history.replay | **不变更字段**，仅扩展新事件不破坏旧的 |
| `memory` | 已有 | `memory.context.supply.*` | 不变更 |
| `model` | 已有 | `model.capability.*` | 不变更 |
| `plugin` | 已有 | 插件管理 | 不变更 |
| `workflow` | **新增** | Workflow 模式 7+ 个 `workflow.*` 事件 + history.replay | 全新协议，无历史包袱 |

### 12.2 Workflow 事件族

```text
workflow.task.started        ← 任务开始（含 mode、members、initialInput）
workflow.task.planned        ← 计划生成（Magentic 专属）
workflow.task.replanned      ← 重规划（Magentic 专属）
workflow.step.started        ← 单步开始（含 agentId、stepIndex）
workflow.step.completed      ← 单步完成（含 output、tokens）
workflow.task.completed      ← 任务结束（含 finalReport、status）
workflow.task.cancelled      ← 任务取消
workflow.task.failed         ← 任务失败
workflow.history.replay      ← 历史回放请求
workflow.history.batch       ← 历史回放批次
workflow.history.completed   ← 历史回放完成
```

**关键设计决策**（用户已确认）：

- **决策 3-A**：流式 token 不走 PluginBus，只在内部 WebSocket 推给 UI。PluginBus 只发 `step.completed` 含完整 output。
- **决策 4-A**：独立 `PluginBusWorkflowHistoryDispatcher`，回放粒度是 Task 级别（不是 Step 级别）。

### 12.3 Memory 插件自动入库（决策 2-B + 决策 5-B）

| 事件 | Memory 入库行为 |
| --- | --- |
| `workflow.task.completed` | ✅ **自动入库**：`FinalReport` 写入 ObservationRecord，`role=assistant`、`eventType=workflow.task.completed`、`agentId=ManagerAgentId`、`content=FinalReport` |
| `workflow.task.*` 其他 | ❌ 不入库（避免噪音） |
| `workflow.step.*` | ❌ 不入库 |
| `conversation.*`（Chat 模式） | ✅ 入库（现状不变） |

详细字段映射见 [07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md) §4。

### 12.4 版本协调要求

由于用户选择决策 5-B，Memory 插件需在 Workflow 上线前同步更新：

- **阶段 4 之前**：宿主侧完成 `workflow` topic 与 dispatcher，Memory 插件完成 `MemoryWorkflowEventHandler`。
- **阶段 4 验收清单**：宿主与 Memory 插件版本号同步发版，避免插件断开重连后协议不匹配。

详见 [04-实施阶段.md](./04-实施阶段.md) Track B 阶段 4 与 [07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md) §6。
