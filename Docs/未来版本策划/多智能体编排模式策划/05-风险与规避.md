> **目的**：列出多智能体编排引入过程中的全部已知风险，明确触发条件、影响范围、规避措施与所属阶段。

## 风险等级图例

- 🔴 **高**：会导致功能完全无法使用或数据损坏，必须在对应阶段开始前规避
- 🟡 **中**：会降低体验或局部失效，应在对应阶段内规避
- 🟢 **低**：可观察可监控，按需要在后续阶段规避

## 风险 1：Magentic 默认 RequirePlanSignoff = true（🔴 高）

### 触发

直接调用 `new MagenticWorkflowBuilder(manager).AddParticipants(...).Build()`，**未显式调用 `.RequirePlanSignoff(false)`**。

### 影响

Workflow 在生成初始计划后会发出 `MagenticPlanReviewRequest`，然后**永久挂起等待 `MagenticPlanReviewResponse`**。项目侧 UI 在阶段 4B 之前不实现 HITL，会出现"用户提交后毫无响应"的死锁。

### 出处

[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §3.1：`MagenticWorkflowBuilder._requirePlanSignoff = true` 是默认值。

### 规避

**阶段 4B** 实现 Magentic 时**必须**：

```csharp
var workflow = new MagenticWorkflowBuilder(manager)
    .AddParticipants(team)
    .RequirePlanSignoff(false)   // ← 强制 false 直到阶段 5B
    .Build();
```

`WorkflowExecutorOptions.MagenticRequirePlanSignoff` 默认为 `false`，且阶段 4B 期间禁止前端配置项暴露此开关。阶段 5B 接入 HITL 时再统一开放。

## 风险 2：GroupChatManager 默认 40 轮（🔴 高）

### 触发

构造 `RoundRobinGroupChatManager(agents)` 或自定义 `GroupChatManager` 后**未覆盖 `MaximumIterationCount`**。

### 影响

GroupChat 默认会跑 **40 次** Agent 调用，token 消耗失控、响应时间不可接受、UI 假死。

### 出处

[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §4：`GroupChatManager.MaximumIterationCount = 40`。

### 规避

**阶段 3B** 实现 GroupChat 时**必须**：

```csharp
var manager = new RoundRobinGroupChatManager(agents)
{
    MaximumIterationCount = _options.GroupChatMaxIterations  // 第一阶段建议 = 3
};
```

阶段 3B 在代码评审清单中强制检查"是否显式设置 MaximumIterationCount"，未设置不允许合并。

## 风险 3：ChatHistoryDataProvider 时间戳归一化在并发下失效（🟢 低，已大幅缓解）

> **状态变更**：用户决策双层模式分离后，本风险从 🔴 高降为 🟢 低。

### 触发（缓解后）

只剩唯一一种触发场景：**违反数据隔离原则**——把 Workflow 内部成员 Agent 中间消息错误地写入 `ChatMessages` 表。

### 影响

`StoreChatHistoryAsync` 的"基于已有最大 `CreatedTimestamp` + 1ms"策略只在单线程顺序写入下成立。多线程同时计算 `baseTs` 会：

- 产生重复时间戳（可能让 `assistant tool_calls` 与 `tool` 顺序错乱）
- 破坏 OpenAI 协议要求的工具调用对邻接关系
- 历史恢复后无法继续对话（OpenAI 拒收乱序 tool 消息）

### 出处

[02-当前代码现状.md](./02-当前代码现状.md) §4.2、SDK 笔记 §13（"AgentSession 跨 Agent 不可复用"是 SDK 强约束）、[03-编排模式与边界约束.md](./03-编排模式与边界约束.md) §5。

### 缓解原理

阶段 3B 起的双层数据流隔离：

| 数据流 | 写入表 | 写入特征 |
| --- | --- | --- |
| Chat 模式 | `ChatMessages` | 单线程顺序写（每条 turn 独占） |
| Workflow 模式 | `OrchestrationMessage` | `(TaskId, StepId, Sequence)` 三元组定序，不依赖时间戳归一化 |

Workflow 任务的多 Agent 中间消息**根本不进 `ChatMessages`**，归一化策略前提保持成立；并发风险消失。

### 仍需规避（残留风险）

- 即使数据隔离已落地，代码评审仍需检查：禁止 `IWorkflowExecutor` 内部代码直接写 `ChatMessages` 表
- 阶段 5B "Workflow → Chat 回灌"必须由用户显式触发，且**单条**写入 `ChatMessages`，不并发
- Chat 模式 `BuildConcurrent` / 真正并行场景仍受此约束，但 Chat 模式不允许出现真正并行（Concurrent 策略只在 Workflow 内使用）

## 风险 4：AIAgentFactory.ChatClient 工厂级状态被串行/并行覆盖（🟡 中）

### 触发与影响（分类）

| 场景 | 当前是否会发生 | 影响 |
| --- | --- | --- |
| **串行多 Agent**（切换提供商/模型/@子智能体导致 ChatClient 重建） | ✅ 已经发生 | 上一个 Agent 的 token 用量被新 Agent 覆盖；UI 进度条只反映最近一次调用 |
| **Workflow 内并行多 Agent**（阶段 4B 起 Magentic / ParallelAnalysis 内部 fan-out） | 阶段 4B 起会发生 | 多线程同时 `Interlocked.Exchange` 同一字段，进度条数据完全乱序 |

### 出处

[02-当前代码现状.md](./02-当前代码现状.md) §5.1：`_lastInputTokens` / `_maxContextTokens` 是工厂级字段，每次 `CreateTrackingClient` 都覆盖。

### 规避

- **串行覆盖**（阶段 1-2A 已存在）：第一阶段 UI 进度条接受"显示最近一次调用"的现状；阶段 2A 起在 `AgentOrchestrationResult.TokenUsage` 中独立累加每个成员 Agent 的 token，不再让 UI 依赖工厂级单值。
- **并行覆盖**（阶段 4B 起）：Workflow 模式下 `IWorkflowExecutor` 内置 `TokenAggregator`，从 SDK 事件流提取 `UsageDetails` 累加到 `OrchestrationStep.TokenUsage`，最终统一上报到 `AIAgentFactory.TokenUsageChanged`。
- 并发成员 Agent 不直接写工厂级 `ChatClient`，统一通过 `IWorkflowExecutor` 收集后再回写。
- Chat 模式与 Workflow 模式的 token 统计独立，UI 分别展示。

## 风险 5：长期记忆按主会话注入，子智能体串读（🟡 中）

### 触发

阶段 3B+ 想为 Workflow 内成员 Agent 启用 `LongMemoryContextProvider`，但仍复用 Workflow 任务级别的 `agentid` / `sessionid` / `traceid`。

### 影响

`LongMemoryContextProvider` 使用 `StateBag.agentid` 作为记忆 owner key。如果 Workflow 任务下所有成员 Agent 共用同一组 StateBag 键，多个成员会读到同一份记忆，违反"每个 Agent 独立长期记忆"预期。

### 出处

[02-当前代码现状.md](./02-当前代码现状.md) §4.3 / §9.5。

### 规避

- 阶段 1-3B：Workflow 内成员 Agent **不带 memory**（沿用 `BuildSubAgent` 当前行为），不修改。
- 阶段 4B+：如需为成员 Agent 启用长期记忆，必须为每个成员 Agent 创建独立 session，并在子 session 的 StateBag 中显式覆盖：
  - `agentid` = 该成员 Agent 自己的 Id（不是 Manager Id）
  - `sessionid` = `OrchestrationTask.Id` 或专门的"成员级 sub session id"
  - `agentname` / `sessiontitle` 同理独立
- 阶段 5+：UI / 系统设置层定义"成员智能体是否允许读写长期记忆"开关，纳入 `AgentEntity` 或 `OrchestrationProfileEntity`。

## 风险 6：UI 与用户预期不一致（🟡 中，分双层模式）

> **状态变更**：双层模式后，Workflow 模式自带任务卡片即"过程可见"。仅 Chat 模式仍存在该风险。

### 6A：Chat 模式（阶段 1-3A）

**触发场景**：Chat 模式内部已多智能体编排（ToolDelegation / HandoffChat），但 UI 只显示最终回复，用户不知道发生了什么。

**影响表现**：

- 用户看到响应慢但不知道为什么
- 用户怀疑系统卡死
- 用户分不清最终回答是单 Agent 还是多 Agent 协作的结果

**规避策略**：

- 阶段 1：在最终回复中附简短"参与智能体"列表（模型自述）
- 阶段 2A：由宿主侧 `UsedAgentIds` 记录真实参与智能体
- 阶段 3A：HandoffChat 切换瞬间不暴露给用户（除非 instructions 明确要求让用户感知）
- **不要**一开始就改 Chat 消息气泡结构

### 6B：Workflow 模式（阶段 3B+）

**当前状态**：

✅ **设计已规避**。Workflow 模式从一开始就以任务卡片为唯一 UI 形式：

- 任务卡片明示"运行中"/"已完成"/"已失败"状态
- 折叠式"执行过程"区域展示每个 Step 的 Agent / 起止时间 / 输出摘要
- Magentic 任务在用户提交前给一次性 token 成本提示
- 进度通过 WebSocket workflow 事件实时刷新

详见 [06-工作模式独立模块设计.md](./06-工作模式独立模块设计.md) §5 UI 设计。

## 风险 7：误判工具边界导致重复设计（🟡 中）

### 触发

多智能体编排层重新设计一套独立工具权限模型。

### 影响

- 与现有插件管理、系统设置和智能体绑定逻辑重复甚至冲突
- 同一插件可能被注入两次（全局 + 编排层指派）
- 高风险工具的开关分散在多处

### 出处

[02-当前代码现状.md](./02-当前代码现状.md) §7。

### 规避

- 第一阶段直接复用现有工具绑定机制（全局插件 + 智能体绑定 + MCP）
- 公共工具使用全局插件或宿主公共能力
- 特殊工具绑定到专用智能体
- Chat 模式 `IAgentOrchestrator` 与 Workflow 模式 `IWorkflowExecutor` 都只**记录**实际使用者、步骤和风险，不重新管理插件/MCP 绑定
- 后续如需任务级授权（阶段 6 收口），只在现有绑定结果之上做临时收窄

## 风险 8：技能无法按智能体约束（🟢 低）

### 触发

阶段 1-4B 接受技能全局可见现状。

### 影响

- 不同主智能体看到同一套技能集合
- 不能像插件和 MCP 一样精确限制到某个智能体
- 高敏感技能（如"读取系统密码"）一旦放入技能目录，所有主 Agent 都能用

### 出处

`AgentSkillsProvider` 是 Microsoft.Agents.AI 1.x SDK 提供的 **`sealed`** 类（[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §9），不能继承。

### 规避

- 阶段 1-4B：接受技能全局可见现状
- 通过 Agent instructions 约束智能体优先使用符合角色的技能
- 阶段 5+：评估自定义 `AgentSkillsSource` 子类（`AgentSkillsSource` 是 `abstract`，可继承），在 `GetSkillsAsync` 里按 AgentId 过滤
- 阶段 5+：再考虑技能元数据 `allowedAgentIds` / `deniedAgentIds`、UI 管理、技能冲突
- **不**把技能隔离作为多智能体编排第一阶段目标

## 风险 9：附件和文件并发访问冲突（🟢 低，已部分缓解）

> **状态变更**：双层模式 + Workflow 数据隔离后，此风险显著降低。

### 触发（缓解后）

仅在 Workflow 模式下，多个成员 Agent 同时**写**同一个附件路径时仍存在风险。读操作不冲突。

### 影响

- 文件锁冲突（Windows 平台尤其严重）
- 写入竞争导致附件内容损坏
- 资源引用追踪丢失

### 规避

- Chat 模式：附件只直接进入主 Agent；子智能体通过主 Agent 摘要或明确任务描述间接获得附件信息——**不会触发此风险**
- Workflow 模式（阶段 3B+）：禁止多个成员 Agent 同时写同一文件
- 写操作必须由 Manager Agent 串行调度（Magentic 内置）
- ParallelAnalysis 模式下，成员 Agent 默认只读附件，不允许写
- 阶段 5+ 引入文件锁、资源引用追踪或工作副本机制

## 风险 10：ChatClientAgentOptions.ThrowOnChatHistoryProviderConflict（🟢 低）

### 触发

未来接入 OpenAI Responses API 或其他自管历史的服务，且仍配置 `ChatHistoryProvider`。

### 影响

`ChatClientAgent` 构造时直接抛异常，拒绝创建 Agent。

### 出处

[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §10：`ThrowOnChatHistoryProviderConflict` 默认 **true**。

### 规避

- 项目当前使用 ChatCompletions 路径，**不会触发**此风险
- 未来切换到 Responses API 时：
  - 在 `AIAgentFactory` 里检测 `provider.UsesServiceManagedHistory`
  - 命中时显式设置 `ChatClientAgentOptions.ThrowOnChatHistoryProviderConflict = false`
  - 或不传 `ChatHistoryProvider`
- 在 csproj 添加 `<NoWarn>$(NoWarn);MAAI001</NoWarn>` 屏蔽 Experimental 警告

## 风险 11：AIContextProvider StateKeys 重名（🟡 中）

### 触发

为多个 Agent 注入同类型 Provider（如每个成员 Agent 一个独立的 `OrchestrationInstructionsProvider`），但没有重写 `StateKeys`。

### 影响

`ChatClientAgent` 构造时调用 `ValidateAndCollectStateKeys`，**直接抛异常**：

> Two or more AIContextProviders share the same StateKey "OrchestrationInstructionsProvider".

### 出处

[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §8。

### 规避

凡是可能多实例的 Provider，必须重写 `StateKeys` 加入 AgentId 等区分维度：

```csharp
public override IReadOnlyList<string> StateKeys => [$"OrchestrationInstructions_{_agentId}"];
```

阶段 1 的 `OrchestrationInstructionsProvider` 默认只在主 Agent 上注入，不会触发；但阶段 3B+ 给每个 Workflow 成员注入时必须注意。

## 风险 13：Magentic 单任务 token 成本（🟡 中）

### 触发

任意 Magentic 任务执行。

### 影响

单次 Magentic 任务最少 13 次 LLM 调用（Manager 算 facts → plan → 每轮 progress → final answer，加上每个成员各自的执行调用）。一个用户问"总结一下我们项目的 README"如果走 Magentic，可能跑出 5-10 倍 token 成本。

### 出处

[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §3.2。

### 规避

- 阶段 4B 启用 Magentic 时：**UI 在用户提交前给一次性 token 成本提示**："本次任务将启动多智能体协作，预计消耗 X 倍 token，预估时长 Y 分钟。是否继续？"
- 双层模式后这个提示不会出现在 Chat 模式（避免简单对话被误推到 Magentic）；只在用户主动选择 Workflow / Execution / Magentic 子模式时弹出
- `WorkflowExecutorOptions.MaxRounds = 5` + `MagenticMaxResets = 2` 强制上限
- Coordinator instructions 要求 Manager Agent 在简单问题上**主动退化**为单 Agent 回答（不强制走完整 Magentic 流程）
- 阶段 5+ 在 UI 增加 token 预算配置和"超出预算时打断"开关

## 风险 14：AsAIFunction 默认单参数限制（🟡 中）

### 触发

直接使用 `subAgent.AsAIFunction()` 而不通过自定义委托。

### 影响

- 子 Agent 拿不到附件路径
- 子 Agent 拿不到主 Agent 的上下文摘要
- 主 Agent 转述时容易丢信息

### 出处

[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §7：`AsAIFunction` 默认产生 `query: string` 单参数。

### 规避

阶段 1 `BuildWithSubAgents` 改造：用 `AIFunctionFactory.Create` + 自定义委托构造工具，支持 `attachmentPaths` / `attachmentDescriptions` 等扩展参数（详见 [03-编排模式与边界约束.md](./03-编排模式与边界约束.md) §7.2）。

## 风险 15：AgentSession 跨 Agent 复用（🔴 高）

### 触发

把主 Agent 的 `AgentSession` 直接传给子 Agent 的 `RunAsync`。

### 影响

SDK 文档明确：

> An AgentSession may not be reusable across different agents, since each agent may add different behaviors to the AgentSession it creates.

具体表现：

- 子 Agent 可能拿到主 Agent 的 ChatHistoryProvider，污染主历史
- 多 Agent Provider 的 StateKeys 冲突
- Workflow 内部 session 状态被错位

### 出处

[01-SDK源码研究笔记.md](./01-SDK源码研究笔记.md) §13。

### 规避

- 项目当前 `BuildSubAgent` 不传 session，**符合此约束**
- 阶段 1 `AsAIFunction` 不传 session，让 SDK 默认每次调用创建新 session
- 阶段 3B+ Workflow.AsAIAgent 内部由 SDK 自动管理 session，不要手动传
- 代码评审清单：禁止 `subAgent.RunAsync(... session: someSharedSession ...)` 模式
- 阶段 5B HITL 接入时尤其注意：`Run.ResumeAsync` 是对 Workflow run 的恢复，不要错把它当成对 Agent session 的恢复

## 风险 16：Memory 插件版本兼容性（🔴 高，新增）

> 用户决策 5-B 引入此风险：Memory 插件需在 Workflow 上线前同步更新。

### 触发

宿主升级到阶段 4B 版本（包含 workflow topic 与 `workflow.task.completed` 事件），但部署环境的 Memory 插件仍是旧版（不订阅 workflow topic、不实现 `MemoryWorkflowEventHandler`）。

### 影响

- ⚠️ Workflow 任务完成事件被 Memory 插件忽略，FinalReport 不入 ObservationRecord
- ⚠️ 用户长期记忆缺失 Workflow 任务结果
- ⚠️ Chat 模式继续工作（仍走 conversation topic），但用户感知"Workflow 不被记住"
- ⚠️ 严重时（运维误升级）数据缺失静默发生

反向组合（旧宿主 + 新插件）兼容：旧宿主不发 workflow 事件，新插件订阅了也收不到，无副作用。

### 出处

[02-当前代码现状.md](./02-当前代码现状.md) §9.6 / §9.8、[04-实施阶段.md](./04-实施阶段.md) 阶段 4B、[07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md) §6。

### 规避

#### 上线门控

阶段 4B 发版必须把宿主与 Memory 插件作为**同一发布单元**：

1. 宿主与 Memory 插件 PR 同步 review、同步 merge
2. Release 流程把两者打包到同一 Release 发布说明
3. 升级文档明示用户："必须同时升级 Cortana 主程序与 Memory 插件至 vX.Y.Z"

#### 协议层版本握手

握手响应增加协议版本字段：

```json
{
  "type": "connected",
  "clientId": "...",
  "topics": ["conversation", "memory", "model", "workflow"],
  "protocol": "cortana.plugin-bus",
  "version": "1.0.0",
  "capabilities": {
    "workflow.topic.minVersion": "1.1.0"   // ← 新增
  }
}
```

Memory 插件订阅 workflow topic 前比对 `capabilities.workflow.topic.minVersion`：

- 满足 → 订阅 + 处理
- 不满足 → 仅记录警告日志，不订阅 workflow topic（降级为 conversation 模式）

详细握手协议见 [07-事件分流与插件兼容设计.md](./07-事件分流与插件兼容设计.md) §6.2。

#### 启动检查

宿主在阶段 4B 起检测连接的 Memory 插件版本：

- 插件版本 ≥ 4B 要求版本：正常
- 插件版本 < 4B 要求版本：宿主侧显示 SystemNotice："检测到 Memory 插件版本过低，Workflow 任务结果将不会进入长期记忆。请升级 Memory 插件到 vX.Y.Z+"

#### 评审清单

合并阶段 4B 相关 PR 前必须确认：

- [ ] 宿主侧 `workflow.*` 事件已实现并测试通过
- [ ] Memory 插件 `MemoryWorkflowEventHandler` 已实现并测试通过

- [ ] 双方版本号同步
- [ ] 集成测试覆盖：宿主发 workflow.task.completed → Memory 插件入库 ObservationRecord
- [ ] 集成测试覆盖：旧 Memory 插件连接到新宿主时，宿主能正确显示 SystemNotice
