# 07 - 事件分流与插件兼容设计

> **目的**：定义 Madorin 多智能体编排引入"工作模式"后的事件分流协议、PluginBus 端点扩展、Memory 插件接入策略以及版本兼容性处理。  
> **依赖**：[02-当前代码现状.md](./02-当前代码现状.md) §9（事件总线现状）、[03-编排模式与边界约束.md](./03-编排模式与边界约束.md) §12（事件分流原则）、[04-实施阶段.md](./04-实施阶段.md) 阶段 4B（同步发版要求）、[06-工作模式独立模块设计.md](./06-工作模式独立模块设计.md)。  
> **关键决策对应**：
>
> - 决策 1-A：Workflow 事件走新 topic `workflow`
> - 决策 2-B：Memory 自动入库 `workflow.task.completed` 的 FinalReport
> - 决策 3-A：流式 token 不走 PluginBus，只给 UI
> - 决策 4-A：独立 `PluginBusWorkflowHistoryDispatcher` + 独立回放路径
> - 决策 5-B：Memory 插件需在 Workflow 上线前同步更新
> - 决策 6-A：Workflow 任务标题异步生成，独立事件 `workflow.task.title.updated`

## 1. 总体架构

### 1.1 双层事件流

```text
┌─────────────────────────────────────────────────────────────────┐
│ AiChatHostedService (Chat 模式)                                  │
│   │                                                              │
│   ├── publisher.Publish(OnConversationTurnStarted, ...)          │
│   ├── publisher.Publish(OnConversationUserMessage, ...)          │
│   ├── publisher.Publish(OnConversationAssistantDelta, ...)       │
│   └── publisher.Publish(OnConversationTurnCompleted, ...)        │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
                ┌────────────────┐
                │   EventHub     │ ← 进程内事件总线
                └────┬───────────┘
                     │
        ┌────────────┴────────────┐
        ▼                         ▼
┌───────────────────────┐  ┌──────────────────────┐
│ Conversation Relay    │  │ Workflow Relay (新增) │
│ Service               │  │ Service               │

     │                          │
     │ topic=conversation       │ topic=workflow

     ▼                          ▼

│ Memory 插件 (订阅 conversation + workflow + memory + model)       │

│   └── MemoryWorkflowEventHandler (阶段 4B 新增)                   │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ IWorkflowExecutor (Workflow 模式)                                │
│   │                                                              │
│   ├── publisher.Publish(OnWorkflowTaskStarted, ...)              │

│   ├── publisher.Publish(OnWorkflowTaskFailed, ...)               │
│   └── publisher.Publish(OnWorkflowTaskCancelled, ...)            │

                         │
                         ▼
                  上方 EventHub
```

### 1.2 关键隔离边界

| 边界 | Chat | Workflow |
| --- | --- | --- |
| EventHub 事件名 | `conversation.*` | `workflow.*` |
| EventArgs 类型继承 | `ConversationEventArgs` | `WorkflowEventArgs` |
| PluginBus topic | `conversation`（现状不变） | `workflow`（新增） |
| op 命名空间 | `conversation.event.publish` / `conversation.history.*` | `workflow.event.publish` / `workflow.history.*` |
| Relay 服务 | `WebSocketConversationFeedRelayService`（现状不变） | `WebSocketWorkflowFeedRelayService`（新增） |
| 历史 dispatcher | `PluginBusConversationHistoryDispatcher`（现状不变） | `PluginBusWorkflowHistoryDispatcher`（新增） |
| Memory 插件 handler | `MemoryConversationEventHandler`（现状不变） | `MemoryWorkflowEventHandler`（阶段 4B 新增） |
| 流式 token 是否上 PluginBus | ✅ 是（`conversation.assistant.delta`） | ❌ 否（决策 3-A） |

**Conversation topic 的协议字段冻结**：所有 `conversation.*` 事件的字段 schema 在阶段 0-5 任一阶段都不允许调整，避免破坏 Memory 插件已落库的 ObservationRecord 解析逻辑。

#### 2.3.7 WorkflowTaskTitleUpdatedArgs（决策 6-A：标题异步生成）

任务标题异步更新事件，**仅服务 UI**，与 Chat 模式 `OnSessionTitleUpdated` 行为对齐（[02-当前代码现状.md](./02-当前代码现状.md) §10.2）。

```csharp
public record WorkflowTaskTitleUpdatedArgs(
    string TaskId,
    string? SourceSessionId,
    string TraceId,
    string? WorkspaceId,
    string Mode,
    string SubMode,
    DateTimeOffset OccurredAt,
    string Title,                  // 新标题（已截断 32 字、清洗引号）
    string Source                  // "llm" = LLM 生成；"user" = 用户重命名
) : WorkflowEventArgs(TaskId, SourceSessionId, TraceId, WorkspaceId, Mode, SubMode, OccurredAt);
```

**触发场景**：

| 场景 | Source 值 |
| --- | --- |
| 任务完成后 LLM 异步生成标题（`IsTitleAutoGenerated=1` 且 FinalReport 非空） | `"llm"` |
| 用户在工作台 Tab 右键 → "重命名"或编辑标题字段 | `"user"` |
| 用户复制任务时新任务标题再次走 LLM 生成 | `"llm"` |

**关键设计点**：

- **Memory 插件不订阅此事件**（决策 2-B 仅订阅 `workflow.task.completed`）：标题变化不影响长期记忆
- 仅 UI 关心：用于"占位 → 正式"的局部刷新
- Source 字段帮助 UI 决策："llm" 时可加渐变动效，"user" 时静默更新

## 4. Memory 插件接入策略

### 4.1 总体策略

**决策 2-B**：Memory 插件订阅 `workflow` topic，仅处理 `workflow.task.completed`，把 FinalReport 写入 ObservationRecord。其他 workflow 事件（task.started / step.* / task.failed / task.cancelled / task.paused 等）**全部不入库**。

### 4.2 不入库事件清单

| 事件 | 不入库理由 |
| --- | --- |
| `workflow.task.started` | 任务刚开始无内容可入库 |
| `workflow.task.planned` | Manager 计划属于内部状态，污染长期记忆 |
| `workflow.task.replanned` | 同上 |
| `workflow.step.started` | 步骤开始无内容 |
| `workflow.step.completed` | 步骤中间输出属于过程性数据，最终汇总在 FinalReport 里 |
| `workflow.task.failed` | 失败任务不进入长期记忆 |
| `workflow.task.cancelled` | 取消任务不进入长期记忆 |
| `workflow.task.paused` / `resumed` | HITL 协议帧，与记忆无关 |
| `workflow.task.title.updated` | 标题变化属于 UI 元数据，与长期记忆无关（决策 6-A）；Memory 插件主动 swallow 不报错 |

### 4.3 入库字段映射（workflow.task.completed → ObservationRecord）

```csharp
// Plugins/Src/Madorin.Plugins.Memory/Services/MemoryWorkflowEventHandler.cs

private void IngestTaskCompleted(JsonElement payload)
{
    // 仅成功任务入库
    var status = GetString(payload, "status");
    var task = new ObservationRecord
    {
        // ──── 主键映射 ────
        Id = GetString(payload, "taskId") ?? Guid.NewGuid().ToString("N"),
        // 复用 taskId 作为 ObservationRecord.Id 防止重复入库

        // ──── Agent 字段映射 ────
        AgentId = GetString(payload, "managerAgentId"),
        AgentName = GetString(payload, "managerAgentName"),
        // Manager 作为"作者"——这与 Magentic 内部 Manager 拥有最终汇总语义一致

        // ──── Workspace 映射 ────
        WorkspaceId = ResolveWorkspaceId(payload),

        // ──── Session 映射 ────
        SessionId = GetString(payload, "sourceSessionId") ?? string.Empty,
        // SourceSessionId 可空：从工作台直接发起的任务无 ChatSession 关联

        // ──── Turn / Message 映射 ────
        TurnId = null,                                    // Workflow 没有 turn 概念
        MessageId = GetString(payload, "taskId"),         // 复用 taskId 作为 messageId

        // ──── 事件类型 ────
        EventType = "workflow.task.completed",            // ← Memory 检索时可按此过滤
        Role = "assistant",                                // Manager 视为 assistant 角色

        // ──── 内容 ────
        Content = GetString(payload, "finalReport"),
        AttachmentsJson = "[]",                            // 任务级不带附件

        // ──── 时间戳 ────
        CreatedTimestamp = GetInt64(payload, "completedAt") ?? DateTimeOffset.UtcNow.ToUnixTimeMilliseconds(),
        CreatedAt = ToIsoTime(...),

        // ──── 模型 / 追踪 ────
        ModelName = null,                                  // 任务可能涉及多模型，不归一
        TraceId = GetString(payload, "traceId"),

        // ──── 原始事实 ────
        SourceFactsJson = payload.GetRawText()
    };

    store.InsertObservation(task);
    logger.LogInformation("Workflow task 已入记忆：{TaskId}", task.Id);
```

### 4.4 防止双重入库（Workflow → Chat 回灌场景）

阶段 5B 用户点击"附加到对话"后会通过 `conversation.turn.completed` 链路触发第二条 ObservationRecord 写入。

**两条记录都保留**，因为它们承载不同语义：

| 记录 | EventType | Id | 含义 |
| --- | --- | --- | --- |
| 第一条 | `workflow.task.completed` | `taskId` | 任务级别完成事实 |
| 第二条 | `conversation.turn.completed` | `assistantMessageId`（来自 Chat 链路新生成） | 用户认可并将报告纳入对话的行为 |

**如何避免内容完全重复检索**：

- Memory 插件在召回时按 `EventType` 优先过滤；Chat 上下文召回时优先取 `conversation.*`，Workflow 历史召回时优先取 `workflow.*`
- Embedding 去重交给现有 `IMemoryStore.UpsertObservation` 内部的语义去重逻辑（如有）
- 阶段 5B+ 评估：如果同一 taskId 已经入过 `workflow.task.completed`，再次通过 `conversation.turn.completed` 写入时附加 `RelatedTaskId` 字段，便于后续合并展示

阶段 4B 仅落地"两条记录都保留"，去重优化作为后续课题。

```csharp
public sealed class MemoryWorkflowEventHandler(
    PluginSettings settings,
    ILogger<MemoryWorkflowEventHandler> logger,
    IMemoryStore store)
{
    public bool TryHandle(JsonElement root, string? op)
    {
        var eventType = root.TryGetProperty("eventType", out var et) ? et.GetString() : null;

        // 处理实时事件（仅 workflow.task.completed）
        if (string.Equals(eventType, "workflow.task.completed", StringComparison.Ordinal))
        {
            if (root.TryGetProperty("payload", out var payload) && payload.ValueKind == JsonValueKind.Object)
            {
                IngestTaskCompleted(payload);
            }
            return true;
        }

        // 处理历史回放批次
        if (string.Equals(op, "workflow.history.batch", StringComparison.Ordinal))
        {
            if (root.TryGetProperty("payload", out var payload) && payload.ValueKind == JsonValueKind.Object)
            {
                IngestHistoryBatch(payload);
            }
            return true;
        }

        // 历史回放完成
        if (string.Equals(op, "workflow.history.completed", StringComparison.Ordinal))
        {
            return true;
        }

        // 其他 workflow.* 事件主动 swallow（避免落到通用错误日志）
        if (eventType?.StartsWith("workflow.", StringComparison.Ordinal) == true)
        {
            return true;
        }

        return false;
    }

    private void IngestHistoryBatch(JsonElement payload)
    {
        if (!payload.TryGetProperty("items", out var items) || items.ValueKind != JsonValueKind.Array) return;

        var records = new List<ObservationRecord>();
        foreach (var item in items.EnumerateArray())
        {
            if (!string.Equals(GetString(item, "status"), "completed", StringComparison.OrdinalIgnoreCase))
                continue;
            records.Add(BuildRecordFromPayload(item));
        }

        if (records.Count > 0)
        {
            store.BulkInsertObservations(records);
            logger.LogInformation("回放 Workflow 历史批次 {Count} 条", records.Count);
        }
    }

    private void IngestTaskCompleted(JsonElement payload)
    {
        var record = BuildRecordFromPayload(payload);
        store.InsertObservation(record);
        logger.LogInformation("Workflow task 已入记忆：{TaskId}", record.Id);
    }
```

### 5.2 PluginBusWorkflowHistoryDispatcher

```csharp
internal sealed class PluginBusWorkflowHistoryDispatcher(
    MadorinDbContext db,
    ILogger logger,
    Func<string, string, CancellationToken, Task> sendAsync)
{
    public async Task ReplayAsync(
        string clientId,
        string? requestId,
        long sinceTimestamp,
        int batchSize,
        string? workspaceId,
        IReadOnlyList<string>? statuses,
        CancellationToken cancellationToken)
    {
        try
        {
            var total = 0;
            var last = sinceTimestamp;
            var statusFilter = statuses ?? ["completed"];

            while (true)
            {
                var queryLimit = batchSize + 1;

                // 仅查询 OrchestrationTask 表，不联 OrchestrationStep / OrchestrationMessage
                var rows = db.Query(
                    """
                    SELECT
                        Id, SourceSessionId, Title, Mode, SubMode, Status,
                        StartedAt, CompletedAt, FinalReport, TraceId, CreatedBy,
                        ManagerAgentId, ManagerAgentName, WorkspaceId
                    FROM OrchestrationTask
                    WHERE CompletedAt >= @Since
                      AND Status IN (@Statuses)
                      AND (@WorkspaceId IS NULL OR WorkspaceId = @WorkspaceId)
                    ORDER BY CompletedAt
                    LIMIT @Limit
                    """,
                    OrchestrationTaskMapper.Read,
                    cmd =>
                    {
                        cmd.Parameters.AddWithValue("@Since", last);
                        cmd.Parameters.AddWithValue("@Statuses", string.Join(",", statusFilter));
                        cmd.Parameters.AddWithValue("@WorkspaceId", workspaceId ?? (object)DBNull.Value);
                        cmd.Parameters.AddWithValue("@Limit", queryLimit);
                    });

                if (rows.Count == 0)
                {
                    await SendCompletedAsync(clientId, requestId, total, cancellationToken).ConfigureAwait(false);
                    break;
                }

                var hasMore = rows.Count > batchSize;
                var items = hasMore ? rows.Take(batchSize).ToArray() : rows.ToArray();
                total += items.Length;
                last = items[^1].CompletedAt + 1;

                await SendBatchAsync(clientId, requestId, hasMore, items, cancellationToken).ConfigureAwait(false);
                if (!hasMore)
                {
                    await SendCompletedAsync(clientId, requestId, total, cancellationToken).ConfigureAwait(false);
                    break;
                }
            }

            logger.LogInformation("Workflow replay 完成：Since={Since}, Total={Total}", sinceTimestamp, total);
        }
        catch (Exception ex)
        {
            logger.LogWarning(ex, "Workflow replay 失败");
            await sendAsync(
                clientId,
                PluginBusMessageFactory.CreateWorkflowReplayError(clientId, requestId, ex.Message),
                cancellationToken).ConfigureAwait(false);
        }
    }

    private Task SendBatchAsync(...) { /* 与 conversation 分发器结构对称 */ }
    private Task SendCompletedAsync(...) { /* 同上 */ }
}
```

## 8. 不在本设计范围

以下能力作为后续课题，本文档不展开：

- **跨 PluginBus 协议版本号双方协商**：当前只支持 1.0.0 和 1.1.0 两档，如果未来要支持 2.0.0+ 重大变更需要更复杂协商
- **多插件同时订阅 workflow topic**：当前 Memory 是唯一订阅者，未来如果有"任务监控""任务报表"等插件需要单独评估订阅过滤
- **Workflow 事件的 schema 演进**：阶段 4B 字段冻结，如果未来需要扩展（如增加 `costEstimate` 字段）需要新版本号
- **Memory 插件之外的其他插件订阅 conversation topic**：现状假设 Memory 是唯一订阅者；未来如有竞争需要重新评估广播策略
- **PluginBus 端点性能调优**：当前广播是顺序的，如果 Workflow 高频事件造成阻塞需要异步化改造
