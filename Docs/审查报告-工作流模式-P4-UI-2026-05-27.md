# 工作流模式 UI 审查报告（P4）

## 二、主流程（摘要）

主要流程：用户首次输入 → WorkflowInputVm.StartAsync → TaskExecutionEngine.StartTaskAsync → ExecuteFullLifecycle (requirements → planning → WaitForPlanConfirmation → executing → validating → completed) → 发布 result 消息并进入 WaitForUserSatisfaction（永续对话）→ 用户说 satisfied 后任务真正结束。

UI 显示要点：所有内容以单一 Feed（对话流）呈现；执行阶段通过 progress 类型消息显示时间线并创建 step_card 卡片；输入区由共享 InputAreaView 控制，工作/对话/群聊三种模式通过 InputMode 决定按钮与走马灯样式。

## 三、发现的问题（按优先级）

1) P0-1（高）—— 完成事件清空 CurrentTaskId，破坏永续对话

  - 位置：WorkflowDetailView.axaml.cs (OnTaskEngineCompleted 调用 `_inputVm.OnTaskFinished()`)
  - 描述：`OnTaskFinished()` 会将 `CurrentTaskId = null`。但引擎设计（TaskExecutionEngine.WaitForUserSatisfactionAsync）期望在执行完成后保留 taskId 并等待用户表达满意，后端通过 UserSatisfactionSignal 结束生命周期。视图在收到 completed 事件时即清空 taskId，会导致后续用户在同一任务上下文中继续对话失败（消息会被当作新任务或被拒绝）。
  - 建议：不要在 completed 事件处理处直接清空 CurrentTaskId；仅将 IsRunning=false、输入解锁。仅在确认是 cancelled/failed 或用户显式 satisfied 时清空。

2) P1-1（中高）—— 缺少显式双模态状态订阅

  - 位置：UI 未订阅 `OnWorkflowModeChanged` 事件以驱动模态切换，当前靠 Feed 中是否有时间线消息间接体现执行模式。
  - 描述：文档 08 要求“对话模式 ↔ 执行模式”由事件自动切换并影响输入区（灰显/可编辑）与视图（时间线/对话流）。当前实现只有部分表现，缺少明确状态，可能在边界条件（断点恢复、暂停/恢复）出现错位。
  - 建议：在 WorkspaceTabVm / WorkflowDetailView 或 InputAreaView 上加入订阅 `OnWorkflowModeChanged`，维护明确的 enum 状态并统一 UI 行为（输入框只读/走马灯颜色/时间线可见性）。

3) P1-2（中高）—— 危险工具授权与文档不一致

  - 位置：WorkflowDetailView.axaml 中 `tool_auth` 作为 Feed 内联消息，代码中提供确认/全部授权/拒绝按钮。
  - 描述：文档 08 要求在输入框上方弹出“浮动意图面板”并支持“自定义输入”以让用户自然语言修改授权参数。当前实现把授权内联为消息卡片，且没有自定义输入路由。
  - 建议：确认产品决策（浮动面板或内联消息）。若遵循文档，应实现浮动面板（InputAreaView 增强态）并把按钮与自定义输入路由到 TaskExecutionEngine 的 ToolAuth 信号；若保留内联，需在文档中说明并支持自定义输入选项。

4) P1-3（中）—— 工具授权缺少自定义输入实现

  - 描述：当前授权只有三按钮交互，无法捕获用户自然语言的“改一下收件人”类定制化决策。
  - 建议：在授权交互路径上增加“自定义输入”状态（UI 打开输入框或把输入路由到 Orchestrator），引擎接收并将其作为修改请求给子智能体。

5) P1-4（中）—— 执行完成结果在 UI 可能重复展示

  - 位置：P4TaskDetailVm 在 completed 时追加 step_card（任务总结）、AppendTimeline(Lifecycle) 以及引擎还会发 result 消息。
  - 描述：UI 可能同时显示总结卡片与 result 消息，造成重复和视觉噪声。
  - 建议：收敛为单一富内容结果块（result）并把任务总结合并，时间线仅保留一条 Lifecycle 行。

6) P2-1（中）—— 结果文件交互不足

  - 描述：Result 文件项仅显示标签/tooltip，无“打开/在文件夹中显示”动作。
  - 建议：为每个 ResultFile 增加打开文件与定位操作并记录用户选择的保存路径（配合 `specify_path` 意图）。

7) P2-2（中）—— 步骤子过程展示与文档细节不完全一致

  - 描述：文档要求每次 AI 输出追加为时间线下的缩进子过程行；当前把多数输出累积进 step_card 折叠卡片内部。
  - 建议：确认设计取舍：若要严格遵守时间线/子过程行模式，需要把增量 AI 输出映射到 progress.Detail 类型消息；若 step_card 是首选展示，应在文档中更新规范并调整术语/示例。

## 四、运行验证说明

- 已在本地环境执行 `dotnet build` 并 `dotnet run` 启动 UI。窗口进程启动成功，说明配置和依赖正常。
- 由于无法自动化 UI 交互，本次未进行“人工点击并观察行为”的端到端验证；建议团队在本地以手工方式复现以下场景验证修复：
  1. 启动工作流，发起任务并确认生成计划；在计划确认后发“开始”观察执行模式切换与输入框行为；
  2. 在执行阶段触发危险工具授权，验证浮动面板/内联消息、以及“自定义输入”是否能路由到引擎；
  3. 执行完成后在 Feed 中发问（例如“第三章数据来源是什么？”），观察 AI 是否在原任务上下文回答并等待用户表达满意；
  4. 测试 satisfied 意图后任务是否真正结束并从运行上下文中移除。

## 五、修复建议（简要优先次序）

1. 修复 completed 事件导致 `CurrentTaskId` 被清空的问题（高优先）。
2. 在 UI 中引入显式 `WorkflowMode` 状态（conversation/execution），订阅后端 `OnWorkflowModeChanged` 事件以统一行为（中高）。
3. 决策浮动面板交互实现（或更新文档以说明内联实现），并补充“自定义输入”通路（中）。
4. 收敛执行完成结果的展示逻辑，合并重复项并增加文件打开交互（中）。
5. 将 `specify_path` 意图与系统设置/TaskMeta 绑定，使用户路径偏好在后续保存中生效（中）。
