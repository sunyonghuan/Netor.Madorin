# 桌面 3D 宠物项目构架

> 阶段：策划 / 验证后架构基线  
> 日期：2026-05-24  
> 目标形态：Windows 独立单 EXE，内置 MCP 工具与 WebSocket 实时对话流，主窗口不使用 WebView2 / WPF / Avalonia。

## 1. 架构结论

当前推荐基线：

```text
DesktopPet.exe
  ├─ Win32 原生透明窗口
  ├─ Direct3D11 渲染循环
  ├─ Live2D Cubism Native Core 薄绑定
  ├─ glTF / GLB / VRM 模型管线
  ├─ 动画状态机
  ├─ 字幕与气泡渲染
  ├─ 内置 MCP 工具服务
  ├─ WebSocket 实时对话流客户端
  ├─ 托盘 / 右键菜单 / 配置
  └─ AOT-first 工程约束
```

第一版坚持单应用进程：

- 用户启动和退出都只面对一个应用。
- MCP 能力内置在同一个 EXE 中。
- 实时对话流通过 WebSocket 进入应用内事件队列。
- 不引入 WebView2 多进程模型。
- 不用 WPF 承担主窗口。
- Avalonia 只作为未来设置面板候选，不进入主桌宠窗口闭环。

## 2. 已验证事实

### 2.1 Live2D Native Core

已下载并验证官方 `Cubism SDK for Native R5`。

已验证：

- `Live2DCubismCore.dll` 可被 Native AOT EXE 加载。
- `Haru.moc3` 可做 consistency check。
- 可 revive moc。
- 可 initialize model。
- 可读取 parameter / drawable 数量。
- 可写入 `ParamMouthOpenY` 并执行 `csmUpdateModel`。

结论：

```text
Live2D Core 本身不阻塞 Native AOT。
推荐自研 LibraryImport 薄绑定。
第三方 Live2DCSharpSDK 可参考，但不作为长期核心绑定。
```

### 2.2 Direct3D11 与 EXE 预览

已编译 Live2D 官方 Native SDK D3D11 Minimum Demo：

```text
runner_data/live2d-native-exe-preview/Demo.exe
```

该 EXE 是原生窗口 + Direct3D11 + Live2D Native Framework，已能作为效果与渲染路径参考。

### 2.3 C# Win32 / D3D11 AOT

已验证：

- C# `LibraryImport` 创建 Win32 窗口可 Native AOT。
- D3D11 device 创建可 Native AOT。
- HWND + swap chain + render target + present 可运行。

Vortice 当前风险：

- 快速原型可用。
- 完整 swap chain 路径会触发 SharpGen.Runtime 的 IL 反射/vtable 警告。
- 若要求严格 0 IL warning，应后续替换为自研极薄 D3D11/DXGI COM vtable 绑定。

```text

│ DesktopPet.exe                                            │
├──────────────────────────────────────────────────────────┤
│ App Host / Composition Root                               │
│  ├─ 单实例守护                                             │
│  ├─ 配置加载                                               │
│  ├─ 日志初始化                                             │
│  └─ 模块生命周期管理                                       │
├──────────────────────────────────────────────────────────┤
│ Platform Layer                                            │
│  ├─ Win32Window                                            │
│  ├─ TrayIcon / ContextMenu                                 │
│  ├─ InputHitTest / ClickThrough                            │
│  └─ Monitor / DPI / Placement                              │
├──────────────────────────────────────────────────────────┤
│ Render Layer                                              │
│  ├─ D3D11Device                                            │
│  ├─ SwapChain / RenderTarget                               │
│  ├─ Texture / Shader / Buffer                              │
│  └─ FrameScheduler                                         │
├──────────────────────────────────────────────────────────┤
│ Model Layer                                               │
│  ├─ Live2D Runtime                                         │
│  ├─ GLB / glTF Runtime                                     │
│  ├─ VRM Runtime                                            │
│  └─ ModelProfile / ModelRegistry                           │
├──────────────────────────────────────────────────────────┤
│ Behavior Layer                                            │
│  ├─ PetStateMachine                                        │
│  ├─ MotionController                                       │
│  ├─ MouthMotionController                                  │
│  ├─ SubtitleController                                     │
│  └─ InteractionController                                  │
├──────────────────────────────────────────────────────────┤
│ AI / Control Layer                                        │
│  ├─ MCP Tool Server                                        │
│  ├─ WebSocket Conversation Client                          │
│  ├─ EventQueue                                             │
│  └─ CommandRouter                                          │
└──────────────────────────────────────────────────────────┘
```

核心原则：

- 渲染线程只消费状态快照，不直接等待 MCP / WS。
- MCP / WS 只投递命令事件，不直接操作 D3D11 资源。
- 模型加载可以异步，但 GPU 资源创建必须回到渲染线程或设备拥有线程。
- 配置、模型 profile、动作状态都要可序列化，方便后续热加载和调试。

### 4.1 DesktopPet.App

职责：

- EXE 入口。
- 单实例控制。
- 生命周期编排。
- 根据启动参数进入桌宠模式或 MCP 模式。

启动模式：

| 模式 | 命令 | 说明 |
|---|---|---|
| 桌宠模式 | `DesktopPet.exe` | 启动窗口、托盘、渲染、WS |
| MCP 模式 | `DesktopPet.exe --mcp` | 同一 EXE 进入 MCP stdio |
| 控制模式 | `DesktopPet.exe --show` | 唤醒已运行实例 |

### 4.2 DesktopPet.Abstractions

职责：

- 定义核心接口与数据结构。
- 不引用 Win32、D3D11、Live2D、AI SDK。
- 作为跨模块契约层。

核心类型：

```text
IPetRuntime
IModelRuntime
IModelInstance
IRenderScene
IRenderTarget
IPetCommand
PetState
PetEvent
ModelProfile
SubtitleFrame
```

### 4.3 DesktopPet.Platform.Win32

职责：

- Win32 窗口创建。
- 透明 / 无边框 / 置顶。
- 点击穿透开关。
- 托盘图标。
- 右键菜单。
- 多显示器 / DPI / 右下角定位。
- 窗口拖拽移动。
- 窗口大小调整。
- 窗口位置与大小恢复。

技术要求：

- 优先 `LibraryImport`。
- 回调使用 `UnmanagedCallersOnly` 或静态委托保活策略。
- 不依赖 WPF / WinForms 作为主窗口框架。

### 4.4 DesktopPet.Rendering.D3D11

职责：

- D3D11 device。
- swap chain。
- render target。
- shader / texture / buffer。
- Live2D drawable 提交。
- 3D mesh 提交。

第一版策略：

- 可以用 Vortice 快速闭环。
- 把 Vortice 使用范围限制在 `Rendering.D3D11` 内部。
- 后续若要 0 IL warning，可用自研 D3D11/DXGI 绑定替换，不影响上层模型与行为模块。

### 4.5 DesktopPet.Models.Live2D

职责：

- 官方 `Live2DCubismCore.dll` 绑定。
- `.model3.json` 解析。
- `.moc3` 加载。
- texture 加载。
- motion / expression / physics / pose。
- 参数读写。
- drawable 数据转换为渲染命令。

第一版能力：

- 加载一个模型。
- idle motion。
- expression 切换。
- `ParamMouthOpenY` 随说话状态随机/周期变化。
- 不做音素级 lipsync。

AOT 策略：

- Core 绑定使用 `LibraryImport`。
- JSON 使用 System.Text.Json source generation。
- 避免反射扫描模型类型。

### 4.6 DesktopPet.Models.Gltf

职责：

- glTF / GLB 基础加载。
- mesh / material / texture。
- skeleton / skinning。
- animation clip。

第一版策略：

- 先支持标准 GLB。
- 扩展与特殊材质延后。
- 任何库接入前必须跑 AOT smoke test。

### 4.7 DesktopPet.Models.Vrm

职责：

- VRM 0.x / 1.0 profile。
- humanoid mapping。
- expression / blendshape。
- spring bone。
- look-at。

第一版策略：

- 先建立接口与 profile 结构。
- 加载能力可以落在 M3。
- 不引入 Unity 依赖型 VRM 库作为主线。

### 4.8 DesktopPet.Behaviors

职责：

- 桌宠状态机。
- 动作优先级。
- 说话 / 思考 / 待机 / 隐藏 / 显示。
- 字幕生命周期。
- 鼠标交互。

推荐状态：

```text
Hidden
Idle
Thinking
Speaking
Interacting
Dragging
Sleeping
Error
```

事件流：

```text
MCP / WS / UI
  -> PetEvent
  -> EventQueue
  -> PetStateMachine
  -> MotionController
  -> RenderSceneSnapshot
  -> RenderLoop
```

### 4.9 DesktopPet.Ai

职责：

- MCP tools。
- WebSocket 实时对话流。
- AI 事件转换为桌宠事件。
- 连接状态与重连。

MCP 工具建议：

| 工具 | 说明 |
|---|---|
| `pet.show` | 显示桌宠 |
| `pet.hide` | 隐藏桌宠 |
| `pet.say` | 显示文字并进入 Speaking |
| `pet.think` | 进入 Thinking |
| `pet.idle` | 回到 Idle |
| `pet.set_model` | 切换模型 |
| `pet.set_expression` | 切换表情 |
| `pet.status` | 返回当前状态 |

WebSocket 事件建议：

```text
conversation.started
assistant.thinking
assistant.delta
assistant.completed
assistant.error
audio.started
audio.completed
```

第一版只需要文本流驱动字幕与嘴部动作。

### 4.10 DesktopPet.Configuration

职责：

- 配置文件读写。
- 模型 profile。
- 窗口位置。
- 用户偏好。

建议配置：

```text
config/settings.json
config/model-profiles.json
config/ai.json
models/live2d/
models/gltf/
models/vrm/
logs/
```

配置序列化必须 AOT 友好，优先 source-generated `JsonSerializerContext`。

`settings.json` 中建议保留窗口状态：

```json
{
  "window": {
    "monitorDeviceName": "\\\\.\\DISPLAY1",
    "x": 1320,
    "y": 620,
    "width": 420,
    "height": 560,
    "scale": 1.0,
    "opacity": 1.0,
    "topMost": true,
    "clickThrough": false,
    "locked": false,
    "rememberPlacement": true
  }
}
```

窗口状态保存策略：

- 用户拖拽结束后延迟保存。
- 用户调整大小结束后延迟保存。
- 通过右键菜单切换缩放、透明度、置顶、点击穿透后立即保存。
- 启动时先尝试恢复上次显示器；如果显示器不存在，则回退到主屏右下角。
- 保存坐标时记录显示器名称，避免多显示器布局变化后窗口跑到屏幕外。

## 5. 线程与消息模型

建议线程模型：

| 线程 / 循环 | 职责 |
|---|---|
| Main | 启动、单实例、生命周期 |
| Window | Win32 message pump |
| Render | D3D11 frame loop |
| AI IO | MCP / WS 异步 IO |
| Loader | 模型文件解析、图片解码、CPU 侧准备 |

第一版可以合并 Main + Window + Render，但代码边界要按未来拆分设计。

禁止：

- 在渲染循环里等待网络 IO。
- 在 MCP tool 回调里直接操作 GPU 资源。
- 在 Win32 window proc 中做耗时模型加载。
- 使用全局静态状态绕过事件队列。

## 6. AOT-first 规则

项目从第一版开始按 Native AOT 约束写：

- 所有项目启用 `IsAotCompatible`。
- AOT smoke test 作为每个里程碑验收项。
- JSON 使用 source generation。
- P/Invoke 使用 `LibraryImport`。
- 禁止运行时反射扫描插件。
- 插件化暂不作为第一版目标。
- 模型格式扩展通过显式 registry 注册。

发布模式：

| 模式 | 用途 |
|---|---|
| Native AOT | 首选发布模式 |
| self-contained IL | 关键依赖无法 AOT 时回退 |
| ReadyToRun | IL 回退时可选优化 |

回退原则：

- 即使回退 IL，源码仍保持 AOT 友好。
- 不因为 IL 模式引入反射式架构。
- AOT 测试继续保留，等待依赖成熟后切回。


3D：

```text

  -> Scene / Mesh / Material / Texture / Skeleton
  -> Animation Mixer

```

统一渲染提交：

```text

IModelInstance.BuildRenderItems()
Renderer.Submit(renderItems)
```


第一版优先级：

| 优先级 | 菜单项 |
|---|---|
| P0 | 显示/隐藏、退出、切换模型、置顶、点击穿透、拖拽移动、调整大小、自动记住位置和大小、重置到右下角 |
| P1 | 缩放、透明度、锁定位置、保存当前位置和大小、重新加载模型、表情、说话测试、字幕开关 |
| P2 | 显示器选择、打开目录、查看状态、开机启动、AI 连接控制 |

菜单行为原则：

- 菜单项必须直接改变本地状态，不依赖 AI。
- 所有状态变化都进入统一 `PetEvent`，避免 UI 和 AI 控制两套逻辑。
- 危险操作很少，第一版只需要对“退出”做确认可选项。
- 菜单文本保持短，复杂配置以后放到设置面板。

暂不做：

- 精确 lipsync。
- 音频驱动口型。
- PMX / MMD 全兼容。
- 复杂模型商店。
- 热插拔插件体系。
- WebView2 UI。
- WPF 主窗口。

## 10. 主要风险与决策

| 风险 | 当前判断 | 应对 |
|---|---|---|
| Live2D Core AOT | 已验证可行 | 自研薄绑定 |
| D3D11 绑定 IL warning | Vortice 有警告 | 第一阶段可用，后续可替换 |
| VRM 生态偏 Unity | 风险高 | 先做 GLB，再做 VRM 子集 |
| Avalonia 体积 | 可接受但不进主窗口 | 后续设置面板再评估 |
| WebView2 多进程 | 不符合目标 | 禁用为主线 |
| WPF 性能/体积 | 不符合目标 | 禁用为主线 |
| Native AOT 依赖不兼容 | 可能发生 | self-contained IL 回退 |

## 12. 最终建议

第一版正式开发不要从“完整引擎”开始，而是从主链路闭环开始：

```text
Win32 透明窗口
  -> D3D11 present
  -> Live2D 单模型渲染
  -> 状态机驱动嘴部动作
  -> 字幕
  -> MCP / WS 控制
  -> Native AOT 发布
```

这条路线已经通过关键风险验证，足够进入正式工程搭建阶段。
