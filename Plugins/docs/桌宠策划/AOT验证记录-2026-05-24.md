## 6. AOT 警告

发布时出现 SharpGLTF 相关警告：

```text
SharpGLTF.Core produced trim warnings
SharpGLTF.Core produced AOT analysis warnings
```

展开后主要集中在：

- `SharpGLTF.IO.UnknownNode.DeserializeProperty`
- `SharpGLTF.IO.UnknownNode._Properties`
- `System.Text.Json.Nodes.JsonNode.ReplaceWith<T>`
- `System.Text.Json.Nodes.JsonNode.ConvertFromValue<T>`
- `System.Text.Json.Nodes.JsonArray.Add<T>`
- `System.Text.Json.Nodes.JsonValue.Create<T>`

警告类型：

- `IL2026`
- `IL3050`
- `IL2075`

判断：

- SharpGLTF 当前版本可以 AOT 发布并运行最小 GLB 探针。
- 但其未知扩展节点处理依赖 JSON DOM 动态值，Native AOT 下存在明确风险。
- 如果第一版只使用 glTF/GLB 的稳定子集，可以继续验证。
- 如果要完整支持 VRM 扩展、未知扩展、复杂材质扩展，需要进一步实测或考虑自研/替换加载层。

## 9. 当前结论

纯 C# 单 EXE + Native AOT 是可行方向，但需要带着约束推进：

- Avalonia、Silk.NET、Vortice、MCP 基础链路初步通过。
- SharpGLTF 可运行，但存在 AOT 警告，是 3D 模型链路第一风险点。
- Live2D 是最大未知项，必须尽快单独做 SDK 绑定探针。
- 第一版建议保留 AOT-first 代码结构，同时准备 IL self-contained 回退发布。

## 10. 追加验证：真实窗口

### 10.1 Silk.NET.Windowing

追加 `--silk-window` 模式，创建真实窗口、渲染 5 帧后自动关闭。

第一次运行结果：

```text
Unhandled exception. System.PlatformNotSupportedException:
Couldn't find a suitable window platform. (none registered)
```

原因：

- `Silk.NET.Windowing` 默认通过反射加载 GLFW / SDL 平台。
- Native AOT 后平台自动发现失败。

修复：

```csharp
GlfwWindowing.Use();
```

修复后运行结果：

```text
silk-window: loaded
silk-window: closing after 5 frames
silk-window: closing
silk-window: completed
```

剩余警告：

- `Silk.NET.Windowing.Window.TryAdd` 仍触发 `Activator.CreateInstance` 相关 AOT 警告。
- `Silk.NET.Core.Loader.DefaultPathResolver` 触发 `Assembly.Location` / `DependencyContext` 单文件发布相关警告。

判断：

- Silk.NET.Windowing 可以通过显式 `GlfwWindowing.Use()` 在 Native AOT 下运行。
- 但作为 AOT-first 主窗口壳并不干净。
- 如果使用 Silk.NET，建议只使用更底层绑定，避免依赖高层反射式窗口平台发现。

### 10.2 Avalonia

## 12. 修正后的技术判断

当前建议优先级调整为：

1. 主窗口与渲染：C# Win32 `LibraryImport` + Vortice.Direct3D11。
2. 设置面板：可选 Avalonia，独立开关或延后。
3. 窗口库：不优先使用 Silk.NET.Windowing 高层 API。
4. 模型加载：SharpGLTF 可先用于 GLB 子集，但必须继续压测 VRM/扩展。
5. Live2D：继续作为最大未知项，需要官方 SDK 后单独验证。

新增 AOT/Trim 警告：

```text
IL2067 SharpGen.Runtime.TypeDataStorage.GetTargetVtbl(...)

```

判断：

- Vortice.Direct3D11 的基础设备创建是 AOT 干净的。
- 进入 swap chain / back buffer / COM helper 路径后，SharpGen.Runtime 会触发反射式 vtable 相关警告。
- 当前运行期实测成功，但如果项目要求“Native AOT 发布 0 IL 警告”，需要进一步处理。

后续路线：

| 路线 | 说明 |
|---|---|
| 继续使用 Vortice | 开发效率高，当前实测可运行；需要接受或隔离 SharpGen 的 IL2067/IL2072 风险，并持续做运行期 smoke test。 |
| 自写极薄 D3D11 绑定 | 使用 `LibraryImport` + 最小 COM vtable wrapper，只覆盖桌宠需要的 D3D11/DXGI API；工作量更大，但更符合严格 AOT。 |

当前建议：

- 第一阶段可以继续用 Vortice 验证渲染闭环和模型管线。
- 如果后续 Live2D、VRM、透明合成均跑通，再评估是否把 D3D11 绑定收敛为自研极薄层。
- 文档中“Vortice 零警告”结论应修正为“设备创建零警告，完整 swap chain 路径有 SharpGen 反射警告但可运行”。

### 15.3 Live2D 当前结论

回答“Live2D 有没有 NuGet 包”：

- 有第三方 NuGet 包。
- 没有发现官方 `Live2DCubismCore` NuGet 包。
- 第三方包仍需要手动提供官方 `Live2DCubismCore.dll`。

当前最可行路线：

```text
Live2DCSharpSDK.Framework / 自写薄绑定
  + 官方 Live2DCubismCore.dll
  + 我们自己的 Win32/D3D11 主窗口和渲染管理
```

AOT 风险判断：

- 官方 Core DLL 作为原生依赖随包分发，在 Native AOT 下是可行的。
- 直接用 `LibraryImport` 写薄绑定是 AOT 友好的。
- `Live2DCSharpSDK` 的基础绑定可以加载 Core，但它标注为 `Cubism 5 SDK for Native R1`，而当前官方 SDK 是 R5；长期建议逐步收敛到自研薄绑定，避免第三方绑定版本落后。
- 真正的后续风险不在 Core 加载，而在渲染层、motion/expression 解析层、贴图加载层是否继续保持 AOT 友好。
