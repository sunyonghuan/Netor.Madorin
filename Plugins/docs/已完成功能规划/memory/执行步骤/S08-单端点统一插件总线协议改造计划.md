# S08 - 单端点统一插件总线协议改造计划

> 日期：2026-05-13  
> 状态：实施复查后继续收敛  
> 范围：先修改主程序，再迁移 Memory 插件  
> 决策：清除旧的双通道方案，不做旧插件兼容；当前只有 Memory 插件依赖该内部通道。

## 1. 背景

当前主程序同时存在两套 WebSocket 通道：

- `WebSocketInteractionServerService`：聊天交互通道，路径 `/ws/`。
- `WebSocketFeedServerService`：内部 conversation-feed 通道，路径 `/internal/conversation-feed/`。

Memory 插件实际通过 conversation-feed 接收两类数据：

1. 对话事实数据：实时事件、历史回放批次。
2. 控制面请求：长期记忆上下文供应请求与响应。

这证明单条内部 WebSocket 连接已经可以同时承载“消息数据”和“控制面”。继续保留两个插件相关通道会造成语义混乱，也容易出现宿主向一个服务发送、插件连接另一个服务的错配问题。

因此后续改造目标不是继续兼容旧方案，而是直接收敛为：

```text
一个端口号
一个端点
一套协议
通过 type / topic / op 区分数据面与控制面
```

## 2. 总体目标

新增并采用统一内部插件总线：

```text
ws://localhost:{port}/internal
```

协议名：

```text
cortana.plugin-bus
```

协议版本：

```text
1.0.0
```

统一承载：

- 连接握手
- 主题订阅
- 对话事实实时事件
- 对话历史回放
- 长期记忆上下文供应请求/响应
- 模型能力请求/响应
- 插件状态、心跳与未来扩展控制消息

## 3. 核心原则

1. **不保留旧插件兼容**  
   当前没有其它插件使用旧内部 feed 通道，允许直接清除旧计划和旧入口。

2. **一个内部插件端点**  
   插件不再同时接收 `chatWsEndpoint`、`conversationFeedEndpoint`、`modelCapabilityEndpoint` 作为内部通信入口。统一为 `pluginBusEndpoint`。

3. **一个端口号**  
   主程序只为内部插件总线下发一个端口：`pluginBusPort`。

4. **控制面和数据面不拆端口**  
   通过 envelope 字段区分：
   - 数据面：`type = "event"`，常用 `topic = "conversation"`。
   - 控制面：`type = "request" | "response" | "error"`，通过 `op` 表示操作。

5. **命名语义清晰**  
   操作名采用 `domain.resource.action` 风格。

## 5. 统一 Envelope 字段

所有插件总线消息都应包含或允许包含以下字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `type` | string | 消息类型 |
| `protocol` | string | 固定为 `cortana.plugin-bus` |
| `version` | string | 协议版本 |
| `topic` | string? | 主题，例如 `conversation`、`memory`、`model` |
| `op` | string? | 操作名，例如 `memory.context.supply.request` |
| `requestId` | string? | 请求响应关联 ID |
| `traceId` | string? | 链路追踪 ID |
| `source` | string? | 来源，例如 `host`、`plugin.memory` |
| `target` | string? | 目标，例如 `plugin.memory`、`host` |
| `timestamp` | long? | Unix 毫秒 |
| `payload` | object? | 业务载荷 |
| `error` | object? | 错误对象 |

## 6. `type` 语义

| type | 方向 | 说明 |
|---|---|---|
| `connected` | host → plugin | 插件连接成功后的握手 |
| `subscribe` | plugin → host | 插件订阅主题 |
| `subscribed` | host → plugin | 订阅确认 |
| `event` | host → plugin | 数据面事件 |
| `request` | 双向 | 控制面请求 |
| `response` | 双向 | 控制面响应 |
| `error` | 双向 | 控制面或协议错误 |
| `replay` | plugin → host | 历史数据回放请求，等价于特殊控制请求 |
| `ping` | host → plugin | 心跳探测 |
| `pong` | plugin → host | 心跳响应 |

## 7. `topic` 语义

| topic | 说明 |
|---|---|
| `conversation` | 对话事实流和历史回放 |
| `memory` | 长期记忆供应、状态、策略 |
| `model` | 模型能力申请与响应 |
| `plugin` | 插件状态、心跳、诊断 |
| `system` | 系统级控制消息 |

## 8. `op` 命名规范

统一使用：

```text
domain.resource.action
```

推荐操作名：

| op | 说明 |
|---|---|
| `conversation.event.publish` | 发布实时对话事件 |
| `conversation.history.replay` | 请求历史回放 |
| `conversation.history.batch` | 历史回放批次 |
| `conversation.history.completed` | 历史回放完成 |
| `memory.context.supply.request` | 请求长期记忆上下文供应 |
| `memory.context.supply.response` | 返回长期记忆上下文供应包 |
| `memory.context.supply.error` | 长期记忆供应错误 |
| `model.capability.request` | 请求模型能力 |
| `model.capability.response` | 返回模型能力结果 |
| `plugin.status.report` | 插件状态上报 |
| `plugin.heartbeat` | 插件心跳 |

旧名称直接废弃：

| 旧名称 | 新名称 |
|---|---|
| `memory.supply.request` | `memory.context.supply.request` |
| `memory.supply.package` | `memory.context.supply.response` |
| `memory.supply.error` | `memory.context.supply.error` |
| `conversation.export.batch` | `conversation.history.batch` |
| `conversation.export.completed` | `conversation.history.completed` |

## 10. 主程序实施步骤

### S08-1：清理端点规划

- 更新 `CortanaWsEndpoints.cs`，加入 `PluginBus` 常量。
- 明确 `/internal/conversation-feed/` 为废弃路径。
- 明确 `/ws/` 只属于外部聊天交互，不再作为插件内部通道。

### S08-2：新增统一协议模型

- 新增 `PluginBusEnvelope` 和必要 DTO。
- 更新 `WebSocketJsonContext`，加入 AOT JSON 源生成类型。
- 将 Memory supply 请求/响应挂载到 `payload` 下，避免顶层字段污染。

### S08-3：新增或重命名服务

推荐直接新增：

```text
WebSocketPluginBusServerService
```

然后把 `WebSocketFeedServerService` 中与 conversation-feed、replay、memory supply、model capability 相关的逻辑迁移进去。

迁移完成后删除或停止注册 `WebSocketFeedServerService`。

### S08-4：统一连接管理

`WebSocketPluginBusServerService` 需要维护：

- 插件连接集合
- 订阅主题集合
- pending request 集合
- 串行发送锁

所有控制请求都通过 `requestId` 关联响应。

### S08-5：长期记忆供应改为新协议

`LongMemoryContextProvider` 仍依赖：

```text
ILongMemorySupplyClient
```

但 DI 绑定到：

```text
WebSocketPluginBusServerService
```

`SupplyAsync` 发出：

```text
memory.context.supply.request
```

等待：

```text
memory.context.supply.response
memory.context.supply.error
```

### S08-6：PluginLoader 只下发插件总线端点

初始化扩展改为：

```text
pluginBusEndpoint
pluginBusPort
pluginBusProtocol
pluginBusVersion
```

清除内部通信相关旧字段：

```text
conversationFeedEndpoint
conversationFeedPort
modelCapabilityEndpoint
modelCapabilityPort
```

`chatWsEndpoint` 如果仅用于外部聊天交互，不应再作为 Memory 插件内部通道字段下发。

### S08-7：UI 启动流程改造

`App.axaml.cs` 中只启动并传递一个内部插件总线端口：

```text
pluginBusServer.Port -> pluginLoader.PluginBusPort
```

不再同时传：

```text
WsPort
FeedPort
```

如果外部聊天通道仍存在，则它不进入 Memory 插件内部通信配置。

### S08-8：日志与诊断

主程序需要明确输出：

- 插件总线端口
- 插件连接数
- Memory 插件是否已订阅 `memory` / `conversation`
- 供应请求发送条数
- 供应响应收到条数
- 供应返回 items 条数
- prompt 实际注入条数

## 13. 验收标准

### 构建验收

- 主程序构建成功。
- Memory 插件构建成功。
- 发布脚本成功。

### 运行验收

启动主程序后日志必须能看到：

- `PluginBus` 监听端口。
- Memory 插件连接 `/internal`。
- Memory 插件订阅 `conversation` 与 `memory`。
- 宿主发送 `memory.context.supply.request`。
- 宿主收到 `memory.context.supply.response`。

### 数据验收

- 新对话事实能够写入 Memory SQLite。
- replay 能补齐历史数据。
- 指定 `agentId` + `workspaceId` 能供应长期记忆。
- `LongMemoryContextProvider` 日志能区分：
  - 请求供应数量
  - 插件返回数量
  - 通过过滤数量
  - 实际注入 prompt 数量

### 负向验收

- Memory 插件未连接时，主程序不崩溃，长期记忆注入降级为空。
- 供应请求超时时，主程序不阻塞主对话。
- 协议错误时日志能打印 `requestId` / `traceId`。

## 15. 当前结论

本次改造不再延续“双端点、双服务、兼容旧插件”的设计。

主线方案是：

```text
/internal
  ├─ type=event    用于消息数据面
  ├─ type=request  用于控制面请求
  ├─ type=response 用于控制面响应
  └─ type=error    用于错误返回
```

通过 `topic` 区分业务域，通过 `op` 区分具体动作。

这能满足 Memory 插件当前全部需求，也为未来插件扩展保留清晰协议边界。

## 16. 复查后的继续收敛步骤

> 目标：彻底消除 `/ws/` 与 `/internal` 双路由、短连接模型能力调用和无 topic 过滤分发，统一由 `WebSocketPluginBusServerService` 承载 WebSocket 工作。

### S08-13：主程序端统一 WebSocket 入口

- `WebSocketPluginBusServerService` 同时实现：
  - `IChatTransport`
  - `IPluginBusBroadcaster`
  - `ILongMemorySupplyClient`
- 停止注册并删除 `WebSocketInteractionServerService`。
- `CortanaWsEndpoints.ChatPath` 改为指向 `PluginBusPath`，保留常量仅用于源码兼容，不再暴露第二个路由。
- 前端聊天消息也走 PluginBus envelope：
  - `topic = conversation`
  - `op = chat.message.send` / `chat.generation.stop` / `chat.token` / `chat.done` / `chat.error`
- 旧 `type=send/stop/token/done/error` 在实现中只允许作为过渡输入格式解析，不再作为新协议主线。

### S08-14：主程序端按 topic 订阅分发

- 订阅表从 `clientId -> byte` 改为 `clientId -> topics`。
- `conversation` 实时事件只发给订阅 `conversation` 的连接。
- `memory.context.supply.request` 只发给订阅 `memory` 的连接。
- `model.capability.response` 只返回发起请求的连接。
- 后续新增 topic 时不再新增 WebSocket 端点，只扩展 topic/op 分发器。

### S08-15：主程序端模块化拆分

避免 `WebSocketPluginBusServerService` 继续膨胀，按职责拆分内部模块：

| 模块 | 职责 |
|---|---|
| `PluginBusSubscriptionRegistry` | 维护连接订阅 topic、按 topic 查询目标连接 |
| `PluginBusChatDispatcher` | 处理前端聊天输入，输出 token/done/error |
| `PluginBusConversationHistoryDispatcher` | 处理 `conversation.history.replay` |
| `PluginBusMemorySupplyDispatcher` | 下发并完成长期记忆供应请求 |
| `PluginBusModelCapabilityDispatcher` | 处理 `model.capability.request` |

所有新增类型必须补充 XML 标准注释。

### S08-16：Memory 插件端统一长连接分发

- `MemoryIngestService` 仅负责连接生命周期、读写循环和心跳。
- 新增插件端分发器：
  - `MemoryPluginBusDispatcher`
  - `MemoryConversationEventHandler`
  - `MemorySupplyRequestHandler`
  - `MemoryHostModelCapabilityClient` 或复用现有客户端但改为通过共享连接发送请求。
- 模型能力调用不再创建临时 WebSocket 连接，统一复用 Memory 插件到 PluginBus 的同一长连接。
- 通过 `requestId` 完成模型能力响应与调用任务的匹配。

### S08-17：心跳与请求响应完整性

- Memory 插件收到 `type=ping` 必须回复 `type=pong`。
- 历史回放 `conversation.history.batch/completed` 必须带回请求的 `requestId`。
- `ConversationExportBatch.HasMore` 必须反映真实后续批次状态。

### S08-18：清理旧协议残留

- 删除 `conversation.export.batch/completed` 兼容分支。
- 删除 `settings.ConversationFeedEndpoint` 的业务兜底。
- 清理 `conversation-feed` 标签、注释、说明文案。
- 清理不再使用的 `CreateChatPingPayload` / `CreateModelCapabilityPingPayload` 等旧双协议辅助方法。

### S08-19：验收标准补充

- 代码中不再注册独立聊天 WebSocket 服务。
- 运行时只监听一个 WebSocket 端点：`/internal`。
- Memory 插件使用一条长连接完成：
  - conversation 订阅与摄取
  - memory supply 响应
  - model capability 请求/响应
  - ping/pong 心跳
- 历史回放 batch/completed 都携带 `requestId`。
- 根据 topic 分发消息，不向未订阅 topic 的连接广播。

## 17. 复查问题修复步骤

> 目标：按只读复查报告优先修复 P0/P1 问题，再推进协议一致性与模块化落地。

### S08-20：修复长期记忆供应 pending 竞态

- `memory.context.supply.request` 发送前必须先登记 pending request。
- pending 生命周期由 `PluginBusMemorySupplyDispatcher` 统一管理。
- 响应先于等待任务进入阻塞状态时也必须能正确完成。

### S08-21：统一模型能力协议常量

- 宿主、插件、DTO 默认值统一使用：
  - `model.capability.request`
  - `model.capability.response`
- 删除或替换残留的旧 `llm.invoke` 命名。
- 宿主处理模型能力请求时必须校验 `topic = model` 与 `op = model.capability.request`。

### S08-22：修复历史回放 `hasMore` 语义

- 查询时使用 `batchSize + 1`。
- 实际发送时只发送前 `batchSize` 条。
- 仅当查询结果超过 `batchSize` 时设置 `hasMore = true`。
- `conversation.history.batch` 与 `conversation.history.completed` 均必须回带原始 `requestId`。

### S08-23：补齐主程序端模块化

- 新增 `PluginBusConversationHistoryDispatcher`，接管历史回放 SQL、分页、batch/completed 组包。
- 新增 `PluginBusModelCapabilityDispatcher`，接管模型能力请求处理和错误响应。
- `WebSocketPluginBusServerService` 仅保留连接生命周期、心跳、订阅和路由分发。

### S08-24：补齐插件端模块化

- 新增 `MemoryPluginBusDispatcher`，按 `type/topic/op` 分发消息。
- 新增 `MemoryConversationEventHandler`，处理历史批次与实时 conversation 事件入库。
- 新增 `MemorySupplyRequestHandler`，处理长期记忆供应请求与错误响应。
- `MemoryIngestService` 仅保留连接、订阅、心跳、读循环和生命周期。

### S08-25：协议 Envelope 标准化

- 新生成消息优先使用统一 envelope + `payload` 结构。
- 顶层业务字段只作为临时兼容解析，不再作为新消息生成格式。
- connected/subscribed/error 后续补齐 `source`、`target`、`payload` 或结构化 `error`。
