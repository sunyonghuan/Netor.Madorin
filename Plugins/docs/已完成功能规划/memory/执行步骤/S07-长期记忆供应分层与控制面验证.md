# S07-长期记忆供应分层与控制面验证

> 日期：2026-05-13  
> 范围：`Cortana.Plugins.Memory`、宿主长期记忆上下文注入默认值  
> 状态：已实现并完成控制面验证

## 1. 背景

长期记忆召回工具可以按 agent 或跨 workspace 查到历史记忆，但自动注入链路在真实会话中经常拿不到可供应的记忆。

排查后确认：

1. 供应服务原先只按单一 `agentId + workspaceId` 召回。
2. 历史记忆库中的 workspaceId 与当前宿主传入的 workspace 路径 MD5 不一定一致。
3. `MemoryStore.SearchRecallCandidates` 中 `agentId = null` / `workspaceId = null` 原语义是“不限制过滤”，不能直接代表“全局记忆”。
4. 如果直接跨 agent 或全库 fallback，容易把其它智能体的记忆注入当前上下文。

因此采用“分层供应 + 严格全局语义 + 同 agent 兜底”的方案。

## 2. 本次修改

### 2.1 分层供应策略

`MemorySupplyService` 改为按以下顺序收集候选：

1. 精确层：`agentId = 当前 agent` 且 `workspaceId = 当前 workspace`
2. 智能体级：`agentId = 当前 agent` 且 `workspaceId IS NULL`
3. 工作区级：`agentId IS NULL` 且 `workspaceId = 当前 workspace`
4. 全局级：`agentId IS NULL` 且 `workspaceId IS NULL`

全局级补充数量限制为最多 3 条，避免无差别注入全局记忆。

如果以上严格分层完全没有命中，则启用安全兜底：

- 仅回退到同一个 `agentId`
- 不限制 workspace
- 不跨 agent

这样可以解决历史 workspaceId 与当前 workspaceId 不一致导致自动注入为空的问题，同时避免跨智能体污染。

### 2.2 严格作用域查询

新增 `IMemoryStore.SearchRecallCandidatesInScope(...)`。

目的：显式表达：

- `agentId = value`
- `agentId IS NULL`
- `workspaceId = value`
- `workspaceId IS NULL`

避免沿用旧查询里 `null` 表示“不限制过滤”的语义。

### 2.3 控制面 workspace 归一化

`MemorySupplyControlHandler` 增加 workspace 归一化：

- 如果传入值已经是 32 位十六进制 MD5，直接使用。
- 否则将传入路径按 UTF-8 计算 MD5。

这与历史摄取链路保持一致。

### 2.4 默认供应数量调整

主动供应默认数量从 8 提高到 12：

- `MemoryOptions.MemorySupplyOptions.MaxMemoryCount`
- `MemorySettingsService.GetSupplyOptions`
- `MemoryStore` 默认 seed setting：`global.supply.maxMemoryCount`
- `MemoryWriteToolHandler` 配置说明默认值
- 宿主 `LongMemoryContextProvider` 默认 `memory.contextInjection.maxMemoryCount`

注意：已有数据库中如果已经存在 `supply.maxMemoryCount=8`，seed 的 `INSERT OR IGNORE` 不会覆盖旧值；但宿主自动注入路径会显式传入默认 12。

### 2.5 召回工具 agentId 参数

`memory_recall` 增加可选 `agentId` 参数。

- 空字符串：不按 agent 过滤，便于排查与跨作用域验证。
- 非空：限定指定 agent。

### 2.6 P1 配置读取 AOT 修复

`memory_get_settings` 改为强类型 DTO + source generation 序列化，避免 Native AOT 禁用反射序列化时失败。

新增/调整：

- `MemorySettingsResult`
- `MemorySettingItem`
- `MemoryToolJsonContext` 注册 `MemorySettingsResult`

## 4. 结论

本次修复后，自动注入路径不再只依赖当前 workspace 精确命中。

新的行为是：

1. 优先使用严格分层，避免乱注。
2. 全局记忆只做小量补充。
3. 严格分层完全无命中时，只在同 agent 内跨 workspace 兜底。
4. 实测控制面请求可以拿到相关长期记忆。

因此当前问题的根因是 workspace 作用域过严与历史 workspaceId 不一致；本次修改已经通过控制面 runner 验证修复。
