# 04 - 插件接入与初始化扩展

> 状态：第一版已落地

## 目标

为新标准插件定义初始化扩展槽位和内部事件端点下发方式。

## 当前现状

当前插件 init 参数仅包含：

- `dataDirectory`
- `workspaceDirectory`
- `pluginDirectory`
- `wsPort`

这只能满足旧聊天 WebSocket 接入需求，无法表达新的内部事件流端点。

## 当前已落地槽位

当前插件 init 配置已新增：

- `extensions`：字符串字典扩展槽位，类型语义为 `Dictionary<string, string>`

当前结构：

```json
{
	"dataDirectory": "xxx",
	"workspaceDirectory": "xxx",
	"pluginDirectory": "xxx",
	"wsPort": 52841,
	"extensions": {
		"chatWsEndpoint": "ws://localhost:52841/ws/",           // Deprecated（由 wsPort 取代）
		"conversationFeedEndpoint": "ws://localhost:65321/internal/conversation-feed/", // Deprecated（由 conversationFeedPort 取代）
		"conversationFeedPort": "65321",
		"conversationFeedProtocol": "conversation-feed",
		"conversationFeedVersion": "1.0.0"
	}
}
```

## 设计原则

- `wsPort` 保留为旧聊天 WebSocket 端口字段
- 新标准插件不再依赖单一 `wsPort` 推导内部事件地址
- 内部事件流通过明确字段传入
- Native / Process SDK 都会把完整扩展字段保留为 `PluginSettings.Extensions`
- 已有常用扩展字段可以继续通过兼容属性读取，但新增扩展不再修改 `PluginSettings` 属性集合
- 扩展值统一为字符串；如果插件需要传递复杂 JSON，由宿主写入 JSON 字符串，插件侧自行反序列化

### 字段去重决策（补充）

- 顶层字段：
	- `wsPort` 作为聊天通道主字段，继续保留与使用。
- 扩展字段（extensions）：
	- `conversationFeedPort` 为对话事实流的主字段（新加）。
	- `conversationFeedProtocol`、`conversationFeedVersion` 保留。
	- `chatWsEndpoint`、`conversationFeedEndpoint` 标注为 Deprecated，仅为兼容旧插件，后续可在 N+1/N+2 版本下线。
- 插件消费建议：
	- 聊天通道优先 `wsPort`；
	- 事实流优先 `conversationFeedPort` 组装 Endpoint；缺失时回退到 `conversationFeedEndpoint`（兼容路径）。

## 当前宿主下发链路

- `ExternalProcessPluginHostBase.InitializePluginAsync(...)` 负责组装 `extensions`
- `NativePluginInitConfig` 增加 `extensions` 字符串字典槽位并走源生成序列化
- `CortanaWsEndpoints` 统一产出 `chatWsEndpoint` 与 `conversationFeedEndpoint`

## 当前插件运行时可见字段

- `PluginSettings.Extensions`
- `PluginSettings.ChatWsEndpoint`
- `PluginSettings.ConversationFeedEndpoint`
- `PluginSettings.ConversationFeedProtocol`
- `PluginSettings.ConversationFeedVersion`

## 当前兼容策略

- 旧字段 `wsPort` 继续保留
- 如果扩展字段缺失，Native / Process SDK 仍会基于 `wsPort` 生成默认 endpoint，避免调试场景直接断裂
