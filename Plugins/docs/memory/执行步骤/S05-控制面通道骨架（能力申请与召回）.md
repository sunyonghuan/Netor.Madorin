# S05 - 控制面通道骨架（能力申请与召回）

> 目标：在不影响现有对话与内部事件 feed 的前提下，提供最小可用的“控制面”通道，用于：
> 1) 插件向宿主申请用途级模型能力；
> 2) 宿主向插件请求 RecallPackage。
> 本步骤仅落地“协议与骨架”，先打通调用与降级路径，不实现复杂业务裁决。

## 范围
- 新增内部 WS 控制面端点与消息协议（版本化）。
- 能力申请：`CapabilityRequest` → `CapabilityGrant`（用途级模型路由）。
- 召回请求：`RecallRequest` → `RecallPackage`（最小字段集）。
- 降级策略：超时/异常直接降级为“仅规则记忆 + 聊天历史压缩”。
- 观测：记录 traceId、耗时、超时/降级计数。

不在本步范围：
- 长期记忆 SQLite 落库与检索（移至 S06）。
- 复杂治理/冲突裁决/抽象记忆形成（移至后续步骤）。

## 交付清单
- 宿主侧：
  - `/internal/capability-broker/` 控制面 WS 路径与消息处理骨架。
  - `/internal/recall/` 控制面 WS 路径与消息处理骨架。
  - DTO（源生成）：`CapabilityRequest/Grant`、`RecallRequest/RecallPackage(min)`。
  - DI：`IHostCapabilityBroker` 已可用，控制面调用该接口完成用途级模型解析。
- 插件侧（Memory 插件）：
  - 增加控制面客户端壳（可接收 RecallRequest 并回包最小 RecallPackage）。
  - 如未准备好检索能力，返回 `hits=[]` 且 `confidence=0` 的占位包，带 `producerVersion`。

## 协议（1.0.0）

通用包头：
```json
{
  "type": "request|response|error",
  "protocol": "control-plane",
  "version": "1.0.0"
}
```

1) 能力申请（宿主→宿主本地解析，无需经插件）：
- 请求（宿主内部调用，不经网络，仅定义形态）：
```json
{
  "type": "request",
  "op": "capability.request",
  "requestId": "uuid",
  "purpose": "memory-processing",
  "maxTokens": 4096,
  "timeoutMs": 10000,
  "traceId": "trace"
}
```
- 响应：
```json
{
  "type": "response",
  "op": "capability.grant",
  "requestId": "uuid",
  "granted": true,
  "providerId": "prov-1",
  "modelId": "model-1",
  "expiresAt": "2026-04-25T10:00:00Z",
  "rateLimitPolicy": "shared",
  "traceId": "trace"
}
```

2) 召回请求（宿主→插件，经 `/internal/recall/`）：
- 请求：
```json
{
  "type": "request",
  "op": "recall.request",
  "requestId": "uuid",
  "agentId": "agent-1",
  "workspaceId": "optional-ws",
  "queryText": "本轮任务提示或问题",
  "budget": { "maxTokens": 800 },
  "traceId": "trace"
}
```
- 响应（最小字段集，与 N-2 兼容）：
```json
{
  "type": "response",
  "op": "recall.package",
  "requestId": "uuid",
  "agentId": "agent-1",
  "hitFragments": [],
  "abstractions": [],
  "links": [],
  "rules": [],
  "confidence": 0.0,
  "producerVersion": "1.0.0",
  "traceId": "trace"
}
```
- 错误：
```json
{ "type":"error", "op":"recall", "requestId":"uuid", "message":"reason", "traceId":"trace" }
```

超时：宿主在 `timeoutMs` 内未收到响应 → 直接降级（仅规则记忆 + 历史压缩），记录一次降级事件。

## 实施任务
- 宿主
  - [ ] 在 `WebSocketServerService` 注册新增前缀：`/internal/capability-broker/`、`/internal/recall/`。
  - [ ] 增加控制面消息分发处理器（connected/subscribe/subscribed/error 与 request/response）。
  - [ ] 新增 `ControlPlaneJsonContext`（源生成 DTO）。
  - [ ] 召回编排：超时与异常降级实现，并计入指标。
  - [ ] 观测：traceId 透传、计时、降级/错误计数。
- 插件（Memory）
  - [ ] 新增控制面客户端连接 `/internal/recall/`，接收 `recall.request` 并回包最小 `recall.package`。
  - [ ] 暂不接 DB，命中列表留空，占位返回（带 `producerVersion`）。
  - [ ] 记录请求日志与统计（便于后续接检索能力）。

## 非功能约束
- AOT 安全：所有 DTO 采用强类型 + 源生成 JsonContext。
- 版本：`major.minor`，minor 仅追加可选字段，未识别字段忽略。
- 可用性：控制面异常不影响主对话通路；默认降级可用。
- 安全边界：控制面仅在本机环回地址暴露（同现有 WS），不透出公网。

## 验收清单（DoD）
- [ ] 宿主启动后，`/internal/capability-broker/` 与 `/internal/recall/` 支持 WS 握手（connected/subscribed）。
- [ ] 触发一次 `recall.request`，插件返回最小 `recall.package`（空命中，`confidence=0`，含 `producerVersion`）。
- [ ] 宿主端设置超时为 50ms 时，能观察到降级执行，且不影响主回复完整性。
- [ ] 日志含 `traceId`、请求耗时、降级/错误计数。
- [ ] 任一端异常关闭，另一端不崩溃，宿主自动降级并复用原通路。

## 验收用例
1. 正常召回
   - 前置：宿主与 Memory 插件均运行。
   - 步骤：宿主向插件 `/internal/recall/` 发送 `recall.request`。
   - 期望：返回最小 `recall.package`，宿主将其并入上下文（数量为 0 时仅记录“无命中”）。
2. 超时降级
   - 前置：将 Memory 插件模拟延迟 > 超时阈值。
   - 期望：宿主在阈值内降级执行，日志出现降级计数 + traceId。
3. 错误恢复
   - 步骤：在请求过程中让插件断开连接。
   - 期望：宿主不抛未处理异常，直接降级，主通路可继续。
