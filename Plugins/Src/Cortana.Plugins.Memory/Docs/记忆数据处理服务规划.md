# 记忆数据处理服务规划

本文档规划 `Cortana.Plugins.Memory` 的记忆数据处理服务。当前系统边界已经明确：上层负责实时推送事实片段，插件负责存储、挖掘、提取、归档、抽象和后续查询。当前阶段不做主动推送式记忆供应，优先建设事实片段到长期记忆、抽象记忆的处理链路。

处理相关代码建议统一放入 `Processing/` 目录，便于把数据处理过程、服务、抽取器、摘要器、分类器、合并器、抽象生成器和后续向量化处理集中管理。

## 1. 背景

当前插件已经具备以下基础能力：

1. `MemoryIngestService` 接收宿主推送的实时事实片段和历史回放批次。
2. `observation_records` 保存原始观察记录。
3. `memory_fragments` 保存长期记忆片段。
4. `memory_abstractions` 保存抽象记忆。
5. `memory_events`、`memory_mutations` 已具备记录内部事件和审计变更的表结构。
6. 配置读取服务和召回服务已经完成基础验收。

但当前缺少核心处理链路：如何把持续产生的 `observation_records` 稳定处理为长期记忆，再进一步生成抽象记忆，并维护处理状态、处理进度和失败重试。

## 2. 当前结论

下一阶段的核心服务应调整为：**记忆数据处理服务**。

暂缓：

1. 主动推送式记忆供应服务。
2. 依赖上层主动接收记忆的能力。
3. 复杂查询体验优化。

优先：

1. 事实片段处理。
2. 长期记忆提取。
3. 记忆去重、合并、评分。
4. 抽象记忆生成。
5. 处理状态和进度管理。
6. 可恢复、可重试、可审计的后台处理链路。

## 3. 服务定位

记忆数据处理服务位于写入链路后半段。

```text
宿主实时事实流
	↓
MemoryIngestService
	↓
observation_records
	↓
MemoryProcessingService
	↓
memory_fragments
	↓
MemoryAbstractionProcessing
	↓
memory_abstractions
```

核心职责：

1. 从原始观察记录中识别可长期保存的信息。
2. 将低层事实转为结构化长期记忆片段。
3. 对重复、相近、冲突的记忆进行整理。
4. 将多个稳定片段归纳为抽象记忆。
5. 记录处理事件、变更审计、进度游标和失败信息。

## 4. 非目标

本阶段不做以下内容：

1. 不做主动推送记忆。
2. 不依赖上层新增交互协议。
3. 不做复杂向量检索。
4. 不做模型驱动的复杂语义抽取。
5. 不做最终提示词拼接。
6. 不做完整冲突裁决系统。
7. 不做记忆可视化界面。

## 5. 总体链路

建议链路如下：

```text
定时触发或入库后触发
	↓
读取待处理 observation_records
	↓
按 agentId / workspaceId / sessionId / turnId 分组
	↓
清洗文本和过滤无效片段
	↓
候选长期记忆抽取
	↓
记忆类型识别：fact / preference / task / constraint / summary
	↓
打分：importance / confidence / novelty / salience / retention
	↓
去重和合并
	↓
写入 memory_fragments
	↓
记录 memory_events / memory_mutations
	↓
更新处理进度
	↓
满足条件时触发抽象记忆生成
	↓
写入 memory_abstractions / memory_links
```

## 6. 第一版处理策略

第一版应使用确定性规则，不依赖大模型。

该结论需要调整：第一版可以先打通确定性框架，但真正可用的摘要生成、主题生成、记忆类型生成和抽象记忆生成高度依赖大模型。此前规划中已经存在插件向宿主申请用途级模型能力的通道，因此记忆处理服务应通过该授权能力调用模型，而不是在插件内直接绑定具体模型或供应商。

原则：

1. 程序负责清洗、过滤、切分、批处理、幂等、状态维护和审计。
2. 模型负责摘要生成、主题生成、类型判断、候选记忆提取、抽象归纳和必要的语义质量判断。
3. 插件只申请用途级模型能力，不直接指定最终 Provider 或 Model。
4. 所有模型输出都必须带来源证据，能够追溯到原始 `observation_records` 和原文内容。
5. 模型失败时可以降级为保守规则处理，但不能生成不可追溯的记忆。

第一版处理内容：

| 阶段 | 第一版策略 |
| --- | --- |
| 文本清洗 | 去空白、过滤空内容、限制最大长度。 |
| 候选识别 | 基于角色、事件类型、关键词和文本长度。 |
| 类型识别 | 关键词规则识别偏好、事实、任务、约束。 |
| 摘要生成 | 优先使用大模型生成，并保留原文证据；失败时使用原文截断。 |
| 主题生成 | 优先使用大模型生成，并保留推断依据；失败时使用关键词规则。 |
| 打分 | 确定性加权规则。 |
| 去重 | 基于规范化文本哈希和近似标题匹配。 |
| 合并 | 增加来源证据、提高置信度或强化次数。 |
| 抽象 | 按主题聚合后调用大模型归纳，满足数量和置信度阈值后生成。 |

## 7. 记忆类型识别

建议第一版识别以下类型：

| 类型 | 说明 | 示例信号 |
| --- | --- | --- |
| `fact` | 稳定事实。 | “我是……”、“项目使用……”、“当前环境是……” |
| `preference` | 用户偏好。 | “我喜欢……”、“我希望……”、“以后都……” |
| `constraint` | 约束和规则。 | “必须……”、“不要……”、“禁止……”、“统一……” |
| `task` | 任务和计划。 | “下一步……”、“待办……”、“计划……” |
| `summary` | 会话总结。 | turn completed、assistant response 中的总结性内容。 |

不确定内容默认进入 `fact` 或跳过，避免过度分类。

## 8. 候选抽取规则

第一版建议规则：

1. 空内容不处理。
2. 过短内容不处理，例如少于 4 个有效字符。
3. `user` 角色优先级高于 `assistant`。
4. 明确偏好、规则、约束类表达优先入库。
5. 工具调用日志和纯状态事件默认不入长期记忆。
6. assistant 增量片段不直接入长期记忆，优先使用 turn completed 汇总内容。
7. 对同一 messageId 的 assistant delta 应避免碎片化入库。

## 9. 评分策略

建议初始评分：

```text
salienceScore = importance * 0.35
			  + confidence * 0.30
			  + novelty * 0.20
			  + roleWeight * 0.15
```

建议默认值：

| 类型 | importance | confidence | novelty | roleWeight |
| --- | --- | --- | --- | --- |
| preference | 0.85 | 0.75 | 0.70 | user=1.0 |
| constraint | 0.90 | 0.80 | 0.70 | user=1.0 |
| fact | 0.65 | 0.65 | 0.60 | user=0.9 |
| task | 0.70 | 0.65 | 0.65 | user=0.9 |
| summary | 0.55 | 0.60 | 0.50 | assistant=0.7 |

第一版不追求精确，只要求稳定、可解释、可调整。

## 10. 去重与合并策略

### 10.1 稳定指纹

建议基于以下字段生成稳定指纹：

```text
agentId + workspaceId + memoryType + normalizedSummary
```

`normalizedSummary` 应执行：

1. Trim。
2. 转小写。
3. 合并连续空白。
4. 去除常见标点。

### 10.2 合并规则

如果发现相同或高度相似记忆：

1. 不新增重复 `memory_fragments`。
2. 合并 `sourceObservationIdsJson`。
3. 合并 `sourceSessionIdsJson` 和 `sourceTurnIdsJson`。
4. 增加 `reinforcementCount`。
5. 更新 `confidence`、`retentionScore`、`updatedAt`。
6. 写入 `memory_mutations`。

## 11. 生命周期策略

第一版建议状态流转：

```text
observation_records
	↓
candidate / pending
	↓ 多次证据或高置信度
active / confirmed
	↓ 后续抽象支撑
reinforced / confirmed
```

暂不自动进入 `forgotten`，遗忘和衰减后续单独实现。

## 12. 抽象记忆生成

抽象记忆不是每条事实都生成，而是基于多个稳定片段归纳。

第一版触发条件：

1. 同一 `agentId`。
2. 同一 `workspaceId` 或全局范围。
3. 同一 `memoryType` 或相近 `topic`。
4. 支撑片段数 >= `abstraction.minimumSupportCount`。
5. 平均置信度 >= `abstraction.minimumConfidence`。

第一版生成策略：

| 字段 | 生成方式 |
| --- | --- |
| `abstractionType` | 来自主要 memoryType。 |
| `title` | “关于 {topic} 的长期记忆”。 |
| `statement` | 使用模板拼接稳定结论。 |
| `summary` | 合并前几条支撑片段摘要。 |
| `supportingMemoryIdsJson` | 支撑片段 ID 集合。 |
| `confidence` | 支撑片段平均值。 |
| `stabilityScore` | 支撑数量和平均置信度加权。 |

## 13. 大模型能力申请

摘要、主题、类型、候选长期记忆抽取和抽象归纳应通过宿主授权的大模型能力完成。

建议用途：

| 用途 | Purpose | 场景 |
| --- | --- | --- |
| 事实片段处理 | `memory-processing` | 从 observation 中抽取候选长期记忆、判断类型、生成摘要和主题。 |
| 抽象记忆生成 | `memory-abstraction` | 从多个长期记忆片段中归纳稳定抽象记忆。 |
| 冲突判断 | `memory-conflict-resolution` | 后续用于判断新旧记忆是否冲突。 |

调用原则：

1. 插件只向宿主申请 Purpose，不直接选择供应商或模型。
2. 请求必须带 `requestId`、`traceId`、预算和超时。
3. 模型输出必须是强类型结构，避免自由文本难以解析。
4. 模型输出必须包含来源 observation id、原文片段位置或引用文本。
5. 模型调用失败时，当前批次应可重试或降级，不能产生不可解释结果。

## 14. 原文追溯和证据链

所有记忆处理结果必须有迹可查。

长期记忆片段至少应能追溯到：

1. `sourceObservationIdsJson`。
2. `sourceSessionIdsJson`。
3. `sourceTurnIdsJson`。
4. 原始 `ObservationRecord.SourceFactsJson`。
5. 原始文本 `ObservationRecord.Content`。

抽象记忆至少应能追溯到：

1. `supportingMemoryIdsJson`。
2. 支撑长期记忆的 `sourceObservationIdsJson`。
3. 生成抽象时的模型请求摘要。
4. 模型输出结构。

建议后续增加处理证据字段或处理日志，用于保存：

| 字段 | 用途 |
| --- | --- |
| `processorVersion` | 处理器版本。 |
| `modelPurpose` | 使用的模型用途。 |
| `modelProviderId` | 宿主授权的模型供应方。 |
| `modelId` | 实际模型标识。 |
| `promptHash` | 提示词哈希，避免保存完整敏感提示词。 |
| `inputObservationIdsJson` | 输入观察记录集合。 |
| `outputMemoryIdsJson` | 输出记忆集合。 |
| `traceId` | 链路追踪标识。 |

## 15. 向量化处理

如果后续需要语义检索、近似去重、联想查询，应将向量化纳入记忆处理链路，而不是单独散落在查询服务中。

建议位置：

```text
ObservationRecord
	↓
候选长期记忆抽取
	↓
MemoryFragment 写入
	↓
向量化 MemoryFragment.Summary / Detail / Keywords
	↓
保存向量索引或向量引用
	↓
抽象记忆生成后再向量化 MemoryAbstraction.Statement / Summary
```

第一版可以不实现向量化，但接口和目录应预留：

1. `Processing/Vectorization/`。
2. 向量化任务状态。
3. 向量生成失败重试。
4. 向量与原始记忆 ID 的映射。
5. 向量模型授权用途。

## 16. 处理状态和进度

这是本服务的关键能力。

当前表结构已有 `memory_events`，但它更偏事件记录，不适合作为完整进度表。建议新增处理进度表。

建议新增：`memory_processing_states`

| 字段 | 说明 |
| --- | --- |
| `id` | 状态记录标识。 |
| `processorName` | 处理器名称，例如 fragment-extraction、abstraction-generation。 |
| `agentId` | 智能体标识，可空表示全局处理器。 |
| `workspaceId` | 工作区标识，可空。 |
| `state` | idle、running、completed、failed。 |
| `lastObservationTimestamp` | 已处理到的观察记录时间戳。 |
| `lastObservationId` | 已处理到的观察记录 ID。 |
| `processedCount` | 累计处理数量。 |
| `createdFragmentCount` | 累计创建片段数量。 |
| `mergedFragmentCount` | 累计合并片段数量。 |
| `createdAbstractionCount` | 累计创建抽象数量。 |
| `lastError` | 最近一次错误。 |
| `lockedUntil` | 处理锁过期时间。 |
| `createdAt` | 创建时间。 |
| `updatedAt` | 更新时间。 |

说明：如果暂时不想新增表，也可以先用 `memory_events` 记录处理事件。但为了“状态和进度”可查询、可恢复，独立表更合适。

## 17. 处理服务接口建议

建议新增：

```text
IMemoryProcessingService
	MemoryProcessingResult Process(MemoryProcessingRequest request)
	MemoryProcessingState GetState(string processorName, string? agentId, string? workspaceId)
```

请求模型：

```text
MemoryProcessingRequest
	RequestId
	AgentId
	WorkspaceId
	MaxObservationCount
	TriggerSource
	TraceId
```

结果模型：

```text
MemoryProcessingResult
	RequestId
	ProcessedObservationCount
	CreatedFragmentCount
	MergedFragmentCount
	CreatedAbstractionCount
	FailedObservationCount
	State
	Summary
```

## 18. 后台服务建议

建议新增一个后台服务：

```text
MemoryProcessingHostedService
```

职责：

1. 定时触发 `IMemoryProcessingService.Process`。
2. 控制扫描间隔和批量大小。
3. 避免并发重复处理。
4. 记录异常但不影响插件进程。

第一版可先实现手动 `Process`，再接 hosted service。

## 19. 存储契约建议

建议扩展 `IMemoryStore`：

```text
IReadOnlyList<ObservationRecord> GetUnprocessedObservations(...)
MemoryProcessingState GetProcessingState(...)
void UpsertProcessingState(...)
IReadOnlyList<MemoryFragment> SearchSimilarFragments(...)
```

现有方法可以继续复用：

1. `UpsertMemoryFragment`。
2. `UpsertMemoryAbstraction`。
3. `UpsertMemoryLink`。
4. `InsertMemoryEvent`。
5. `InsertMemoryMutation`。

## 20. 配置建议

当前已有：

| 配置 | 用途 |
| --- | --- |
| `abstraction.minimumSupportCount` | 抽象记忆最小支撑数量。 |
| `abstraction.minimumConfidence` | 抽象记忆最低置信度。 |
| `governance.auditEnabled` | 是否记录审计。 |

建议后续新增：

| 配置 | 默认值 | 用途 |
| --- | --- | --- |
| `processing.enabled` | `true` | 是否启用记忆处理。 |
| `processing.batchSize` | `100` | 每轮最多处理观察记录数。 |
| `processing.scanIntervalSeconds` | `30` | 后台扫描间隔。 |
| `processing.minimumContentLength` | `4` | 最小有效文本长度。 |
| `processing.maxContentLength` | `4000` | 单条处理最大文本长度。 |
| `processing.includeAssistantMessages` | `true` | 是否处理助手消息。 |

第一版可以先硬编码保守默认值，后续再补配置表默认数据。

## 21. 错误处理

建议策略：

1. 单条 observation 失败不影响整批。
2. 失败记录写入 `memory_events`，事件类型为 `processing.failed`。
3. 批处理失败更新 `memory_processing_states.state = failed`。
4. 下一轮允许从上一个成功游标继续。
5. 不吞掉关键异常，后台服务记录日志后继续下一轮。

## 22. 审计策略

创建或合并记忆时写入 `memory_mutations`：

| 操作 | MutationType |
| --- | --- |
| 创建片段 | `create` |
| 合并片段 | `merge` |
| 更新评分 | `update` |
| 创建抽象 | `abstract` |

同时写入 `memory_events`：

| 事件 | EventType |
| --- | --- |
| 批处理开始 | `processing.started` |
| 批处理完成 | `processing.completed` |
| 单条失败 | `processing.failed` |
| 创建片段 | `fragment.extracted` |
| 创建抽象 | `abstraction.created` |

## 23. 第一版实施顺序

建议按以下顺序实施：

1. 新建 `Processing/` 目录，并把处理过程和服务统一放入该目录。
2. 新增处理状态模型。
3. 新增 `memory_processing_states` 表和迁移版本。
4. 扩展 `IMemoryStore` 查询待处理观察记录和读写处理状态。
5. 新增 `IMemoryProcessingService`。
6. 新增 `MemoryProcessingService`，先实现批处理编排、文本清洗和基础过滤。
7. 接入宿主模型能力授权，用于摘要、主题、类型和候选长期记忆生成。
8. 实现基础去重和合并。
9. 写入 `memory_events` 和 `memory_mutations`。
10. 实现基于模型的基础抽象生成。
11. 预留 `Processing/Vectorization/` 和向量化任务状态。
12. 新增 `MemoryProcessingHostedService` 定时触发。
13. 注册 DI。
14. 构建验证。
15. 编写处理服务验收文档。

## 24. 验收标准

第一版完成后应满足：

1. 新入库的 `observation_records` 能被处理为 `memory_fragments`。
2. 空内容、过短内容不会生成长期记忆。
3. 同一观察记录重复处理不会重复创建相同记忆。
4. 相似记忆会合并来源和强化计数。
5. 处理进度可查询，并能在重启后继续。
6. 处理失败可记录，不阻塞后续记录处理。
7. 满足条件的片段能生成 `memory_abstractions`。
8. 关键创建、合并、失败操作有 `memory_events` 或 `memory_mutations` 记录。
9. 后台服务不会阻塞插件启动和事实流入库。
10. 摘要、主题、类型和抽象生成通过宿主授权模型能力完成，且有超时和失败降级策略。
11. 每条长期记忆都能追溯到原始 `observation_records` 和原文内容。
12. 每条抽象记忆都能追溯到支撑长期记忆和原始观察记录。
13. 如启用向量化，向量记录必须能反查对应记忆 ID 和原文来源。
14. 项目构建通过。

## 25. 风险

| 风险 | 说明 | 缓解 |
| --- | --- | --- |
| 噪声入库 | 规则抽取可能把无价值内容保存为记忆。 | 设置最小长度、关键词规则和置信度阈值。 |
| 重复记忆 | 指纹过弱会产生重复。 | 使用规范化摘要和来源合并。 |
| 错误合并 | 相似但语义不同的内容可能被合并。 | 第一版只合并完全指纹一致或高度确定内容。 |
| 抽象过早 | 支撑证据不足时生成不稳定抽象。 | 使用最小支撑数和最低平均置信度。 |
| 后台处理影响写入 | 大批处理可能占用 SQLite。 | 限制批量大小，使用短事务。 |
| 模型输出不稳定 | 摘要、主题和类型可能存在幻觉或漂移。 | 使用强类型输出、来源引用、置信度和失败降级。 |
| 模型授权不可用 | 宿主未授予模型能力或超时。 | 当前批次进入待重试或使用保守规则，不生成不可追溯记忆。 |
| 向量与原文脱节 | 向量索引可能无法追溯来源。 | 向量记录必须保存 memoryId、sourceObservationIds 和模型信息。 |
