# 记忆数据处理服务验收

## 1. 验收范围

本次验收范围为 `Cortana.Plugins.Memory` 第一版记忆数据处理服务，重点核对以下内容：

1. 规划要求是否已有对应实现。
2. 数据库字段是否能支撑处理状态、原文追溯、片段生成和审计。
3. 上游摄取服务是否能提供处理服务需要的输入字段。
4. 处理服务是否能把 `observation_records` 转换为 `memory_fragments`。
5. 当前未完成项是否明确记录。

验收基准文档：

- `Docs/记忆数据处理服务规划.md`

核心实现文件：

- `Processing/MemoryProcessingModels.cs`
- `Processing/IMemoryProcessingService.cs`
- `Processing/MemoryProcessingService.cs`
- `Processing/IMemorySemanticProcessor.cs`
- `Processing/FallbackMemorySemanticProcessor.cs`
- `Storage/IMemoryStore.cs`
- `Storage/MemoryStore.cs`
- `Services/MemoryIngestService.cs`
- `Startup.cs`

## 2. 总体验收结论

当前实现可以作为第一版处理链路的基础版本通过阶段验收。

已满足的核心能力：

1. 已新增 `Processing/` 目录集中存放处理相关代码。
2. 已新增处理请求、结果、状态和候选记忆模型。
3. 已新增 `memory_processing_states` 处理状态表。
4. 已扩展 `IMemoryStore`，支持待处理观察记录查询、处理状态读写和相似片段查询。
5. 已实现 `MemoryProcessingService.Process` 的第一版批处理链路。
6. 已实现文本清洗、空内容和过短内容过滤。
7. 已实现从 `observation_records` 生成 `memory_fragments`。
8. 已实现基于摘要和类型的保守去重、合并和强化计数。
9. 已写入 `memory_events` 和 `memory_mutations`。
10. 已保留 observation、session、turn 的追溯链路。
11. 已注册 DI。
12. 构建已通过。

需要标记为后续继续完善的能力：

1. 宿主授权大模型能力尚未真正接入，目前仅有 `IMemorySemanticProcessor` 适配接口和 `FallbackMemorySemanticProcessor` 降级实现。
2. 抽象记忆生成尚未实现。
3. `MemoryProcessingHostedService` 定时处理尚未实现。
4. 向量化目录和向量化任务状态尚未实现。
5. 当前没有专门的处理日志表，处理证据主要依赖 `memory_events`、`memory_mutations` 和记忆来源字段。

## 5. 上游服务验收

上游服务为 `MemoryIngestService`。

### 5.1 历史批次链路

链路：

```text
conversation-feed replay
	↓
conversation.export.batch
	↓
IngestExportBatch
	↓
ObservationRecord
	↓
BulkInsertObservations
	↓
observation_records
```

验收结论：通过。

说明：历史批次会保存原始 item JSON 到 `SourceFactsJson`，满足原文和来源事实追溯要求。

### 5.2 实时事件链路

链路：

```text
conversation-feed event
	↓
TryIngestLiveEvent
	↓
ObservationRecord
	↓
InsertObservation
	↓
observation_records
```

当前支持的实时事件：

1. `conversation.user.message`
2. `conversation.assistant.delta`
3. `conversation.turn.completed`

验收结论：通过。

说明：实时事件同样保存 `SourceFactsJson`，并尽量解析 `AgentId`、`WorkspaceId`、`TurnId`、消息 ID、内容和 TraceId。

### 5.3 上游到处理服务字段链路

| 上游字段 | ObservationRecord | MemoryFragment | 验收结论 |
| --- | --- | --- | --- |
| 原始文本 | `Content` | `Summary`、`Detail` | 通过 |
| 原始 JSON | `SourceFactsJson` | 可通过 `SourceObservationIdsJson` 回查 | 通过 |
| 会话 ID | `SessionId` | `SourceSessionIdsJson` | 通过 |
| 轮次 ID | `TurnId` | `SourceTurnIdsJson` | 通过 |
| 观察记录 ID | `Id` | `SourceObservationIdsJson` | 通过 |
| TraceId | `TraceId` | `TagsJson`、events、mutations | 部分通过，fragment 中为标签形式，不是独立字段 |

结论：上游字段能够支撑第一版处理服务；TraceId 如后续需要精确查询，建议增加独立处理证据表或扩展字段。

## 6. 处理服务验收

### 6.1 输入查询

`MemoryProcessingService.Process` 通过以下接口读取待处理数据：

- `GetUnprocessedObservations(processorName, agentId, workspaceId, limit)`

查询基于：

1. `memory_processing_states.lastObservationTimestamp`
2. `memory_processing_states.lastObservationId`
3. `createdTimestamp, id` 顺序

验收结论：通过。

说明：该方式支持重启后继续处理。

### 6.2 文本清洗和过滤

当前由 `FallbackMemorySemanticProcessor` 完成：

1. 空文本返回空候选。
2. 小于 4 个字符的文本不生成候选。
3. 合并连续空白字符。
4. 摘要最大截断到 240 字符。

验收结论：通过。

### 6.3 语义生成

当前实现：

1. `MemoryType` 基于关键词推断。
2. `Topic` 基于关键词推断。
3. `Summary` 使用清洗后的原文截断。
4. `Keywords` 使用分隔符切词。

验收结论：部分通过。

原因：规划要求摘要、主题、类型和候选长期记忆生成应通过宿主授权大模型完成。当前仅实现适配接口和降级规则，未实际调用宿主模型授权能力。

### 6.4 片段创建

当前写入字段包括：

1. 稳定 fragment id。
2. `agentId`、`workspaceId`。
3. `memoryType`、`topic`、`title`、`summary`、`detail`。
4. `keywordsJson`、`tagsJson`。
5. `sourceObservationIdsJson`。
6. `sourceSessionIdsJson`。
7. `sourceTurnIdsJson`。
8. 评分和生命周期字段。

验收结论：通过。

### 6.5 去重和合并

当前策略：

1. 查询同一 agent、workspace、memoryType、summary 的片段。
2. 未命中则创建。
3. 命中则追加来源 observation/session/turn。
4. 更新置信度、重要性、新颖度、显著性、保留分和强化次数。

验收结论：通过。

说明：当前是保守合并策略，符合第一版避免错误合并的要求。

### 6.6 处理状态

当前状态更新点：

1. 开始处理时写入 `running`。
2. 批次完成后写入 `completed`。
3. 整体异常时写入 `failed` 和 `lastError`。
4. 每条 observation 成功经过处理循环后更新游标。

验收结论：通过。

### 6.7 失败记录

当前失败记录方式：

1. 单条 observation 处理异常时写入 `processing.failed` 事件。
2. 整体批次异常时写入 `processing.failed` 事件。
3. 整体批次异常时更新状态表 `lastError`。

验收结论：通过。

## 8. 当前风险

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| 语义质量依赖降级规则 | 摘要、主题、类型可能不准确 | 下一步接入宿主授权模型能力。 |
| TraceId 未作为 fragment 独立字段 | 按 TraceId 查询片段不方便 | 后续增加处理证据表或扩展字段。 |
| 状态锁字段未实际使用 | 并发调用可能重复处理同一批数据 | 后续实现 `lockedUntil` 占用和释放逻辑。 |
| 未实现抽象生成 | 无法形成高层稳定记忆 | 下一阶段实现 `memory-abstraction` 用途模型调用。 |
| 未实现后台处理服务 | 当前需要手动调用处理服务 | 后续新增 `MemoryProcessingHostedService`。 |
| 未实现向量化 | 无法做语义近似检索和向量去重 | 后续增加 `Processing/Vectorization/`。 |

## 9. 后续整改清单

建议按以下顺序继续：

1. 实现宿主授权模型适配器，用 `memory-processing` purpose 替换降级语义生成。
2. 为模型输出定义强类型 JSON 契约，要求包含来源 observation id 和引用文本。
3. 实现 `memory-abstraction` 抽象记忆生成。
4. 实现 `lockedUntil` 并发处理锁。
5. 新增 `MemoryProcessingHostedService`，默认可配置开启。
6. 新增 `Processing/Vectorization/`，预留向量生成、失败重试和向量来源映射。
7. 增加处理证据表，保存 processorVersion、modelPurpose、modelId、promptHash、traceId、输入输出记录集合。
8. 为处理服务补充单元测试和集成测试。

## 10. 验收结论

第一版记忆数据处理服务已完成“数据生产链路打通”的目标：

```text
conversation-feed
	↓
MemoryIngestService
	↓
observation_records
	↓
MemoryProcessingService
	↓
memory_fragments
	↓
memory_events / memory_mutations / memory_processing_states
```

当前可以进入下一阶段：宿主授权大模型适配、抽象记忆生成、后台定时处理和向量化链路实现。
