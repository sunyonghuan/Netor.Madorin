# 长期记忆上下文自动注入方案

> 当前状态：已选择并落地“宿主 `LongMemoryContextProvider` + 内部 WebSocket/feed 扩展协议 `memory.context.supply.*`”方案。现有 `memory_supply_context` 工具继续保留给 AI 主动查询、MCP、调试和验收，不承担默认注入职责。

## 1. 背景

增强记忆插件已经具备对话事实摄取、观察记录处理、长期记忆生成、抽象记忆生成、召回和供应能力，并已暴露 `memory_supply_context` 工具。

当前缺口是：主对话默认上下文构建时不会主动请求长期记忆，是否使用记忆依赖 AI 自己主动调用工具。模型如果没有调用 `memory_supply_context`，相关记忆就不会进入本轮上下文。

本方案将“宿主自动注入 Provider”和“插件内部控制面供应通道”结合：

```text
主对话构建 AIContext 前
    ↓
宿主 LongMemoryContextProvider
    ↓
宿主向 Memory 插件请求 memory.supply
    ↓
Memory 插件复用 IMemorySupplyService
    ↓
返回 MemorySupplyResult
    ↓
宿主格式化为 AIContext.Instructions
```

核心原则：**插件负责供应结构化记忆包，宿主负责决定是否注入以及如何拼接 prompt。**

## 2. 当前基础能力检查

### 2.1 插件已有内部 WebSocket 客户端

当前 Memory 插件已经通过内部 PluginBus 接收宿主事实流，宿主在插件初始化时传入：

- `conversationFeedEndpoint`
- `conversationFeedPort`
- `conversationFeedProtocol`
- `conversationFeedVersion`
- `modelCapabilityEndpoint`

因此，后续不需要额外发明一套与宿主通信的底层连接机制，可以在现有内部 WebSocket 连接体系上扩展控制面消息，或复用同一 feed 端点增加请求/响应操作。

### 2.2 现有记忆工具

Memory 插件已经暴露读取类工具：

| 工具 | 用途 |
| --- | --- |
| `memory_recall` | 根据查询文本召回相关长期记忆。 |
| `memory_supply_context` | 根据当前任务和最近消息生成结构化记忆供应包。 |
| `memory_get_status` | 查看记忆系统基础状态。 |

其中 `memory_supply_context` 最接近自动注入需求。

当前工具入口：

- `Tools/MemoryReadTools.cs`
- `ToolHandlers/MemoryReadToolHandler.cs`
- `Services/IMemorySupplyService.cs`
- `Services/MemorySupplyService.cs`

工具调用最终会复用：

```text
memory_supply_context
    ↓
MemoryReadToolHandler.SupplyContext(...)
    ↓
IMemorySupplyService.Supply(...)
```

## 3. 是否可以直接调用工具完成注入

### 3.1 技术上可以调用

宿主插件系统会把插件工具包装成 `AITool`：

- Native 插件通过 `NativePluginWrapper` 构建 `AITool`。
- 每个工具内部通过 `ExternalProcessPluginHostBase.SendRequestAsync(...)` 向插件子进程发送 `invoke` 请求。
- `memory_supply_context` 暴露给 AI 时，工具参数以 `argsJson` 字符串形式传给插件。

因此，从宿主进程角度看，理论上可以找到 Memory 插件的 `AITool`，构造参数 JSON，然后主动执行该工具。

### 3.2 但不建议把“宿主默认注入”建立在工具调用层上

直接调用工具有几个问题：

1. **工具层面向 AI，不面向宿主控制面。**
   `memory_supply_context` 的工具签名是为了模型调用而设计，参数被压缩成少数字符串，不能完整表达宿主上下文。

2. **工具返回是工具结果包装字符串。**
   当前工具返回 `MemoryToolResult.Ok(...)` 包装后的字符串，宿主还要再解析内部 JSON，链路多一层协议壳。

3. **工具调用超时时间偏长。**
   现有外部插件请求默认读响应超时约 30 秒，不适合主对话构建上下文前的快速注入。自动注入应该使用 150ms～300ms 级别的短超时。

4. **工具参数缺少关键作用域字段。**
   详见下一节。

5. **工具名可能受暴露名和冲突过滤影响。**
   `AITool.Name` 可能是短名，也可能因冲突过滤被排除。宿主内部关键链路不应依赖 AI 工具暴露策略。

结论：**可以作为过渡验证手段，不建议作为正式自动注入链路。**

## 4. 现有工具参数是否满足条件

### 4.1 `memory_supply_context` 当前参数

当前工具参数为：

```text
scenario: string
currentTask: string
recentMessages: string
workspaceId: string
maxMemoryCount: int
maxTokenBudget: int
triggerSource: string
```

对应 `MemorySupplyRequest` 当前模型字段：

```text
RequestId
AgentId
WorkspaceId
Scenario
CurrentTask
RecentMessages
TriggerSource
MaxMemoryCount
MaxTokenBudget
TraceId
```

### 4.2 缺失或不充分字段

| 需求字段 | 当前工具是否支持 | 说明 |
| --- | --- | --- |
| 当前智能体 `agentId` | 不支持显式传入 | 工具内部使用 `runtimeContext.ResolveAgentId(null)`，默认是 `default`。这会影响长期记忆隔离。 |
| 当前工作目录 / 工作区路径 | 不支持显式传入 | 只支持 `workspaceId` 字符串，无法区分“路径”和“稳定工作区 ID”。 |
| 当前会话 ID `sessionId` | 不支持 | 不能让供应服务基于当前会话做更可靠过滤或排除。 |
| 当前 Turn ID `turnId` | 不支持 | 无法精确追踪本轮注入。 |
| 当前消息 ID `messageId` | 不支持 | 无法关联具体用户消息。 |
| 当前模型 / provider | 不支持 | 暂非第一版必须，但对预算和策略有帮助。 |
| 请求 ID `requestId` | 不支持显式传入 | 工具内部由模型默认生成。 |
| Trace ID `traceId` | 不支持显式传入 | 工具内部新建 Guid，宿主无法串联主对话 trace。 |
| 最近消息结构化列表 | 弱支持 | 当前是按换行拆分的字符串，不能携带 role、messageId、createdAt。 |

### 4.3 关键问题：`agentId` 当前不可靠

`MemoryReadToolHandler.SupplyContext(...)` 当前构造请求时使用：

```text
AgentId = runtimeContext.ResolveAgentId(null)
```

而 Native 插件启动时注册默认上下文：

```text
new MemoryRuntimeContext("default", null, "plugin")
```

也就是说，如果没有其他链路调用 `SetScope(...)` 设置当前 agent，工具供应默认会以 `default` 作为智能体标识。这对长期记忆召回是不可靠的。

### 4.4 当前工具不足以承担正式自动注入

现有 `memory_supply_context` 可以提供“任务文本 + 最近消息 + 工作区”的粗粒度供应，但缺少自动注入需要的可靠作用域：

- agentId
- sessionId
- turnId
- messageId
- workspaceDirectory
- traceId

因此，正式自动注入不应直接使用现有工具签名。

## 5. 方案决策

### 5.1 不直接用 AI 工具作为正式链路

保留 `memory_supply_context` 用于：

1. AI 主动查询长期记忆。
2. 用户显式要求检查记忆。
3. 调试、验收和 MCP 独立模式。
4. 临时验证供应服务质量。

但宿主默认注入链路不依赖它。

### 5.2 新增宿主内部供应请求

新增宿主侧 `LongMemoryContextProvider`，在构建 AI 上下文前自动发起内部供应请求。

请求不要走 AI 工具暴露层，而是走插件内部控制面。控制面可以直接复用当前内部 WebSocket 客户端连接宿主 feed 的能力，在已有连接体系上增加请求/响应操作。

推荐控制面操作名：

```text
memory.context.supply.request
memory.context.supply.response
memory.context.supply.error
```

当前第一版实际复用 `/internal` 连接：Memory 插件作为 feed 客户端保持长连接，宿主通过该连接下发 `memory.context.supply.request`，插件回传 `memory.context.supply.response` 或 `memory.context.supply.error`。宿主以 `requestId` 建立短等待映射，超时或错误时静默降级为空上下文。

### 5.3 插件侧复用同一个业务服务

控制面处理器不重新实现召回逻辑，只做协议适配：

```text
memory.context.supply.request
    ↓
MemorySupplyControlHandler
    ↓
IMemorySupplyService.Supply(...)
    ↓
memory.context.supply.response
```

这与现有工具共享同一底层能力：

```text
AI 工具 memory_supply_context ─┐
                              ├─ IMemorySupplyService
控制面 memory.context.supply.request ──┘
```

## 6. 控制面请求模型

建议新增完整请求模型，字段面向宿主默认注入：

```text
MemoryContextSupplyRequest
- RequestId: string
- AgentId: string
- AgentName: string?
- WorkspaceId: string?
- WorkspaceDirectory: string?
- SessionId: string?
- TurnId: string?
- MessageId: string?
- Scenario: string?
- CurrentTask: string?
- RecentMessages: IReadOnlyList<MemoryContextMessage>
- TriggerSource: string? = before-prompt
- MaxMemoryCount: int? = 8
- MaxTokenBudget: int? = 1200
- TraceId: string?
```

最近消息建议结构化：

```text
MemoryContextMessage
- MessageId: string?
- Role: string
- Content: string
- CreatedAt: string?
```

插件内部第一版可以只映射到现有 `MemorySupplyRequest`：

```text
AgentId            -> AgentId
WorkspaceId        -> WorkspaceId
Scenario           -> Scenario
CurrentTask        -> CurrentTask
RecentMessages     -> RecentMessages.Select(m => $"{m.Role}: {m.Content}")
TriggerSource      -> TriggerSource
MaxMemoryCount     -> MaxMemoryCount
MaxTokenBudget     -> MaxTokenBudget
TraceId            -> TraceId
```

`SessionId`、`TurnId`、`MessageId` 第一版可先只用于 trace 和日志，后续再进入召回策略。

## 7. 宿主侧实现方案

### 7.1 新增 `LongMemoryContextProvider`

职责：

1. 在 `ProvideAIContextAsync(...)` 中被自动调用。
2. 从 `InvokingContext`、`AgentSession.StateBag`、`IAppPaths`、数据库或当前请求中提取上下文。
3. 调用 `ILongMemorySupplyClient`。
4. 将返回的供应包格式化为 `AIContext.Instructions`。
5. 失败时返回空 `AIContext`。

建议注入位置：与 `ProjectSettingsProvider` 类似，作为宿主内置 `AIContextProvider` 注册。

### 7.2 新增 `ILongMemorySupplyClient`

职责：

1. 找到 Memory 插件控制面连接。
2. 发送 `memory.context.supply.request`。
3. 等待 `memory.context.supply.response`。
4. 超时或错误时返回空结果。

第一版可以使用短超时：

```text
TimeoutMs = 250
```

### 7.3 新增 `LongMemoryPromptFormatter`

格式化输出：

```text
--long-term-memory--

以下是与当前任务相关的长期记忆，仅在有帮助时参考；如与用户当前明确指令冲突，以当前指令为准。

[项目约束]
- ...

[用户偏好]
- ...

[历史事实]
- ...

[长期模式]
- ...

> 以上内容来自长期记忆系统，可能不完整或过期。
```

不注入内部 ID、trace、策略 JSON、数据库字段。

## 8. 插件侧实现方案

### 8.1 扩展内部 feed 客户端处理能力

当前插件已有内部 WebSocket 客户端连接宿主 feed。后续可以在该连接体系上支持：

```text
type = request
op = memory.context.supply.request
```

插件收到后：

1. 解析请求。
2. 校验 `agentId`。
3. 调用 `IMemorySupplyService.Supply(...)`。
4. 返回结构化供应包。

### 8.2 不把业务逻辑写进 WebSocket 层

WebSocket 层只做协议适配。业务继续由：

- `IMemorySupplyService`
- `IMemoryRecallService`
- `IMemorySettingsService`

承担。

### 8.3 可选：为工具补齐参数，但不作为主链路

可以后续新增一个工具版本：

```text
memory_supply_context_v2
```

或者扩展现有工具参数，加入：

- `agentId`
- `sessionId`
- `turnId`
- `messageId`
- `workspaceDirectory`
- `traceId`

但这只是为了 AI 手动查询更可靠，不替代宿主控制面自动注入。

## 9. 降级策略

默认注入不能阻塞主对话。

| 场景 | 行为 |
| --- | --- |
| Memory 插件未加载 | 返回空上下文。 |
| 控制面未连接 | 返回空上下文。 |
| 请求超时 | 返回空上下文，记录降级日志。 |
| 插件返回错误 | 返回空上下文。 |
| `Enabled = false` | 不注入。 |
| `Items.Count = 0` | 不注入。 |
| `Confidence <= 0` | 不注入。 |
| 格式化异常 | 不注入。 |

## 10. 推荐实施顺序

### Step 1：宿主侧上下文 Provider 骨架

- 新增 `LongMemoryContextProvider`。
- 暂时返回空上下文。
- 注册到内置 `AIContextProvider`。
- 验证主对话构建时会调用。

### Step 2：定义宿主内部供应 DTO

- 新增 `MemoryContextSupplyRequest`。
- 新增 `MemoryContextSupplyResponse`。
- 新增 `MemoryContextMessage`。
- 配置 AOT JSON 源生成。

### Step 3：新增 `ILongMemorySupplyClient`

- 先实现 no-op 或本地 mock。
- 再接入内部 WebSocket 控制面。
- 加 250ms 超时。

### Step 4：插件侧处理 `memory.context.supply.request`

- 在现有内部 WebSocket 客户端体系上增加 op 分发。
- 调用 `IMemorySupplyService.Supply(...)`。
- 返回 `memory.context.supply.response`。

### Step 5：格式化并注入

- 新增 `LongMemoryPromptFormatter`。
- 将供应包转为 `AIContext.Instructions`。
- 控制最大条数、最大字符数和分组顺序。

### Step 6：补充工具参数（可选）

- 为 `memory_supply_context` 增加显式 `agentId` 等参数，或新增 v2 工具。
- 仅作为手动查询和调试增强。

## 11. 当前结论

1. **第一，能不能调工具：技术上可以。**
   宿主能拿到插件暴露的 `AITool`，工具最终可通过插件子进程 invoke 机制执行。

2. **第二，工具参数满不满足条件：当前不满足正式自动注入。**
   关键缺口是不能显式传入 `agentId`、`sessionId`、`turnId`、`messageId`、`workspaceDirectory`、`traceId`。尤其 `agentId` 当前会回退到 `default`，会影响记忆召回可靠性。

3. **正式方案不走 AI 工具层。**
   宿主自动注入应走内部控制面，插件复用 `IMemorySupplyService`。现有工具保留给 AI 主动查询、MCP 和调试验收。

4. **可以复用现有内部 WebSocket 客户端连接宿主 feed。**
   不需要新增完全独立的底层连接，只需要扩展请求/响应 op 和 DTO。
