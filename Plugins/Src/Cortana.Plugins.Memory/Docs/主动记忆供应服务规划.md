# 主动记忆供应服务规划

> 当前状态：暂缓实施。
>
> 暂缓原因：当前插件体系中，上层不会接收插件主动推送的记忆；上层只会实时推送事实片段，并在打开对话或需要上下文时请求记忆包。记忆插件当前更核心的职责是完成事实片段存储、长期记忆提取、归档、抽象记忆生成、处理状态维护和记忆包查询。因此本规划仅作为后续上层具备配合能力后的参考，不作为当前实施步骤。

本文档用于规划 `Cortana.Plugins.Memory` 的主动记忆供应服务。当前仅定义业务目标、链路边界、数据模型、算法策略、实施步骤和验收标准，不直接实施代码。

## 1. 背景

当前记忆插件已经完成两个基础能力：

1. 配置读取服务：从 `memory_settings` 读取记忆系统运行参数。
2. 记忆召回服务：基于查询文本从 `memory_fragments` 和 `memory_abstractions` 中召回相关记忆，并写入 `recall_logs`。

主动记忆供应服务是读取链路上的关键核心服务。它不是简单的查询召回，而是在上层尚未明确提出检索请求时，根据当前对话、任务、工作区、智能体状态和预算，主动判断“此刻应该供应哪些记忆”。

## 2. 目标

主动记忆供应服务的目标是：

1. 在上层构建提示词或执行任务前，主动提供高价值记忆上下文。
2. 减少用户反复说明偏好、规则、历史事实和项目约束的成本。
3. 在 Token 预算内输出可直接注入上下文的记忆包。
4. 避免低可信、冲突、过期、无关或敏感记忆污染当前任务。
5. 复用现有召回服务和配置服务，不重复建设检索逻辑。
6. 为后续衰减、强化、抽象和治理服务提供稳定读取入口。

## 3. 非目标

本阶段不做以下事项：

1. 不实现后台定时推送。
2. 不直接修改宿主提示词系统。
3. 不实现复杂语义向量检索。
4. 不实现模型级意图识别。
5. 不实现冲突裁决服务。
6. 不实现衰减、归并或抽象生成。
7. 不新增数据库表，除非后续验收发现必须持久化供应日志。

## 4. 与召回服务的区别

| 项目 | 记忆召回服务 | 主动记忆供应服务 |
| --- | --- | --- |
| 触发方式 | 上层提供查询文本后召回。 | 上层进入任务或构建上下文前主动请求供应。 |
| 核心问题 | “哪些记忆与这个查询相关？” | “当前场景应该给上层哪些记忆？” |
| 输入重点 | `QueryText`、`QueryIntent`。 | 当前任务、场景、最近消息、预算、供应模式。 |
| 输出形态 | 召回窗口和命中项。 | 可注入的供应包、分组、预算和策略说明。 |
| 日志语义 | 一次查询召回记录。 | 一次上下文供应决策记录，可复用召回日志或后续扩展。 |
| 是否依赖召回 | 自身提供检索。 | 应优先复用召回服务。 |

## 5. 总体链路

主动记忆供应链路建议如下：

```text
宿主或上层编排器
	↓
MemorySupplyRequest
	↓
MemorySupplyService
	↓ 读取 supply / recall 配置
IMemorySettingsService
	↓
构造一个或多个 MemoryRecallRequest
	↓
IMemoryRecallService
	↓
MemoryRecallResult
	↓
供应过滤：状态、可信度、预算、重复、主题、场景
	↓
供应分组：规则/事实/偏好/项目/任务/抽象
	↓
MemorySupplyPackage
	↓
宿主提示词上下文注入或任务执行上下文
```

核心原则：**供应服务负责“何时给、给多少、以什么结构给”，召回服务负责“从库里找什么”。**

## 6. 服务边界

### 6.1 输入边界

主动供应服务只接收上层传入的上下文快照，不主动读取宿主内部状态。

建议输入包括：

| 字段 | 说明 |
| --- | --- |
| `RequestId` | 供应请求标识。 |
| `AgentId` | 智能体标识，必填。 |
| `WorkspaceId` | 工作区标识，可选。 |
| `Scenario` | 场景，例如 chat、coding、planning、tool-call。 |
| `CurrentTask` | 当前任务描述。 |
| `RecentMessages` | 最近对话摘要或文本片段。 |
| `TriggerSource` | 触发来源，例如 before-prompt、before-tool、manual。 |
| `MaxMemoryCount` | 本次最多供应记忆数，可选。 |
| `MaxTokenBudget` | 本次供应的最大 Token 预算，可选。 |
| `TraceId` | 链路追踪标识。 |

### 6.2 输出边界

主动供应服务输出一个结构化供应包。

建议输出包括：

| 字段 | 说明 |
| --- | --- |
| `RequestId` | 供应请求标识。 |
| `Enabled` | 本次是否启用供应。 |
| `Groups` | 分组后的供应记忆。 |
| `Items` | 扁平化供应记忆列表。 |
| `Summary` | 供应摘要。 |
| `Confidence` | 整体可信度。 |
| `Budget` | 实际使用的数量和预算。 |
| `AppliedPolicy` | 使用的策略说明。 |
| `TraceId` | 链路追踪标识。 |

## 7. 建议模型

### 7.1 MemorySupplyRequest

建议新增模型：

```text
MemorySupplyRequest
- RequestId: string
- AgentId: string
- WorkspaceId: string?
- Scenario: string?
- CurrentTask: string?
- RecentMessages: IReadOnlyList<string>
- TriggerSource: string?
- MaxMemoryCount: int?
- MaxTokenBudget: int?
- TraceId: string?
```

设计说明：

1. `AgentId` 必填，用于隔离记忆。
2. `CurrentTask` 与 `RecentMessages` 用于生成主动召回查询文本。
3. `MaxMemoryCount` 优先级高于配置默认值，但不能超过系统限制。
4. `MaxTokenBudget` 第一版可以只记录和粗略估算，不做精确 tokenizer。

### 7.2 MemorySupplyResult

建议新增模型：

```text
MemorySupplyResult
- RequestId: string
- Enabled: bool
- Groups: IReadOnlyList<MemorySupplyGroup>
- Items: IReadOnlyList<MemorySupplyItem>
- Confidence: double
- Summary: string?
- Budget: MemorySupplyBudget
- AppliedPolicy: MemorySupplyPolicy
- TraceId: string?
```

### 7.3 MemorySupplyGroup

```text
MemorySupplyGroup
- GroupKey: string
- Title: string
- Items: IReadOnlyList<MemorySupplyItem>
- Priority: int
```

分组建议：

| GroupKey | 含义 |
| --- | --- |
| `preference` | 用户偏好。 |
| `constraint` | 项目、工作区或任务约束。 |
| `fact` | 稳定事实。 |
| `task` | 与当前任务相关的历史任务记忆。 |
| `abstraction` | 抽象记忆或长期模式。 |
| `other` | 其他记忆。 |

### 7.4 MemorySupplyItem

```text
MemorySupplyItem
- Id: string
- Kind: string
- Topic: string
- Title: string?
- Content: string
- Reason: string
- Confidence: double
- Score: double
- SourceRecallScore: double
- LifecycleState: string
- ConfirmationState: string
- UpdatedAt: string
```

设计说明：

1. `Content` 用于上层直接注入。
2. `Reason` 用于解释为什么供应该记忆。
3. `Score` 是供应服务最终排序分。
4. `SourceRecallScore` 保留召回服务原始评分，方便排查。

### 7.5 MemorySupplyBudget

```text
MemorySupplyBudget
- MaxMemoryCount: int
- UsedMemoryCount: int
- MaxTokenBudget: int?
- EstimatedTokens: int?
```

### 7.6 MemorySupplyPolicy

```text
MemorySupplyPolicy
- SupplyEnabled: bool
- MaxMemoryCount: int
- RecallMinimumConfidence: double
- Ranking: string
- Grouping: string
```

## 8. 配置策略

当前 `MemorySupplyOptions` 已包含：

| 配置 | 默认值 | 用途 |
| --- | --- | --- |
| `supply.enabled` | `true` | 是否启用主动供应。 |
| `supply.maxMemoryCount` | `8` | 主动供应最大记忆数量。 |

第一版实现建议只使用这两个配置，避免过度设计。

后续可扩展配置：

| 配置 | 建议默认值 | 用途 |
| --- | --- | --- |
| `supply.maxGroupCount` | `4` | 最大供应分组数。 |
| `supply.minimumScore` | `0.35` | 最低供应分数。 |
| `supply.maxEstimatedTokens` | `1200` | 默认供应 Token 预算。 |
| `supply.includeAbstractions` | `true` | 是否供应抽象记忆。 |
| `supply.includeFragments` | `true` | 是否供应片段记忆。 |

这些扩展配置不建议第一版立即加入，除非实现中确有必要。

## 9. 查询构造策略

主动供应不是无条件全库扫描，第一版应将输入上下文压缩成召回查询文本。

建议拼接顺序：

```text
Scenario
CurrentTask
RecentMessages 最近 N 条
TriggerSource
```

规则：

1. 空字段跳过。
2. 总长度限制在合理范围内，例如 2000 字符。
3. `CurrentTask` 权重最高，放在最前。
4. `RecentMessages` 只取最近几条，避免长上下文稀释匹配。
5. 如果没有任务和最近消息，则可以执行“无查询文本供应”，只召回高分稳定记忆。

## 10. 供应排序策略

供应服务不应完全复用召回排序，而应在召回分数基础上叠加“可供应性”。

建议第一版供应分数：

```text
supplyScore =
  recallScore * 0.60
+ confidence * 0.20
+ freshnessScore * 0.10
+ stateScore * 0.10
```

### 10.1 freshnessScore

第一版可用简化策略：

| 条件 | 分数 |
| --- | --- |
| `UpdatedAt` 为空 | `0.5` |
| 30 天内 | `1.0` |
| 90 天内 | `0.7` |
| 更早 | `0.4` |

### 10.2 stateScore

| 状态 | 分数 |
| --- | --- |
| `active + confirmed` | `1.0` |
| `reinforced + confirmed` | `1.0` |
| `candidate + pending` | `0.55` |
| `dormant` | `0.45` |
| 其他允许召回状态 | `0.3` |

已经被召回服务排除的 `forgotten`、`rejected` 不应进入供应排序。

## 11. 供应过滤策略

供应服务需要在召回结果上再做一层过滤。

建议过滤规则：

1. 空内容过滤。
2. 重复 ID 过滤。
3. `Confidence <= 0` 过滤。
4. `LifecycleState = forgotten` 过滤。
5. `ConfirmationState = rejected` 过滤。
6. 超过 `MaxMemoryCount` 后截断。
7. 超过 `MaxTokenBudget` 后截断。
8. 同一主题过多时限流，避免单主题挤占全部预算。

第一版可以先实现 1 到 6；Token 预算可以先用字符数粗略估算。

## 12. 分组策略

建议按以下优先级决定分组：

1. 如果 `Kind = abstraction`，优先进入 `abstraction` 分组。
2. 如果 `Topic` 或 `Title` 包含偏好类关键词，进入 `preference`。
3. 如果包含约束、规则、必须、禁止等关键词，进入 `constraint`。
4. 如果包含任务、计划、待办等关键词，进入 `task`。
5. 其他进入 `fact` 或 `other`。

第一版可以使用确定性关键词规则，不调用模型。

## 13. 输出格式建议

上层最终注入提示词时，建议转换为如下文本结构：

```text
以下是与当前任务相关的长期记忆，仅在有帮助时参考：

[项目约束]
- ...

[用户偏好]
- ...

[历史事实]
- ...

[长期模式]
- ...
```

供应服务自身只输出结构化结果，不负责最终提示词拼接。提示词拼接应由宿主或上层编排器负责。

## 14. 日志和审计策略

第一版可复用 `recall_logs`，因为主动供应内部会调用召回服务，召回服务已经写入召回日志。

建议在 `MemorySupplyResult.AppliedPolicy` 中记录：

1. 供应是否启用。
2. 最大供应数量。
3. 查询构造来源。
4. 排序策略。
5. 分组策略。
6. 是否发生预算截断。

是否新增 `supply_logs` 表建议延后决定。只有当需要独立统计主动供应命中率、注入率、用户采纳率时，再新增供应日志表。

## 15. 错误和降级策略

主动供应服务不能阻塞主对话链路。

建议策略：

| 场景 | 行为 |
| --- | --- |
| `supply.enabled = false` | 返回 `Enabled = false` 的空结果。 |
| `AgentId` 为空 | 抛出 `ArgumentException`。 |
| 召回结果为空 | 返回空供应包，`Confidence = 0`。 |
| 召回服务异常 | 允许异常向上冒泡，由上层降级；后续可增加日志。 |
| 预算为 0 | 返回空供应包。 |

第一版保持简单，不吞异常，避免静默失败。

## 16. 接口设计建议

建议新增接口：

```text
IMemorySupplyService
- MemorySupplyResult Supply(MemorySupplyRequest request)
```

命名说明：

1. 使用 `Supply` 表示主动供应决策。
2. 仍保持同步接口，与当前 `IMemoryRecallService.Recall` 一致。
3. 后续如控制面走异步通信，再单独增加异步适配层。

## 18. 第一版算法流程

```text
Supply(request)
	↓
校验 request 和 AgentId
	↓
读取 supplyOptions 和 recallOptions
	↓
如果 supply.enabled = false，返回禁用结果
	↓
构造 queryText
	↓
调用 recallService.Recall(...)
	↓
将 MemoryRecallItem 转换为 MemorySupplyItem
	↓
计算 supplyScore
	↓
过滤无效项和重复项
	↓
按 supplyScore 排序
	↓
按 MaxMemoryCount 截断
	↓
按 Topic / Kind 分组
	↓
计算 Confidence、Budget、AppliedPolicy、Summary
	↓
返回 MemorySupplyResult
```

## 19. 与现有服务的依赖关系

```text
MemorySupplyService
	├── IMemorySettingsService
	│       └── 读取 supply / recall 配置
	└── IMemoryRecallService
			└── 复用召回查询、召回日志、访问统计
```

依赖原则：

1. 不直接依赖 `MemoryStore`。
2. 不重复实现数据库查询。
3. 不绕过召回日志和访问统计。
4. 不引入新抽象层。

## 20. 验收标准

第一版完成后，应满足：

1. `supply.enabled = false` 时返回禁用空结果。
2. `AgentId` 为空时抛出明确参数异常。
3. 能根据 `CurrentTask` 和 `RecentMessages` 构造召回查询。
4. 能复用 `IMemoryRecallService` 获取候选记忆。
5. 能输出 `Groups` 和 `Items`。
6. `Items.Count <= supply.maxMemoryCount`。
7. 输出项不包含空内容、重复 ID、rejected 或 forgotten 记忆。
8. 输出包含预算信息和策略信息。
9. DI 注册完成。
10. 项目构建通过。

## 21. 测试建议

如后续创建测试项目，建议覆盖：

1. 禁用供应返回空结果。
2. 空 `AgentId` 抛出异常。
3. 空上下文时仍可基于高分记忆供应。
4. 有任务文本时正确传入召回请求。
5. 重复召回项被去重。
6. 最大供应数量限制生效。
7. 分组结果稳定。
8. 预算信息正确。

当前仓库尚未为 `Cortana.Plugins.Memory` 建立专用测试项目，第一版实现时可以先通过构建和人工验收验证。

## 22. 风险和后续演进

| 风险 | 说明 | 后续处理 |
| --- | --- | --- |
| 供应过多 | 可能污染提示词。 | 增加 Token 预算和主题限流。 |
| 供应不相关 | 查询构造过弱时可能召回泛化记忆。 | 引入更强意图识别或语义检索。 |
| 候选记忆过早注入 | `candidate/pending` 可能不稳定。 | 供应排序中降低状态分，后续接确认服务。 |
| 缺少独立供应日志 | 无法统计供应采纳率。 | 后续根据需要新增 `supply_logs`。 |
| 与宿主注入格式耦合 | 插件输出结构化结果，宿主负责注入，可降低耦合。 | 保持边界不变。 |

## 23. 推荐实施顺序

1. 新增 `MemorySupplyModels.cs`。
2. 新增 `IMemorySupplyService.cs`。
3. 新增 `MemorySupplyService.cs`。
4. 在 `Startup.cs` 注册服务。
5. 构建验证。
6. 编写阶段验收文档或更新当前验收文档。

## 24. 当前决策

本阶段建议采用以下决策：

1. 第一版主动供应服务复用现有召回服务。
2. 第一版不新增数据库表。
3. 第一版不新增存储接口。
4. 第一版不新增配置项，只使用 `supply.enabled` 和 `supply.maxMemoryCount`。
5. 第一版输出结构化供应包，不负责拼接最终提示词。
6. 第一版使用确定性规则进行排序、过滤和分组。

上述决策可以降低实现风险，让主动供应服务尽快接入读取链路，并为后续控制面和宿主注入链路预留空间。
