# 查询与召回处理验证报告

## 1. 验证目标

本次验证目标是进入记忆系统的查询与召回处理阶段，确认已生成的 `memory_fragments` 与 `memory_abstractions` 可以根据查询文本被召回，并写入 `recall_logs` 形成可审计记录。

## 2. 验证范围

- 召回服务：`IMemoryRecallService` / `MemoryRecallService`
- 存储查询：`MemoryStore.SearchRecallCandidates`
- 访问记录：`MemoryStore.RecordMemoryAccesses`
- 召回日志：`recall_logs`
- 本地验证工具：`Tools\MemoryRecallRunner`

## 3. 本次实现与修复

### 3.1 SQL 层查询词过滤

原逻辑先按基础评分取候选，再在内存中做文本匹配。当记忆数量较多时，相关但基础分较低的记忆可能在候选截断前被丢弃。

已修复为：

- 有 `queryText` 时，先将查询词拆分为 terms。
- SQL 层对 `topic/title/summary/detail` 执行 `LIKE` 过滤。
- `memory_fragments` 使用真实列：`topic/title/summary/detail`。
- `memory_abstractions` 使用真实列：`abstractionType/title/summary/statement`。
- 有查询时扩大候选池：`max(limit * 50, 500)`。
- 对 `%`、`_`、`\` 做 LIKE 转义。

### 3.2 本地召回验证工具

新增工具：

```text
Tools\MemoryRecallRunner
```

用途：

- 从命令行接收查询文本。
- 调用 `IMemoryRecallService.Recall`。
- 输出召回摘要、窗口数、条目数、置信度、前 10 条结果。
- 触发 `recall_logs` 写入。

示例：

```powershell
dotnet run --project Tools\MemoryRecallRunner\MemoryRecallRunner.csproj --configuration Debug -- TikTok 连接器
```

## 7. 验收结论

本阶段验证通过。

已确认：

1. 查询文本可以召回 `memory_fragments`。
2. 查询文本可以召回 `memory_abstractions`。
3. 召回结果可以按 topic 分组形成窗口。
4. 召回结果会更新访问次数与最后访问时间。
5. 每次召回会写入 `recall_logs`。
6. SQL 层查询过滤已避免基础分截断导致的漏召回问题。

## 8. 已知限制

1. 当前仍是关键词匹配，不具备向量语义相似召回能力。
2. `tool` 角色输出仍会产生部分噪声召回，例如 PowerShell 执行日志。
3. 当前召回评分仍偏简单，尚未区分用户偏好、约束、任务、事实的业务优先级。
4. 抽象记忆目前以 topic-summary 为主，尚未形成更细粒度的规则记忆、偏好记忆、长期任务记忆。

## 9. 下一步建议

1. 增加 tool 输出降权或过滤策略。
2. 查询时提高 abstraction 权重，使摘要型记忆优先进入上下文。
3. 增加召回预算控制，例如最大 token、最大 detail 长度。
4. 增加主动记忆供应服务，将召回结果组装成上层 AI 可直接消费的上下文块。
5. 后续接入 embedding 或大模型重排，提升语义查询质量。
