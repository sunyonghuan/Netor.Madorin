## 当前结论

当前阶段结论：**基础验收通过，可以进入下一阶段主动记忆供应服务开发**。

已完成的能力包括：

1. 从 `memory_settings` 读取运行配置。
2. 将配置转换为强类型选项对象。
3. 基于 `memory_fragments` 和 `memory_abstractions` 执行本地召回。
4. 按主题形成召回窗口。
5. 写入 `recall_logs` 召回日志。
6. 更新被召回记忆的 `accessCount`、`lastAccessedAt` 和 `updatedAt`。
7. 支持工作区、智能体和智能体工作区组合级配置覆盖。

### 配置作用域验收

当前读取顺序支持以下优先级：

1. 全局配置：`agentId IS NULL AND workspaceId IS NULL`。
2. 工作区配置：`agentId IS NULL AND workspaceId = @workspace`。
3. 智能体配置：`agentId = @agent AND workspaceId IS NULL`。
4. 智能体工作区配置：`agentId = @agent AND workspaceId = @workspace`。

`MemorySettingsService.GetString` 按 `settingKey` 分组后取最后一条记录，因此较高优先级配置会覆盖较低优先级配置。

结论：**通过**。

### 召回过滤验收

当前过滤规则：

1. 必须匹配 `agentId`。
2. 如果传入 `workspaceId`，允许召回全局工作区记忆和当前工作区记忆。
3. `confidence >= recall.minimumConfidence`。
4. 排除 `lifecycleState = 'forgotten'`。
5. 排除 `confirmationState = 'rejected'`。
6. 默认只召回 `active + confirmed`。
7. 当 `recall.includeCandidateMemories = true` 时允许候选记忆参与召回。
8. 当存在明确 `QueryText` 且文本命中时，允许 `candidate + pending` 参与召回。

结论：**通过**。

该策略解决了当前入库默认 `candidate/pending` 与默认不召回候选记忆之间的阶段性冲突，同时仍然排除了已拒绝和已遗忘记忆。

### 排序和匹配验收

当前基础评分公式：

```text
confidence * 0.35
+ salienceScore * 0.30
+ retentionScore * 0.25
+ min(accessCount, 10) * 0.01
```

文本匹配策略：

1. 对查询文本进行分词。
2. 按词项匹配主题、标题、摘要和详情。
3. 每个命中词项提供少量 boost。
4. boost 上限为 `0.24`。
5. SQL 先扩大候选池到 `limit * 5`，再在内存中应用文本 boost 和最终截断。

结论：**通过**。

该策略修复了长句整体 `LIKE` 匹配弱，以及 SQL 过早 `LIMIT` 导致高文本相关项被排除的问题。

## 剩余注意事项

以下事项不阻塞当前阶段验收，但建议在后续阶段处理：

1. 为 `valueType = json` 增加 JSON 类型读取方法，或明确当前配置服务只提供基础标量读取。
2. 在数据库说明文档中补充抽象记忆排序时 `stabilityScore` 映射为统一召回模型 `SalienceScore` 的说明。
3. 后续接入真实回答生成后，可以将 `supportingMemoryIdsJson` 从“窗口项”细化为“最终回答实际使用的支撑记忆”。
4. 当前召回是同步接口；如果后续外部调用频繁或存储访问变重，应再评估异步化。
5. 当前尚未创建单元测试项目，本阶段通过构建验证。后续公开 API 稳定后建议补充测试。
