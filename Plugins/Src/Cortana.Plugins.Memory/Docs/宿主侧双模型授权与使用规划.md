# 宿主侧插件能力授权、全局插件与 LLM 模型绑定规划

## 1. 背景

Memory 插件长期记忆质量依赖 LLM 能力，但插件不应该直接配置供应商、密钥或最终模型 ID。更通用的插件生态也会遇到同样问题：不同插件可能需要摘要、分类、结构化提取、代码分析、参数生成等 LLM 能力，这些能力都应该由宿主统一授权、统一绑定模型、统一预算和审计。

此前的规划以 Memory 插件的 `Memory.ModelId` 为中心。这个方向适合作为早期占位，但不适合作为长期架构：

1. `Memory.ModelId` 只是设置端预留，当前未真正进入 Memory 插件模型调用链路。
2. 未来不只有 Memory 插件需要 LLM，继续追加 `Xxx.ModelId` 会导致设置分散。
3. 插件授权需要按 `pluginId + capability + purpose` 精确控制，而不是按某个模块专用设置控制。
4. 插件协议需要兼容 Native AOT，宿主不能为每个插件业务输入定义强 DTO。

因此，本方案将“Memory 双模型”升级为“宿主侧插件能力授权与 LLM 模型绑定”。Memory 插件只是第一批落地用例。

## 2. 总体目标

1. 在宿主系统设置中新增“插件设置 / 插件授权”专区。
2. 支持对插件进行能力级授权，例如 LLM、Embedding、conversation-feed、recall、文件、网络、后台任务等。
3. LLM 能力按 `pluginId + purpose` 绑定 Provider 和 Model。
4. 支持全局插件：全局插件无需智能体侧逐个勾选绑定，所有智能体默认可用。
5. 全局插件必须安装在用户数据目录的全局插件目录下，工作区插件、临时插件、开发调试插件默认不能设为全局插件。
6. 插件 init 阶段下发该插件的授权摘要，让插件启动时即可判断可用能力。
7. 运行时仍由宿主控制面二次校验授权，防止伪造或过期授权。
8. LLM 控制面协议采用 AOT 友好的通用 envelope，插件业务输入使用 opaque 字符串，不让宿主绑定插件专用 DTO。
9. 支持插件通过内部控制面主动向宿主推送信息包，由宿主按授权类型和操作类型决定如何处理；Memory Context Package 是第一批落地用例。
10. 不扩展 `get_info / init / invoke / destroy` 子进程基础协议；模型调用和插件主动推送走独立内部 WebSocket 控制面。
11. 不复用 Ollama/OpenAI 兼容代理，因为该代理面向外部软件直接调用模型，不具备插件授权语义。

## 3. 废弃 `Memory.ModelId`

`Memory.ModelId` 不再作为新方案的主入口。

处理建议：

1. 从系统设置种子和设置页中删除或标记废弃 `Memory.ModelId`。
2. 不再新增 `Memory.AbstractionModelId` 这类 Memory 专用设置。
3. Memory 插件所需模型改由插件授权配置提供。
4. 如用户已有历史 `Memory.ModelId`，可在升级时迁移为 Memory 插件的默认 LLM 授权配置。

迁移目标示例：

```text
Plugin:memory_engine:Capability:llm:memory-processing:Enabled = true
Plugin:memory_engine:Capability:llm:memory-processing:ProviderId = ...
Plugin:memory_engine:Capability:llm:memory-processing:ModelId = ...

Plugin:memory_engine:Capability:llm:memory-abstraction:Enabled = true
Plugin:memory_engine:Capability:llm:memory-abstraction:ProviderId = ...
Plugin:memory_engine:Capability:llm:memory-abstraction:ModelId = ...
```

如果没有历史值，则默认不授权 LLM，插件继续使用 fallback。

## 4. 插件设置与授权模型

宿主新增插件设置面板，用于管理每个插件的全局状态和能力授权。

建议面板分组：

| 设置项 | 说明 |
| --- | --- |
| 启用插件 | 是否允许加载该插件。 |
| 设为全局插件 | 是否对所有智能体默认可用。 |
| 安装来源 | 用户全局目录、工作区目录、开发调试目录等。 |
| LLM 授权 | 是否允许插件使用 LLM。 |
| LLM 用途绑定 | 每个 purpose 绑定 Provider / Model / 预算 / 超时。 |
| 后台任务 | 是否允许插件在后台定时或持续运行。 |
| conversation-feed | 是否允许插件订阅宿主对话事实流。 |
| recall | 是否允许宿主主动向插件请求召回结果。 |
| 网络访问 | 是否允许插件访问外部网络。 |
| 文件访问 | 是否允许插件访问指定文件范围。 |

授权配置逻辑键建议：

```text
Plugin:{pluginId}:Enabled
Plugin:{pluginId}:GlobalEnabled
Plugin:{pluginId}:Capability:{capability}:Enabled
Plugin:{pluginId}:Capability:{capability}:{purpose}:Enabled
Plugin:{pluginId}:Capability:{capability}:{purpose}:ProviderId
Plugin:{pluginId}:Capability:{capability}:{purpose}:ModelId
Plugin:{pluginId}:Capability:{capability}:{purpose}:MaxInputTokens
Plugin:{pluginId}:Capability:{capability}:{purpose}:MaxOutputTokens
Plugin:{pluginId}:Capability:{capability}:{purpose}:TimeoutMs
Plugin:{pluginId}:Capability:{capability}:{purpose}:MaxConcurrency
Plugin:{pluginId}:Capability:{capability}:{purpose}:AllowBackground
```

第一阶段可以用系统设置表保存，后续可迁移到专用插件授权表。

## 5. 全局插件规则

当前插件模式主要与智能体绑定。新增全局插件后，智能体最终可用插件集合为：

```text
可用插件 = 全局插件 + 智能体显式绑定插件
```

全局插件规则：

1. 全局插件对所有智能体默认可用。
2. 全局插件不需要在智能体侧逐个勾选。
3. 智能体侧仍可以绑定自己的局部插件。
4. 全局插件必须安装在用户数据目录的全局插件目录下。
5. 工作区插件不能设为全局插件。
6. 临时加载插件不能设为全局插件。
7. 开发调试插件默认不能设为全局插件，除非显式开启开发模式。
8. 如果插件移动、删除或安装来源校验失败，应自动撤销全局状态。

全局插件校验建议：

```text
plugin.InstallScope == UserGlobal
plugin.PluginDirectory 位于宿主用户数据插件目录
plugin.Manifest.Id 与安装记录一致
plugin.Manifest.Hash 或签名校验通过（后续支持）
```

界面行为建议：

| 插件来源 | 是否允许设为全局 |
| --- | --- |
| 用户数据目录全局插件 | 允许 |
| 工作区插件 | 不允许 |
| 临时插件 | 不允许 |
| 开发调试插件 | 默认不允许，开发模式可放开 |

## 6. LLM 用途预定义

LLM 授权分为两层：

```text
capability = llm
purpose = 具体用途
```

宿主按 `pluginId + capability + purpose` 做授权和模型绑定。

通用 LLM purpose 预留：

| Purpose | 用途 |
| --- | --- |
| `llm.chat` | 通用对话。 |
| `llm.complete` | 文本生成。 |
| `llm.extract` | 结构化信息提取。 |
| `llm.summarize` | 摘要。 |
| `llm.classify` | 分类。 |
| `llm.rewrite` | 改写。 |
| `llm.reason` | 推理和分析。 |
| `llm.code-analyze` | 代码分析。 |
| `llm.tool-args` | 工具参数生成。 |
| `llm.validate` | 结果校验。 |

Memory 插件第一阶段 purpose：

| Purpose | 用途 | 推荐 taskKind |
| --- | --- | --- |
| `memory-processing` | 从 observation 中提取候选长期记忆。 | `structured-extraction` |
| `memory-abstraction` | 对同主题 fragment 做抽象归纳。 | `structured-extraction` |

Memory 后续可选 purpose：

| Purpose | 用途 |
| --- | --- |
| `memory-conflict-resolution` | 冲突判断。 |
| `memory-importance-scoring` | 重要性评分。 |
| `memory-tagging` | 标签和主题生成。 |

不纳入当前阶段：

1. `embedding`：作为独立 capability，不混入 `llm`。
2. `rerank`：作为独立 capability 或后续专用 purpose 规划。
3. 外部 Ollama/OpenAI 兼容代理：不作为插件授权通道。

## 7. Init 阶段授权下发

插件 init 时，宿主通过 `PluginSettings.Extensions` 下发该插件的授权摘要。

为降低 Native AOT 风险，授权下发不使用插件专用强 DTO。建议仍使用字符串字典，并将复杂授权清单压缩为 JSON 字符串或分隔字符串。

建议字段：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| `pluginAuthorizationVersion` | `1.0.0` | 授权摘要格式版本。 |
| `pluginGlobalEnabled` | `true` | 是否为全局插件。 |
| `pluginInstallScope` | `UserGlobal` | 插件安装来源。 |
| `modelCapabilityEnabled` | `true` | 是否允许该插件申请 LLM 能力。 |
| `modelCapabilityEndpoint` | `ws://localhost:65321/internal/model-capability/` | 控制面完整 endpoint。 |
| `modelCapabilityPort` | `65321` | 控制面端口。 |
| `modelCapabilityPath` | `/internal/model-capability/` | 控制面路径。 |
| `modelCapabilityProtocol` | `model-capability` | 协议名。 |
| `modelCapabilityVersion` | `1.0.0` | 协议版本。 |
| `modelCapabilityPurposes` | `memory-processing,memory-abstraction` | 已授权 purpose 列表。 |
| `modelCapabilityGrantsJson` | `{...}` | AOT 友好的授权摘要 JSON 字符串。 |
| `modelCapabilityToken` | `...` | 短期 token，随插件进程生命周期有效。 |
| `pluginPushEnabled` | `true` | 是否允许插件主动向宿主推送信息包。 |
| `pluginPushPurposes` | `memory-context` | 已授权主动推送 purpose 列表。 |
| `pluginPushMaxItems` | `20` | 单次推送最大条目数。 |
| `pluginPushMaxChars` | `12000` | 单次推送最大字符数。 |

`modelCapabilityGrantsJson` 是字符串，不要求宿主和插件共享插件业务 DTO。它只描述授权元数据，不描述业务输入结构。

示例：

```json
{
  "version": "1.0.0",
  "capability": "llm",
  "grants": [
    {
      "purpose": "memory-processing",
      "enabled": true,
      "taskKinds": "structured-extraction",
      "defaultTimeoutMs": 30000,
      "maxInputTokens": 6000,
      "maxOutputTokens": 1200,
      "maxConcurrency": 1,
      "allowBackground": true
    },
    {
      "purpose": "memory-abstraction",
      "enabled": true,
      "taskKinds": "structured-extraction",
      "defaultTimeoutMs": 30000,
      "maxInputTokens": 6000,
      "maxOutputTokens": 1200,
      "maxConcurrency": 1,
      "allowBackground": true
    }
  ]
}
```

AOT 约束：

1. 宿主下发授权时只使用 `Dictionary<string, string>`。
2. `modelCapabilityGrantsJson` 可由插件用 `JsonDocument` / `JsonElement` 读取。
3. 宿主不为每个插件生成授权 DTO。
4. 插件可在自己程序集内定义自己的授权解析 DTO，但这不影响宿主 AOT 边界。
5. init 阶段授权用于功能开关和 DI 决策；运行时 WebSocket Broker 仍必须重新校验。

## 8. 推荐通道

新增内部 WebSocket 控制面路径：

```text
ws://localhost:{controlPlanePort}/internal/model-capability/
```

第一阶段可与 conversation-feed 使用同一个内部端口：

```text
ws://localhost:{conversationFeedPort}/internal/model-capability/
```

实现上建议优先在当前实际启动的内部 WebSocket 服务中挂载该路径，避免 `WebSocketFeedServerService` 与 `WebSocketServerService` 两套实现分叉。

路径职责划分：

| 路径 | 协议 | 方向 | 职责 |
| --- | --- | --- | --- |
| `/internal/conversation-feed/` | `conversation-feed` | 宿主 -> 插件为主 | 对话事实流、历史 replay、实时 delta 和 completed 事件。 |
| `/internal/model-capability/` | `model-capability` | 插件 -> 宿主请求，宿主 -> 插件响应 | 插件用途级 LLM 调用授权和执行。 |
| `/internal/plugin-push/` | `plugin-push` | 插件 -> 宿主推送，宿主 -> 插件确认 | 插件主动推送信息包，例如记忆上下文、索引更新、诊断状态、上下文建议等。 |
| `/internal/recall/` | `memory-recall` | 宿主 -> 插件请求，插件 -> 宿主响应 | 后续宿主主动向 Memory 插件请求 RecallPackage。 |

## 10. AOT 友好的 LLM 请求协议

采用“方案 B”：插件直接传 prompt，宿主不理解插件业务 input 结构。

宿主只强类型识别通用 envelope：

1. `type`
2. `protocol`
3. `version`
4. `op`
5. `requestId`
6. `traceId`
7. `pluginId`
8. `purpose`
9. `taskKind`
10. `prompt`
11. `output`
12. `budget`
13. `options`

插件业务输入必须 opaque 化：

1. 使用字符串承载，例如 `prompt.input`。
2. 可以是普通文本、JSON 字符串、Markdown、代码片段等。
3. 宿主不反序列化插件业务输入。
4. 宿主不绑定 `MemoryProcessingInput`、`SomePluginInput` 等插件专用 DTO。
5. 插件自己负责构造 prompt、解释输出和校验 schema。

请求帧：

```json
{
  "type": "request",
  "protocol": "model-capability",
  "version": "1.0.0",
  "op": "llm.invoke",
  "requestId": "...",
  "traceId": "...",
  "pluginId": "memory_engine",
  "purpose": "memory-processing",
  "taskKind": "structured-extraction",
  "prompt": {
    "system": "你是长期记忆提取器。",
    "instructions": "只输出 JSON，不要输出解释性文本。",
    "inputFormat": "application/json",
    "input": "{...插件自定义 JSON 字符串...}"
  },
  "output": {
    "format": "json",
    "schemaId": "memory.processing.output.v1"
  },
  "timeoutMs": 30000,
  "budget": {
    "maxInputTokens": 6000,
    "maxOutputTokens": 1200
  },
  "options": {
    "temperature": 0.1
  }
}
```

成功响应：

```json
{
  "type": "response",
  "protocol": "model-capability",
  "version": "1.0.0",
  "op": "llm.invoke.result",
  "requestId": "...",
  "traceId": "...",
  "pluginId": "memory_engine",
  "purpose": "memory-processing",
  "providerId": "...",
  "providerName": "...",
  "modelId": "...",
  "modelName": "...",
  "content": "{...模型输出文本...}",
  "contentFormat": "json",
  "schemaId": "memory.processing.output.v1",
  "usage": {
    "inputTokens": 1234,
    "outputTokens": 456
  },
  "durationMs": 3210
}
```

失败响应：

```json
{
  "type": "error",
  "protocol": "model-capability",
  "version": "1.0.0",
  "op": "llm.invoke",
  "requestId": "...",
  "traceId": "...",
  "pluginId": "memory_engine",
  "purpose": "memory-processing",
  "code": "MODEL_NOT_CONFIGURED",
  "message": "插件 purpose 未绑定有效模型。",
  "retryable": false
}
```

取消请求：

```json
{
  "type": "cancel",
  "protocol": "model-capability",
  "version": "1.0.0",
  "requestId": "...",
  "traceId": "...",
  "pluginId": "memory_engine"
}
```

## 11. 通用插件主动推送协议

当插件完成自身计算后，可以通过通用 `plugin-push` 控制面主动向宿主推送信息包。Memory 插件推送 Memory Context Package 只是第一批落地用例；后续其他插件也可以推送索引更新、诊断状态、上下文建议、任务进度、外部系统事件等信息。

该能力用于减少宿主反向查询插件的复杂度：

1. 插件负责判断哪些信息已经成熟、可信、适合上报宿主。
2. 插件按授权 purpose 主动推送给宿主。
3. 宿主按 `pluginId + capability=plugin-push + purpose + op` 校验授权和路由。
4. 宿主根据 purpose 和 op 决定处理器，例如写入上下文候选池、更新状态面板、记录诊断事件或触发后续任务。

主动推送不等于插件可以直接修改宿主核心状态。宿主仍是最终决策者。

### 授权要求

主动推送需要独立授权：

```text
Plugin:{pluginId}:Capability:plugin-push:{purpose}:Enabled
Plugin:{pluginId}:Capability:plugin-push:{purpose}:AllowedOps
Plugin:{pluginId}:Capability:plugin-push:{purpose}:MaxItems
Plugin:{pluginId}:Capability:plugin-push:{purpose}:MaxChars
Plugin:{pluginId}:Capability:plugin-push:{purpose}:AllowBackground
```

Memory 插件第一阶段使用：

```text
Plugin:memory_engine:Capability:plugin-push:memory-context:Enabled
Plugin:memory_engine:Capability:plugin-push:memory-context:AllowedOps = memory.context.upsert
```

init extensions 中 `pluginPushEnabled=true` 且 `pluginPushPurposes` 包含目标 purpose 时，插件才允许启用主动推送。

运行时宿主仍需校验：

1. pluginId 是否为已启用插件。
2. token 是否有效。
3. 插件是否具备对应 `plugin-push` purpose 授权。
4. `op` 是否属于该 purpose 的允许操作。
5. 推送包是否超过条目数、字符数和频率限制。
6. 推送内容是否包含必要 traceId、sourceIds 和证据引用。

### 通用推送帧

```json
{
  "type": "request",
  "protocol": "plugin-push",
  "version": "1.0.0",
  "op": "memory.context.upsert",
  "requestId": "...",
  "traceId": "...",
  "pluginId": "memory_engine",
  "purpose": "memory-context",
  "targetKind": "agent-context",
  "targetId": "...",
  "workspaceId": "...",
  "sessionId": "...",
  "packageId": "...",
  "createdAt": "2026-05-02T00:00:00Z",
  "contentFormat": "application/json",
  "contentSchemaId": "memory.context-package.v1",
  "content": "{...插件自定义 JSON 字符串...}"
}
```

为保持 AOT 友好，宿主只识别 envelope 和少量通用元数据，`content` 是 opaque 字符串。宿主可以只做大小、schemaId、hash、基础 JSON 合法性检查；插件专用结构由插件自己维护。

### 确认帧

```json
{
  "type": "response",
  "protocol": "plugin-push",
  "version": "1.0.0",
  "op": "plugin.push.accepted",
  "requestId": "...",
  "traceId": "...",
  "pluginId": "memory_engine",
  "purpose": "memory-context",
  "packageId": "...",
  "accepted": true,
  "acceptedItems": 12,

}
```

### 拒绝帧


### Purpose 与 op 示例

| Purpose | Op | 处理方式 |
| --- | --- | --- |
| `memory-context` | `memory.context.upsert` | 写入智能体上下文候选池。 |
| `plugin-diagnostics` | `diagnostics.report` | 写入宿主诊断日志或状态页。 |
| `index-status` | `index.status.update` | 更新索引进度或索引状态。 |
| `context-suggestion` | `context.suggestion.add` | 添加通用上下文建议候选。 |

### Memory Context Package 示例

当 `purpose=memory-context` 且 `op=memory.context.upsert` 时，`content` 内部结构由 Memory 插件维护，建议包含：

```json
{
  "items": [
    {
      "memoryId": "...",
      "type": "preference",
      "title": "...",
      "summary": "...",
      "detail": "...",
      "confidence": 0.82,
      "importance": 0.76,
      "sourceObservationIds": ["..."],
      "sourceFragmentIds": ["..."],
      "evidenceQuotes": ["..."]
    }
  ],
  "reason": "为什么推送这些记忆",
  "promptHint": "建议注入上下文时使用的简短提示"
}
```

宿主不直接信任 `content`，只将其作为候选来源。对于 Memory 推送，宿主侧可根据智能体、会话、token 预算、用户设置和隐私策略决定是否注入。

### 宿主侧上下文接入

Memory 推送到达后建议进入“上下文候选池”，而不是立即拼接进当前请求。

处理流程：

1. 校验授权与大小限制。
2. 记录 packageId、pluginId、purpose、op、targetId、traceId、schemaId 和 contentHash。
3. 根据 purpose 和 op 路由到对应处理器。
4. `memory-context` 写入上下文候选池。
5. 当前或下一轮智能体请求构建上下文时，从候选池中按相关性、重要性、置信度和 token 预算选择注入。
6. 记录实际注入事件，便于追溯某条回复使用了哪些插件推送信息。

## 12. 宿主侧设计

宿主新增内部服务：

```text
IPluginModelCapabilityBroker
    InvokeAsync(PluginModelInvokeRequest request, CancellationToken cancellationToken)
```

职责：

1. 校验连接来源，只允许本机环回地址。
2. 校验 pluginId、短期 token、插件启用状态。
3. 校验 purpose 是否在该插件授权列表内。
4. 校验该 purpose 是否允许后台调用、并发、预算和超时。
5. 根据 `pluginId + capability=llm + purpose` 找到 ProviderId 和 ModelId。
6. 解析实际 Provider、Model 和 AI Driver。
7. 构造宿主内部 `IChatClient`。
8. 将 `prompt.system`、`prompt.instructions`、`prompt.input` 转换为模型消息。
9. 调用模型并返回文本结果。
10. 返回 providerId、modelId、usage 和 duration。
11. 写宿主审计日志。

复用现有能力：

1. 可复用模型服务、Provider 服务、Driver Registry 和 `IChatClient` 调用链。
2. 当前 `ModelPurposeResolver` 以 setting key 为中心，后续应抽取或新增基于 `ProviderId + ModelId` 的解析方法。
3. 不建议让 Broker 依赖 `Memory.ModelId`。

宿主 AOT DTO 边界：

```text
ModelCapabilityEnvelope
PluginModelInvokeRequest
PluginModelPrompt
PluginModelOutputContract
PluginModelBudget
PluginModelOptions
PluginModelInvokeResponse
PluginModelError
```

不允许宿主 DTO 边界包含：

```text
MemoryProcessingInput
MemoryAbstractionInput
AnyPluginBusinessInput
```

宿主还需要新增通用插件推送接收服务：

```text
IPluginPushReceiver
  AcceptAsync(PluginPushRequest request, CancellationToken cancellationToken)
```

职责：

1. 校验 pluginId、token 和 `plugin-push:{purpose}` 授权。
2. 校验 targetKind、targetId、workspaceId、sessionId 的可见范围。
3. 校验 content 大小、schemaId、packageId 幂等性。
4. 根据 purpose 和 op 路由到对应宿主处理器。
5. 对 `memory-context` 写入上下文候选池。
6. 返回 acceptedItems、receiptId 或拒绝错误。

## 13. 插件侧设计

Memory 插件新增客户端抽象：

```text
IHostModelCapabilityClient
    InvokeAsync(HostModelInvokeRequest request, CancellationToken cancellationToken)
```

Memory 插件新增两个模型驱动实现：

```text
HostModelMemorySemanticProcessor : IMemorySemanticProcessor
HostModelMemoryAbstractionGenerator : IMemoryAbstractionGenerator
```

建议不要在 DI 阶段强依赖复杂授权 DTO。更稳妥的方式是注册代理实现：

```text
ModelAwareMemorySemanticProcessor : IMemorySemanticProcessor
ModelAwareMemoryAbstractionGenerator : IMemoryAbstractionGenerator
```

代理实现运行时根据 init extensions 中的字符串授权摘要判断：

1. `modelCapabilityEnabled=true` 且 purpose 已授权时，尝试调用宿主模型控制面。
2. WebSocket 未连接时，首次请求前延迟连接并握手。
3. 授权缺失、MCP 独立模式、连接失败、模型错误时使用 fallback。
4. 单批处理遇到模型错误时，按错误类型重试或降级，不影响 observation 摄取。

连接策略：

1. 插件启动后延迟连接模型控制面，避免阻塞插件 init。
2. 首次模型请求前若未连接，尝试连接并握手。
3. 连接断开后指数退避重连。
4. requestId 与 pending task 映射，收到响应后完成对应请求。
5. 插件停止时发送 cancel 或取消所有 pending 请求。
6. 连接关闭时，宿主取消该连接上的 pending 调用。

主动推送策略：

1. 插件完成自身计算后，按授权、purpose、op 和阈值判断是否生成推送包。
2. 插件将推送内容作为 opaque 字符串发送给宿主。
3. 插件记录 packageId、traceId、推送状态和宿主确认结果。
4. 宿主拒绝或断线时，不影响插件本地状态；后续可按策略重试或等待下一次计算。
5. Memory 插件第一阶段使用 `purpose=memory-context` 与 `op=memory.context.upsert` 推送 Memory Context Package。

## 14. Memory 插件输出契约

宿主不校验 Memory 专用输出结构。Memory 插件自己校验模型输出。

### `memory-processing` 输出契约

模型只输出 JSON，不输出解释性自由文本。

```json
{
  "candidates": [
    {
      "sourceObservationIds": ["..."],
      "quotedText": "原文引用",
      "memoryType": "preference",
      "topic": "coding-style",
      "title": "...",
      "summary": "...",
      "detail": "...",
      "keywords": ["..."],
      "importance": 0.8,
      "confidence": 0.75,
      "reason": "为什么值得记忆"
    }
  ]
}
```

校验规则：

1. `sourceObservationIds` 必须来自本次输入或可回查窗口。
2. `quotedText` 必须能在原文中找到。
3. `memoryType` 必须属于允许枚举。
4. `importance`、`confidence` 必须在 0 到 1 之间。
5. 任何不合法候选必须丢弃，并记录处理事件。

### `memory-abstraction` 输出契约

```json
{
  "abstractions": [
    {
      "supportingMemoryIds": ["..."],
      "statement": "稳定抽象结论",
      "summary": "简要说明",
      "abstractionType": "constraint",
      "confidence": 0.82,
      "stabilityScore": 0.76,
      "evidenceQuotes": ["..."]
    }
  ]
}
```

校验规则：

1. `supportingMemoryIds` 必须来自本次输入 fragment 集合。
2. `evidenceQuotes` 必须能追溯到 fragment detail、summary 或原始 observation。
3. `statement` 不能引入无证据的新事实。
4. `confidence`、`stabilityScore` 必须在 0 到 1 之间。
5. 不满足契约的 abstraction 不入库。

## 15. 审计和证据

宿主侧审计至少记录：

1. pluginId。
2. capability。
3. purpose。
4. taskKind。
5. providerId 和 modelId。
6. traceId 和 requestId。
7. token 使用量。
8. 耗时、成功、失败和错误码。

插件侧第一阶段可通过 `MemoryEvent.PayloadJson` 记录模型调用证据，不强制新增专表。

建议事件类型：

| EventType | 说明 |
| --- | --- |
| `model_call_started` | 模型调用开始。 |
| `model_call_completed` | 模型调用完成。 |
| `model_call_failed` | 模型调用失败。 |
| `model_call_fallback` | 使用 fallback。 |
| `model_output_invalid` | 模型输出不满足插件契约。 |
| `plugin_push_started` | 插件主动推送开始。 |
| `plugin_push_accepted` | 插件主动推送被宿主接受。 |
| `plugin_push_rejected` | 插件主动推送被宿主拒绝。 |

插件证据字段建议：

| 字段 | 说明 |
| --- | --- |
| `requestId` | 插件侧请求 ID。 |
| `traceId` | 宿主、插件、数据库贯穿链路。 |
| `purpose` | LLM purpose。 |
| `taskKind` | 任务类型。 |
| `providerId` | 宿主返回的实际 Provider ID。 |
| `modelId` | 宿主返回的实际 Model ID。 |
| `promptHash` | 提示词哈希。 |
| `inputIdsJson` | 输入 observation 或 fragment ID。 |
| `outputIdsJson` | 生成的 fragment 或 abstraction ID。 |
| `status` | completed、failed、fallback。 |
| `errorCode` | 失败原因。 |
| `durationMs` | 调用耗时。 |
| `createdAt` | 记录时间。 |

插件主动推送证据字段建议：

| 字段 | 说明 |
| --- | --- |
| `packageId` | 推送包 ID。 |
| `receiptId` | 宿主返回的推送回执 ID。 |
| `purpose` | 推送 purpose。 |
| `op` | 推送操作类型。 |
| `targetKind` | 目标类型。 |
| `targetId` | 目标 ID，例如智能体 ID。 |
| `workspaceId` | 工作区 ID。 |
| `sessionId` | 会话 ID。 |
| `contentHash` | 推送内容 hash。 |
| `acceptedItems` | 宿主接受的条目数。 |

完整 prompt 和完整原文是否保存由隐私策略决定。第一阶段建议保存 hash、输入 ID、输出 ID 和短摘要，不默认保存完整 prompt。

## 16. 错误码

宿主错误码：

| 错误码 | retryable | 说明 |
| --- | --- | --- |
| `UNAUTHORIZED_PLUGIN` | false | 插件未被允许使用模型控制面。 |
| `UNAUTHORIZED_PURPOSE` | false | 插件未被授权申请该 purpose。 |
| `MODEL_NOT_CONFIGURED` | false | 该插件 purpose 未绑定模型。 |
| `MODEL_NOT_FOUND` | false | 绑定的模型 ID 已失效。 |
| `PROVIDER_NOT_AVAILABLE` | true | Provider 暂不可用。 |
| `MODEL_TIMEOUT` | true | 模型调用超时。 |
| `MODEL_RATE_LIMITED` | true | 命中限流。 |
| `BUDGET_EXCEEDED` | false | 请求预算超过授权范围。 |
| `CONCURRENCY_LIMITED` | true | 超过该插件 purpose 并发限制。 |
| `INVALID_REQUEST` | false | 请求格式、预算或 prompt 为空。 |
| `PLUGIN_PUSH_UNAUTHORIZED` | false | 插件未被授权执行该主动推送。 |
| `PLUGIN_PUSH_TOO_LARGE` | false | 推送包超过条目数或字符数限制。 |
| `PLUGIN_PUSH_INVALID_TARGET` | false | targetId、workspaceId 或 sessionId 不可见或无效。 |
| `PLUGIN_PUSH_DUPLICATE` | false | packageId 已处理。 |
| `INTERNAL_ERROR` | true | 宿主内部异常。 |

插件侧处理事件：

| 事件 | 说明 |
| --- | --- |
| `INVALID_MODEL_OUTPUT` | 模型输出不是合法 JSON。 |
| `OUTPUT_CONTRACT_VIOLATION` | 模型输出不满足插件契约。 |
| `EVIDENCE_NOT_FOUND` | 输出引用的证据不存在。 |

插件降级策略：

1. `MODEL_TIMEOUT`、`PROVIDER_NOT_AVAILABLE`、`MODEL_RATE_LIMITED` 可按批次重试一次。
2. `MODEL_NOT_CONFIGURED`、`UNAUTHORIZED_PURPOSE` 不重试，直接 fallback。
3. `INVALID_MODEL_OUTPUT` 不重试同一输出，记录失败并进入保守规则。
4. 连续失败达到阈值后，短时间熔断模型调用。

## 18. 验收标准

1. `Memory.ModelId` 不再作为新模型授权入口。
2. 插件设置面板可管理插件启用、全局插件和 LLM 授权。
3. 全局插件无需智能体绑定，所有智能体默认可用。
4. 只有用户数据目录的全局插件可设为全局插件。
5. 工作区插件、临时插件默认不能设为全局插件。
6. Memory 插件可获得 `memory-processing` 和 `memory-abstraction` 两个 purpose 授权。
7. 插件 init 阶段可通过 extensions 判断自己是否具备 LLM 能力。
8. init 授权下发使用 AOT 友好的字符串字典和 JSON 字符串，不引入插件业务 DTO。
9. LLM 请求协议使用 prompt 字符串承载插件业务输入，宿主不反序列化插件业务 input。
10. 宿主只绑定通用 model-capability DTO。
11. 宿主按 `pluginId + purpose` 精确校验授权和模型绑定。
12. 未授权 purpose 返回 `UNAUTHORIZED_PURPOSE`。
13. 未绑定模型返回 `MODEL_NOT_CONFIGURED`。
14. 模型调用结果带 providerId、modelId、usage、duration 和 traceId。
15. 不合法 Memory 模型输出不会写入 fragment 或 abstraction。
16. WebSocket 控制面断开不会影响 conversation-feed 摄取。
17. 插件退出时 pending 模型请求会被取消。
18. MCP 独立模式没有宿主授权时继续使用 fallback。
19. 插件在获得 `plugin-push:{purpose}` 授权后，可以主动向宿主推送信息包。
20. Memory 插件的 `memory-context` 推送进入上下文候选池，不让插件直接修改智能体上下文。
21. 智能体上下文构建阶段可从候选池选择记忆注入，并记录注入 trace。
22. 未授权主动推送返回 `PLUGIN_PUSH_UNAUTHORIZED`。

## 19. 当前决策

第一阶段采用内部 WebSocket 控制面，不扩展插件子进程基础协议。

插件授权从 Memory 专用模型设置升级为 `pluginId + capability + purpose` 模型。`Memory.ModelId` 不再作为新方案入口。

Memory 插件第一阶段使用两个 LLM purpose：

1. `memory-processing`
2. `memory-abstraction`

LLM 请求协议采用方案 B：插件直接传 prompt，业务输入作为 opaque 字符串交给宿主，宿主只负责授权、模型选择、预算、调用和审计，不绑定插件业务 DTO。

宿主 init 下发授权也采用 AOT 友好方式：使用 `PluginSettings.Extensions` 字符串字典和 `modelCapabilityGrantsJson` 字符串，避免为每个插件新增授权 DTO。

插件可以在获得 `plugin-push:{purpose}` 授权后主动向宿主推送信息包。推送内容同样使用 opaque 字符串，宿主只做授权、大小、目标和审计校验，并按 purpose / op 路由处理。Memory 插件的 `memory-context` 推送会进入上下文候选池；最终是否注入智能体上下文由宿主决定。

全局插件作为插件设置的一部分实现。全局插件必须来自用户数据目录的全局插件目录，不允许工作区插件直接提升为全局插件。
