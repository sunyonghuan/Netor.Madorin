# 记忆插件数据库表与关系说明

本文档描述 `Cortana.Plugins.Memory` 记忆插件当前 SQLite 数据库 `memory.db` 中已经由 `MemoryStore.EnsureInitialized()` 初始化的表结构、表用途、核心字段和表之间的关系。

## 总览

当前数据库用于承载记忆插件的私有长期记忆数据。宿主只负责通过内部事实流提供会话事件，记忆插件负责将事实流落库、提取候选记忆、维护记忆图、记录召回日志和审计变更。

当前已定义 9 张表：

| 表名 | 作用 |
| --- | --- |
| `observation_records` | 保存从宿主会话事实流同步过来的原始观察记录。 |
| `schema_migrations` | 保存数据库结构迁移版本和应用记录。 |
| `memory_fragments` | 保存从观察记录中提取出的长期记忆片段。 |
| `memory_abstractions` | 保存由多个记忆片段归纳出的抽象记忆。 |
| `memory_links` | 保存记忆片段和抽象记忆之间的图关系。 |
| `memory_events` | 保存记忆系统内部处理事件。 |
| `recall_logs` | 保存每一次记忆召回请求及命中结果。 |
| `memory_mutations` | 保存记忆内容、状态、评分等变更审计记录。 |
| `memory_settings` | 保存记忆系统运行参数和策略配置。 |

## 数据流关系

```text
宿主会话事实流
	↓
observation_records
	↓ 提取、评分、去重、合并
memory_fragments
	↓ 归纳、抽象
memory_abstractions
	↘
	  memory_links  ← 连接 fragment / abstraction 之间的支持、冲突、相关、派生关系
	↙
recall_logs         ← 记录召回请求命中的 fragments / abstractions
memory_mutations    ← 记录 fragments / abstractions / links 的创建、更新、合并、遗忘等变更
memory_events       ← 记录提取、强化、衰减、遗忘等内部事件
memory_settings     ← 控制衰减、召回窗口、主动供应、抽象、审计等运行策略
schema_migrations   ← 记录数据库结构版本
```

## 1. observation_records

### 作用

`observation_records` 是记忆系统的原始输入表，用于保存从宿主会话事实流接收到的原始观察记录。该表不直接代表长期记忆，而是长期记忆提取、分析、归纳的事实来源。

### 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 观察记录唯一标识，主键。 |
| `agentId` | 产生该观察记录的智能体标识。 |
| `workspaceId` | 观察记录所属工作区标识。 |
| `sessionId` | 会话标识。 |
| `turnId` | 会话轮次标识。 |
| `messageId` | 来源消息标识。 |
| `eventType` | 事实流事件类型。 |
| `role` | 消息角色，例如用户、助手或系统。 |
| `content` | 观察记录文本内容。 |
| `attachments` | 附件信息 JSON。 |
| `createdTimestamp` | 来源事件创建时间戳，通常为 Unix 毫秒。 |
| `modelName` | 生成消息或事件时使用的模型名称。 |
| `traceId` | 链路追踪标识。 |
| `sourceFacts` | 原始来源事实 JSON 快照。 |
| `schemaVersion` | 数据结构版本号。 |
| `recordVersion` | 当前记录版本号。 |
| `createdAt` | 写入记忆库的时间。 |

### 关系

- 一条或多条 `observation_records` 可以支撑一个 `memory_fragments`。
- `memory_fragments.sourceObservationIdsJson` 通过 JSON 数组保存来源观察记录 ID。
- `sessionId`、`turnId` 可以辅助追溯记忆片段的对话上下文。

### 索引

| 索引 | 作用 |
| --- | --- |
| `IX_obs_session_time` | 按会话和时间顺序读取观察记录。 |
| `IX_obs_messageId` | 按消息标识查找观察记录。 |
| `IX_obs_eventType` | 按事件类型过滤观察记录。 |
| `IX_obs_agent_session_turn` | 按智能体、会话、轮次定位上下文。 |
| `IX_obs_trace` | 按链路追踪标识排查问题。 |

## 2. schema_migrations

### 作用

`schema_migrations` 用于记录数据库结构迁移历史，避免重复执行迁移，并为后续升级和回滚提供依据。

### 关键字段

| 字段 | 说明 |
| --- | --- |
| `version` | 迁移版本号，主键。 |
| `appliedAt` | 迁移应用时间。 |
| `description` | 迁移说明。 |
| `isRollbackSafe` | 是否可安全回滚，`1` 表示可安全回滚。 |

### 当前版本

| 版本 | 说明 |
| --- | --- |
| `1` | 初始记忆数据库基线。 |
| `2` | 扩展长期记忆数据库结构。 |
| `3` | 增加记忆系统配置表。 |

### 关系

- 该表不参与业务数据关系。
- 该表用于管理所有其他表的结构版本。

## 3. memory_fragments

### 作用

`memory_fragments` 保存从原始观察记录中提取出来的候选或已确认长期记忆片段，是记忆系统的核心事实层。

### 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 记忆片段唯一标识，主键。 |
| `agentId` | 记忆所属智能体标识。 |
| `workspaceId` | 记忆所属工作区标识。 |
| `memoryType` | 记忆类型，例如事实、偏好、任务或约束。 |
| `topic` | 记忆主题，用于聚类、检索和召回排序。 |
| `title` | 记忆标题。 |
| `summary` | 记忆摘要。 |
| `detail` | 记忆详细内容。 |
| `keywordsJson` | 关键词 JSON。 |
| `tagsJson` | 标签 JSON。 |
| `entitiesJson` | 实体识别结果 JSON。 |
| `sourceObservationIdsJson` | 支撑该记忆的观察记录 ID 集合 JSON。 |
| `sourceSessionIdsJson` | 来源会话 ID 集合 JSON。 |
| `sourceTurnIdsJson` | 来源轮次 ID 集合 JSON。 |
| `importance` | 重要性评分。 |
| `confidence` | 可信度评分。 |
| `emotionalWeight` | 情绪权重。 |
| `novelty` | 新颖度评分。 |
| `salienceScore` | 显著性评分。 |
| `retentionScore` | 保留评分。 |
| `decayRate` | 衰减率。 |
| `accessCount` | 被访问次数。 |
| `reinforcementCount` | 被强化次数。 |
| `contradictionCount` | 被矛盾证据命中的次数。 |
| `clarityLevel` | 清晰度等级。 |
| `confirmationState` | 确认状态。 |
| `lifecycleState` | 生命周期状态。 |
| `lastAccessedAt` | 最近访问时间。 |
| `lastReinforcedAt` | 最近强化时间。 |
| `expiresAt` | 过期时间。 |
| `schemaVersion` | 数据结构版本号。 |
| `recordVersion` | 当前记录版本号。 |
| `compatibilityTagsJson` | 兼容性标签 JSON。 |
| `createdAt` | 创建时间。 |
| `updatedAt` | 更新时间。 |

### 关系

- 来源于 `observation_records`。
- 可以支撑 `memory_abstractions`。
- 可以通过 `memory_links` 与其他 `memory_fragments` 或 `memory_abstractions` 形成图关系。
- 可以被 `recall_logs.hitMemoryIdsJson` 或 `recall_logs.supportingMemoryIdsJson` 引用。
- 变更过程由 `memory_mutations` 记录。

### 索引

| 索引 | 作用 |
| --- | --- |
| `IX_fragment_agent_topic_score` | 按智能体、主题和显著性评分检索候选记忆。 |
| `IX_fragment_agent_workspace` | 按智能体和工作区过滤记忆。 |
| `IX_fragment_state_confirm` | 按生命周期状态和确认状态筛选记忆。 |
| `IX_fragment_updated` | 按更新时间扫描记忆。 |

## 4. memory_abstractions

### 作用

`memory_abstractions` 保存由多个记忆片段归纳形成的高层抽象记忆，例如稳定偏好、长期事实、行为模式和项目约束。

### 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 抽象记忆唯一标识，主键。 |
| `agentId` | 所属智能体标识。 |
| `workspaceId` | 所属工作区标识。 |
| `abstractionType` | 抽象类型。 |
| `title` | 抽象标题。 |
| `statement` | 抽象记忆核心陈述。 |
| `summary` | 抽象摘要。 |
| `supportingMemoryIdsJson` | 支撑该抽象的记忆 ID 集合 JSON。 |
| `counterMemoryIdsJson` | 反例或冲突记忆 ID 集合 JSON。 |
| `keywordsJson` | 关键词 JSON。 |
| `tagsJson` | 标签 JSON。 |
| `importance` | 重要性评分。 |
| `confidence` | 可信度评分。 |
| `stabilityScore` | 稳定性评分。 |
| `retentionScore` | 保留评分。 |
| `decayRate` | 衰减率。 |
| `accessCount` | 被访问次数。 |
| `reinforcementCount` | 被强化次数。 |
| `contradictionCount` | 被矛盾证据命中的次数。 |
| `clarityLevel` | 清晰度等级。 |
| `confirmationState` | 确认状态。 |
| `lifecycleState` | 生命周期状态。 |
| `lastValidatedAt` | 最近校验时间。 |
| `lastAccessedAt` | 最近访问时间。 |
| `expiresAt` | 过期时间。 |
| `schemaVersion` | 数据结构版本号。 |
| `recordVersion` | 当前记录版本号。 |
| `compatibilityTagsJson` | 兼容性标签 JSON。 |
| `createdAt` | 创建时间。 |
| `updatedAt` | 更新时间。 |

### 关系

- 主要由 `memory_fragments` 归纳而来。
- `supportingMemoryIdsJson` 保存支撑抽象的记忆 ID。
- `counterMemoryIdsJson` 保存与抽象冲突或削弱抽象的记忆 ID。
- 可以通过 `memory_links` 与 `memory_fragments` 或其他 `memory_abstractions` 建立关系。
- 可以被 `recall_logs` 命中或作为支撑记忆引用。
- 变更过程由 `memory_mutations` 记录。

### 索引

| 索引 | 作用 |
| --- | --- |
| `IX_abstraction_agent_type` | 按智能体和抽象类型检索抽象记忆。 |
| `IX_abstraction_state_confirm` | 按生命周期状态和确认状态筛选抽象记忆。 |

## 5. memory_links

### 作用

`memory_links` 保存记忆图中的关系边，用于描述记忆片段和抽象记忆之间的支持、冲突、相关、派生等关系。

### 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 关系唯一标识，主键。 |
| `agentId` | 关系所属智能体标识。 |
| `sourceMemoryId` | 源记忆 ID。 |
| `sourceMemoryKind` | 源记忆类型，例如 `fragment` 或 `abstraction`。 |
| `targetMemoryId` | 目标记忆 ID。 |
| `targetMemoryKind` | 目标记忆类型，例如 `fragment` 或 `abstraction`。 |
| `relationType` | 关系类型，例如 `supports`、`contradicts`、`related`、`derived_from`。 |
| `weight` | 关系权重。 |
| `evidenceCount` | 支撑该关系的证据数量。 |
| `confidence` | 关系可信度。 |
| `schemaVersion` | 数据结构版本号。 |
| `recordVersion` | 当前记录版本号。 |
| `createdAt` | 创建时间。 |
| `updatedAt` | 更新时间。 |

### 关系

- `sourceMemoryId` 和 `targetMemoryId` 可以指向 `memory_fragments.id` 或 `memory_abstractions.id`。
- 具体指向哪张表由 `sourceMemoryKind` 和 `targetMemoryKind` 决定。
- 当前 SQLite 表未声明外键，关系由应用层维护。
- `memory_mutations` 可以记录关系边的创建、更新、删除或权重变化。

### 索引

| 索引 | 作用 |
| --- | --- |
| `IX_link_source` | 从源记忆查找所有关系。 |
| `IX_link_target` | 从目标记忆反查所有关系。 |
| `IX_link_relation` | 按智能体和关系类型筛选关系。 |

## 6. memory_events

### 作用

`memory_events` 保存记忆系统内部事件，例如记忆提取、强化、衰减、遗忘、重新校验等操作事件。该表用于后台流程追踪和后续异步处理。

### 关键字段

| 字段 | 说明 |
| --- | --- |
| `eventId` | 事件唯一标识，主键。 |
| `agentId` | 事件所属智能体标识。 |
| `eventType` | 事件类型。 |
| `payloadJson` | 事件负载 JSON。 |
| `processedAt` | 事件处理完成时间。 |
| `schemaVersion` | 数据结构版本号。 |
| `recordVersion` | 当前记录版本号。 |

### 关系

- 可以在 `payloadJson` 中引用 `observation_records`、`memory_fragments`、`memory_abstractions` 或 `memory_links`。
- 当前表不声明外键，事件关联由 `payloadJson` 和应用层逻辑维护。

### 索引

| 索引 | 作用 |
| --- | --- |
| `IX_event_agent_type` | 按智能体和事件类型检索事件。 |

## 7. recall_logs

### 作用

`recall_logs` 保存每次记忆召回请求的输入、命中、抑制和最终摘要，用于审计召回行为、调试召回质量和后续强化学习。

### 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 召回日志唯一标识，主键。 |
| `requestId` | 召回请求标识。 |
| `agentId` | 发起召回的智能体标识。 |
| `workspaceId` | 所属工作区标识。 |
| `queryText` | 召回查询文本。 |
| `queryIntent` | 查询意图。 |
| `triggerSource` | 触发来源。 |
| `hitMemoryIdsJson` | 命中的记忆 ID 集合 JSON。 |
| `supportingMemoryIdsJson` | 用于支撑回答的记忆 ID 集合 JSON。 |
| `suppressedMemoryIdsJson` | 被策略、权限或冲突抑制的记忆 ID 集合 JSON。 |
| `recallSummary` | 本次召回摘要。 |
| `confidence` | 本次召回整体可信度。 |
| `budgetJson` | 召回预算 JSON。 |
| `appliedPolicyJson` | 已应用策略 JSON。 |
| `traceId` | 链路追踪标识。 |
| `schemaVersion` | 数据结构版本号。 |
| `recordVersion` | 当前记录版本号。 |
| `createdAt` | 日志创建时间。 |

### 关系

- `hitMemoryIdsJson` 可以引用 `memory_fragments.id` 或 `memory_abstractions.id`。
- `supportingMemoryIdsJson` 可以引用实际进入回复上下文的记忆。
- `suppressedMemoryIdsJson` 记录被策略过滤掉的记忆。
- 当前表不声明外键，召回引用关系由 JSON 字段和应用层维护。

### 索引

| 索引 | 作用 |
| --- | --- |
| `IX_recall_agent_created` | 按智能体和创建时间查询召回历史。 |

## 9. memory_settings

### 作用

`memory_settings` 保存记忆系统的运行配置和策略参数，用于控制记忆衰减、保留、召回窗口、主动供应、抽象生成和治理审计等行为。该表支持全局默认配置，也支持后续按智能体或工作区覆盖配置。

### 关键字段

| 字段 | 说明 |
| --- | --- |
| `id` | 配置记录唯一标识，主键。 |
| `agentId` | 配置所属智能体标识；为空表示全局默认配置。 |
| `workspaceId` | 配置所属工作区标识；为空表示不限定工作区。 |
| `settingKey` | 配置键，例如 `decay.defaultRate` 或 `recall.maxWindowCount`。 |
| `settingValue` | 配置值，以字符串保存。 |
| `valueType` | 配置值类型，例如 `string`、`int`、`double`、`bool` 或 `json`。 |
| `category` | 配置分类，例如 `decay`、`recall`、`supply`、`abstraction` 或 `governance`。 |
| `description` | 配置项说明。 |
| `isEnabled` | 是否启用该配置项，`1` 表示启用。 |
| `schemaVersion` | 数据结构版本号。 |
| `recordVersion` | 当前记录版本号。 |
| `createdAt` | 配置创建时间。 |
| `updatedAt` | 配置最后更新时间。 |

### 默认配置项

| 配置键 | 默认值 | 类型 | 分类 | 作用 |
| --- | --- | --- | --- | --- |
| `decay.enabled` | `true` | `bool` | `decay` | 是否启用记忆衰减。 |
| `decay.defaultRate` | `0.015` | `double` | `decay` | 默认记忆衰减率。 |
| `decay.scanIntervalMinutes` | `60` | `int` | `decay` | 衰减扫描间隔，单位分钟。 |
| `retention.minimumScore` | `0.2` | `double` | `retention` | 参与保留计算的最低保留分数。 |
| `retention.forgetThreshold` | `0.05` | `double` | `retention` | 低于该阈值的记忆可进入遗忘候选。 |
| `recall.maxWindowCount` | `6` | `int` | `recall` | 每次向上层提供的最大记忆窗口数量。 |
| `recall.maxMemoryCount` | `20` | `int` | `recall` | 每次召回最多返回的记忆数量。 |
| `recall.minimumConfidence` | `0.35` | `double` | `recall` | 允许参与召回的最低记忆可信度。 |
| `recall.includeCandidateMemories` | `false` | `bool` | `recall` | 是否允许候选记忆参与召回。 |
| `supply.enabled` | `true` | `bool` | `supply` | 是否启用主动记忆供应。 |
| `supply.maxMemoryCount` | `8` | `int` | `supply` | 主动供应给上层的最大记忆数量。 |
| `abstraction.enabled` | `true` | `bool` | `abstraction` | 是否启用自动抽象记忆生成。 |
| `abstraction.minimumSupportCount` | `3` | `int` | `abstraction` | 生成抽象记忆需要的最小支撑记忆数。 |
| `abstraction.minimumConfidence` | `0.55` | `double` | `abstraction` | 抽象记忆入库所需的最低可信度。 |
| `governance.auditEnabled` | `true` | `bool` | `governance` | 是否启用记忆治理审计。 |

### 关系

- `memory_settings` 不直接引用业务数据表，而是控制 `memory_fragments`、`memory_abstractions`、`recall_logs`、`memory_events` 和 `memory_mutations` 相关流程。
- `decay` 和 `retention` 类配置影响记忆片段、抽象记忆的衰减、保留和遗忘。
- `recall` 和 `supply` 类配置影响每次主动提供给上层的记忆窗口数量和召回数量。
- `abstraction` 类配置影响抽象记忆的生成开关、支撑记忆数量和可信度阈值。
- `governance` 类配置影响审计和变更记录策略。

### 索引

| 索引 | 作用 |
| --- | --- |
| `IX_setting_scope_key` | 保证同一作用域下配置键唯一，并支持按全局、智能体、工作区查找配置。 |
| `IX_setting_category` | 按配置分类和启用状态筛选配置项。 |

## 关系总结

### 主数据链路

```text
observation_records
	→ memory_fragments
	→ memory_abstractions
```

- `observation_records` 是原始事实层。
- `memory_fragments` 是从事实中提取出来的长期记忆片段。
- `memory_abstractions` 是多个记忆片段归纳出的高层结论。

### 图关系链路

```text
memory_fragments ─┐
				  ├─ memory_links ─ memory_fragments
memory_abstractions ┘              └ memory_abstractions
```

- `memory_links` 用于连接任意两个记忆对象。
- 记忆对象可以是 `fragment`，也可以是 `abstraction`。
- 关系类型由 `relationType` 表示。

### 召回与审计链路

```text
memory_fragments / memory_abstractions
	→ recall_logs
	→ memory_mutations
```

- `recall_logs` 记录记忆被召回、被使用、被抑制的情况。
- `memory_mutations` 记录记忆对象在生命周期中的所有重要变化。

### 事件链路

```text
memory_events
	→ payloadJson 引用 observation / fragment / abstraction / link
```

- `memory_events` 用于记录或驱动后台记忆处理流程。
- 事件和业务对象的关系暂由 `payloadJson` 承载。

### 配置控制链路

```text
memory_settings
	→ 控制 decay / retention / recall / supply / abstraction / governance 策略
	→ 影响 memory_fragments / memory_abstractions / recall_logs / memory_events / memory_mutations
```

- `memory_settings` 提供全局默认运行参数。
- 后续可以通过 `agentId` 和 `workspaceId` 增加智能体级或工作区级覆盖配置。
- 配置值统一以字符串保存，由上层读取后按 `valueType` 转换。

## 当前设计注意事项

1. 当前数据库没有声明 SQLite 外键，跨表关系主要通过 JSON 字段和应用层逻辑维护。
2. `memory_links`、`recall_logs`、`memory_mutations` 中的记忆引用都是多态引用，需要结合 `memoryKind`、`sourceMemoryKind` 或 `targetMemoryKind` 判断目标表。
3. `memory_settings` 的 `agentId` 和 `workspaceId` 均为空时表示全局默认配置；后续覆盖策略应按工作区级、智能体级、全局级顺序回退。
4. 所有核心业务表都包含 `schemaVersion` 和 `recordVersion`，用于后续兼容升级。
5. 时间字段当前使用字符串保存 ISO 8601 UTC 时间，便于跨平台和日志排查。
6. JSON 字段用于保存集合、快照、策略和事件负载，后续如查询压力增大，可以再拆分为规范化子表。
