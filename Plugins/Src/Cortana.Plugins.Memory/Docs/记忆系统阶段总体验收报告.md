## 4. 代码逻辑验收

### 4.1 历史同步逻辑

文件：`Tools\DataSync\Program.cs`

已确认：

1. 默认源数据库为 `D:\Contrna\cortana.db`。
2. 目标数据库为 `runner_data\memory.db`。
3. 导入前清理 `observation_records`、`memory_fragments`、`memory_abstractions`、`memory_mutations`、`memory_events`、`memory_processing_states`。
4. 优先从 `ChatMessages` 按 `CreatedTimestamp DESC` 导入最新 2000 条。
5. 通过 `ChatSessions` 回填 `agentId`，缺失时回退 `global`。

验收结果：通过。

### 4.2 记忆处理逻辑

文件：`Src\Cortana.Plugins.Memory\Processing\MemoryProcessingService.cs`

已确认：

1. 处理前确保数据库初始化。
2. 使用 `fragment-extraction` 处理器状态记录处理游标。
3. 从 `GetUnprocessedObservations` 获取待处理记录。
4. 调用 `IMemorySemanticProcessor.ExtractCandidates` 生成候选。
5. 创建或合并 `memory_fragments`。
6. 写入 `memory_mutations` 和 `memory_events`。
7. 批次完成后触发 `IMemoryAbstractionService.RunAbstractionPass`。

验收结果：通过。

已知问题：`CreatedAbstractionCount` 当前没有在处理结果中准确累计，数据库实际已经生成抽象记录。

### 4.3 抽象生成逻辑

文件：`Src\Cortana.Plugins.Memory\Processing\MemoryAbstractionService.cs`

已确认：

1. 从 `IMemoryStore.GetDistinctAgentIds` 获取 agent。
2. 读取可抽象的 fragment。
3. 按 topic 分组。
4. 支撑数量达到阈值后调用 `IMemoryAbstractionGenerator`。
5. 写入 `memory_abstractions`、`memory_mutations`、`memory_events`。

验收结果：通过。

已知限制：当前以 topic-summary 为主，尚未形成更细粒度的偏好、约束、任务、事实分层抽象。

### 4.4 查询召回逻辑

文件：`Src\Cortana.Plugins.Memory\Storage\MemoryStore.cs`、`Src\Cortana.Plugins.Memory\Services\MemoryRecallService.cs`

已确认：

1. `SearchRecallCandidates` 对 queryText 进行 terms 拆分。
2. SQL 层对 fragment 的 `topic/title/summary/detail` 执行 `LIKE` 过滤。
3. SQL 层对 abstraction 的 `abstractionType/title/summary/statement` 执行 `LIKE` 过滤。
4. 查询时候选池扩大到 `max(limit * 50, 500)`。
5. LIKE 特殊字符已转义。
6. 召回服务按 `RecallScore` 排序并按 topic 分窗口。
7. `RecordMemoryAccesses` 会更新 accessCount 与 lastAccessedAt。
8. 每次召回写入 `recall_logs`。

验收结果：通过。

已知限制：当前仍是关键词匹配，不具备 embedding 或大模型语义重排能力；抽象记忆尚未设置明显优先权重。

## 7. 验收结论

本阶段总体验收：通过。

通过范围：

1. 真实历史会话同步链路通过。
2. observation 到 fragment 的处理链路通过。
3. fragment 到 abstraction 的基础抽象链路通过。
4. 处理状态、事件、变更审计链路通过。
5. queryText 到 fragment/abstraction 的召回链路通过。
6. 召回访问计数与 recall_logs 审计链路通过。
7. 当前 runner 验证库已具备进入下一阶段的真实数据基础。

有条件通过项：

1. 记忆质量：tool 日志噪声仍明显，需要优先治理。
2. 抽象质量：当前 fallback topic-summary 可用，但不等同于模型级抽象。
3. 召回质量：关键词检索可用，但语义召回和重排尚未实现。
4. 主动供应：仍处于规划阶段，尚未实现上层上下文包组装。

## 8. 必须保留的已知问题

1. `tool` 角色输出、PowerShell 日志、文件读取结果仍会生成低价值 fragment。
2. `MemoryProcessingResult.CreatedAbstractionCount` 未准确反映数据库实际新增抽象数量。
3. 抽象记忆默认是 `candidate/pending`，召回时依赖查询命中放行，尚未形成独立优先级策略。
4. 召回排序未按业务类型提升 `constraint/preference/task`。
5. 未实现 token 预算、detail 截断和上下文组装。
6. 未接入宿主大模型能力，摘要、类型、主题和抽象均为 fallback。
7. 未实现后台处理服务，当前依赖 runner 手动触发。

## 9. 下一阶段建议

建议按以下优先级继续：

1. P0：过滤或降权 tool 输出，避免 PowerShell 日志污染长期记忆和召回结果。
2. P1：提高 abstraction、constraint、preference、task 在召回排序中的权重。
3. P2：实现主动记忆供应服务，将召回结果组装为上层可注入的上下文包。
4. P3：增加召回预算控制，包括最大 token、最大 detail 长度、窗口内条数限制。
5. P4：修正 `CreatedAbstractionCount` 统计，并补充处理结果测试。
6. P5：接入宿主授权的大模型能力或 embedding，用于语义抽取、摘要、抽象和重排。
